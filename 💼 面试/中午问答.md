# 面试笔记

## Java底层原理

### JS中的 = 、 == 、 === 的区别：

- = ：是赋值运算符，比如：let num = 100;

- ==：是比较运算符，要求值相等，类型可以不一样（会自动的进行类型转换），比如：1 == '1' 为true，1 == 1 也为true 。

- ===：是比较运算符，要求值相等，类型也必须相同，比如：1 == '1' 为false，1 == 1 也为true 。

  

###  Java语言中的封装、继承、多态怎么理解？有什么好处？

- 封装：把类的属性（数据）和方法（行为）藏起来，不让外部随便乱改，只通过规定的方法访问。
- 继承：子类继承父类，直接拥有父类的属性和方法，不用重复写代码，还能扩展自己的功能。
- 多态：对于同一个行为，不同的对象会有不同的表现形式。

 

### final关键字能加在哪些地方？分别代表什么？

- 加在变量前，这个变量的值不能被修改，代表常量（常量名通常全字母大写）
- 加在方法前，这个方法不能被重写。
- 加在类前，这个类不能被继承，代表最终类。

 

### static关键字能加在哪些地方？分别代表什么？

- 修饰成员变量时，不属于对象，而是属于类，所有对象共享一份。
- 修饰成员方法时，无需创建对象，直接用类名就能调用。
- 修饰内部类时，独立于外部类，不依赖外部对象实例。
- 修饰代码块时，类加载时只执行一次。

static修饰的本质是，将它修饰的变量、方法、内部类、代码的加载时间提升到类加载器执行时，会将static修饰的内容加载到方法区。



## 数据库

### 什么是数据库连接池，数据库连接池有什么作用？

回答要点：

- 数据库连接池是个容器，负责分配、管理数据库连接(Connection)。
- 它允许应用程序重复使用一个现有的数据库连接，而不是再重新建立一个。
- 释放空闲时间超过最大空闲时间的连接，来避免因为没有释放连接而引起的数据库连接遗漏。
- 优势：资源重用、提升系统响应速度、避免数据库连接遗漏。

### Mybatis中的 #{} 号 与 ${} 号的区别：

| 符号   | 说明                                               | 场景                       | 优缺点         |
| ------ | -------------------------------------------------- | -------------------------- | -------------- |
| #{...} | 占位符。执行时，会将#{...}替换为?，生成预编译SQL   | 参数值传递                 | 安全、性能高   |
| ${...} | 拼接符。直接将参数拼接在SQL语句中，存在SQL注入问题 | 表名、字段名动态设置时使用 | 不安全、性能低 |



### XML映射配置的规则

默认规则：

1. XML映射文件的名称与Mapper接口名称一致，并且将XML映射文件和Mapper接口放置在相同包下（同包同名）。
2. XML映射文件的namespace属性为Mapper接口全限定名一致。
3. XML映射文件中sql语句的id与Mapper接口中的方法名一致，并保持返回类型一致。

如果xml映射文件的位置不满足同包同名规则，那么需要在application.properties配置文件中手动指定xml映射文件的位置：

```properties
mybatis.mapper-locations=classpath:mapper/*.xml  # 扫描mapper文件夹里所有以.xml结尾的文件
```



### 标准隔离级别与解决的问题

- 读未提交(Read Uncommitted):最低级别，允许读取未提交的数据，可能发生脏读、不可重复读和幻读。
- 读已提交(Read Committed):允许读取并发事务已提交的数据，阻止脏读，但可能发生不可重复读和幻读。
- 可重复读(Repeatable Read):保证同一事务内多次读取结果一致，阻止脏读和不可重复读，标准中允许幻读，但MySQLInnoDB通过锁机制避免了幻读。
- 可串行化(Serializable):最高级别，强制事务串行执行，防止所有并发异常，但性能最低。



### where与having区别

- 执行时机不同：where是分组之前进行过滤，不满足where条件，不参与分组；而having是分组之后对结果进行过滤。
- 判断条件不同：where不能对聚合函数进行判断，而having可以。



## Web

### 什么是反向代理？反向代理有什么好处？

反向代理是一种网络架构技术，通过反向代理服务器为后端做代理。

反向代理的好处：

- 安全：隐藏后端真实服务器
- 负载均衡：请求分给多台服务器，分摊压力，不会单台卡死
- 统一入口：多个服务只用一个域名对外，用户访问简单
- 动静分离：静态图片、页面交给代理处理，动态业务走后端，提速减负
- 缓存加速：代理缓存静态资源，重复访问不用反复查后端，打开更快
- 统一限流、SSL证书、跨域：统一做HTTPS、拦截恶意请求、解决前端跨域



Nginx中反向代理的配置

location：定义路径匹配方式

rewrite：路径重写指令

proxy_pass：代理转发指令



### 你对会话跟踪技术的了解有多少？聊一聊它们的底层原理。

目前主流的会话跟踪技术有三种，分别是Cookie技术、Session技术，以及JWT令牌技术，其中使用最多的是JWT令牌技术。

底层涉及两个头：响应头Set-Cookie和请求头Cookie。当我们在服务器端设置Cookie之后，它会自动在响应头中将Cookie的数据响应给客户端浏览器，而浏览器一旦见到这个响应头是Set-Cookie，它就会自动将Cookie保存在浏览器本地。并且在后续的每一次请求当中，它都会将当前服务器所涉及的Cookie通过请求头一并携带到服务器端。

### 如果两个相同的类都交给了IOC容器管理，那么如何注入呢？

- @Primary：在@Component上加Priamry注解，声明这个bean为主要的，注入时默认注入它。

- @Qualifier(value = “”)：在@Autowired上加Qualifier注解，value属性指定注入的bean的名称

- @Resource(name = “”)：通过name属性指定要注入的bean的名称（这种方法是Java的，用它不需要用@Autowired）



## Spring Boot 自动装配原理

pring Boot 自动装配的核心作用：**让你无需手动配置大量 XML/JavaConfig，引入依赖后自动帮你配置 Bean**（比如引入`spring-boot-starter-web`就自动配好 SpringMVC、Tomcat）。

它的底层是**SPI 机制 + 条件注解 + 启动注解**三者配合实现的，我用最清晰的流程给你讲透。
### 核心入口：@SpringBootApplication

启动类上的这个注解，是自动装配的**总开关**，它是一个组合注解：
@SpringBootApplication // 组合注解 public class DemoApplication { public static void main(String[] args) { SpringApplication.run(DemoApplication.class, args); } }
拆解后关键注解：

1. **@SpringBootConfiguration**：标记为配置类（Spring 原生注解）
2. **@ComponentScan**：自动扫描包（默认扫描启动类所在包及子包）
3. **@EnableAutoConfiguration** → **自动装配的核心注解**
4. 核心注解：@EnableAutoConfiguration
5. 这个注解是自动装配的**发动机**，底层做了两件事：
6. @Import(AutoConfigurationImportSelector.class) public @interface EnableAutoConfiguration {}
7. 读取自动配置文件
8. Spring Boot 规定：

所有自动配置类，都定义在 **META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports**（Spring Boot 2.7+ 新版路径）

或 `META-INF/spring.factories`（旧版）文件中。
### 核心流程：自动装配全步骤
### 1. 导入核心配置处理器
- **启动项目** → 触发`@SpringBootApplication`
- **开启自动装配** → `@EnableAutoConfiguration`生效
- **加载配置类名单** → 通过`AutoConfigurationImportSelector`读取上述配置文件，拿到所有自动配置类全类名
- **条件过滤** → 用**条件注解**判断：当前项目是否引入对应依赖？符合条件才加载该配置类
- **创建 Bean** → 符合条件的配置类生效，自动向容器注册 Bean
- **完成装配** → 无需手动配置，直接使用组件
### - 关键支撑：条件注解（@Conditional）
自动装配**不会无脑加载所有配置**，而是按需加载，靠的就是 Spring 的条件注解：
|注解|作用|
|---|---|
|@ConditionalOnClass|类路径下存在指定类时生效|
|@ConditionalOnMissingBean|容器中没有该 Bean 时生效|
|@ConditionalOnProperty|配置文件存在指定属性时生效|
|@ConditionalOnWebApplication|是 Web 应用时生效|

### 一句话总结原理
Spring Boot 通过 @EnableAutoConfiguration 注解，利用 SPI 机制读取预置的自动配置类清单，再通过条件注解按需加载配置，最终自动向 Spring 容器注册 Bean，实现零配置开箱即用。
### 总结
- **总入口**：`@SpringBootApplication`（组合了`@EnableAutoConfiguration`）
- **核心类**：`AutoConfigurationImportSelector`（加载配置类）
- **配置源**：`META-INF/spring/xxx.imports`（自动配置类清单）
- **按需加载**：`@Conditional`系列条件注解
- **最终效果**：引入 starter → 自动配置 → 直接使用

### Spring MVC
我按**请求流转顺序**一步步说，核心围绕 Spring MVC 九大核心组件，流程分**6 步**：

1. **请求进入 DispatcherServlet（前端控制器）**
    
    所有请求统一被它拦截，它是整个流程的入口、总调度，**不处理业务，只分发请求**。
    
2. **调用 HandlerMapping（处理器映射器）**
    
    DispatcherServlet 拿请求地址，去 HandlerMapping 中匹配映射关系，找到**对应的 Controller（处理器）**，同时组装拦截器链，返回执行链对象。
    
3. **调用 HandlerAdapter（处理器适配器）**
    
    因为处理器写法多样（普通 Controller、注解 Controller 等），需要适配器**统一调用规则**，由它真正执行目标 Controller 的方法。
    
4. **执行 Controller 并返回 ModelAndView**
    
    Controller 处理业务、封装数据（Model）+ 视图名（View），封装成 `ModelAndView` 对象返回给 DispatcherServlet。
    
5. **调用 ViewResolver（视图解析器）**
    
    视图解析器拼接**前缀 + 视图名 + 后缀**，把逻辑视图名解析为物理视图对象。
    
6. **视图渲染 & 响应**
    
    框架把 Model 中的数据填充到视图，渲染页面，最终把响应结果返回给客户端。
    

> 补充一句收尾：整个流程是**请求→拦截→找处理器→执行控制器→解析视图→响应**，DispatcherServlet 贯穿全程做统一调度。


## 事务的传播行为
## 一、先一句话理解

**事务传播行为**：当**带事务的方法 A 调用 另一个带事务的方法 B**时，**B 该如何沿用 / 创建 / 挂起事务**。

---

## 二、7 种传播行为（面试标准回答 + 场景）

### 1. REQUIRED（**默认、最常用**）

- 有当前事务：**加入当前事务**
- 无当前事务：**新建事务**
- 特点：**同生共死**，一个回滚全部回滚。

### 2. SUPPORTS

- 有当前事务：加入事务
- 无当前事务：**以非事务运行**
- 特点：随缘事务。

### 3. MANDATORY

- 有当前事务：加入事务
- 无当前事务：**直接抛异常**
- 特点：**强制要求外层必须有事务**。

### 4. REQUIRES_NEW（**高频考点**）

- **不管外层有没有事务，都新建独立事务**
- 外层事务**挂起**，内层执行完再恢复
- 特点：**互相独立、互不影响**。内层回滚不影响外层，外层回滚也不影响内层。

### 5. NOT_SUPPORTED

- **始终以非事务运行**，外层事务直接挂起
- 特点：强制无事务。

### 6. NEVER

- 检测到外层有事务：**抛异常**
- 无事务：正常非事务执行
- 特点：**强制不能有事务**。

### 7. NESTED（**嵌套事务，高频考点**）

- 外层有事务：**嵌套在当前事务中，有独立保存点**
- 外层无事务：新建事务
- 和 `REQUIRES_NEW` 区别（必问）：
    1. **外层回滚 → 内层一定跟着回滚**
    2. **内层单独回滚 → 外层不受影响**
    3. 属于**同一个大事务**，不是完全独立。
