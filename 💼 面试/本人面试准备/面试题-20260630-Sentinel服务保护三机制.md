# 面试题：Sentinel 服务保护三机制 — 2026.06.30

---

> 石化平台微服务架构中，商品服务接口响应变慢时，如何用 Sentinel 保护系统？
> 三道小题：① 雪崩/级联失败 ② 熔断降级（断路器） ③ 限流 + 线程隔离

## 标准答案（三合一，60秒）

**① 雪崩问题**：商品服务慢 → 调用方的 Tomcat 线程被占、等待返回 → 线程越堆越多 → 线程池耗尽 → 调用方自己瘫痪 → 上游再被拖垮 → 一层一层 → 整条链路崩溃（雪崩）。

**② 熔断降级**：断路器三个状态——关闭（正常）、打开（熔断，拒绝所有请求）、半开（冷却后放少量试探）。错误比例或慢调用比例达阈值→打开，走降级（返回默认值/抛异常）。半开试探成功→关闭；失败→回到打开重新计时。

**③ 限流 + 线程隔离**：
- 限流：限制 QPS 或并发线程数，超阈值直接拒绝。三种流控效果：快速失败、预热、排队等待。
- 线程隔离：给每个下游分配独立线程池，满了就拒绝走降级。好处：只拖这一个池，不连累主服务和其他下游。

### 追问：线程隔离 vs 信号量隔离

> 核心差异：线程池隔离有独立线程，调用方线程 ≠ 执行线程，能 interrupt 杀；信号量隔离是 Tomcat 线程自己上，只有计数器没有独立线程，杀不了，只能等下游超时。

| | 线程池隔离 | 信号量隔离 |
|---|---|---|
| 谁干活 | 线程池里独立的兵 | Tomcat 自己的兵 |
| 线程切换 | 有（T1≠T2） | 无（从头到尾 T1） |
| 能强杀吗 | ✅ `interrupt()` | ❌ 只能等超时 |
| 资源成本 | 高（独立线程） | 低（只是计数器） |

---

## 结构速记

```
① 雪崩：慢响应 → 占线程 → 池满 → 瘫痪 → 传染 → 全崩
② 熔断：关闭→打开(拒绝+降级)→半开(试探)→成功关/失败开
③ 限流：QPS/并发 → 超→拒 → 快速失败/预热/排队
④ 隔离：独立池 → 满→拒+降级 → 不影响主服务
```
