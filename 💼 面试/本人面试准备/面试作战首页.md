# 面试作战首页

## 第一性原则

通过面试不取决于背了多少材料，而取决于面试官能否在有限时间内确认四件事：

1. 经历可信：事实、数字和职责经得住连续追问。
2. 能立即干活：Java 基础、业务开发和排障能力足够稳定。
3. 有岗位差异：能把 AI 能力可靠地接入 Java 企业系统。
4. 风险可控：不夸大、不越权，知道方案边界和失败处理。

因此复习顺序固定为：`事实核验 -> 主线口述 -> 两层追问 -> 真实案例 -> 基础补漏`。任何未经确认的数字和案例都不能靠背诵变成事实。

## 一句话定位

> Java 后端工程师，做过复杂企业业务系统，能把 RAG、Function Calling、缓存、会话、权限和流式输出接进现有 Java 业务系统。

这页只解决一个问题：面试前打开哪里、先背什么、怎么复盘。

## 回答结构

任何技术问题都按三段回答：

1. 先说结论。
2. 再说原理。
3. 最后落到项目、踩坑或取舍。

示例：

> RAG 不是让模型自由回答，而是先检索企业内部知识库，再让模型基于检索结果生成回答。项目里文档会经过解析、切分、向量化、Milvus 检索和 Reranker 重排。它能降低幻觉，但不能彻底消灭幻觉，所以还要做引用来源、阈值兜底、权限过滤和用户反馈回流。

## 红线

- 不说 AI 底座全是自己从零做。
- 不说 Function Calling 100% 准确。
- 不说 RAG 彻底解决幻觉。
- 不说 MCP 是石化项目当前生产主链路。
- 不说没有统计依据的提效百分比。
- 不把中州养老课程项目包装成主生产项目。
- 不主动堆 BERT、BM25、MCP、Agent 产品对比，除非面试官追问。

## 模拟面试复盘

- [[Java+AI面试作战手册/99-整改与红线/模拟面试评分表|模拟面试评分表]]

每次面试后只补四类问题：

1. 哪个问题没答出来？
2. 哪句话说过头了？
3. 哪个项目细节不够真实？
4. 哪个 Java 基础被打穿？

复盘动作：

- 能补成逐字稿的，补到逐字稿。
- 能补成三层追问的，补到追问文档。
- 能补成红线的，补到整改 SPEC。
- 能补成真实案例的，补到 Bug 排查文档。
