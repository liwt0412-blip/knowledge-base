# 面试准备：AI Agent 三层架构——Codex / Hermes / Claude 对比

## 这张牌为什么是你的核心弹药

你的面试方向是 AI 应用开发。面试官大概率知道 Agent 是什么、甚至自己用过。他不需要你告诉他「AI Agent 能提效」——这话谁都说得出来。

他需要看到的是：**你对 Agent 产品有系统级的判断力。** 不是用过，是理解为什么这么设计。

三层架构分析就是用来打这个点的。

---

## 三个产品，三层架构

市面上的 AI Agent 产品不是横向竞品。它们纵向分三层：

| 层 | 代表 | 核心能力 | 设计取舍 |
|----|------|---------|---------|
| 模型能力层 | Claude | 理解、推理、表达。长文档总结、方案讨论、代码解释。 | 单独一个模型不等于 Agent——得有工具环境。 |
| 工程执行层 | Codex | 进代码仓库，读代码、改文件、跑测试、看结果、迭代修。 | 窄而深。牺牲了长期记忆和多端能力，换来了工程链路的完整和可控。 |
| Agent 平台层 | Hermes | 长期记忆、跨会话、多端接入、定时任务、技能沉淀、模型可替换。 | 宽而全。牺牲了执行链路的聚焦，换来了 Agent 基础设施的灵活性。 |

三层不是谁替代谁。Codex 可以用 Claude 当模型，Hermes 也可以用 Claude 当 provider。

一个成熟的 Agent 产品要解决几个基本问题——上下文、工具、入口、模型策略、运行环境、安全边界。三者在这几个维度上权重完全不同：

| 维度 | Codex | Hermes | Claude |
|------|-------|--------|--------|
| 上下文来源 | 当前 workspace | 长期记忆 + 跨会话 + skills | 对话 + 上传文档 |
| 入口 | CLI、开发环境 | CLI/桌面/Telegram/Slack 等多端 | 聊天/文档/企业工具 |
| 模型策略 | 产品管理 | **Provider 可替换、多模型** | 绑定 Claude 模型族 |
| 运行环境 | 当前本机 | **本机/容器/SSH/云** | 偏对话，执行靠外部工具 |
| 适合任务 | 修 bug、写测试、review、重构 | 长期助理、定时任务、跨端协作 | 分析、写作、方案讨论、代码解释 |
| 主要风险 | 长期记忆弱、多端能力弱 | 系统复杂、安全面大 | 执行链路弱、平台依赖外部 |

## 面试话术

### 主话术：面试官问「你怎么理解 AI Agent」

这是你的核心回答。30 秒把三个产品 + 三层架构 + 你的判断全串起来。

> "我实际用了三类 Agent 产品，它们不是竞争对手，是三层架构。
>
> 最底层是模型能力——Claude 这类，强在理解、推理、表达。但它不等于完整 Agent，单独一个模型没有工具环境，只能聊不能干。
>
> 往上是工程执行层——Codex 这种。它的设计哲学是窄而深：你给它代码仓库，它改文件、跑测试、看结果、迭代修，工程链路从头到尾做干净。代价是长期记忆和跨会话能力弱。
>
> 最上是平台层——Hermes 的方向。它的设计哲学是宽而全：长期记忆、跨会话、多端接入、定时任务、技能沉淀、模型随便换。它不只想解决单次任务，想做的是 Agent 基础设施。
>
> 我的判断是：三层解决不同问题。日常开发里工程执行层更直接——修 bug 快。但工作流越复杂，平台层的优势越明显——我需要 Agent 记住我的偏好、沉淀技能、接消息平台、跑定时任务。
>
> 我在石化项目的 AI 模块里也跑了类似的架构思路——RAG 系统不是调一次模型就完了，它需要处理文档切分、检索精度、延迟控制、成本优化这整套链路。Agent 不是调一个 API，是把模型装进可运行的系统里。"

### 如果面试官追问：你实际怎么用的

面试官听到上面那段之后，大概率会追一句「你具体怎么用的」。这是好事——说明他感兴趣了。

> "我日常开发用平台层的方案。具体几件事：
>
> 一是把重复劳动交出去——写单元测试、生成 CRUD 代码、做代码审查。Agent 帮我提效大概一半，省下来的时间我花在架构设计和业务逻辑上。
>
> 二是长期沉淀——Agent 会记住我的项目偏好、技术栈、代码风格。下次开新会话它不需要重新学习，直接进入状态。
>
> 三是自动化——我配了定时任务，每天自动拉技术热点做简报。Agent 是我工作流的一部分，不是我每次去打开的网页。
>
> 但有一条底线：所有 AI 生成的代码我都 review。Stack Overflow 2025 调查说 66% 的开发者对 AI 准确率感到沮丧——我也是。提效是 Agent 的，把关必须是人的。"

### 如果面试官追问：你觉得 Codex 和 Hermes 谁更好

这是一个陷阱问题。任何直接回答「X 更好」都显示你没理解设计取舍。

> "看场景。修一个 bug、补一个模块的测试——Codex 更快，因为它不需要知道我是谁、什么历史，直接进仓库干活。
>
> 但如果你需要一个 Agent 长期跟着你——跨项目、跨会话、记住你的偏好、接你的消息平台、跑你的定时任务——Codex 不是为这个设计的。这时候 Hermes 的平台能力就值回来了。
>
> 它们不是 '更好' 的选择，是 '更对' 的选择。"

### 如果面试官追问：模型层 Claude 被你放在最底层，是不是说它不重要

> "不是不重要，是位置不一样。Claude 是大脑——在我用的平台层方案里，Claude 就是我接的 provider 之一。模型层是能力底座，工程执行层和平台层是把这个底座装进可运行系统里的方式。
>
> 没有强模型，上面两层跑不起来。但没有上面两层，强模型只能聊天。"

## 你自己跑了哪些东西（可以说的）

面试官问「你怎么验证这些判断的」——不要空谈，说实际跑过的东西：

| 类别 | 实际跑过 | 可以聊的点 |
|------|---------|-----------|
| 多模型切换 | DeepSeek / Claude / 通义千问 等 provider 切换 | 模型不是绑死的，不同任务选不同模型 |
| Skills 沉淀 | 写了多个技能——知识库操作、面试话术、代码审查 | Agent 可以积累领域能力，越用越顺手 |
| 定时任务 | 每天拉技术热点、定时检查 | Agent 不需要人守着，自己跑 |
| 多端接入 | CLI + 微信消息入口 | Agent 跟着工作流走，不是你去适应它 |
| CDP 桥接 | Chrome DevTools Protocol 接浏览器 | Agent 不止写代码，还能操作浏览器 |
| 跨系统 | WSL ↔ Windows 混合环境 | Agent 的运行环境可以跨操作系统 |

上面这张表不要主动背。面试官追问才展开。

---

## 什么情况下点到为止

你面试方向是 AI 应用开发，大部分场景下三层架构可以主动打。但有两种情况需要收着：

| 情况 | 策略 |
|------|------|
| 面试官明显是传统后端背景、被临时拉来面 AI 岗位 | 提三层架构的核心判断，不展开具体产品名 |
| 面试官明显对 Agent 有偏见（「AI 写代码不靠谱」） | 不争论。落到「我自己也 review 所有代码，AI 是辅助、人是把关」 |
