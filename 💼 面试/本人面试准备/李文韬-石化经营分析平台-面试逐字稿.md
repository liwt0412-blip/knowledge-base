17|17|## 二、项目概述（1-2分钟，必须背到流畅）
18|18|
19|19|这个项目是某能源央企某省全省的经营分析平台，覆盖 14 个地市、近 2000 座加油站。平台 2023 年 2 月正式上线，中间不断迭代——从初期的核心业务跑稳，到扩展报表和数据分析能力，再到最近一年引入了 AI 智能助手。
20|20|
21|21|我所在的是项目实施团队，驻场在省中心机房。跟甲方信息部在同一个办公室，同一套代码仓库、同一套集团基线——日常工作内容跟甲方开发人员没有区别。
22|22|
23|23|我主要负责了两个模块——AI 智能助手模块（全链路设计和实现），还有设备 IoT 接入模块（数据采集、状态监控、告警处理）。技术架构是 Spring Cloud Alibaba 微服务体系，服务间通过 Nacos 做注册发现、OpenFeign 同步调用、RabbitMQ 异步通信，Gateway 统一入口。
24|24|
25|25|AI 模块是我参与最深的部分，整个架构是我主导设计的。如果你想深入了解，我可以展开说一下。
26|26|
27|27|*（如果面试官点头）*
28|28|
29|29|AI 助手最大的设计决策是——我们没做独立的路由层。我把业务函数注册成 Spring AI 的 @Tool，大模型自己判断该走哪条路。调了函数就拿数据库数据直接回答，没调函数就扔给 Python 那边跑 RAG 检索知识库。判断标准不是 LLM 的文字输出，那个不可靠——是靠 Function Calling 的 API 返回，模型调了就是调了，没调就是没调，100% 确定。
30|30|
31|31|Python 侧也不简单——整个 RAG 检索管道是我一手搭的。文档加载用了 PaddleOCR 处理扫描件，切分用了父子块双层策略——子块 300 字做精确匹配、父块 900 字喂给大模型保证上下文完整。向量库是 Milvus，索引是 IVF_FLAT 加 SPARSE_INVERTED_INDEX 双索引，稠密和稀疏两路并行检索然后加权融合。最后再加一层 BGE Reranker 做交叉编码器重排序，检索精度有明显提升。
32|32|
33|33|同时 Java 和 Python 两个服务共享 MySQL 的 conversations 表——Java 侧 Function Calling 完了写入，Python 侧 RAG 生成完了写入，两边读取历史取最近 5 轮。用户感觉不到跨服务。
34|34|
35|35|*（面试官如果继续追问记忆架构）*
36|36|
37|37|会话记忆这块，除了 conversations 表做跨会话持久化，我还基于 Spring AI 的 ChatMemory 接口用 Redis 做单次对话的短期记忆——保留最近 10 轮。这里面有个坑——Spring AI 如果你不显式传 conversationId，所有用户的记忆就串到一起了。必须在对话入口把每个会话的 conversationId 传进去做隔离。

41|41|## 三、AI 模块深度展开（按面试官追问顺序，熟读到能即兴组织）
42|42|
43|43|### 3.1 为什么 Java + Python 双语言？
44|44|
45|45|Java 做企业集成——Function Calling 调业务数据、认证鉴权、配置管理、VectorStore 管理面。Python 做 NLP 流水线——文档加载、切分、BGE-M3 嵌入、混合检索、重排序。BGE-M3 和 BGE Reranker 都是 PyTorch 模型，无法在 JVM 上运行，所以嵌入和检索管道天生归属 Python 侧。两边各做自己最擅长的事，不是重复。
46|46|
47|47|### 3.2 为什么不用 BERT 做意图分类？
48|48|
49|49|一开始确实想过。BERT 微调了二分类模型，准确率能到 90% 以上。但后来发现没必要——垂类 Agent 场景下用户几乎只问业务问题。一个 90% 准确率的分类器放在 LLM 前面做路由，每 100 个问题误判 10 个，在业务安全场景下不可接受。Function Calling 自路由的判断标准是 API 返回，100% 确定，不需要分类器。
50|50|
51|51|### 3.3 Prompt 为什么三个 temperature？
52|52|
53|53|业务操作顾问 temperature=0.1，最关键的设计。业务操作——盘点、调拨、校验——错了会造成实际损失。低 temperature 保证同样的输入产生同样的输出。系统运维顾问 0.3，允许轻度变体。智能答题助手 0.7，出题需要多样性。
54|54|
55|55|### 3.4 数据流怎么走的？
56|56|
57|57|加油机 MQTT → 油站管控机 → 市公司前置机房（只转发，断网缓存 48 小时）→ 省中心。省侧 RabbitMQ 接收后经 Flink 实时清洗，分两路写入——交易数据进 MySQL 做清算，时序数据进 ClickHouse 做实时大屏分析。平台直连省中心 MySQL 和 ClickHouse 读取数据——我们是外包实施团队，数据库由省中心 DBA 统一运维，不自己另起 MySQL 实例。
58|58|
59|59|### 3.5 表怎么设计的？
60|60|
61|61|按集团标准的四层数仓建模——ODS 贴源层、DWD 明细清洗层、DWS 预聚合汇总层、ADS 应用层。查询直接走 ADS 层，毫秒级响应。底层用 MySQL 数据库，分析引擎是 ClickHouse。device_data 表覆盖全省约 8.3 万 IoT 终端——加油机 8100 多台、加油枪 16400 多支、液位仪、充电桩、IoT 网关等等。

75|75|### 非油模块有什么技术难点？ *（平台公共模块，同事负责，本人了解架构）*
76|76|
77|77|调拨审批的库存变更必须放在同一个事务里——扣了 A 站的库存就一定要加到 B 站。并发调拨用乐观锁 version 字段防止超扣。盘点的时候用 Redis 分布式锁隔离销售出库，不用数据库行锁——因为盘点锁要持有几十分钟，长事务锁数据库会耗尽连接池。
78|78|
79|79|### 设备模块有什么技术难点？
80|80|
81|81|8000 台加油机早晚高峰集中上报，高峰期 500 到 1000 条每秒。通过 RabbitMQ 削峰缓冲。设备状态判断用 Redis TTL 机制——设备在线就续期，离线 TTL 自动过期，XXL-Job 每 5 分钟扫一次发现离线就触发告警。比轮询数据库高效得多。
82|82|
83|83|### 核算模块有什么技术难点？ *（平台公共模块，同事负责，本人了解架构）*
84|84|
85|85|全省每天近 20 万笔交易，不能实时聚合做报表——数据库扛不住。每天凌晨 XXL-Job 按四层数仓建模汇总到 ADS 应用层，查询走汇总表。详情页加了当日实时 tab 直接查当天流水补时效性。
