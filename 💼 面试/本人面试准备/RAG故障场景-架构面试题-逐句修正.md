# RAG故障场景-架构面试题-逐句修正

> Hermes 出题 · 深度（架构级） · 1轮修正 · 41→可背版本

## 原题

**面试官：**

「你们 AI 助手是 Java 调 Python RAG 服务，HTTP 调用对吧？那我问你一个实际场景——

下午两点，业务高峰期，Python RAG 服务所在机器的内存爆了，Milvus 连接全部超时。这时候日调用 4.5 万次，DAU 1200。

我问你三个问题：

1. 用户在前端**会看到什么**？
2. 你的系统内部**会发生什么连锁反应**？
3. 你**怎么设计的**，才能让这个故障不拖垮整个经营分析平台？

边想边说，两分钟。」

## 考察维度

| 维度 | 面试官想听到 |
|------|------------|
| 超时控制 | HTTP 调用不设超时 = Tomcat 线程被拖死 |
| 降级策略 | RAG 挂了，用户看到什么？能不能只答 FC 实时数据？ |
| 业务隔离 | 设备告警、报表、CRUD 能不能被 RAG 拖垮？ |
| 可观测 | traceId、错误码、耗时——你怎么发现是它挂了？ |

## 我的原始回答

> 确实，我这个是我的弱点。但是以我的判断的话，前端应该会返回服务器繁忙这种兜底策略，而不是单纯的去用FC兜底。然后，我们内部的话，因为我们用了一个微服务的架构，并且我们用了那个sentinel去进行一个熔断降级，所以说它并不会拖累我们其他的微服务。

## 逐句修正

| 我说的 | 问题 | 正确说法 |
|--------|------|---------|
| "前端返回服务器繁忙" | 太粗——接口级降级，一刀切 500 | **功能级降级**：FC 能答的实时数据正常返回，只有知识库问答走兜底文案 |
| "不是单纯用FC兜底" | ✅ 判断方向对 | FC 只能答实时业务数据，知识库问题必须兜底文案 |
| "Sentinel 熔断降级，不会拖累其他微服务" | ❌ 因果搞反 | 设备/非油/核算不受影响，是因为**调用链不依赖 Python RAG**，不是 Sentinel 的功劳。Sentinel 保护的是 **AI 助手服务自己** |
| 连锁反应 | ❌ 完全没答 | 见下文故障链 |
| 可观测性 | ❌ 没提 | traceId、熔断告警、错误码打点 |

## 核心错误分析

### 错误一：真正的受害者是 AI 助手服务自己

```
Python RAG 挂了（OOM，Milvus 连接超时）
→ Java 侧每个 HTTP 调用阻塞等待响应
→ 一个请求占一个 Tomcat 线程
→ 下午两点高峰，并发上来，200 个 Tomcat 线程全部堵在等 Python
→ AI 助手服务自身不可用（新请求进不来）
```

所以**第一道防线不是 Sentinel，是超时控制**——HTTP 客户端设连接超时 3 秒、读超时按 SSE 首字时间定。超时就断开，线程释放。

Sentinel 是**第二道防线**：异常比例超过阈值 → 熔断，熔断期间直接走 fallback，根本不打下游，给 Python 喘息时间。

### 错误二：降级粒度

「服务器繁忙」把整个接口打死了。但系统里有两类问题：

- 「430101 站现在什么状态」→ 走 FC 查 MySQL 实时数据，**根本不经过 Python**，应该正常回答，用户无感知
- 「操作规程第 5 章怎么说」→ 走 RAG，挂了 → SSE 流里返回明确文案：「知识库问答暂时不可用，设备实时查询正常」

面试官追问降级粒度时，**功能级降级**比接口级 500 高一个段位。

## 可背版本（三段式，90秒）

**problem**：Python RAG 挂了，真正的风险不是「别的服务被拖累」——设备、非油、核算跟 Python 没有调用链——真正的风险是 **AI 助手服务自己的 Tomcat 线程被全部占满**。

**approach**：三道防线。

1. **超时控制**：HTTP 客户端连接超时 3 秒，读超时按 SSE 首字时间定，超时断连释放线程。
2. **Sentinel 熔断**：异常比例超阈值触发熔断，熔断期间走 fallback 直接返回兜底文案，不再打下游。
3. **资源隔离**：调 Python 的 HTTP 客户端用独立连接池，就算打满也不吃 Tomcat 主线程池。

**mechanism**：降级是**功能级**的。FC 能查的实时数据问题正常回答，用户无感知；只有知识库问答类问题走兜底文案。同时 traceId 贯穿，错误码、耗时、熔断事件全部打点——出故障时从监控上先看到熔断告警，不是等用户报障。

## 关键数字速查

| 指标 | 数值 |
|------|------|
| 日调用量 | 4.5 万次 |
| DAU | 1200 |
| HTTP 连接超时 | 3 秒 |
| SSE 读超时 | 按首字时间 |
| Tomcat 默认线程池 | ~200 |
| 功能级降级粒度 | FC 实时查询 → 正常回答；知识库问答 → 兜底文案 |
