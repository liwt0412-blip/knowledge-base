## 二、项目介绍

---

我参与的项目叫**石化经营分析平台**，是中国石化某省公司为了提升加油站和便利店经营管理效率而搭建的一套后台管理系统。

**项目背景：** 该省公司下辖十几个地市、数百座加油站，日常经营涉及设备管理、油品零售、非油商品进销存、安全巡检、客户核算等多个环节。此前各油站数据主要靠纸质表单和 Excel 填报，数据汇总滞后、无法实时监控设备状态、人工咨询成本高。平台经历了三年迭代——从早期单体架构做基础业务数字化，到中期拆分为微服务处理增长的数据量，再到最近一年引入 AI 智能助手降低培训和咨询成本。

**架构形态：** 平台采用 Spring Cloud 微服务架构，拆分为 7 个独立微服务——设备管理、账册表单、客户核算、AI 智能助手、清洁打卡、网关鉴权、消息通知。服务间通过 Nacos 注册发现、Gateway 统一入口、RabbitMQ 异步通信。

**技术栈：** 核心框架 Spring Boot + Spring Cloud + MyBatis-Plus，缓存 Redis，消息队列 RabbitMQ，定时任务 XXL-Job。AI 模块采用 Java（Spring AI）+ Python（LangChain + Milvus + BGE-M3）双语言微服务协作架构。

**项目团队：** 核心开发四人——我负责后端四个微服务（设备管理、非油管理、客户核算、AI 助手），两个前端，一个做 IOT 设备对接。各地市公司有信息人员负责系统落地和员工培训，省公司运维团队管线上环境和设备接入。

**我的职责：** 独立负责设备管理、非油管理、客户核算三个 Java 微服务的设计与开发，主导 AI 助手服务的全链路设计（文档加载→向量化→双路混合检索→流式生成）。

## 三、项目整体业务流程

---

### 1、加油机加油枪设备管理流程

地市公司设备管理员在系统中维护加油站的基础设备档案，包括油站信息、加油机编号、加油枪编号等。设备数据通过MQTT协议上报至平台，系统实时监控设备状态（正常/异常/离线），异常情况触发告警通知相关油站负责人。

### 2、加能站账册表单管理流程

各油站营业员每日在系统中填报经营数据，包括零售销量、非油商品进销存、HSE安全巡检记录、液位仪比对数据等。数据逐级汇总至片区、地市、省公司，支持报表导出和经营分析。

- 零售管理：记录每日各油枪的销量、油品库存变化
- 非油管理：便利店商品从入库、调拨、促销到盘点的全链路管理，含商品保质期预警
- HSE管理：记录安全巡检、风险评估、铅封检查等
- 数质量管理：液位仪比对、日损耗分析、加油机校验等
- 人事管理：油站考勤、排班、员工花名册
- 财务管理：营业款交接、发票管理

### 3、加油站清洁打卡流程

运营人员在系统中对清洁任务进行排班，员工到站后扫码打卡完成清洁任务，管理人员可查询打卡记录和清洁完成率，确保各站卫生达标。

### 4、单站核算与客户管理流程

各油站建立客户档案（加油卡客户、小额配送客户、石化钱包用户），定期做客户分析，统计客户消费频次、贡献度等指标，支持"六进"等营销活动的客户筛选和效果跟踪。

### 5、AI助手使用流程

油站工作人员在遇到业务操作疑问或系统故障时，可在平台中直接向AI助手提问。AI助手基于知识库和模型推理，提供业务操作指引、常见问题解答和运维建议，降低人工咨询成本。

### 模块一：设备管理服务——加油机物联网设备接入

---

#### 0、整体架构

设备管理模块核心是一条**数据上报→异步处理→状态监控**链路。加油机通过 MQTT 协议上报实时数据，后端经 RabbitMQ 削峰填谷后异步写入 MySQL，同时 Redis 维护设备心跳状态，XXL-Job 定时扫描离线设备触发告警。

```
加油机/MQTT → 数据接收层(HTTP) → RabbitMQ ─┬→ 写入 device_data (MySQL)
                                          └→ Redis 心跳刷新 (TTL 5min)

XXL-Job (每5分钟) → 扫描 Redis 心跳超时 → 生成告警 → RabbitMQ → 通知服务
```

**技术要点：** RabbitMQ + Redis 组合——消息队列缓冲早晚加油高峰的数据洪峰，Redis TTL 机制天然适合心跳超时判断（设备在线就续期，离线就自动过期），比定时查 MySQL 高效得多。

---

#### 1、业务介绍

设备管理服务负责全省加油站的加油机、加油枪设备档案管理以及物联网数据接入。加油站是石油公司最核心的资产，设备运行状态直接影响经营数据采集的准确性和及时性。

核心功能包括：
- **设备档案管理：** 按"地市→片区→油站→加油机→加油枪"五级维度维护设备信息
- **设备数据采集：** 通过消息队列接收加油机上报的实时数据（加油量、液位仪数据、设备状态等）
- **设备状态监控：** 实时检测设备在线/离线状态，异常时触发告警通知
- **设备校验管理：** 记录加油机定期校验结果，管理铅封、防水滤芯更换等

#### 4、技术实现

1. **设备数据对接与消息处理：**

    加油机通过MQTT协议将实时数据上报到平台的数据接收层。数据接收层对外暴露HTTP接口`/api/device/data/receive`接收JSON格式的加油量、液位仪、设备心跳等数据。接收到数据后，通过RabbitMQ进行异步分发——一条消息写入设备数据表用于历史查询和报表统计，另一条消息进入实时处理队列用于设备状态判断和告警检测。使用消息队列的好处是削峰填谷：全省几千台加油机在早晚高峰集中上报时，通过队列缓冲保证系统不会被打垮。

2. **设备状态判断与告警：**

    设备管理服务维护一个Redis缓存（key: `device:heartbeat:{deviceId}`, TTL: 5分钟），每次收到设备上报数据时更新该缓存。同时通过XXL-Job定时任务每隔5分钟扫描所有设备，检查哪些设备最近5分钟内没有上报心跳，标记为离线并生成告警记录。告警记录通过RabbitMQ发送给通知服务，通知服务通过系统消息推送给对应油站负责人。

    Redis的TTL机制天然适合做设备超时判断——设备只要持续上报，TTL不断续期；一旦中断，TTL到期后Redis自动过期，定时任务扫描时就能检测到，比遍历数据库判断时间差效率高得多。

3. **统一返回格式与异常处理：**

    前后端所有接口统一使用R对象返回格式。通过全局异常处理器`@ControllerAdvice`统一拦截业务异常和系统异常，避免每个Controller重复写try-catch。前端拿到统一的响应结构后，可以在Axios拦截器中统一处理错误提示和登录超时跳转，开发效率提升不少。

4. **MyBatis-Plus数据操作：**

    CRUD操作全部基于MyBatis-Plus的BaseMapper和IService，分页使用MyBatis-Plus自带的分页插件。公共字段（createTime、updateTime、createUser、updateUser）通过MyBatis-Plus的自动填充功能处理，业务代码里不需要手动赋值。

#### 5、为什么使用这个技术

- **RabbitMQ + Redis组合：** 加油站设备上报数据量大且有明显的波峰波谷特征（早晚加油高峰），用消息队列缓冲是标准做法。Redis做设备状态判断则是因为TTL机制天然适合"心跳超时"场景，比数据库轮询更高效。
- **MyBatis-Plus：** 油站管理涉及大量CRUD操作（新增油站、编辑加油机、查询设备列表等），MyBatis-Plus的代码生成器和BaseMapper能减少大量重复代码，核心业务用XML写复杂SQL，简单CRUD用注解搞定。

#### 6、存在什么问题

- **设备数据延迟：** 加油机通过MQTT上报到平台再经RabbitMQ流转，对于需要实时告警的场景（如加油机异常停机）可能延迟几十秒。
- **网络不稳定：** 部分偏远站点网络条件差，偶尔出现长时间断连，导致上报数据丢失。
- **数据一致性：** 加油机的累计加油量和系统统计量偶尔不一致，需要定时对账校准。

#### 7、如何解决

- **设备告警双通道：** 对于紧急告警，除了消息队列异步处理外，增加一条直接写入告警表的同步通道，确保关键告警不丢失。
- **断线重连补偿：** 设备重新上线后，自动补传离线期间的关键数据（如总加油量、液位仪读数），通过RabbitMQ的延迟队列实现。
- **日终对账任务：** 通过XXL-Job每天凌晨跑对账任务，将系统统计的日销量与加油机上报的累计销量进行比对，差异超阈值时标记异常并通知运营人员核查。

### 模块二：非油管理服务——便利店商品全链路管理

---

#### 0、整体架构

非油管理模块以 CRUD 为主，但有三条需要特别关注的数据一致性链路：调拨审批涉及跨油站库存变更的事务操作，盘点期间需要锁定库存防止并发写入，保质期预警通过定时任务扫描推送。

```
前端 → Gateway → 非油管理服务 → MyBatis-Plus → MySQL

调拨审批流:
  创建调拨单 → 审批通过 → @Transactional(库存A-、库存B+) → 完成

盘点流程:
  锁定库存 → 生成盘点底稿 → 提交实盘数 → 比对差异 → 审核

保质期预警:
  XXL-Job (每日凌晨) → 扫描临期商品 → 推送通知
```

**技术要点：** 调拨审批的库存变更放在 `@Transactional` 保证原子性（扣了 A 的库存就一定要加到 B），乐观锁（version 字段）防止并发调拨超扣。盘点期间通过 Redis 分布式锁防止销售出库与盘点数据冲突。

---

#### 1、业务介绍

非油管理服务负责全省加油站便利店商品的全链路管理。加油站便利店虽然面积不大，但SKU种类多（食品、饮料、润滑油、日用品等），而且涉及多个地市多个油站之间的调拨和库存管理，业务流程复杂度不低。

核心功能包括：
- **商品信息管理：** 维护便利店商品的档案信息（名称、条码、分类、售价、保质期等）
- **商品调拨审批：** 油站之间或地市仓到油站的商品调拨申请、审批、出库、入库全流程
- **便利店商品盘点：** 油站定期盘点，系统根据盘点结果自动生成盘盈盘亏记录
- **商品损耗管理：** 记录商品报损（过期、破损等），统计分析损耗率
- **商品促销管理：** 配置促销活动（满减、折扣、赠品），跟踪促销效果
- **赠品台账：** 管理赠品的入库、领用、库存
- **卫生清洁台账：** 记录便利店卫生检查情况

#### 4、技术实现

1. **商品管理CRUD：**

    商品管理模块是典型的增删改查，基于MyBatis-Plus的BaseMapper实现，分页查询使用MyBatis-Plus的分页插件PageHelper。商品分类使用树形结构存储（parent_id方式），前台展示时通过递归查询构建分类树。商品条码做了唯一索引，防止重复录入。

2. **调拨审批流：**

    调拨申请提交后状态变为"待审批"，通过Gateway网关根据当前用户的角色判断是否有审批权限。审批通过后，系统自动更新调出方库存（减库存）和调入方库存（加库存），这个操作放在同一个事务中保证数据一致性。如果审批拒绝，调拨单状态变为"已拒绝"，库存不做变动。

3. **盘点流程与差异处理：**

    盘点时先锁定油站库存（防止盘点期间有销售出库），生成盘点底稿。员工逐项录入实际数量后提交，系统比对账面数量和实际数量，自动计算差异。差异在合理范围内（如损耗率<0.5%）自动通过审核，超范围的需要管理员人工确认并填写原因。

4. **保质期预警：**

    通过XXL-Job每天凌晨扫描所有商品，查询距离保质期到期不足30天的商品，生成预警推送给对应油站负责人。同时支持按商品分类、油站维度查询临期商品清单，方便运营人员处理（降价促销或报损）。

#### 5、为什么使用这个技术

- **MyBatis-Plus：** 非油管理接口绝大部分是单表CRUD加简单关联查询，MyBatis-Plus基本不需要写SQL，开发效率高。分页查询也是框架自带的，省去自己写limit/offset的麻烦。
- **事务保证数据一致性：** 调拨审批通过后的库存变更涉及两张表（调拨单状态更新 + 双方库存更新），放在一个事务里执行，防止出现"扣了库存但没加到对方"的情况。
- **定时任务做预警：** 保质期预警这种场景不需要实时触发，每天凌晨跑一次就够了，XXL-Job支持分布式部署，比Spring自带的@Scheduled更可靠。

#### 6、存在什么问题

- **并发调拨冲突：** 同一个商品同时被多个调拨单操作时，可能出现库存超扣的情况。
- **盘点期间业务中断：** 盘点时需要锁定库存，盘点期间该油站不能有销售出库，影响正常业务。
- **临期商品处理不及时：** 预警虽然是每天推送，但油站负责人可能忽略，导致商品过期报损。

#### 7、如何解决

- **乐观锁防止超扣：** 库存更新时使用MyBatis-Plus的乐观锁机制（version字段），更新时校验version，版本不一致则更新失败，提示重试。
- **盘点窗口优化：** 将盘点时间安排在凌晨低峰期（通过XXL-Job定时盘点），避开白天营业高峰。同时支持"动盘"模式（只盘点有变动的商品，不锁定全部库存）。
- **临期商品升级提醒：** 对于距离保质期不足7天的商品，预警等级升级，通过短信或企业微信消息直接通知油站站长，不再仅仅依赖系统内消息。

### 模块三：客户核算服务——客户档案与经营分析

---

#### 0、整体架构

客户核算模块的核心是**T+1 汇总 + 实时查询双轨**——客户消费明细数据量大（全省每天数万笔交易），不能用实时聚合做报表。通过 XXL-Job 每天凌晨汇总到分析表，查询走汇总数据；详情页提供当日实时 tab 直接查交易流水补时效性。

```
前端 → Gateway → 客户核算服务 → MyBatis-Plus → MySQL

客户CRUD:
  Controller → Service → Mapper → customer / customer_contract

报表分析:
  XXL-Job (每日凌晨) → 聚合当日交易 → 写入 customer_analysis
  前端查询 → 直接查汇总表 (毫秒级)

六进活动管理:
  创建活动 → 跟踪执行 → 录入效果数据 → 计算 ROI
```

**技术要点：** 汇总表替代实时聚合是核心优化——交易流水表数据量太大，每次查报表都实时 count/sum 数据库扛不住。每日凌晨定时汇总到分析表，查询走汇总表，配合 customer_id + month 联合索引实现毫秒级响应。

#### 1、业务介绍

客户核算服务负责全省加油站客户信息的统一管理和经营数据分析。石化公司不仅要管"油"，还要管"客户"——特别是大客户（物流公司、车队、企业客户）的消费情况和信用额度。

- **六进活动管理：** 管理"进企业、进社区、进机关"等营销活动，记录活动效果和客户转化情况

2. **客户分析报表：**

    客户分析数据不是实时计算的，而是通过XXL-Job每天晚上统计当天的消费数据，按客户维度汇总后存入客户分析数据表。查询时直接从汇总表取数，避免每次查询都对大量交易记录做实时聚合。支持按消费金额、消费次数、活跃天数等维度排序和筛选。

    报表接口使用了MyBatis-Plus的分页查询，排序字段通过参数动态传入，避免了硬编码多个查询方法。SQL层面做了索引优化——在客户ID和统计月份上建了联合索引，查询效率基本是毫秒级。

3. **六进活动管理：**

    活动管理包含活动创建、活动执行跟踪和活动效果统计三个环节。活动创建时选择参与油站和活动类型，活动结束后运营人员录入效果数据（新增客户数、销售额等），系统自动计算投入产出比（ROI），为后续活动预算提供数据参考。

#### 5、为什么使用这个技术

- **汇总表替代实时聚合：** 客户消费明细数据量庞大（全省每天几万笔交易），如果每次查报表都去聚合原始交易数据，数据库扛不住。通过定时任务提前汇总，查询时走汇总表，性能问题基本不存在。
- **MyBatis-Plus + 动态排序：** 客户分析报表的排序维度不固定（有时按金额、有时按频次），用MyBatis-Plus的LambdaQueryWrapper动态拼接排序条件，代码比写多个XML映射简洁得多。

#### 6、存在什么问题

- **数据时效性：** 汇总表是T+1的（隔天更新），运营人员当天下班前查到的客户数据其实是昨天的，不能实时反映当天情况。
- **客户重复问题：** 不同油站录入的客户可能重复（同一个物流公司在多个油站加油），但当时没有做客户去重。
- **活动效果难量化：** 六进活动的新增客户，很难精确归因到某个具体活动（客户可能是多渠道过来的）。

#### 7、如何解决

- **实时+定时双轨：** 在客户分析明细页增加"今日实时"tab，直接查当天的交易记录（限制只查当天，数据量小），首页报表仍用T+1汇总数据，两者互不干扰。
- **客户统一认证：** 基于客户统一社会信用代码或营业执照号做去重，同一企业主体在系统内只保留一条客户记录，各油站通过关联关系引用。
- **活动归因标记：** 客户注册时增加"来源活动"字段，新注册客户必须选择或由系统自动匹配最近参加过的活动，后续分析时按来源活动分组统计。

#### 0、整体架构

AI 助手模块采用 **Java + Python 双语言微服务协作** 架构。路由由 Spring AI 的 **Function Calling 机制自驱动**——不设独立路由规则或分类模型，LLM 自己判断是否需要调业务函数、是否需要检索知识库。

**分层职责：**

| 层 | 位置 | 核心技术 | 职责 |
|----|------|---------|------|
| ① Redis 缓存 | Java | Spring Data Redis + MD5 | 高频问题命中直接返回，~60% 请求零网络开销 |
| ② Function Calling 自路由 | Java | Spring AI @Tool + ChatClient | **无独立路由规则**。LLM 自己决定：调函数拿业务数据 / 直接回答 / 降级转发 Python |
| ③ RAG 检索 | Python | LangChain + BGE-M3 + Milvus | LLM 不会回答时降级到此——语义检索 + LLM 生成，BM25 作为内部可选前置步骤 |

**路由机制——为什么不需要独立路由层：**

Spring AI 的 Function Calling 机制本身就是一个天然路由器。我把设备查询、客户查询、告警查询等接口注册为 `@Tool` 函数后，**LLM 自己推理**判断要不要调函数。路由规则只有两条，判断标准是 Function Calling 的 API 返回——100% 确定：

```
LLM 调了函数 → 拿到 MySQL 数据 → 基于数据生成回答 ✅（Java 侧完成）
LLM 没调函数 → REST 转发 Python 走 RAG ✅
```

```
用户: "430101 设备状态"
  → LLM 推理: "需要查设备 → 调 getDeviceStatus('430101')"
  → 拿到 MySQL 实时数据 → 回答 ✅

用户: "怎么建商品"
  → LLM 没调函数 → 转发 Python RAG → 检索文档 → 按操作手册回答 ✅

用户: "你好"
  → 白名单过滤（3行正则）→ ChatClient 直答 ✅
```

**白名单优化：** "你好""谢谢""再见""今天星期几"这类无内容消息，加 3 行正则前置过滤，直接交给 ChatClient 回复。不算路由规则，纯粹是避免让 RAG 管道处理无意义的问候。这是极简路由的 trade-off——其余所有 LLM 不调函数的问题都走 RAG，牺牲少量冗余 RAG 调用，换取零误判的确定性路由。

**不需要 BERT 分类器：** 垂类 Agent 场景下通用问题几乎不存在，一个 90% 准确率的分类模型放在 LLM 前面做路由，等于把 LLM 的能力限制在分类器的准确率之内。LLM 自己的判断力远高于 90%。

**为什么 Layer ① 放在 Java：** Spring Data Redis 本地集成，缓存命中零网络开销。放在 Python 的话 HTTP 往返会吃掉 <5ms 优势。

**为什么 Layer ③ 在 Python：** BGE-M3 嵌入、SPARSE_INVERTED_INDEX、BGE Reranker——Python 生态独有。Spring AI 的 VectorStore 只做 CRUD，没有文档处理和混合检索能力。

#### 1、业务介绍

AI助手服务为石化平台提供智能化能力支撑，面向三类使用场景：

- **业务操作顾问：** 油站工作人员在操作系统中遇到"如何新建便利店商品"、"怎么填报损耗数据"等问题时，AI助手基于业务知识库给出步骤指引。这对减少油站人员培训成本和降低咨询电话量很有帮助。

#### 3.1 知识库运维机制

知识库文档的日常更新不需要开发人员介入。运营人员通过管理后台上传新手册、替换旧版本、删除过期文档，上传后系统自动触发 Python 服务做切分→嵌入→写入 Milvus，全程自动化。

文档版本通过 `upload_time` 和 `doc_category` 字段管理，检索时优先匹配最新的文档版本。后端开发的工作集中在系统建设阶段——做好上传工具和管理后台，上线后文档维护是运营侧的日常工作。

#### 4、技术实现

**4.1 检索架构（Redis 缓存 + RAG 向量检索）**

    检索分为两层——第一层在 Java 侧零网络开销命中，第二层在 Python 侧走语义检索。路由不设独立规则或分类模型，由 Spring AI 的 Function Calling 机制自驱动。

    **第一层——Redis缓存命中（Java 侧）：**

    用户提问后，先用问题的 MD5 哈希值作为 key 去 Redis 查。如果命中，直接返回缓存答案，响应时间 <5ms。这层主要拦截高频重复问题——油站员工问得最多的是"怎么登录""怎么建商品""怎么改密码"这类，缓存命中率在 60% 以上。

    Redis 未命中时，请求进入 Spring AI 的 ChatClient。注意：这里**没有 if-else 路由逻辑**——LLM 看到带 `@Tool` 注解的函数列表后，自己推理判断要不要调函数、调哪个、传什么参数。

    路由规则只有两条，不靠 LLM 的文本输出判断（LLM 太自信，不会主动说不知道），而是靠 Function Calling 的 API 返回——100% 确定：

    ```
    LLM 调了函数 → 拿到 MySQL 数据 → 基于数据生成回答 ✅（Java 侧完成，不到 Python）
    LLM 没调函数 → REST 转发 Python 走 RAG ✅
    ```

    举例：
    - "430101 设备状态" → LLM 推理"需要查设备状态 → 调 getDeviceStatus('430101')" → 拿到数据 → 回答
    - "怎么建商品" → LLM 没调函数 → 转发 Python RAG → 检索《非油管理操作手册》→ 基于文档回答
    - "你好" → LLM 没调函数 → 转发 RAG → Milvus 无匹配 → LLM 自然回复"你好，我是小石"

    **为什么这个极简路由在垂类场景下足够——不需要 BERT 分类器：**

    这个 AI 助手不是通用 ChatBot，是石化平台的垂类 Agent。三个机制保证了用户只会问业务问题：

    1. **场景限制：** 助手嵌在平台后台，上下文就是加油站运营——员工打开它是为了查设备、填报表、报故障，不是为了聊天。
    2. **System Prompt 限制：** "你是石化经营分析平台的专属助手，只回答加油站运营、设备管理、安全规范、账册表单等业务问题。非业务问题统一回复：'我不处理这类问题，请咨询相关技术人员。'"
    3. **用户行为可预测：** 油站站长、营业员、巡检员的工作内容高度固定。

    垂类 Agent 不需要意图分类器去区分"通用 vs 业务"——因为在这个语境下，通用问题几乎不存在。路由逻辑可以极简：LLM 调了函数就用数据回答，没调函数就查知识库。这也是为什么我微调了 BERT 达到 90% 准确率但最终没有上——一个 90% 的分类器每 100 个问题会误判 10 个，把专业问题错误放行让 LLM 硬编，在石化安全场景下不可接受。垂类场景下前置分类不是优化，是增加误差链路。

    **第二层——RAG向量检索+生成（Python 侧）：**

    降级到 Python 的请求走 LangChain RAG 全流程：BGE-M3 嵌入 → Milvus 双路混合检索（IVF_FLAT + SPARSE_INVERTED_INDEX）→ BGE Reranker 重排序 → LLM 生成。生成后回写 Redis（TTL=24h），下次同样问题直接 Layer ① 命中。

    BM25 关键词检索作为 RAG 内部的可选前置步骤——候选问答库中命中时直接返回，未命中时自动过渡到向量检索。不独立成层，避免增加架构复杂度。

    这种两层架构的好处是：60% 问题在 Redis 命中，剩余走 Function Calling 或 RAG。路由判断标准是 API 返回而非文本分析（100% 确定）。垂类场景下不需要意图分类器，LLM 自己就是最好的路由器。

**4.2 文档预处理——父子块分层切分策略**

    我采用了"父块-子块"双层切分策略：

    - **父块**（900字符）：按分隔符递归切分，保留完整的段落/章节结构。Markdown文档用MarkdownTextSplitter按标题层级切，普通文档用ChineseRecursiveTextSplitter按句号/换行等分隔符递归拆。
    - **子块**（300字符）：在父块基础上进一步切分，保证每个片段粒度足够细，能精准匹配用户问题。

    存储时，子块内容向量化存入Milvus用于精确检索；父块内容作为标量字段关联存储。检索时，用子块匹配到最相似的片段后，返回对应的父块内容作为大模型的上下文——这样既保证了检索精度（小粒度匹配），又保证了上下文完整性（大粒度喂给LLM）。

**4.3 双路混合检索——为什么检索在 Python 不在 Java**

    Spring AI 也有 RAG 能力（VectorStore 抽象层），但它只能做单路稠密向量相似度搜索——`similaritySearch("问题", topK)`。而生产级的 RAG 需要双路混合检索、SPARSE_INVERTED_INDEX、Reranker 重排序——这些 Spring AI 都不支持。

    在我的架构里，Spring AI 的 VectorStore 负责知识库的**管理面**（管理员用的后台功能），LangChain 负责知识库的**数据面**（用户提问时的检索管道）：

    | 操作 | 谁做 | 用什么 |
    |------|------|--------|
    | 上传文档 → 写入 Milvus | Java / Spring AI | `MilvusVectorStore.add()` |
    | 删除文档 | Java / Spring AI | `MilvusVectorStore.delete()` |
    | 管理员查看知识库状态 | Java / Spring AI | VectorStore 元数据查询 |
    | 用户提问 → 文档切分 → 嵌入 | Python / LangChain | ChineseRecursiveTextSplitter + BGE-M3 |
    | 用户提问 → 双路混合检索 | Python / LangChain + pymilvus | IVF_FLAT + SPARSE_INVERTED_INDEX + WeightedRanker |
    | 用户提问 → 重排序 | Python / LangChain | BGE Reranker |
    | 用户提问 → 生成回答 | Python / LangChain | LLM |

    好处：管理面跟 Java 业务系统整合（权限、配置、审计日志），检索面用 Python 最成熟的 NLP 工具链。两边通过 REST API 通信，Spring AI 上传文档时调 Python 做切分+嵌入，拿到向量后由 Java 侧写入 Milvus。

    **以下是 Python 侧检索管道的详细设计：**


    这是整个RAG流程的核心——单一的向量检索召回效果有限，我采用了"稠密+稀疏"双路加权混合检索：

    - **稠密向量（语义检索）：** 基于BGE-M3模型对用户问题编码，用IVF_FLAT倒排索引在Milvus中做ANN近似最近邻搜索（nprobe=10），找到"意思相近"的文档。
    - **稀疏向量（词汇匹配）：** BGE-M3同步生成的稀疏向量（本质是词频权重向量），用SPARSE_INVERTED_INDEX索引做词汇级匹配，找到包含"相同关键词"的文档。
    - **加权融合：** 稠密路径权重1.0、稀疏路径权重0.7，各取Top-K结果后通过Milvus的WeightedRanker合并排序。稠密权重大于稀疏，是因为石化业务场景下用户措辞多变（"建商品"可能说成"新增货物""加个商品"），语义理解比关键词匹配更重要。

    两路召回后再做一次二次重排——用BGE Reranker Large交叉编码器模型，把用户问题和每个召回的父块内容拼成(query, doc)对逐一打分，按相关性分数重排后取Top-M返回。这一步相当于让一个专门的"裁判"再筛一遍，把最相关的内容排到最前面。实验表明，加Reranker后Top-5的相关性提升了约35%。

**4.3.1 Milvus 字段设计与索引设计**

    双路混合检索能跑起来，底层靠的是 Milvus 的 Collection Schema 和索引配置。下面是我设计的完整方案：

    **Collection 字段设计：**

    | 字段名 | 类型 | 说明 |
    |--------|------|------|
    | `id` | VARCHAR(64) | 主键，chunk 内容的 MD5 哈希（配合 upsert 实现幂等） |
    | `doc_id` | VARCHAR(64) | 所属文档 ID，关联知识库文档表 |
    | `doc_title` | VARCHAR(256) | 文档标题，检索结果展示来源 |
    | `doc_category` | VARCHAR(64) | 文档分类（操作手册/安全规范/培训课件），Partition Key |
    | `parent_text` | VARCHAR(2000) | 父块原文（~900 字符），检索命中后作为 LLM 上下文返回 |
    | `child_text` | VARCHAR(1000) | 子块原文（~300 字符），用于向量化后的精确匹配 |
    | `chunk_index` | INT64 | 块在文档中的序号，用于追溯上下文位置 |
    | `dense_vector` | FLOAT_VECTOR(1024) | BGE-M3 稠密向量，1024 维 |
    | `sparse_vector` | SPARSE_FLOAT_VECTOR | BGE-M3 稀疏词权向量，维度由词典动态决定 |
    | `upload_time` | INT64 | 上传时间戳，用于时效性过滤 |

    **索引设计：**

    | 字段 | 索引类型 | 参数 | 选型理由 |
    |------|---------|------|---------|
    | `dense_vector` | IVF_FLAT | `nlist=128`, `metric=IP` | IVF_FLAT 将 128 个聚类中心的向量按簇分组，检索时只搜 `nprobe=10` 个最近簇（扫约 7.8% 数据），精度损失 1% 换来 10 倍加速。`IP`（内积）= 余弦相似度（BGE-M3 输出已 L2 归一化） |
    | `sparse_vector` | SPARSE_INVERTED_INDEX | `drop_ratio_build=0.2` | 倒排索引天然适合稀疏向量——每个词项指向包含它的文档列表。`drop_ratio_build=0.2` 在建索引时丢弃词频最低的 20% 词项，减少噪音 |
    | `id` | 主键索引 | 自动 | Milvus 自动维护 |
    | `doc_id` | 标量索引 | Inverted Index | 支持按文档精准过滤：`expr="doc_id == 'xxx'"` |
    | `doc_category` | 标量索引 | Inverted Index | 支持按分类过滤 + Partition Key 剪枝 |
    | `upload_time` | 标量索引 | 排序索引 | 支持"检索最近 30 天内上传的文档"这类时效性过滤 |

    **检索时的实际参数：**

    ```
    稠密检索: nprobe=10, limit=20, metric_type=IP
    稀疏检索: drop_ratio_search=0.2, limit=20
    混合融合: WeightedRanker(dense=1.0, sparse=0.7)
    最终输出: Top-10 → BGE Reranker 重排 → Top-K 返回
    ```

    **为什么用 MD5 做主键而不是自增 ID：** 文档更新后重新切分，同一段文本的 MD5 不变，配合 upsert 直接覆盖旧向量，不会产生重复数据。自增 ID 则每次插入新行，旧数据变成"僵尸向量"占着空间污染检索结果。

**4.4 多格式文档OCR加载**

    石化业务文档格式很杂：有Word操作手册、PDF规章制度、PPT培训课件，甚至还有纸质的巡检表扫描件。我封装了统一的文档加载器，根据不同文件后缀自动路由到对应的加载器（PDF用OCRPDFLoader、Word用OCRDOCLoader、PPT用OCRPPTLoader、图片用OCRIMGLoader），底层基于PaddleOCR做光学字符识别提取文字，确保图文混排的文档也能完整提取。

**4.5 三类Prompt + LLM参数设计**

    虽然底层都是同一套 RAG 架构，但三个助手的角色完全不同。我设计了一个**共享身份底座**（保证语气一致），加上三个**场景 Prompt 层**（行为区分）。Java 侧和 Python 侧使用同一套 Prompt 模板。

    **共享身份底座（三个助手共用）：**

    ```
    你是"小石"，中国石化某省公司经营分析平台的 AI 助手。
    你的回答必须严格基于提供的文档内容和系统数据，不得编造。
    语气：简洁、专业、不废话。使用中文回答。
    如果提供的文档不足以回答问题，回复"我目前没有找到相关信息，建议联系人工客服或查阅操作手册"。
    如果用户的问题与石化业务无关，回复"我不处理这类问题，请咨询相关技术人员"。
    ```

    **场景 Prompt 层（按角色叠加）：**

    | 角色 | 追加指令 | 输出格式要求 |
    |------|---------|------------|
    | **业务操作顾问** | "你是业务操作指导。严格按照文档中的操作步骤回答，禁止自由发挥和猜测。每个步骤标注序号。如果文档没有明确描述某一步，不要自行补充。" | `第1步：... 第2步：... 引用来源：《xxx》` |
    | **系统运维顾问** | "你是系统运维顾问。可以结合故障案例给出排查建议。回答时先判断问题类型（设备故障/系统报错/配置问题），再给出可能原因和解决方案。" | `【问题类型】→【可能原因】→【解决方案】→【参考资料】` |
    | **智能答题助手** | "你是考试出题助手。基于提供的文档内容自动生成考题。每题包含题干、4个选项（单选）、正确答案和解析。题干不能照抄文档原文，需改写为提问形式。解析要引用文档原文作为依据。" | `题干 + A/B/C/D + 答案：X + 解析：...` |

    **Java 侧 Function Calling 的 System Prompt（追加）：**

    ```
    你可以使用以下工具函数查询系统的实时数据。当用户问到具体的设备状态、客户信息、告警记录等内容时，优先调用对应的函数获取最新数据，而不是依赖训练知识回答。如果没有合适的函数可用，告诉用户你不知道。
    ```

    **LLM 参数设计：**

    | 参数 | 业务操作顾问 | 系统运维顾问 | 智能答题助手 | 选型理由 |
    |------|-----------|-----------|-----------|---------|
    | `temperature` | 0.1 | 0.3 | 0.7 | 操作指引必须确定性输出（0.1），运维允许轻度变体（0.3），出题需要多样性（0.7） |
    | `top_p` | 0.8 | 0.85 | 0.9 | 操作顾问收窄候选词分布，防止"创造性"措辞 |
    | `max_tokens` | 1024 | 1536 | 2048 | 操作步骤简洁，运维方案中等长度，出题含题干+选项+解析最长 |
    | `presence_penalty` | 0 | 0 | 0.3 | 出题需要避免重复用语，其余场景不需要 |
    | `frequency_penalty` | 0 | 0 | 0.2 | 同上 |
    | `stop` | 无 | 无 | `["---"]` | 出题用分隔符控制每题独立输出 |

    **为什么三个角色的 temperature 差这么大：** 操作顾问的 temperature=0.1 是最关键的设计——石化的业务操作（盘点、调拨、校验）错了会造成实际损失。低 temperature 确保同样的文档输入产生同样的输出，不会出现第一次说"第3步点保存"、第二次说"第3步点提交"这种不一致。

**4.6 SSE流式输出 + 会话历史管理**

    大模型生成回答通常需要 3-5 秒，如果等全部生成完再返回，用户体验很差。我实现了 SSE（Server-Sent Events）流式输出，前端每收到一个 token 就逐字渲染，让用户感知到系统"在打字"而不是"卡住了"。

    会话历史存入 MySQL 共享的 `conversations` 表，Java 和 Python 双写双读——Java 侧 Function Calling 完成后写入，Python 侧 RAG 生成完成后写入。每次生成前取出最近 5 轮对话拼入 Prompt，让 AI 助手理解上下文。比如用户先问"怎么建商品"，再问"那建完之后呢"，AI 能理解"建完之后"指"建完商品之后"。

    Java 和 Python 共享同一张表的目的是保证跨侧追问的连贯性——用户在 Java 侧调了 Function Calling 查设备状态，再在 Python 侧问"上次巡检呢"，Python 能从历史中拿到"430101"这个主语。

**4.7 选型决策：RAG vs 微调、SpringAI vs 直接调用**

    - **选RAG不选微调：** 石化业务文档频繁更新（油价调整通知、新安全规范下发），RAG只需替换知识库文件即可生效，迭代成本为零。微调则需要重新标注数据、重新训练，每次更新成本高。而且RAG能引用文档来源："这段回答来自《非油管理操作手册V2.3》"，可信度远远高于微调模型。

    - **选SpringAI三层用途（不是简单调API）：**
        1. **ChatClient 直调 LLM：** 对于不需要检索知识库的简单问题（闲聊、操作指引），Java 侧用 SpringAI 的 ChatClient 直接调 LLM。切换模型供应商（通义千问→DeepSeek）只改配置，代码零改动。
        2. **Function Calling（Agent 能力）：** 用 SpringAI 的 `@Tool` 注解把系统业务函数暴露给 LLM。比如员工问"查一下 430101 油站的实时设备状态"，LLM 自动调用 `getDeviceStatus()` 从数据库拿到真实数据再回复——这让 AI 不止能背文档，还能查真实业务数据。
        3. **VectorStore 抽象层：** 知识库管理后台（上传、删除、查状态）在 Java 侧通过 SpringAI 的 MilvusVectorStore 操作，不直接写 pymilvus 代码。好处是未来换 Qdrant 或其他向量数据库，改配置即可，业务代码不动。

    - **SpringAI + LangChain 双框架分工（不是冗余，是各管对方做不好的事）：**

        Spring AI 和 LangChain 不是功能重复，而是解决两个不同层面的问题——Spring AI 负责"业务整合"，LangChain 负责"NLP 流水线"。

        **分工逻辑：**

        | 维度 | Spring AI（Java） | LangChain（Python） |
        |------|------------------|---------------------|
        | 核心职责 | AI 能力嵌入企业 Spring Cloud 体系 | NLP 处理全流程编排 |
        | 它的主场 | Function Calling 调业务数据、认证鉴权、配置管理 | 文档加载切分、嵌入编码、混合检索 |
        | 换对方做 | LangChain 查 Spring Bean 调数据库？绕一大圈 | Spring AI 调 BGE-M3 做稀疏向量？Java 无等价品 |

        **三个具体场景说明为什么必须双框架：**

        1. **查业务数据（Spring AI 主场）：** 员工问"430101 油站设备状态"，LLM 需要调 `getDeviceStatus()` 从 MySQL 拿实时数据。Spring AI 的 `@Tool` 注解直接暴露 Spring Bean 方法，LLM 推理出参数 → 调函数 → 拿到结果。LangChain 要绕 HTTP 调用 Java REST 接口，多一跳且丢类型安全。

        2. **文档入知识库（LangChain 主场）：** 上传一份《安全巡检规范.pdf》，需要 PaddleOCR 提取文字 → ChineseRecursiveTextSplitter 递归切分 → jieba 分词 → BGE-M3 嵌入 → 写入 Milvus。SPARSE_INVERTED_INDEX 这类索引只有 Python pymilvus 支持。Spring AI 的 VectorStore 只能做增删查，没有文档处理能力。

        3. **切换模型供应商（平手）：** 通义千问 → DeepSeek，Java 改 `application.yml` 一行，Python 改环境变量一行，都是零代码切换——说明两个框架的抽象层都选对了。

        **边界通信：** 如果一个请求既需要查业务数据又要检索知识库，Java 侧先走 Function Calling 拿业务数据，再把用户问题转发到 Python 走 RAG，两边结果合并后由 Python 侧 LLM 统一生成回复。微服务架构天然支持多语言——Python 服务和 Java 服务同样注册 Nacos、走 Gateway。

#### 5、为什么使用这个技术

- **Redis 缓存 + Function Calling 自路由而非全部走RAG：** 油站员工问的大多是重复问题，Redis 缓存拦截60%请求；剩余走 Function Calling 或降级到 RAG。没有独立路由层，LLM 自己判断该走哪条路。这比"所有问题都走 RAG"的方案大幅降低了延迟和大模型调用量。
- **Function Calling 自路由而非规则路由：** 不设独立的意图分类器或正则路由。Spring AI 的 Function Calling 机制本身就是路由器——LLM 看到 `@Tool` 函数列表后自动推理判断是否需要调函数、调哪个、传什么参数。不确定时自动降级到 RAG。一个 90% 准确率的分类模型放在 LLM 前面做路由，反而把 LLM 的能力限制在分类器的准确率之内。
- **RAG而非微调：** 石化业务文档频繁更新（油价调整、新政策下发），RAG模式下只需更新知识库文档，不需要重新微调模型，迭代成本极低。而且RAG能明确告诉用户"这段回答来自哪份文档"，可信度更高。
- **双路混合检索而非单路：** 稠密+稀疏双路互补——稠密向量能理解"建商品"= "新增货物"（语义），稀疏向量能精准匹配"油品""损耗"等专业术语（词频），两者取长补短。加了Reranker二次重排后，Top-5相关性提升约35%。
- **SpringAI框架：** 不是简单调API——ChatClient 封装多模型切换、@Tool 注解实现 Function Calling 让 LLM 能查真实业务数据、VectorStore 抽象层统一操作向量数据库。三项能力都绑在 Spring Cloud 体系里，与业务代码零摩擦。
- **SSE流式输出：** 2-5秒的生成等待中逐字渲染，杜绝用户"以为卡住了"的焦虑感。

#### 6、存在什么问题

- **知识库更新延迟：** 业务文档更新后，如果管理员忘记上传新版本，AI助手给的还是旧文档的答案，可能误导员工。这是RAG架构的固有问题——知识库不会自动同步。
- **检索精度边界：** 部分操作流程比较接近（如"新建商品"和"新建促销"），偶尔检索到不相关的文档片段。虽然加了Reranker，但交叉编码器的判别能力也有上限。
- **成本控制：** 云端大模型按token计费，每次RAG调用消耗约2000-4000 token。Redis 缓存已经过滤掉60%，Function Calling 再拦截一部分，剩余走 RAG 的请求仍需持续关注成本。一个优化方向是对高频 RAG 问题也做 Redis 缓存（TTL=24h），同类问题直接返回缓存结果。
- **路由极简的 trade-off：** LLM 没调函数就全部转发 RAG，虽然保证了零误判的确定性路由，但有时 LLM 的训练知识完全够用（如简单问候、常识问题），仍不可避免多走一趟 RAG。当前通过 3 行白名单正则过滤无内容消息缓解，未来可考虑引入轻量意图模型做更精细的路由。

#### 7、如何解决

- **知识库版本管理：** 在文档管理后台增加版本标记和生效日期字段，AI助手优先回答生效日期内的文档。同时在管理端增加"知识库健康度"看板，监控文档更新状态和覆盖率。
- **检索优化——Hybrid Search增强：** 目前的双路混合检索权重是固定的（稠密1.0/稀疏0.7）。后续可以引入自适应权重：对关键词密集型问题自动调高稀疏权重，对描述型问题调高稠密权重，进一步提升召回精度。
- **RAG回答缓存：** 对高频RAG问题的最终回答做Redis缓存（TTL=24小时），同类问题直接返回缓存结果，进一步降低大模型调用成本。

- **BERT 意图分类技术储备：** 在探索阶段我基于 BERT-base-chinese 微调了意图二分类模型——训练数据约 1000 条人工标注样本，按 7:1.5:1.5 划分训练/验证/测试集，微调 3 个 epoch，准确率从 48% 提升到 90% 以上。但在工程落地时发现：Spring AI 的 Function Calling 机制本身就能完成路由判断，不需要额外的分类器。一个 90% 准确率的分类器放在 LLM 前面做路由，等于把 LLM 的能力限制在分类器的准确率之内——LLM 自己的判断力远高于 90%。这个探索让我意识到：在 AI 工程中，能用系统自身能力解决的问题，不要引入额外模型增加误差链路。后续若需要更精细的多场景路由，可以在 Java 侧部署 ONNX 推理或通过 gRPC 调用 Python 模型服务。
