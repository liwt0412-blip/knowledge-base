# MySQL 项目感面试练习（逐句修正）

> 练习时间：2026-07-17 | 题型：石化项目 MySQL 优化实战追问

---

## 原题

**面试官：你们石化项目每天近 20 万笔交易，MySQL 怎么扛住的？**

---

## 第一轮（55 分）

### 原始回答

> 我们会从三个地方来进行优化，第一个是索引设计，第二个是慢查询优化，第三个是读写分离。索引设计的话，我们是在创表的时候会建立一个联合索引，把经常需要查询的数据建立联合索引。然后慢查询的话，我们先是通过线上日志排查到慢 SQL，然后对 SQL 语句去进行 explain，观察它是哪个环节出了问题，是索引失效还是别的什么原因。最后，我们会进行读写分离，就是开始分库分表。

### 修正要点

| 问题 | 严重度 | 修正 |
|------|--------|------|
| 读写分离 = 分库分表 | ❌ 致命 | 读写分离是主从复制 + 路由；分库分表是水平拆分，完全不同 |
| 索引设计太泛 | ⚠️ | 必须说具体表名 + 字段顺序 + 为什么这样排 |
| EXPLAIN 只说名字没展开 | ⚠️ | 三个核心字段：key / type / Extra |
| 缺项目场景 | ⚠️ | 每点都要落到项目里的真实表/真实场景 |

### 读写分离 ≠ 分库分表

| 概念 | 做什么 | 技术手段 |
|------|--------|----------|
| **读写分离** | 主库写、从库读，分摊压力 | MySQL 主从复制 + 应用层路由 |
| **分库分表** | 数据量太大，单库单表扛不住 | 水平拆分（ShardingSphere/MyCat） |

石化项目日 20 万笔、累计千万级，单库能扛，重点在索引和缓存，不是分库分表。

---

## 第二轮（70 分）

### 修正后回答

> 我们会从三个方面进行优化，第一个是索引设计，第二个是排查慢查询，第三个是读写分离。索引的话，比如设备状态表，我们会将设备 ID、设备状态和创建时间建立联合索引，符合最左匹配原则——设备 ID 最常用放最左边。慢 SQL 查询的话，我们通过线上日志，阈值设两秒，超过的用 EXPLAIN 定位——看 key 有没有命中索引、type 是 ALL 还是 index、Extra 有没有 filesort。排查出来后该建索引建索引，该优化 SQL 优化 SQL。读写分离的话，主库写、从库读，通过主从复制分担压力。

### EXPLAIN 三字段速查

| 字段 | 看什么 | 危险信号 |
|------|--------|----------|
| **key** | 实际用了哪个索引 | NULL = 没走索引 |
| **type** | 访问类型 | ALL（全表扫）/ index（全索引扫）→ 需优化 |
| **Extra** | 额外信息 | Using filesort（额外排序）/ Using temporary（临时表）→ 性能杀手 |

### 扣分项

- [x] ~~读写分离=分库分表~~ 已纠正
- [ ] EXPLAIN 三个字段背熟（key/type/Extra）
- [ ] 发音：SQL 不是"收口"，慢 SQL 不是"慢收口"

## 核心教训：项目感

八股回答必须有项目落点。面官要的不是教科书定义，是「你做过的东西里怎么体现这个知识点」。

| 没项目感（❌） | 有项目感（✅） |
|---|---|
| "建联合索引，把常查询的字段放一起" | "设备状态表建了 `(device_id, create_time, status)` 联合索引，device_id 最左" |
| "优化 SQL" | "月汇总报表 SQL 从 3 秒降到 0.1 秒，加了 `(station_id, report_month)` 联合索引" |
| "用到 EXPLAIN 分析" | "看 key/type/Extra 三个字段，有次发现 key=NULL 全表扫，补索引后解决" |
