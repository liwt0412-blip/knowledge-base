# AI Agent 辅助开发流程与代码验证面试回答

## 一、这份文档的定位

这份文档用于回答面试官这类问题：

- 你平时怎么用 AI 写代码？
- AI Agent 生成的代码你怎么验证？
- 你会不会直接用 AI 生成的代码？
- AI 编程工具到底给你带来了什么价值？
- 你怎么避免 AI 生成 bug、安全漏洞或不符合项目规范的代码？

核心口径：

> 我不会把 AI 生成代码当成最终结果，而是把它当成高质量初稿。AI 负责提高实现和分析效率，我负责需求判断、工程边界、测试验证、代码 Review 和最终质量。

不要把重点放在“我用过哪些工具”，而要放在：

> 我有一套可验证的 AI 辅助开发流程。

---

## 二、第一性原理：为什么 AI 写代码必须验证？

AI 写代码的本质不是“替代工程师”，而是“根据上下文生成一个可能正确的实现”。

它的问题在于：

- 它可能误解需求；
- 它可能不知道项目里的隐藏约定；
- 它可能生成看似能跑、但边界错误的代码；
- 它可能忽略权限、事务、幂等、并发、安全；
- 它可能引入和现有架构不一致的写法；
- 它可能只满足 happy path，不覆盖异常分支。

所以我对 AI 代码的态度是：

> AI 负责给方案和初稿，人负责判断能不能进项目。

面试中不要说：

> AI 写的代码基本都能直接用。

更稳的说法是：

> AI 能明显减少重复性工作，但最终还是要经过需求校验、代码 Review、测试和风险检查。

---

## 三、我使用 AI/Agent 开发的完整流程

### 1. 先把需求整理成 PRD

我不会一上来就让 AI 写代码，而是先把需求整理成一个轻量 PRD。

PRD 不一定要很正式，但至少要讲清楚：

- 背景：为什么要做这个功能；
- 目标：解决哪个业务问题；
- 用户：谁会使用；
- 主流程：用户从哪里进入，做什么操作，看到什么结果；
- 异常流程：失败、无权限、数据不存在时怎么办；
- 约束：权限、性能、兼容性、数据范围；
- 验收标准：做到什么程度算完成。

面试话术：

> 我使用 AI 开发时，第一步不是让它写代码，而是先把需求整理成轻量 PRD。因为 AI 很擅长生成实现，但如果需求目标、用户流程和验收标准不清楚，它生成得越快，偏得也可能越快。

---

### 2. 再把 PRD 转成技术 Spec

PRD 解决“做什么”和“为什么做”，Spec 解决“怎么做”。

我会让 AI 帮我把 PRD 拆成技术 Spec，但我会人工审核。

Spec 里一般包括：

- 涉及哪些模块；
- 新增或修改哪些接口；
- 入参、出参、错误码；
- 需要哪些表或字段；
- 是否需要 Redis 缓存；
- 是否涉及 MQ、定时任务、第三方接口；
- 权限怎么校验；
- 事务边界在哪里；
- 日志和审计怎么做；
- 需要哪些测试用例。

面试话术：

> PRD 更偏业务，Spec 更偏工程实现。我会让 AI 帮忙把 PRD 拆成接口、表结构、权限、缓存、事务和测试点，但最终要由我确认技术边界，尤其是权限、数据一致性和兼容性。

---

### 3. 把 Spec 拆成可执行任务

Spec 过大时，我不会让 AI 一次做完，而是拆成小任务。

常见拆法：

- 接口定义；
- DTO / VO；
- 数据库变更；
- Service 业务逻辑；
- Mapper / SQL；
- 权限和日志；
- 单元测试；
- 接口测试；
- 文档更新。

每个任务都要能独立验证。

面试话术：

> 我会把 Spec 拆成小的可验证任务，而不是让 Agent 一次性改完整个功能。比如先完成接口骨架，再补业务逻辑，再补测试和异常分支。这样每一步都能 review 和回滚。

---

### 4. 明确任务边界

我会先确认：

- 这个需求解决什么业务问题；
- 涉及哪些模块；
- 是新增功能、修 bug、重构，还是补测试；
- 输入输出是什么；
- 哪些接口、表、缓存、权限会被影响；
- 有没有兼容旧逻辑的要求。

面试话术：

> 我使用 AI 开发时，第一步不是写 prompt，而是把任务边界讲清楚。因为 AI 的输出质量很依赖上下文，如果需求边界不清楚，它很容易生成看似合理但不符合业务的代码。

---

### 5. 给 AI 足够的项目上下文

AI 最容易出错的地方，是不了解当前项目已有结构。

所以我会让它先看：

- 相关 Controller；
- Service 接口和实现；
- Mapper / Repository；
- DTO / VO；
- 数据库表结构；
- 已有相似功能；
- 项目异常处理方式；
- 权限注解和日志规范；
- 单元测试或接口测试样例。

面试话术：

> 我不会只给 AI 一个孤立需求，而是会让它先读项目里相似功能和相关类。这样生成的代码更容易符合原有工程风格，比如异常返回、权限注解、事务边界、DTO 命名和日志格式。

---

### 6. 让 AI 先出方案，不急着改代码

对于复杂任务，我会先让 AI 输出实现方案，而不是直接生成代码。

我会看它是否说清楚：

- 要改哪些文件；
- 每个文件改什么；
- 数据流怎么走；
- 是否需要事务；
- 是否影响接口兼容；
- 是否需要补测试；
- 有哪些风险点。

面试话术：

> 如果任务稍微复杂，我会先让 AI 给方案。我主要看它有没有理解业务链路、改动范围是否收敛、有没有破坏已有逻辑。方案过关后，再让它生成代码。

---

### 7. 小步生成，不一次性大改

我不会让 AI 一次性重写一大片代码。

更稳的方式是：

- 先改一个最小闭环；
- 跑测试；
- 再补异常分支；
- 再补日志、权限、参数校验；
- 最后整理代码。

面试话术：

> 我更倾向于小步使用 AI。一次让它改太多文件，review 成本会很高，也容易引入隐藏问题。小步生成、小步验证，风险更可控。

---

### 8. 本地验证和测试

AI 生成代码后，我不会直接合并。

我会根据改动类型做验证：

- 编译是否通过；
- 单元测试是否通过；
- 接口测试是否通过；
- 关键 SQL 是否符合预期；
- Redis key 是否正确；
- MQ 是否幂等；
- 权限是否拦截；
- 异常分支是否符合统一返回；
- 日志是否能支持排查。

面试话术：

> AI 生成代码后，我至少会做编译、单测或接口测试。如果是后端业务接口，我会重点测正常流程、参数异常、权限不足、数据不存在、重复提交和事务回滚。

---

### 9. 人工 Code Review

AI 代码必须 Review。

我 Review 的重点不是格式，而是工程风险：

- 是否破坏原有架构；
- 是否绕过权限；
- 是否事务边界不对；
- 是否吞异常；
- 是否 SQL 有性能风险；
- 是否缓存和数据库不一致；
- 是否存在并发问题；
- 是否日志不足；
- 是否把敏感信息写进日志；
- 是否新增了不必要的依赖。

面试话术：

> 我 review AI 代码时，重点看工程风险，不只是看语法。比如权限、事务、幂等、并发、缓存一致性、SQL 性能和异常处理，这些是 AI 最容易忽略但后端项目最容易出事故的地方。

---

### 10. 对照 PRD / Spec 做验收

代码写完后，不能只看测试是否通过，还要回到 PRD 和 Spec 做验收。

我会检查：

- PRD 中的主流程是否全部完成；
- 异常流程是否处理；
- Spec 中的接口、权限、事务、缓存是否都实现；
- 验收标准是否可验证；
- 有没有遗漏非功能要求，比如性能、日志、兼容性；
- 有没有产生 PRD / Spec 之外的无关改动。

面试话术：

> 最后验收时，我会回到 PRD 和 Spec，而不是只看代码。因为测试通过只能说明某些用例没问题，但需求是否完整、边界是否覆盖、是否有无关改动，还是要对照原始需求和技术方案确认。

---

### 11. 最后才提交

通过验证后，我才会整理提交。

提交前会确认：

- 改动范围是否和需求一致；
- 有没有无关格式化；
- 有没有调试代码；
- 有没有敏感配置；
- 测试结果是否可说明；
- 是否需要补充文档或注释。

面试话术：

> AI 可以参与实现，但最终提交还是按正常工程流程走。能不能合并，看的是测试、review 和改动风险，不是看是不是 AI 写的。

---

## 四、我的五层验证法

面试时可以直接说：

> 我一般按五层验证 AI 生成代码：需求、结构、测试、边界、风险。

### 第一层：需求验证

看代码是否真的解决业务问题，而不是只实现了表面功能。

重点看：

- 输入输出是否正确；
- 业务规则是否覆盖；
- 是否兼容旧逻辑；
- 是否遗漏状态流转。

### 第二层：结构验证

看代码是否符合项目原有结构。

重点看：

- Controller 是否只做参数接收；
- Service 是否承载业务逻辑；
- Mapper 是否只处理数据访问；
- DTO / VO 是否清晰；
- 异常处理是否统一；
- 命名和分层是否符合已有风格。

### 第三层：测试验证

看代码能不能被实际验证。

重点看：

- 编译；
- 单元测试；
- 接口测试；
- SQL 验证；
- Mock 外部依赖；
- 回归相关功能。

### 第四层：边界验证

看异常场景是否覆盖。

重点看：

- 参数为空；
- 数据不存在；
- 用户无权限；
- 重复提交；
- 外部接口超时；
- MQ 重复消费；
- 缓存不存在；
- 并发更新。

### 第五层：风险验证

看是否有上线风险。

重点看：

- 权限绕过；
- 事务不一致；
- SQL 慢查询；
- 缓存脏数据；
- 敏感信息泄露；
- 日志不可排查；
- 新增依赖过重；
- 改动范围过大。

## 五、不同任务怎么用 AI？

## 五、Loop Engineering：怎么高质量使用 Agent？

这部分用于回答：

- 你怎么让 Agent 真的参与开发？
- 你和普通“复制粘贴 AI 代码”的区别是什么？
- 你怎么避免 Agent 乱改？

核心口径：

> 我使用 Agent 时，不是把每一步都写死，而是给清晰目标、提供项目规则、让它先探索、再小步实现、最后自我 review。这样 Agent 才能形成工程闭环。

### 1. 给目标，不给步骤

不推荐：

```text
打开 UserController.java，第 45 行加一个 @GetMapping。
```

更推荐：

```text
给用户列表接口加分页，参考项目里其他分页接口的风格。
```

面试话术：

> 我不会把 Agent 当代码补全工具来用，而是给它目标和约束，让它根据项目上下文找实现方式。这样能发挥它读代码和适配项目风格的能力。

### 2. 复杂任务先 Explore

复杂任务不要直接让 Agent 改代码。

先让它做：

- 阅读项目结构；
- 找相似接口；
- 梳理数据流；
- 汇报可能影响的文件；
- 提出实现方案。

面试话术：

> 复杂任务我会先让 Agent explore 项目结构，让它先汇报涉及哪些模块和风险点。我确认方案后，再让它进入实现阶段。这样能降低它盲改的概率。

### 3. 用项目约定约束 Agent

如果项目长期使用 Agent，可以把约定写进类似 `AGENTS.md` 的文件：

- 项目技术栈；
- 分层规范；
- 命名规范；
- 测试命令；
- 不允许改哪些目录；
- 提交前必须检查什么。

面试话术：

> 我理解 Agent 质量不只取决于模型，还取决于项目约束。把代码风格、分层规范、测试命令和风险边界沉淀下来，Agent 每次执行任务时就不会从零猜。

### 4. 让 Agent 自我 Review

做完后不要马上接受结果，可以让 Agent 先 review 自己：

```text
请 review 你刚才的改动，重点检查权限、事务、异常、并发、缓存一致性和测试覆盖。
```

面试话术：

> 我会让 Agent 先自我 review 一轮，再由我人工 review。它能发现一些明显问题，但最终质量还是我负责。

### 5. 复杂任务拆阶段

不要一次做完：

```text
用户注册 + 验证码 + 创建账号 + 登录态 + 前端页面
```

更稳的是：

```text
先设计接口和表结构 -> 确认 -> 后端实现 -> 确认 -> 测试 -> 文档
```

面试话术：

> Agent 最怕一次性大改，所以我会把复杂任务拆阶段。每个阶段都能单独 review 和验证，发现问题也容易回滚。

### 6. 让 Agent 主动发现坑

可以让 Agent 做预审：

```text
读一下这个模块，告诉我可能的风险点和历史风格。
```

面试话术：

> 我会让 Agent 主动扫描潜在问题，比如重复代码、异常处理不一致、权限遗漏、测试缺失。它不是替我做最终判断，而是帮我扩大检查面。

## 六、不同任务怎么用 AI？

### 1. 写 CRUD 或 DTO

AI 适合做：

- DTO / VO 初稿；
- Controller 基础结构；
- Service 方法骨架；
- Mapper XML 初稿；
- 参数校验；
- 简单单元测试。

人工重点验证：

- 字段是否和表结构一致；
- 状态枚举是否正确；
- 权限注解是否补齐；
- 是否符合已有接口风格。

面试话术：

> 对重复性 CRUD，我会让 AI 生成初稿，但字段、权限、状态流转和异常返回必须人工确认。

### 2. 修 bug

AI 适合做：

- 帮我读报错堆栈；
- 定位可能的出错位置；
- 对比相似逻辑；
- 提供修复方案；
- 补回归测试。

人工重点验证：

- 是否复现了 bug；
- 修复是否只针对根因；
- 是否引入新问题；
- 是否覆盖回归测试。

面试话术：

> 修 bug 时，我不会只让 AI 猜原因。我会先复现问题，把日志、堆栈、相关代码给它，让它辅助分析。修完后必须用同样的场景验证 bug 消失，再跑相关回归。

### 3. 写单元测试

AI 适合做：

- 根据已有方法生成测试用例；
- 补正常、异常、边界场景；
- 生成 Mock；
- 帮忙发现漏测分支。

人工重点验证：

- 测试是否真的断言业务结果；
- 不是只测方法被调用；
- Mock 是否过度；
- 边界条件是否覆盖。

面试话术：

> 我觉得 AI 很适合帮忙补测试，尤其是根据业务方法生成正常和异常用例。但我会检查测试有没有有效断言，不能只是为了覆盖率写空测试。

### 4. 重构

AI 适合做：

- 找重复代码；
- 提出抽取方案；
- 批量改命名；
- 辅助拆方法；
- 生成迁移步骤。

人工重点验证：

- 对外行为是否不变；
- 改动范围是否可控；
- 测试是否覆盖；
- 是否引入过度抽象。

面试话术：

> 重构时我会更谨慎。AI 可以帮我找重复和生成改法，但我会优先保证行为不变，小步提交，并通过测试确认没有破坏原有功能。

### 5. 写文档和接口说明

AI 适合做：

- 根据代码生成接口说明；
- 整理字段含义；
- 生成变更说明；
- 总结排查过程。

人工重点验证：

- 文档是否和代码一致；
- 是否泄露内部信息；
- 是否夸大效果；
- 是否保留关键限制。

面试话术：

> 文档类任务我会大量用 AI，但最终会人工核对事实，尤其是接口字段、限制条件和风险说明。

## 七、面试官高频追问

### 1. 你会直接用 AI 生成的代码吗？

推荐回答：

> 不会直接合并。我会把 AI 生成的代码当成初稿，必须经过 review 和测试。尤其是后端项目，权限、事务、异常、并发和数据一致性这些不能靠 AI 自觉保证。

### 2. AI 写错代码怎么办？

推荐回答：

> 我会先定位它错在需求理解、上下文不足，还是实现细节。如果是上下文不足，我会补充相关代码和业务规则；如果是实现问题，我会让它缩小改动范围，然后用测试验证。AI 错了不可怕，关键是不能没有验证闭环。

### 3. 你怎么判断 AI 代码质量？

推荐回答：

> 我主要看五点：是否满足需求，是否符合项目结构，是否有测试验证，是否覆盖异常边界，是否存在权限、事务、性能、安全风险。

### 4. AI 会不会让开发者能力退化？

推荐回答：

> 会不会退化取决于怎么用。如果只是复制粘贴，肯定会退化；但如果把 AI 当成代码初稿、测试生成、方案对比和 review 辅助工具，反而会逼着自己更关注架构、边界和质量。

### 5. 使用 Agent 编程和普通 ChatGPT 写代码有什么区别？

推荐回答：

> 普通对话式 AI 更像给代码片段和建议，Agent 编程更强调进入项目上下文：它能读仓库、改文件、跑命令、看测试结果，再继续迭代。它的价值在于工程闭环，但也意味着更要控制权限和改动范围。

### 6. Agent 自动改代码会不会有风险？

推荐回答：

> 有风险，所以我会限制它的任务范围，并且要求小步修改。涉及权限、支付、删除、审批、数据修复这类高风险逻辑，不会让 Agent 自动决定，必须人工确认。

### 7. 你怎么防止 AI 引入安全问题？

推荐回答：

> 我会重点检查权限绕过、SQL 注入、敏感信息日志、越权数据访问、接口参数校验和依赖安全。Prompt 不是安全边界，最终还是靠后端代码和测试控制。

### 8. AI 生成测试靠谱吗？

推荐回答：

> AI 生成测试很有用，但不能只看覆盖率。我会看测试有没有有效断言，有没有覆盖异常分支、边界条件和回归场景。否则测试只是看起来很多，实际挡不住 bug。

### 9. 你怎么把 AI 用在团队协作里？

推荐回答：

> 我觉得比较适合的方式是让 AI 辅助生成测试、接口文档、代码 review checklist 和问题排查记录。但核心业务判断、架构决策和上线风险评估还是要由开发人员负责。

### 10. 面试官问：你用 AI 提效多少？

推荐回答：

> 我不会给一个绝对百分比，因为不同任务差异很大。重复性代码、测试初稿、文档整理提效明显；复杂业务设计、线上问题排查、权限和事务这种场景，AI 更多是辅助分析，最终还要靠工程判断。

### 11. 你说的 PRD 和 Spec 在 AI 开发里有什么用？

推荐回答：

> PRD 用来约束业务目标，Spec 用来约束工程实现。没有 PRD，AI 可能不知道真正要解决什么问题；没有 Spec，AI 可能生成一个能跑但不符合项目架构的实现。所以我会先让 AI 辅助整理 PRD，再把 PRD 拆成技术 Spec，最后按 Spec 小步实现和验证。

### 12. PRD、Spec、测试之间是什么关系？

推荐回答：

> PRD 定义要解决的业务问题和验收标准，Spec 定义技术实现路径，测试用例验证 PRD 和 Spec 是否真的落地。AI 可以辅助生成这三类材料，但最终要人工确认它们是否一致。

### 13. 你会让 AI 直接从需求写代码吗？

推荐回答：

> 简单重复任务可以，但稍微复杂的业务功能我不会直接从一句需求写代码。我会先让 AI 帮我整理 PRD，再拆技术 Spec 和任务列表，确认边界后再小步实现。这样比直接生成代码更可控。

---

## 八、结合你的石化项目怎么聊？

### 第一层：轻描淡写，不抢主项目

> 我平时会用 AI 辅助开发，比如生成重复性代码、补测试、做代码解释和 review。但我不会直接使用 AI 代码，都会经过人工验证。

### 第二层：结合后端开发

> 比如后端接口开发时，我会让 AI 先参考项目里相似 Controller、Service、DTO 的写法，生成一个初稿。然后我会检查权限注解、参数校验、事务边界、异常返回、SQL 和日志，最后跑接口测试或单元测试。

### 第三层：结合 AI 项目

> 因为我自己也做 AI 助手落地，所以我对 AI 的边界比较敏感。AI 能生成内容，但不能替代后端的权限、校验和审计。这个思路同样适用于 AI 编程：Agent 可以生成代码和跑测试，但最终是否能进项目，要看工程验证闭环。

---

## 九、最适合背的 30 秒回答

> 我使用 AI 编程工具时，不会直接让它从一句话需求开始写代码。我一般先把需求整理成轻量 PRD，再拆成技术 Spec 和可执行任务，然后给它项目上下文，让它小步生成代码。生成后按需求、结构、测试、边界、风险五层验证。AI 对重复代码、测试初稿、文档和代码解释很有帮助，但最终质量还是靠开发者 review、测试和 PRD/Spec 验收闭环。

---

## 十、最适合背的 1 分钟回答

> 我把 AI 生成代码当成初稿，不会直接合并。我的流程一般是先把需求整理成轻量 PRD，明确背景、目标、用户流程、异常流程和验收标准；然后把 PRD 拆成技术 Spec，比如接口、表结构、权限、缓存、事务、日志和测试点；再把 Spec 拆成小任务，让 Agent 小步实现。
>
> 实现时我会先给它项目上下文，比如相似 Controller、Service、DTO、Mapper 和测试样例。生成之后，我会做几层验证：第一看 PRD 需求是否满足；第二看代码结构是否符合项目原有分层；第三跑编译、单测或接口测试；第四检查异常分支，比如参数为空、数据不存在、权限不足、重复提交；第五看工程风险，比如事务、幂等、并发、SQL 性能、缓存一致性和敏感日志。
>
> 所以我觉得 AI 的价值是提高初稿效率和分析效率，但不能替代工程判断。越是企业后端项目，越不能只看代码能不能跑，还要看能不能安全、稳定、可维护地上线。

---

## 十一、红线提醒

不要说：

> AI 写的代码我基本直接用。

不要说：

> 我用 AI 提效 50%。

不要说：

> Agent 能自动完成所有开发任务。

不要说：

> 测试通过就一定没问题。

更稳的表达是：

> AI 是初稿和辅助分析工具，最终要经过人工 review、测试和风险验证。

> Agent 更适合可验证任务，比如补测试、修局部 bug、解释代码和小范围重构。

> 涉及权限、资金、审批、删除、数据修复的高风险操作，必须人工确认。

## AI 协作方法论追问（四象限 / LOOP / 复利模式的翻译口径）

> 使用边界：本章方法论只在面试官主动追问"你平时怎么用 AI 写代码 / 怎么保证 AI 生成代码质量 / AI 协作怎么推广"时展开；自我介绍只讲结果层一句话（重度使用 Claude Code/Cursor，有 Spec 先行、AI 执行、人工验证的协作流程）。"四象限""LOOP""复利模式"是内部术语，面试时不直接说出口，全程用本章的翻译口径。

### 一、流程层（LOOP 的翻译版）

**面试官问法**："你平时怎么用 AI 写代码？"

**稳妥回答**（可直接背）：

> 我有一个固定的闭环。第一步是我自己先写规范，把边界定清楚：这次改动的范围是什么、涉及哪些接口、有什么约束、做到什么程度算验收。第二步才是让 AI 按规范执行，它可以读项目、改文件、跑命令。第三步是验证，编译和测试必须过，而且我有一个原则：交付前必须自己验证过，没验证过的代码我不会说成已经完成。最后一步很多人会忽略——我会把这次协作里确认下来的规范、踩过的坑、验证方法沉淀回知识库，下一轮同类任务直接当上下文复用。所以对我来说 AI 不是一次性的问答工具，是一个越用越顺手的协作流程。

**真实例子**（被追问"举个具体例子"时用）：

> 比如我们 AI 助手的回答反馈功能。我先写了一份 Spec，定清楚反馈入口的接口契约、表结构、一人一判的约束和验收标准，然后让 AI 按 Spec 实现，一共生成和修改了 8 个文件，编译一次通过，整个流程一次跑通。这能跑通的前提不是 AI 多聪明，而是 Spec 把边界定死了，它没有自由发挥的空间。

> 口径注意：这个例子讲的是"Spec 先行 + AI 执行 + 编译验证"的协作流程，不展开为该功能的线上运行状态；反馈入口的项目口径仍以 [[石化经营分析平台-项目深挖追问底稿]] 4.11 为准（方案是我设计的）。

### 二、体系层（四象限的翻译版）

**面试官问法**："什么任务交给 AI、什么自己干？"

**稳妥回答**（可直接背）：

> 我心里有一个分层，按两个维度看：AI 对这类任务的擅长程度，和这类任务出错的代价。
>
> AI 擅长、出错代价低的，比如样板代码、DTO、文档初稿、测试用例，我全放给它，我只看结果。
>
> AI 擅长但出错代价高的，比如核心业务逻辑，我让它起草，但我逐行 review，重点看权限、事务、异常分支这些地方。
>
> AI 不擅长、出错代价又高的，比如架构取舍、技术选型、业务口径决策，这些是我主导的，AI 只负责出参考方案和对比分析，决策权在我。
>
> 剩下 AI 不擅长、代价也低的边角，我顺手就自己写了，不值得协调一轮。
>
> 这个分层标准不是拍脑袋来的，是踩坑踩出来的。举个真实的例子：调用第三方库的 API 之前，我现在必须先在依赖锁定的版本里确认这个类和方法真的存在。因为 AI 的记忆天然偏向用得最多的新版本，我之前按它给的写法调一个老版本的拦截器，编译失败了两次。从那以后，"第三方库 API 先验证版本事实"就成了我的固定规则。

> 内部对照（不对外讲）：本节"擅长程度 × 出错代价"是面试翻译版，方便面试官理解任务怎么分工；内部四象限原文是乔哈里视窗视角——"你知道/不知道 × AI 知道/不知道"，解决的是每一轮协作里"什么时候下指令、什么时候问方案、什么时候给反馈、什么时候让 AI 扫盲区"。两套说的是同一件事的两面：信息分布决定协作方式，协作方式落到任务上就是分层。被深追"这套分层怎么来的"时，可以讲"我按自己和 AI 各自的信息边界来分工，我不确定的让它出方案我来拍板，我俩都没想到的让它主动扫一遍风险"，不要把四象限名词抛出去。

### 三、复利层（复利模式的翻译版）

**面试官问法**："AI 协作怎么持续提效？怎么在团队推广？"

**稳妥回答**（可直接背）：

> 我的做法是每一轮协作的产出都不浪费。规范、排错记录、验证方法，这些东西我会沉淀成可复用的文档，它们会变成下一轮任务的上下文。效果是同类任务的边际成本递减：第一次做要写完整规范，第二次同类任务 AI 直接继承上一轮沉淀的规则，我只需要补差异部分。这才是提效的实质——不是某一次生成得快，而是协作的资产在累积。
>
> 如果在团队推广，我不会先讲理念，只推两个动作：一是规范模板，让任何人在把任务交给 AI 之前先把范围、接口、约束、验收标准写清楚；二是验证清单，编译、测试、权限、事务这几项交付前必须过一遍。我选这两个是因为它们杠杆率最高：规范模板直接决定 AI 输出质量的上限，验证清单直接兜住质量的下限，而且都不依赖个人能力，新人拿来就能用。

### 四、红线

- 不把方法论讲成"我的工作是 AI 替我做的"。口径永远是：规范我定、执行 AI 干、验证我来，主动权在人。
- 不抛出未经翻译的内部术语。"四象限""LOOP""复利模式"这些词面试里一个都不说，只讲本章的方法、动作和结果。
- 不承诺具体提效百分比，除非有统计口径。被追问时按本文档"七、10"的回答：分任务类型讲差异，不给绝对数字。
