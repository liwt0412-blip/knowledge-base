# AI Agent 三层渐进面试回答

> 定位：加分项，不是主线。  
> 使用场景：面试官主动问 AI Agent、AI 工具、MCP、AI 应用趋势时再讲。  
> 原则：先讲抽象理解，再讲产品分层，最后拉回石化项目落地。

---

## 0. 使用边界

这份内容不要在普通 Java 面试一开始主动展开。

优先级永远是：

1. Java 后端能力
2. 石化经营分析平台项目
3. AI 助手本地化落地
4. RAG / Function Calling / 缓存 / 权限 / 会话
5. AI Agent 理解

适合聊的场景：

- JD 明确写 AI Agent / 大模型应用 / MCP
- 面试官主动问“你怎么理解 Agent”
- 面试官问“你平时怎么用 AI 工具”
- 面试官问“你怎么看 AI 应用落地”
- 技术负责人明显懂 AI，想看你的系统判断力

不适合聊的场景：

- 面试官主要问 Java 八股
- 面试官对 AI 工具不感兴趣
- 项目还没讲清楚
- 你已经在 AI 概念上讲太久

---

## 1. 第一层：30 秒，不报工具名

适合普通面试官。目标是稳，不炫技。

**回答：**

我理解 AI Agent 不是单纯的大模型，而是模型、工具、上下文、执行环境和安全边界的组合。

真正有价值的 Agent，不只是能回答问题，而是能接入外部工具、进入业务流程、完成可验证的任务。

这个理解也影响了我做石化 AI 助手的思路：不是只调模型 API，而是把模型接到知识库、业务函数、权限、缓存和会话系统里，让它能在真实业务里工作。

**关键词：**

- 模型
- 工具
- 上下文
- 执行环境
- 安全边界
- 可验证任务

**不要说：**

- 我深入研究过 Codex / Hermes / Claude
- 我配了很多 MCP
- 我用 AI 提效 50%

---

## 2. 第二层：1 分钟，讲三层架构

适合面试官主动问“你怎么理解 Agent / AI 工具”。

**回答：**

我会把 Agent 产品分成三层看。

第一层是模型能力层，比如 Claude、GPT 这类，强在理解、推理和生成。但单独一个模型不等于完整 Agent，因为它没有稳定的工具环境和执行链路。

第二层是工程执行层，比如能进入代码仓库的编程 Agent。它可以读代码、改文件、跑测试、根据结果继续迭代，适合修 bug、写测试、重构这种可验证的开发任务。

第三层是 Agent 平台层，强调长期记忆、多端入口、定时任务、技能沉淀和模型可替换。它解决的不是单次任务，而是长期工作流和工具编排。

所以我不认为 Agent 只是一个模型，而是模型能力接上工具、上下文和执行环境之后形成的系统。

**一句话总结：**

> Agent = 模型能力 + 工具调用 + 上下文管理 + 执行环境 + 安全边界。

---

## 3. 第三层：2 分钟，拉回项目落地

适合 AI 技术负责人继续追问。

**回答：**

这套理解放到企业 AI 应用里也一样。

石化项目的 AI 助手如果只是调用大模型，它只能聊天；接上 RAG 后，它能回答内部知识库问题；接上 Function Calling 后，它能查询真实业务数据；再接上权限、缓存、会话和兜底后，它才像一个能上线的企业助手。

所以我看 Agent 落地，重点不是模型多强，而是工具边界、数据权限、上下文管理、失败降级和结果可验证。

模型只是大脑，后面的工具和工程系统决定它能不能在业务里真正工作。

在项目里我也是按这个思路做的：Java 侧负责业务函数、鉴权、缓存、会话；Python 侧负责 RAG 检索；所有外部数据都通过后端权限和参数校验，不能让模型直接越权操作系统。

---

## 4. 如果面试官问：你实际用过哪些 AI 工具？

**回答：**

我用过一些 AI 编程工具，但面试里我更愿意讲它们解决的问题，而不是报工具名。

日常开发里，我主要用它们做几类事：

1. 生成重复性代码，比如 DTO、CRUD、测试样例。
2. 做代码解释和代码 review。
3. 辅助排查报错和梳理技术方案。
4. 整理文档和面试材料。

但所有 AI 生成的代码我都会 review。AI 可以减少重复劳动，但不能替代工程判断。

**如果面试官继续追问工具差异：**

可以说：

> 有的工具偏对话和分析，有的工具偏进入代码仓库执行，有的工具偏长期记忆和工作流编排。我的判断是：修 bug、写测试这种开发任务，工程执行能力最重要；长期助理和自动化任务，平台能力更重要。

## 5. 如果面试官问：MCP 和 Agent 有什么关系？

**第一层短答：**

MCP 可以理解为让 Agent 连接外部工具和数据源的标准协议。

**项目化回答：**

Agent 真正有价值，不是只聊天，而是能调用工具。MCP 的作用就是把工具能力标准化暴露出去，比如查数据库、搜文档、读文件、调用业务 API。

石化项目当前主要是用 Spring AI Function Calling，没有正式把所有服务 MCP 化。但如果后续要让多个 Agent 或多个客户端统一调用，可以把只读查询类能力封装成 MCP Server，比如设备查询、告警查询、知识库搜索。

**风险边界：**

MCP 不能绕过后端权限。即使工具通过 MCP 暴露，真正执行时也必须做用户身份、参数和数据权限校验。尤其是写操作、审批操作，不能直接交给模型自动执行。

## 6. 如果面试官问：你怎么看 Agent 落地难点？

**回答：**

我觉得 Agent 落地难点不在“模型会不会回答”，而在工程边界。

主要有五个问题：

1. **工具边界**：模型能调哪些工具，哪些只能读，哪些不能自动写。
2. **权限边界**：用户能不能看这个数据，不能靠 prompt 控制。
3. **上下文边界**：历史会话、业务数据、知识库内容不能无限塞，要控制 token 和相关性。
4. **失败边界**：模型超时、工具失败、检索不到时要有降级。
5. **责任边界**：AI 是建议还是执行，关键业务必须保留人工确认。

这也是为什么我觉得企业 AI 应用不是“接个 API”就结束，而是要和现有系统的权限、数据、流程、日志、监控结合起来。

## 7. 面试时的收口句

如果感觉自己讲 Agent 讲多了，用这句话拉回项目：

> 所以我在面试里不想把重点放在我用过哪些 AI 工具，而是放在怎么把 AI 能力落到业务系统里。石化项目里我做的 AI 助手，本质就是把模型、知识库、业务函数、缓存、会话和权限接成一条能上线的链路。

---

## 8. 红线提醒

不要主动长篇比较工具。

不要把自己包装成 Agent 平台架构师。

不要说“我深入研究过某某工具”，除非面试官主动问。

不要说 AI 工具提效百分比，除非有统计依据。

不要把 Agent 讲成万能自动化。必须强调权限、校验、人工确认和安全边界。

---

## 9. 最终记忆版

只背这三句也够用：

1. Agent 不是单纯的大模型，而是模型、工具、上下文、执行环境和安全边界的组合。
2. Agent 的价值不是聊天，而是能接工具、进流程、完成可验证任务。
3. 企业 AI 落地的关键不是模型多强，而是权限、工具边界、缓存、会话、降级和结果可验证。
