# 真实团队开发流程与接口联调

## 一、这份文档的定位

这份文档用于回答：

- 你们平时怎么开发协作？
- 你怎么和前端联调接口？
- 接口不通你怎么排查？
- 你熟悉 Git 分支开发吗？
- 你怎么保证代码提交质量？

核心口径：

> 我不是只写完代码就结束，真实开发还包括拉分支、看接口文档、联调参数、处理 token 和权限、看日志、提交代码和记录问题。

---

## 二、真实开发基本流程

一般流程可以概括为：

```text
拉取代码 -> 切分支 -> 理解需求 -> 看相似接口 -> 写代码 -> 本地自测 -> 接口联调 -> 修问题 -> 提交代码 -> 复盘记录
```

面试话术：

> 我平时不会直接在主分支上改代码，一般先从开发分支拉代码，根据任务建自己的功能分支。开发前先看已有相似接口和接口规范，开发后本地自测，再和前端用 Apifox 或 Postman 联调。

---

## 三、Git 分支和提交怎么讲

### 基础回答

常见操作包括：

- clone 指定分支；
- 从 dev 拉新分支；
- 本地提交；
- push 到远程；
- 提交合并请求；
- 解决冲突。

### 面试话术

> 我理解 Git 在团队里不是只会 `add commit push`，更重要的是分支边界。一般开发会从 dev 或 feature 分支拉出自己的任务分支，完成后提交合并请求。提交前会检查改动范围，避免把无关格式化、临时文件、敏感配置一起提交。

---

## 四、接口联调怎么排查

接口联调出问题时，不要直接说“后端没问题”或“前端传错了”。

可以按这条链路排：

```text
请求地址 -> 请求方法 -> 请求头 -> Token -> 参数 -> Controller -> Service -> SQL -> 返回结构
```

常见问题：

- GET / POST 方法不一致；
- URL 路径写错；
- query 参数和 JSON body 混用；
- DTO 字段名不一致；
- token 没带或过期；
- 用户没有权限；
- 后端返回字段名和前端约定不一致；
- 枚举值前后端不统一；
- 跨域或网关拦截。

面试话术：

> 接口联调时我会先看浏览器 Network 或 Apifox 请求，确认路径、方法、请求头、参数和响应。后端再看 Controller 是否进入、参数是否绑定、SQL 是否执行、返回对象是否有值。这样能避免前后端互相猜。

---

## 五、Token 和权限问题怎么排查

权限问题通常不是一个点，而是一条链：

```text
登录 -> 返回 token -> 前端保存 -> 请求头携带 -> 拦截器解析 -> 用户上下文 -> 权限校验 -> 业务数据过滤
```

排查时看：

- 登录接口是否返回 token；
- 前端请求头有没有 `Authorization`；
- token 是否过期；
- 拦截器是否正确解析用户 ID；
- ThreadLocal 是否 set/remove；
- 当前用户是否有角色和权限标识；
- 数据权限是否过滤掉了结果。

面试话术：

> 如果用户说某个功能访问不了，我不会只看 Controller。我会从 token 是否携带、拦截器是否解析、用户上下文是否注入、权限标识是否匹配一路排查。权限问题不能靠前端判断，最终必须以后端校验为准。

---

## 六、接口返回空怎么排查

接口返回空不一定是代码错。

常见原因：

- 数据库确实没有数据；
- 查询条件过严；
- 关联表缺数据；
- 用户无数据权限；
- 分页参数错误；
- 字段映射不一致；
- 状态枚举不匹配；
- 缓存返回旧结果。

面试话术：

> 如果接口返回空，我会先打印或查看实际 SQL，再拿 SQL 到数据库里执行。确认是 SQL 没查到，还是 Java 映射丢了，或者前端取字段取错。很多时候问题不是代码，而是测试数据不完整。

---

## 七、怎么保证提交质量

提交前检查：

- 编译是否通过；
- 接口是否自测；
- 是否影响旧功能；
- 是否有调试日志；
- 是否提交了本地配置；
- 是否有无关格式化；
- 是否需要补充接口文档；
- 是否记录了特殊风险。

面试话术：

> 我提交前会看 diff，确认只提交和任务相关的改动。尤其是配置文件、测试代码、临时日志、格式化大面积变更都要注意，否则 review 成本会很高。

---

## 八、最终背诵版

> 我理解真实开发不是写完代码就结束，而是从拉分支、看需求、参考相似接口、写代码、本地自测、接口联调、修问题、提交代码到复盘记录的一整套流程。接口联调时我会按请求地址、方法、请求头、token、参数绑定、业务逻辑、SQL 和返回结构逐层排查。权限类问题还要看 token 解析、用户上下文、角色权限和数据权限。这样排查比前后端互相猜要高效很多。
