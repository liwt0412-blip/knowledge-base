# 天机学堂-积分系统设计面试三层回答

> 上级入口：[[../00-Java+AI面试作战MOC|Java+AI面试作战MOC]]
> 项目事实来源：[[🖥 项目笔记/天机学堂/10-tj-learning积分服务实现]]（仓库代码与 Spec 为准，本文数字与之一致）

## 一、这份文档的定位

天机学堂是黑马 Spring Cloud 微服务课程项目，**定位与中州养老相同：补强材料，不是主项目**。主项目仍然是石化经营分析平台。

但积分模块和中州养老的"拆解学习"不同——**这个模块是我在课程骨架上真实动手实现并做了二次设计的**，可以说"直接负责"：

- 签到、五渠道积分、幂等、扣分、月结清零的完整链路是我实现并联调的；
- 设计中有多处**偏离课程参考实现的自主决策**（统一消息信封、消费端规则表、防冤扣规则），每一处都能讲清动机；
- 有四个真实排错案例（MySQL 8 保留字、Redisson 依赖缺失启动失败、前端按名称匹配字段、MyBatis-Plus 版本 API 差异）。

面试时稳的说法：

> 我主项目是石化经营平台。另外我在一个 Spring Cloud 微服务课程项目里完整实现过积分模块——签到、多渠道加分、删除扣分、月度榜单清零，里面幂等和防刷是我重点设计的，还踩过几个真实的坑（依赖传递、保留字、版本兼容都有）。

不要说：

> 这个项目是我主导上线的生产系统。（没有真实用户、真实数据量和线上故障记录，不包装。）

## 二、积分系统整体怎么设计的？

### 第一层：基础回答

积分系统在 learning 微服务内（前端约定积分接口归属学习中心），覆盖五类行为：每日签到、课程学习、课程评论、问答回答、学习笔记。积分当月有效，月底清零并结算月度榜单。

三条存储线各管一段：

- **签到事实**：纯 Redis Bitmap，每月一个 key（`sign:uid:{userId}:{yyyyMM}`），offset 是日期减一，TTL 90 天自然过期，不落 MySQL；
- **积分流水**：MySQL `points_record` 表，当月所有加扣分记录，支撑积分明细页，月底由定时任务清空；
- **榜单**：Redis ZSet 做当月实时榜，月底结算时把榜单**全量归档**进 MySQL `points_board_{season}` 赛季分表永久保留（2026-07-21 起由前 100 改为全量并物理分表，原因见第七·五节），所以历史榜单和"我的排名"任意名次都可查。

### 第二层：结合面试回答

跨服务行为（学完一个小节、问答审核通过）通过 RabbitMQ 的 topic 交换机发积分事件，learning 服务统一消费。评论和笔记两个渠道业务侧还没做，但我把队列、路由键、积分规则全部预声明了——生产侧哪天上线发消息即生效，积分侧零改动。

分值不是写死的常量散在各处，而是一张消费端规则表：路由键 → 渠道 + 分值 + 动作（加/扣）+ 允许的 bizType 白名单。

### 第三层：边界和追问

- 没有累计总积分，产品口径就是"当月有效"，所以不存在总积分一致性问题；
- 评论/笔记渠道是预留通道，面试官追问时如实说"业务侧未上线，通道已验证过：手工往队列发一条消息能正常入账"；
- 积分模块放 learning 而不是独立服务，是前端路由约定决定的；最早我建过独立的 promotion 服务，后来按约定迁入 learning，MQ 队列名和 Redis key 保持字面值不变，已有数据和未消费消息无缝衔接。

## 三、MQ 消费怎么保证不重复加分？（幂等，核心卖点）

### 第一层：基础回答

RabbitMQ 是 at-least-once 投递，重复消息必然存在，所以消费端必须幂等。我的做法是**流水先行、唯一键裁决**：

1. 每条积分事件带统一信封 `{userId, bizId, bizType, occurredAt}`，bizId 是业务唯一标识（签到是 yyyyMMdd，学习是小节 id，问答是问题/回答 id）；
2. 消费时先插 `points_record` 流水，表上有唯一键 `uk_user_type_biz_action`（用户 + 渠道 + 业务 id + 加/扣分）；
3. 插入成功才去动 Redis（榜单 ZINCRBY、当日累计）；插入撞唯一键说明是重复消息，直接丢弃；
4. 如果 Redis 操作异常，回滚刚才的流水（或记录幂等键），消息进 error.queue 等待重放，不会"分加了一半"。

### 第二层：结合面试回答

签到链路更进一步：签到本身不走 MQ，服务内同步完成——先 SETBIT 置签到位（原子判重，重复签到直接报业务错），再发积分；如果加分步骤失败，**回滚当日签定位**允许用户重签，避免"位已置、分未加"导致当天积分永久丢失还补签不了。

课程参考实现的消息体只有一个 `Long userId`，没有 bizId，重复投递就会重复加分，这是我最开始就改掉它的原因。

### 第三层：边界和追问

- **为什么不用 Redis setnx 做幂等？** 流水表本身就是业务必须落的数据，唯一键裁决顺带完成幂等，少维护一套幂等 key 的生命周期；而且流水先行让"加分留痕"和"幂等判重"是同一个原子动作，不存在两套状态不同步的窗口。
- **每日上限是 check-then-act，极端并发会轻微越顶**，我清楚这个边界：影响是用户多拿几分，业务可接受；要严格就把"读当日累计 + 判断 + 累加"收进 Lua 脚本原子化。
- **问答埋点是事务内发 MQ**，存在小窗口幻影分（消息发了、事务回了），学习记录埋点已改成 afterCommit，问答侧后续统一收敛——主动讲这个，比被问出来强。

## 四、分值为什么不让生产侧传？（规则表 + 防御性设计）

### 第一层：基础回答

积分事件的信封里**没有分数字段**。渠道和分值由消费端的规则表（`PointsRouteRule`）按"路由键 + bizType"裁定，生产侧只管说"谁在什么时间做了什么"，无权决定值多少分。

### 第二层：结合面试回答

动机来自对课程参考实现的代码评审，它的写法是每个监听方法各自硬编码，我在评审时发现两个真实问题：

1. 签到监听方法调加分时渠道类型写成了 QA——典型的复制粘贴错误，签到积分全记到问答渠道，每日上限统计和明细页全乱；
2. 消息体判空后只 log.error 不 return，空消息继续往下执行，要么 NPE 要么写脏数据。

这两个问题的根源是同一个：**渠道、分值、校验散落在十几个调用点，靠人肉保证一致性**。所以我把它们收敛成一张规则表，并给每条规则加 bizType 白名单——生产侧发错类型直接拒收记日志，错误在规则层面被堵死，而不是靠大家小心。

### 第三层：边界和追问

- 这也是安全边界：MQ 不是私有通道，任何能连上交换机的服务都能发消息。消费端不信任入参是基本素养，和"前端传的 price 不能信"一个道理；
- 代价是新增渠道要在积分侧动三个文件（队列常量、监听方法、规则映射），但**生产方永远零改动**，这个交换我认为值；
- 参考实现的两个问题我不说成"线上事故"，口径是"对照参考实现做代码评审时发现，我的实现从第一版就规避了"。

## 五、删除/隐藏内容怎么扣分？（防冤扣）

### 第一层：基础回答

问答被删除或隐藏时，业务侧发 `reply.deleted` 事件，消费端按同一规则表裁定为扣分动作。但扣分有一条保护规则：**必须先查到该用户当月、该 bizId 的加分流水，查得到才扣，查不到不扣**。

### 第二层：结合面试回答

这条规则防两类冤扣：

- 内容是在加分通道上线**之前**发的，删除时根本没有对应加分记录，扣分就是无中生有；
- **跨月删除**：上月得的积分上月榜单已结算归档，这月删了不该动这月的分——而流水表月底清空，"查当月流水"这个条件天然把跨月扣分挡掉了。

防冤扣优先于防刷：宁可漏扣一个钻空子的，不能错扣一个正常用户。恢复上架不回补积分，语义同样简单一致。

### 第三层：边界和追问

- 已知缺口（主动交代）：人工过审的内容目前永远不得分（后台 un-hide 不会补发事件）；问题被删时级联删除的回答不扣回答者的分。这两条口径待产品确认，面试时如实说"识别到了，留了待确认项"；
- 扣分与加分共用同一幂等键空间（action 是唯一键的一部分），删除事件重放不会重复扣。

## 六、签到为什么用 Bitmap？连续天数怎么算？

### 第一层：基础回答

签到是典型的"用户 × 日期"布尔矩阵，Bitmap 是最省的结构：一个用户一个月 31 个 bit 不到 4 字节，MySQL 一行都存不下这个成本。每月一个 key，TTL 90 天自然过期，连"清理历史"都不用做。当时也评估过 MySQL 签到表，确认产品不需要签到历史报表后直接砍掉了——要报表时给签到补 MQ 事件出口，由数据服务落库，不回溯。

### 第二层：结合面试回答

连续天数用 `BITFIELD GET u{day} 0` 一次读出当月前 N 位：1 号在最高位、今天在最低位，从低位数连续 1 的个数就是连续签到天数，一次 Redis 往返搞定。签到日历高亮（本月 1 号到今天的 0/1 数组）复用同一次读取，从高位往低位逐位拆。

重复签到靠 SETBIT 返回值原子判重——返回旧值是 1 就说明签过了，并发签到也只成功一次，不需要分布式锁。

连续 7/14/28 天叠加 10/20/40 奖励，仅恰好当天触发一次；跨月 key 换了，连续进度自然从 1 重计，不用写重置逻辑。

### 第三层：边界和追问

- **BITFIELD 的位序是易错点**：offset 0 在最高位，从低位数连续 1 才是"从今天往回数"，写反了就是"从 1 号往今天数"，跨断签就算错——这个我专门写了注释；
- Bitmap 的代价是签到事实 90 天后消失，产品认可"签到无历史"才成立，这是和 MySQL 方案的真正取舍点，不是"Bitmap 一定更好"。
- **被问"其他业务怎么判断能不能用 Bitmap"**：四问框架——二元状态？稠密有界序号？海量数据？位运算查询？四条全中用位图，中两三条配合其他存储，只中一条用表（已提炼：[[☕ Java笔记/Bitmap标志位存储模式-连签统计与验重]]）。

## 七、积分当月有效、月底清零怎么实现？

### 第一层：基础回答

用"赛季"模型：`points_board_season` 表记录每个自然月的起止时间，XXL-Job 每月 1 号 00:11 触发，三段任务链按子任务串联：

1. **准备**：插入本月新赛季（刻意最先做，让新加分立即进入新赛季榜，把跨窗口加错榜的概率压到最小）→ 刷新当前赛季缓存 → 幂等建上赛季分表；
2. **归档**：分片广播，各实例按分片序号切 ZSet 排名区间，把上月 Redis 榜**全量归档**进 `points_board_{season}` 赛季分表，rank 全局连续（历史榜永久可查，"我的排名"任意名次可查）；
3. **清理**：数量闸门确认全部分片归档完成 → 清空上月积分流水 → 删除上月 Redis 榜单 key。

### 第二层：结合面试回答

三个保护设计：

- **破坏性步骤（清流水、删榜）前置校验**：只有上赛季 end_time 确实是昨天才执行——任务被人工误触发或调度重放时，不会把当月数据清掉；
- **数量闸门防分片竞态**：分片广播下每个归档分片完成都会触发清理，先触发的实例比对"分表已归档行数 >= ZSet 成员数"，未齐跳过等下一分片，齐才放行；
- 清理段用 Redisson 分布式锁防重入：手动触发撞链式触发、重试风暴时，拿不到锁的实例静默跳过。

### 第三层：边界和追问

- 清零口径是"流水清零 + 榜单重开"，`points_board` 归档不动，所以"清零后历史榜还能看"是自洽的；
- **被问"定时任务怎么设计的、多实例怎么跑"**：月结从单任务演进为 XXL-Job 子任务链（建表/归档/清理三个 handler 按子任务 ID 串联）。要能说清两种编排哲学：调度层编排（子任务链：每步独立状态/重试/可分片广播，但子任务在控制台可被手动触发、守卫必须在代码里做纵深防御）vs 应用层编排（单任务：失败语义代码内强保证、运维面小）。分片广播是链的真正价值（重步骤多实例并行），不是链本身；清理类步骤无论哪种编排，"仅上赛季 end_time 为昨天才执行"的守卫都必须留在代码里；
- **被问"Redis 没持久化到 DB 就被清理了怎么办"**：顺序上归档成功才删旧榜，归档失败 Job 标错可幂等重跑，不会"删了没存"；真正的风险窗口是三个——①赛季榜整月只有 Redis 一份，实例丢数据（无 AOF/主从切换/逐出）会无声丢失，加固是 AOF + 逐出策略隔离，且流水表月底才清、当月可重放重建榜单；②刷新赛季缓存与归档读取间的毫秒级竞态，代价是个别用户少几分，弱一致可接受；③Job 不跑反而安全（榜 key 无 TTL）。主动讲这个比被问出来强；
- **被问"分布式下月结怎么保证安全"（四层防护模型，按层递进讲）**：①归档并行化但区间不相交——分片广播下各实例按分片序号切 ZSet 排名区间，天然无重叠；②子任务链保证顺序——清理是归档的子任务，XXL-Job 父任务成功回调后才触发；③清理段数量闸门+分布式锁——分片广播下每个分片完成都会触发一次清理，先触发的实例比对"分表已归档行数 >= ZSet 成员数"，未齐则跳过等下一分片触发，齐了之后 Redisson 锁保证只有一台实例真正执行删除；④幂等兜底——Redis 单命令原子读 ZSet + 唯一键 INSERT IGNORE，任何一层失效最差是重复劳动不会半归档。关键认知：**安全不靠某一把锁，而是每层各守一个门、任意单层失效都不丢数据**；从"路由第一个消灭并发"演进到"分片广播+数量闸门"的动因是多实例归档提速，要讲清"引入并发换性能，就必须用闸门把时序保证从调度层补回到应用层"；
- **被问"归档性能怎么考虑"（分批写入 + 幂等代替事务）**：归档插入按 500 行/批拆成多条 INSERT IGNORE。单条全量 SQL 的三个风险：①单赛季上榜用户到十万级时报文撞 max_allowed_packet（默认 4M~64M）；②单条巨型 INSERT 让 binlog/redo 突刺、从库延迟；③Job 超时无法观测进度。批大小的量化依据：每行约 50 字节（雪花 id + user_id + int × 2 + season），500 行单条 SQL 约 30KB，离 max_allowed_packet 三个数量级。批间**故意不开事务**——INSERT IGNORE 靠 idx_season_user 唯一键幂等，第 N 批失败重跑时前 N-1 批被唯一键自动跳过、从断点继续，这是"幂等优先于事务"的典型场景：事务只能保证原子性，但长事务放大锁与日志成本，幂等既保证重试安全又不持锁；
- 这里有一个真实排错案例：课程 DDL 的榜单表有 `rank` 列，MySQL 8.0.2 起 rank 是保留字（窗口函数 RANK()）。只给 DDL 加反引号没用——MyBatis-Plus 自动生成的 SQL 不加反引号，运行时才炸。最终列名改 `rank_no`，DDL、PO、Mapper 自定义 SQL、归档逻辑四处同步。经验：引入旧 DDL 到 MySQL 8 先扫一遍新增保留字（rank、rows、system、recursive）。

## 七·五、榜单存储的三次演进：分表 → 单表 → 工程化分表

### 第一层：基础回答

这个模块的榜单存储经历了三次演进，每一步都有明确约束驱动：

1. **课程参考实现**：每赛季动态建一张 `points_board_{seasonId}`；
2. **我的第一版**：改为单表 `points_board` + `season` 列（当时只归档前 100，一年 1200 行，分表是过度设计）；
3. **全量归档变更后（已落地）**：业务量随全量归档增长，**重新回到物理分表**——但不是照抄课程，而是工程化实现：ThreadLocal + MyBatis-Plus 动态表名拦截器做表名路由，月结任务按模板表自动建分表。

### 第二层：结合面试回答

分表落地的四个工程细节（课程方案没覆盖，我补齐的）：

1. **表名路由**：PO 逻辑名 `points_board`，查询前把目标分表名放进 ThreadLocal，MP `DynamicTableNameInnerInterceptor` 执行前改写——拦截器必须注册在**分页拦截器之前**，否则分页的 count SQL 打到逻辑表上；
2. **ThreadLocal 防串表**：Tomcat 线程池复用，不清理会让下个请求读到上个请求的表名写错表——所有入口统一 `withSeason()` 收口 try/finally；
3. **表从哪来**：月结归档前 `CREATE TABLE IF NOT EXISTS ... LIKE 模板表` 幂等建表，模板表永不删；不新增定时任务，也不需要应用账号常态 DDL 之外的额外操作；
4. **失败要响**：没走路由上下文直接访问逻辑表会立刻报表不存在，漏路由的代码测试期就暴露，而不是静默查错数据。

### 第三层：边界和追问（演进背后的决策依据）

- 第一版改单表的依据：归档量小（前 100/赛季）、跨赛季查询高频（`WHERE user_id=?`）、生产 DDL 权限难——三个约束下单表是唯一解；
- 全量归档后约束变了（行数 = 月得分用户 × 12），但**不是分表唯一可选**——我也评估过分区表（引擎内裁剪、应用零改动、DROP PARTITION 清理红利），最终选物理分表是对齐课程方案并练手 MP 拦截器机制；面试被问"为什么不分区"要能答：分区在跨赛季查询和元数据管理上更优，分表在物理隔离和退役清理上更直观，本场景两者都可，选分表是工程偏好而非分区不行；
- 变更同时引爆 `rank_no` tinyint（255 上限）溢出隐患，已改 int；
- 核心话术：**没有银弹，约束摆清楚方案就是唯一解；约束变了就演进，结构要能跟着走**。

## 七·六、榜单查询接口怎么设计的？（热冷分离，2026-07-21 落地）

### 第一层：基础回答

两个接口：`GET /boards`（赛季前 100 榜单 + 我的排名）和 `GET /boards/seasons/list`（赛季历史列表）。核心是**双数据源路由**：

- 查当前赛季（season 参数为空或等于当前赛季 id）→ Redis ZSet 实时榜，`ZREVRANGE` 取榜单、`ZREVRANK`+`ZSCORE` 取我的排名；
- 查历史赛季 → `points_board_{season}` 赛季分表（物理分表，ThreadLocal + MP 动态表名拦截器路由，见第七·五节）。

### 第二层：结合面试回答

三个设计细节：

1. **路由条件是"等于当前赛季"而非"参数缺省"**——用户显式传当前赛季 id 也必须走 Redis，走归档表查不到（当前赛季月底才归档）；
2. **展示截断与数据归档分离**：榜单列表永远只显示前 100（分页窗口与 [0,100) 取交集，超出返回空不报错），但"我的排名"不受限——当前赛季 ZRevRank 天然全量，历史赛季靠全量归档支撑；只有整赛季一分未得才是 `rank=null, points=0` 的未上榜语义；
3. **姓名回填可降级**：榜单只有 userId，姓名走 UserClient 一次批量查询（防 N+1），user-service 挂了显示"未知用户"，榜单本体不受影响。

### 第三层：边界和追问

- 模式通用性：排行榜、账单、统计类数据都是这个结构——**当前周期高频变动放 Redis，历史周期只读归档进 MySQL，查询按"是否当前周期"路由**（已提炼成通用模式，见 [[☕ Java笔记/热冷分离-实时榜Redis与历史归档表的双数据源查询]]）；
- 响应结构用的是课程契约的自定义结构（`{rank, points, boardList}`），不是项目标准 PageDTO——契约以调用方约定为准，不为统一而统一。

## 八、真实排错案例（四个）

### 案例一：启动即挂的依赖陷阱

新模块照 remark 服务模板建，用到 tj-common 的 `@Lock`（Redisson 分布式锁），编译全绿，IDEA 一启动组件扫描就报 `NoClassDefFoundError: org/redisson/api/RLock`。

根因：tj-common 里 redisson 是 **provided scope，不向下游传递**，仓库惯例是每个用到 @Lock 的服务自己声明依赖，而 remark 恰好不用 @Lock，模板上没有。

一句话版本：

> 编译期不报错、启动期组件扫描才炸——provided 依赖的传递性陷阱。照模板建新服务时，模板没用到的能力要单独核一遍依赖。

这个案例适合回答"讲一个你排查过的非业务 bug"，含金量在"编译过 ≠ 运行过"的认知，不在修复本身。

### 案例二："数据有、接口有、页面没有"（2026-07-20）

现象：提问加分后，`points_record` 流水有、`/points/today` 接口返回问答 10 分，但积分页"课程问答"恒显示 0——而签到却能正常显示。

排查路径：先排除 MQ（消费日志显示流水 INSERT 成功，且幂等键 SETNX 在落库之前，流水成功即反推 Redis 可达）→ 查接口响应（正确）→ 定位前端：**页面写死了一张渠道表按"名称"匹配接口数据填数**，前端名称来自旧版枚举（"课程问答"），后端返回的是新版 desc（"问答回答"），匹配不上落默认 0；"每日签到"恰好同名所以正常——同名行正常、异名行全 0，极具迷惑性。

修复：后端 desc 对齐前端（编码不动，表里已有数据）。

一句话版本：

> "数据有但页面没有"的排查顺序是接口响应 → 页面渲染逻辑 → 匹配字段；按名称匹配是脆弱契约，编码才是匹配键。（已提炼：[[☕ Java笔记/前后端联调排错-接口有数据但页面不显示]]）

### 案例三：MySQL 8 保留字（见第七节第三层）

课程 DDL 的 `rank` 列在 MySQL 8.0.2+ 是保留字，改 `rank_no` 四处同步；经验是引入旧 DDL 先扫 8.0 新增保留字。

### 案例四：MyBatis-Plus 版本 API 差异（2026-07-22）

现象：写分表的动态表名拦截器时按网上教程用 `new DynamicTableNameInnerInterceptor()` + `setTableNameHandler(lambda)`，连续两次编译失败——`setTableNameHandler` 找不到符号、构造器传 lambda 又不兼容 Map 类型。

根因：教程全是 MP 3.5.x 写法，而项目根 pom 锁定 **3.4.3**——该版本这个类只有无参构造 + `setTableNameHandlerMap(Map<表名, handler>)` 一个注册入口（按表名精确路由，其他表自动放行）。最终解析本地 `.m2` jar 的 class 常量池拿到真实签名才修对。

一句话版本：

> 第三方库 API 不能凭"最常见写法"写——教程版本和项目锁定版本是两回事，本地 jar 才是事实。查 pom 版本 → 验证 jar 签名，比试错成本低得多。

| 追问 | 一句话应对 |

| 排行榜怎么查？ | 已实现：当前赛季走 Redis ZSet 实时榜，历史赛季走归档表，按"是否当前赛季"路由；榜单显示前 100，我的排名任意名次可查（见第七·六节） |

| 队列积压/消费者挂了？ | 队列全 durable、消息持久化，消费者重启后继续消费；幂等保证重放安全 |
| 为什么不用 Seata 保证发消息和业务一致？ | 积分是弱一致场景，最终一致即可；幻影分窗口已识别，收敛方案是 afterCommit 发送或本地消息表，为几分上 Seata 不值 |

| 数据量级？ | 课程项目无真实量级，不编数字；按设计讲：Bitmap 每用户每月 <4B，榜单侧已物理分表 + 500 行/批分批归档 + 双实例分片广播并行，流水表单月数据量决定月底清表成本 |
| 数据量真上来了怎么扩？ | 按阶梯讲：索引 → 归档/分区表（DROP PARTITION）→ 读写分离 → 分库分表（Java 首选 ShardingSphere-JDBC，关键是分片键/基因键和 Seata 叠加，见 [[☕ Java笔记/分库分表选型-演进阶梯与ShardingSphere]]） |
