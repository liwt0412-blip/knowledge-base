# 天机学堂-优惠券兑换码与防爆破设计

> 通用方案细节：[[☕ Java笔记/兑换码设计-签名防伪与三层防爆破]]

## 定位与稳的说法

优惠券服务（tj-promotion）是我在课程骨架上从建模块开始真实动手实现的：券模板 CRUD、发放/暂停状态机、兑换码生成算法与防重设计是我直接负责的，其中**签名算法在课程方案基础上做了 S 盒加固**，属于有动机的自主决策，可追问。

稳的说法：

> 我在课程项目里独立搭建了优惠券微服务，兑换码这块我重点设计了生成算法和防刷——签名防伪、位图防重、限流防爆破三层，其中签名我在课程的加权方案上做了非线性加固。

## 一、兑换码怎么设计的？

### 第一层：基础回答

六个约束倒推方案：可读（≤10 位、32 字符集）、10 亿级容量、唯一、不可重兑、防猜测、不压 DB。

生成链路：自增 id（Redis 计数器按券分稠密号段）→ 对 id 生成 14 位签名 → 46 位 payload 做 Base32 编码 → 定长 10 位码。验证时 Base32 解码、重算签名比对，**验签不过直接拒绝，不碰 Redis 和 DB**。

### 第二层：结合面试回答

签名思路和 JWT 同构——都是 MAC：密钥 + 数据生成签名，密钥不泄露就伪造不了，篡改了验签必失败。不用 MD5/RSA 是因为签名 128 位起步，塞不进 10 位码；课程方案是按位加权：id 按 4 位分 8 组、各乘权重求和。

我在此之上做了一个加固：**纯加权是线性函数，收集 8 组码就能列方程解出权重**，所以加权之前每组先查密钥 S 盒（16 项置换表）做非线性替换，方程组解法失效，签名长度不变。这是我识别出课程方案短板后的主动改进。

### 第三层：边界和追问

- 强度定位诚实讲：混淆+实用级，不是密码学级；高价值场景升级 Speck32 分组密码或 HMAC 截断签名；
- 唯一性不靠算法运气：自增 id 由存储层保证，变换是一一对应；
- 容量：32¹⁰≈10¹⁵，10 亿需求余量巨大；int 号段上限 21 亿，再大换 long 重做位预算。

## 二、怎么防爆破/防刷？

### 第一层：基础回答

核心认知先抛：**任何码都能被试，防爆破主力不在编码层，在请求侧——让试错成本高到不划算**。三层防御：编码层签名防伪、请求侧限流锁定、架构层布隆过滤与监控告警。

### 第二层：结合面试回答

- 请求侧：Sentinel 按 userId+IP 限流；Redis `INCR`+TTL 记录连续失败，超阈值锁定兑换功能 N 分钟——暴力枚举会迅速把自己锁死；再往上加滑块人机校验和每日兑换上限；
- 架构层：布隆过滤器把"肯定不存在"的码挡在存储之前；兑换失败率突增告警；
- 编码层的价值是让伪造码在验签处零成本被拒，不进存储。

### 第三层：边界和追问

- 另一路线：真随机码 + 唯一索引，靠 32¹⁰ 大空间稀释使枚举命中率趋零——简单但失去离线验签，每次都进存储；签名方案把非法请求挡在最外层，这是我选它的原因；
- 限流阈值要按券价值分级，高价值券配二次验证（短信确认）。

## 三、怎么防重复兑换？并发安全吗？

### 第一层：基础回答

Redis Bitmap：每券一个键，offset = 码 id − 号段起始 id。兑换时 `SETBIT` 占位——**它返回旧值，旧值 1 就是重复兑换**，一次原子调用完成"查+占"，并发天然安全。

### 第二层：结合面试回答

号段稠密是位图成立的前提，所以 id 分配时就按券分连续段。位图只是加速层：DB 的 exchange_code 表是最终事实，兑换成功异步落库；Redis 故障回退 DB 乐观锁更新，重启后可从 DB 重建位图；key 的 TTL = 券有效期 + 缓冲，生命周期白嫖 Redis。

### 第三层：边界和追问

- 管理端分页看码列表仍走 MySQL——位图只回答"是否"，干不了条件筛选（位图选型四问框架）；
- "位已置、事未办"：SETBIT 成功但落库失败的补偿走异步重试 + 对账，和积分链路的幂等清单同一套思路（[[☕ Java笔记/事件驱动积分发放的幂等、防冤扣与重放清单]]）。

## 四、兑换码怎么生成的？发放接口会变慢吗？

### 第一层：基础回答

管理端发放动作触发，一个券最多 5000 个码，同步插会拖慢接口，所以**异步生成**：发放接口只改状态立即返回，@Async 线程池分批（500/批）插入。

### 第二层：结合面试回答

完整链路：发放事务里 `ApplicationEventPublisher` 发事件 → 监听方法上叠加 `@TransactionalEventListener(AFTER_COMMIT) + @Async("exchangeCodeExecutor")` → **事务提交成功后**才异步生成。两个注解分工：@Async 解决"慢"（不阻塞发放接口），AFTER_COMMIT 解决"脏"（发放事务回滚时事件被丢弃，不会出现码已生成但券没发出去的脏数据）。

### 第三层：边界和追问（异步三考点）

- **@Async 原理与失效**：本质是 Spring AOP 代理，把方法调用包装成任务提交给指定线程池；所以必须 `@EnableAsync` 开启 + **跨 Bean 调用**才生效——同类内部 this 自调用不经过代理，注解静默失效退化为同步，这是最经典的坑。我这里由 Spring 事件机制触发，天然跨 Bean；
- **异步与事务时序**：事务内直接触发异步任务，回滚时任务可能已跑完产生脏数据；解法就是 AFTER_COMMIT 事件（也有 BEFORE_COMMIT/AFTER_ROLLBACK/AFTER_COMPLETION 相位可选）；
- **线程池与拒绝策略**：核心 2/最大 4/队列 100，拒绝策略 CallerRuns——队列满时退化为调用线程同步执行，不丢任务但拖慢主线程；对比 AbortPolicy 直接抛异常丢任务，CallerRuns 是"宁可慢不可丢"的取舍。失败兜底是 ERROR 日志 + 人工补偿，后续可加 XXL-Job 扫描"码数量不足"的券自动补，按 couponId 比对数量缺多少补多少，天然幂等；
- 一致性窗口：券已"发放中"但码未就绪的短暂窗口，C 端兑换按码验签后查记录兜底，兑不到即码未生成，提示稍后重试。

### 附：从直接调用到事件驱动的重构故事（高频追问"为什么这么设计"）

第一版是**直接调用 @Async 方法**，自审时发现触发点在发放事务内：事务后段回滚时异步任务可能已跑完，券没发出去、码却进了库（孤儿数据）。于是改为 Spring 事务事件，决策依据是**正确性**，解耦只是顺带收益。

- 直接调用的优点要肯定：调用链显式、IDE 可跳转、少一个事件类——只有一个消费者且时序能保证时它完全没问题，不是为了用事件而用事件；
- 事件的代价也要能答：调用链隐式（新人找不到消费者）、本地事件不持久（进程崩溃即丢，所以留补偿 Job 兜底；要更强可靠上 RocketMQ 事务消息）；
- 话术关键：承认演进过程 + 讲清决策依据 + 主动暴露另一种方案的适用条件。
