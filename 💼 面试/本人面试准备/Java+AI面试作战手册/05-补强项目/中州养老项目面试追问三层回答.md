它不是单一功能点，而是从用户、角色、业务对象、设备、数据、规则、通知形成完整链路。

## 三、若依 RBAC 权限模型怎么讲？

### 第一层：基础回答

RBAC 是基于角色的权限控制。它不是直接给用户分配权限，而是把菜单、按钮、接口权限打包成角色，再把角色分配给用户。

常见关系包括：

- 用户表；
- 角色表；
- 菜单表；
- 用户角色中间表；
- 角色菜单中间表。

### 第二层：结合项目回答

在若依里，用户登录后会查询用户对应的角色和权限标识。前端根据后端返回的路由和菜单展示不同页面，后端接口通过权限标识控制访问。

比如一个普通护理人员只能看到自己相关的老人、护理计划和报警通知，而管理员能看到系统管理、角色管理、菜单管理等更多功能。

### 第三层：边界和追问

权限不能只靠前端菜单隐藏。前端隐藏按钮只是体验层，真正的安全边界必须在后端。

若依里常见做法是：

- 登录时查询权限集合；
- Token 校验通过后把用户信息放到 Spring Security 上下文；
- Controller 方法上用 `@PreAuthorize`；
- 通过权限标识判断当前用户是否能访问接口。

如果面试官问“为什么需要后端权限校验”，可以回答：

> 因为前端路由和按钮可以被绕过，用户可以直接构造 HTTP 请求访问接口。真正的权限控制必须在服务端完成。

## 四、Spring Security + JWT + Redis 登录流程怎么讲？

### 第一层：基础回答

登录流程大致是：用户提交用户名密码和验证码，后端校验后通过 Spring Security 认证，认证成功后生成 Token 返回给前端。后续请求携带 Token，后端解析并校验用户身份。

### 第二层：结合项目回答

若依的登录流程可以拆成两段：

第一段是登录认证：

- 校验验证码；
- 校验用户名密码；
- 查询用户状态；
- 查询用户权限集合；
- 构建 LoginUser；
- 生成 UUID；
- 把 LoginUser 存入 Redis；
- 生成 JWT 返回前端。

第二段是请求校验：

- Token 过滤器从请求头取 Token；
- 解析 JWT 拿到用户 key；
- 根据 key 从 Redis 取 LoginUser；
- 放入 Spring Security 上下文；
- 访问接口时通过 `@PreAuthorize` 校验权限。

### 第三层：边界和追问

JWT 本身适合做身份凭证，但如果只依赖无状态 JWT，会遇到主动退出、续期、踢人、权限变更不及时等问题。

若依这种 JWT + Redis 的方式，本质是把 JWT 作为客户端凭证，把 Redis 作为服务端登录态存储。这样可以做到：

- 主动退出时删除 Redis 登录态；
- Token 续期；
- 服务端控制登录状态；
- 用户权限变化后更容易处理。

如果面试官问“JWT 不是无状态吗，为什么还要 Redis”，可以回答：

> 纯 JWT 更轻，但服务端控制能力弱。企业后台通常更关心可控性，所以会把用户登录态放 Redis，JWT 只保存登录 key。

## 五、小程序登录怎么讲？

### 第一层：基础回答

微信小程序登录通常是前端调用微信接口拿到临时 code，然后把 code 传给后端。后端调用微信服务端接口，用 code 换取 openid 或 session_key，再完成系统内用户绑定和登录态生成。

### 第二层：结合项目回答

在养老项目里，小程序端用户可能是家属、护理人员或其他业务角色。微信 openid 只能证明微信用户身份，不能直接等于系统用户身份。

所以后端需要把 openid 和业务用户绑定，比如绑定手机号、老人家属关系或员工账号。绑定成功后，再生成系统自己的 Token。

### 第三层：边界和追问

小程序登录的核心不是“拿到 openid 就登录”，而是要解决身份映射问题。

需要注意：

- code 只能使用一次；
- openid 是微信体系身份，不等于业务用户；
- session_key 不能随便返回给前端；
- 后端要维护 openid 和业务账号的绑定关系；
- 敏感接口仍然要走系统自己的权限校验。

如果面试官问“微信登录和普通账号密码登录有什么区别”，可以回答：

> 普通登录校验的是系统账号密码，小程序登录校验的是微信身份。微信身份通过后，还要映射到系统业务用户，否则无法判断他能看哪些老人、哪些护理数据。

## 六、IoT 设备管理怎么讲？

### 第一层：基础回答

IoT 设备管理主要涉及产品、物模型和设备。

- 产品：一类设备的抽象，比如烟雾报警器、睡眠检测带；
- 物模型：描述设备有哪些属性、服务和命令；
- 设备：某个具体硬件实例，比如烟雾报警器 1 号。

### 第二层：结合项目回答

在养老场景中，设备不能只存在于 IoT 平台，还必须和业务系统绑定。比如烟雾报警器绑定到房间，睡眠检测带绑定到床位，手表绑定到老人。

这样设备数据上报后，系统才能知道：

- 这条数据来自哪个设备；
- 设备在哪个房间或床位；
- 影响哪个老人；
- 应该通知哪个护理人员或维修人员。

### 第三层：边界和追问

IoT 平台解决的是设备接入和消息上报，业务系统解决的是业务含义。

比如设备上报了心率、烟雾、温度，IoT 平台只知道这是某个设备的属性值，但业务系统要判断这个设备绑定了谁、是否异常、通知谁、是否生成报警记录。

如果面试官问“为什么产品可以在 IoT 平台维护，设备要在业务后台维护”，可以回答：

> 产品数量少，偏设备类型定义，可以在 IoT 平台维护；设备数量多，并且要绑定房间、床位、老人等业务对象，所以必须在业务后台维护和管理。

## 七、设备数据上报链路怎么讲？

### 第一层：基础回答

设备数据上报链路可以理解为：硬件设备通过 MQTT 等协议把数据上报到 IoT 平台，IoT 平台再把数据转发给业务系统，业务系统保存、展示或触发后续规则。

### 第二层：结合项目回答

中州养老里，智能设备比如烟雾报警器、睡眠检测带，会把温度、烟雾浓度、心率、睡眠状态等数据上报到华为云 IoTDA。

业务系统接收到数据后，会保存设备最新状态，也可以把最新数据放到 Redis，供报警规则定时扫描。

### 第三层：边界和追问

设备数据处理要考虑实时性和可靠性。

不能每次报警规则扫描都从历史表全量查数据，否则性能差。更合理的是：

- 历史数据入库，用于追溯和报表；
- 最新数据放 Redis，用于实时判断；
- 定时任务扫描最新数据；
- 异常触发后再写报警表和通知表。

如果面试官问“为什么用 Redis 存最新设备数据”，可以回答：

> 报警规则关注的是设备最新状态和短时间内的连续异常，用 Redis 存最新数据可以减少数据库压力，提高规则扫描效率。历史数据仍然可以落库用于追溯。

## 八、报警规则系统怎么讲？

### 第一层：基础回答

报警规则系统用于判断设备数据是否异常。规则一般包括监控对象、指标、运算符、阈值、持续周期、生效时间和沉默周期。

比如烟雾报警器温度大于等于 40 度，连续 5 个周期触发报警。

### 第二层：结合项目回答

项目里的报警流程可以这样讲：

- 后台配置报警规则；
- 设备持续上报数据；
- Redis 保存设备最新数据；
- 定时任务每分钟扫描有效规则；
- 判断设备数据是否命中阈值；
- 如果连续命中达到持续周期，则触发报警；
- 进入沉默周期，避免重复报警；
- 写入报警数据和站内通知；
- 通过 WebSocket 推送给护理人员、维修人员或管理员。

### 第三层：边界和追问

报警系统最重要的不是简单 if 判断，而是避免误报和重复通知。

所以要设计：

- 持续周期：避免瞬时抖动导致误报；
- 沉默周期：避免同一异常重复刷屏；
- 生效时间：不同规则只在特定时间段生效；
- 通知对象：老人异常通知护理人员，设备异常通知维修人员；
- 管理员兜底：关键报警通知管理员。

如果面试官问“为什么不一异常就立刻报警”，可以回答：

> 有些设备数据会抖动，单次异常不一定代表真实风险。持续周期可以降低误报，沉默周期可以防止重复报警轰炸用户。

## 九、WebSocket 报警通知怎么讲？

### 第一层：基础回答

WebSocket 用于服务端主动向前端推送消息。和普通 HTTP 请求不同，WebSocket 建立连接后，服务端可以在报警触发时主动把消息推给在线用户。

### 第二层：结合项目回答

在养老项目中，报警触发后不能等护理人员刷新页面才看到。系统需要把报警消息实时推送给相关人员。

流程是：

- 用户登录后台后建立 WebSocket 连接；
- 服务端维护用户和连接的映射；
- 报警规则触发后找到需要通知的人；
- 保存站内信；
- 如果用户在线，通过 WebSocket 实时推送；
- 如果用户不在线，后续登录仍然可以从站内信看到。

### 第三层：边界和追问

WebSocket 解决实时通知，但不能替代消息持久化。

如果只靠 WebSocket，用户离线时消息会丢。因此更稳的设计是：

- 站内信或通知表负责可靠存储；
- WebSocket 负责实时体验；
- 前端收到推送后弹窗或刷新未读数量；
- 用户离线后再次登录可以拉取未读消息。

如果面试官问“WebSocket 推送失败怎么办”，可以回答：

> 推送失败不能影响报警主流程。报警数据和通知记录要先落库，WebSocket 只是实时触达手段。用户离线或推送失败时，可以通过未读通知补偿。

## 十、智能床位和 MySQL 多表查询怎么讲？

### 第一层：基础回答

智能床位功能需要把楼层、房间、床位、老人、设备、设备数据关联起来展示，本质是一个典型的多表业务查询场景。

### 第二层：结合项目回答

比如查询某楼层下所有智能床位，需要关联：

- floor 楼层；
- room 房间；
- bed 床位；
- elder 老人；
- device 设备；
- device_data 设备数据。

如果设备绑定位置不同，查询逻辑也不同。烟雾报警器可能绑定房间，睡眠检测带可能绑定床位，所以 SQL 里要根据设备类型和绑定位置做区分。

### 第三层：边界和追问

多表查询不能只追求一次查完，还要考虑数据量和索引。

需要关注：

- join 字段是否有索引；
- where 条件是否命中索引；
- 是否出现无意义笛卡尔积；
- `left join` 和 `inner join` 的语义区别；
- `union` 和 `distinct` 的去重成本；
- 查询结果是否可以拆分或缓存。

如果面试官问“你怎么优化这类 SQL”，可以回答：

> 我会先看执行计划，确认驱动表、连接顺序、索引命中情况和扫描行数。对于设备绑定位置、楼层、房间、床位这些高频关联字段，需要建立合适索引。对于实时设备状态，能用 Redis 或汇总表减少复杂 join 的场景，就不强行每次实时多表联查。

## 十一、项目部署和线上日志排查怎么讲？

### 第一层：基础回答

Java 后端项目上线后，需要关注部署、配置、日志、异常排查和服务运行状态。常见排查方式包括查看日志、检查端口、确认数据库和 Redis 配置、定位异常堆栈。

### 第二层：结合项目回答

中州养老项目里涉及 MySQL、Redis、后端服务、前端、小程序、第三方接口、IoT 平台配置。部署时最容易出问题的是环境配置不一致，比如数据库端口、Redis 密码、OSS、微信配置、华为云配置。

### 第三层：边界和追问

线上排查不能只会看报错最后一行，要按链路定位：

- 请求有没有到网关或后端；
- Controller 有没有进入；
- 权限是否被拦截；
- 数据库连接是否正常；
- Redis key 是否存在；
- 第三方接口是否超时；
- 日志里有没有业务异常和堆栈。

如果面试官问“线上用户说登录不了你怎么排查”，可以回答：

> 我会先确认是验证码、账号密码、Token、Redis 登录态还是权限拦截问题。看登录接口日志、Redis 中是否写入 LoginUser、JWT 是否正常返回、后续请求头是否携带 Token，再看过滤器是否能从 Redis 取到用户信息。

## 十三、面试高频追问与推荐回答

### 1. 你为什么说这个项目不是普通 CRUD？

推荐回答：

> 因为它不只是单表增删改查，而是有完整业务链路。老人入住后绑定房间、床位、护理人员和设备；设备数据上报后进入业务系统；报警规则定时扫描；异常触发后生成报警和通知，再通过 WebSocket 推给对应人员。

### 2. 若依权限体系的核心是什么？

推荐回答：

> 核心是 RBAC。用户绑定角色，角色绑定菜单和权限标识。登录后查询用户权限集合，前端根据路由控制菜单展示，后端通过 `@PreAuthorize` 和权限标识控制接口访问。

### 3. JWT 为什么还要配 Redis？

推荐回答：

> JWT 负责客户端身份凭证，Redis 负责服务端登录态。这样可以支持主动退出、续期、踢人和服务端控制登录状态，比纯无状态 JWT 更适合企业后台。

### 4. IoT 平台和业务系统怎么分工？

推荐回答：

> IoT 平台负责设备接入、协议和数据上报；业务系统负责设备和老人、房间、床位的绑定关系，以及异常判断、通知和业务闭环。

### 5. 报警规则为什么需要持续周期和沉默周期？

推荐回答：

> 持续周期用于降低瞬时数据抖动带来的误报，沉默周期用于避免同一异常短时间内重复通知，影响用户处理。

### 6. WebSocket 推送失败怎么办？

推荐回答：

> WebSocket 只负责实时触达，不能作为唯一可靠通道。报警和通知要先落库，用户在线时实时推送，离线或推送失败时后续通过未读通知补偿。

### 7. 智能床位多表查询怎么优化？

推荐回答：

> 先看执行计划，确认索引命中、扫描行数和 join 顺序。对楼层、房间、床位、设备绑定位置等关联字段建索引。对于实时设备状态，能用 Redis 或汇总表减少频繁复杂 join 的场景，就不要每次都多表实时联查。

## 十四、面试时不要这样说

不要说：

> 中州养老是我主导上线的生产项目。

更稳说法：

> 这是我系统拆解和学习过的企业业务项目，用来补强 Java 后台系统能力。

不要说：

> 若依权限只要前端隐藏菜单就可以。

更稳说法：

> 前端隐藏菜单只是体验，真正权限必须在后端接口校验。

不要说：

> JWT 有了就不需要 Redis。

更稳说法：

> 企业后台通常用 JWT + Redis，既有 Token 凭证，也保留服务端控制能力。

不要说：

> WebSocket 可以保证消息不丢。

更稳说法：

> WebSocket 负责实时推送，消息可靠性要靠通知表或消息持久化保证。

不要说：

> IoT 设备上报后直接报警就行。

更稳说法：

> 报警要结合规则、生效时间、持续周期、沉默周期和通知对象，否则容易误报或重复通知。

## 十五、最终背诵版

如果面试官问你中州养老项目有什么收获，可以这样回答：

> 这个项目我主要把它作为 Java 企业业务系统来拆解。它基于若依框架，涉及 RBAC 权限、Spring Security、JWT + Redis 登录态、小程序登录、IoT 设备接入、设备数据上报、报警规则、WebSocket 通知和 MySQL 多表查询。
>
> 我认为它的价值不是普通 CRUD，而是完整业务闭环。比如老人入住后绑定房间、床位、护理人员和智能设备；设备数据上报后进入业务系统；定时任务根据报警规则判断异常；触发后写报警数据和站内通知，再通过 WebSocket 实时推送给护理人员或维修人员。
>
> 我不会把这个项目包装成自己的主生产项目，它更多是我补强 Java 后台通用能力的材料。我的主项目还是石化经营分析平台，但中州养老让我对权限体系、IoT 数据处理、实时通知和复杂业务建模有了更完整的理解。
