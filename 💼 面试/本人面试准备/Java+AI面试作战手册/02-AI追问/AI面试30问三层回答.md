# AI 面试 30 问三层回答

> 使用方式：  
> 第一层用于直接回答，不超过 20 秒。  
> 第二层用于面试官继续追问。  
> 第三层用于技术负责人深挖。  
>
> 总原则：主动回答要简单，追问时再展开。不要一上来把所有细节倒出来。

---

## 1. 你这个 AI 模块到底是你做的，还是集团底座自带的？

**第一层：一句话结论**

集团侧提供统一 AI 底座和规范，我负责省公司经营分析平台的本地化落地和关键链路实现。

**第二层：项目做法**

我主要做三块：第一，把设备查询、告警查询、客户查询这些 Java 业务能力封装成 Spring AI 工具函数；第二，把省公司操作手册、安全规范、培训课件接入知识库，联调 Python 侧 RAG 检索链路；第三，做 Redis 缓存、SSE 流式输出和会话历史打通。

**第三层：深挖细节**

我不会把集团底座说成自己从零造的。我的价值在于把底座接进现有业务系统：对接权限、组织范围、业务表、知识库文档、缓存策略和用户体验。企业项目里真正麻烦的不是单独调模型，而是让 AI 能稳定、安全、低成本地服务现有业务。

---

## 2. 为什么用 RAG，不用微调？

**第一层：一句话结论**

因为业务文档经常更新，RAG 更新知识库就能生效，微调成本高、周期长，而且不方便追溯来源。

**第二层：项目做法**

石化业务里的操作手册、安全规范、油品政策、培训资料都会变。RAG 模式下，新文档上传后重新切分、向量化、入库即可。用户回答还能附带来源，知道答案来自哪份文档。

**第三层：深挖细节**

微调更适合让模型学习风格或稳定能力，不适合作为频繁变化的业务知识库。业务知识如果写进模型参数里，版本不可控，也很难解释“这句话来自哪个制度版本”。RAG 把知识放在外部存储里，更适合企业知识库场景。

---

## 3. RAG 流程完整说一下。

**第一层：一句话结论**

RAG 就是先检索内部知识库，再让大模型基于检索结果回答，而不是让模型自由发挥。

**第二层：项目做法**

我们流程是：文档解析和 OCR -> 文档切分 -> BGE-M3 生成向量 -> 写入 Milvus。用户提问时，先把问题向量化，再做稠密和稀疏混合检索，经过 Reranker 重排序，把最相关片段放进 Prompt，最后由大模型生成回答并返回引用来源。

**第三层：深挖细节**

检索阶段重点是召回相关上下文，生成阶段重点是约束模型只基于上下文回答。真正影响效果的是文档质量、切分策略、embedding 模型、Top-K、rerank、阈值和 Prompt 约束，不是简单“把文档丢给模型”。

---

## 4. chunk size 怎么定？为什么用父子块？

**第一层：一句话结论**

子块负责精确检索，父块负责保证上下文完整。

**第二层：项目做法**

业务手册通常有标题、步骤、注意事项。如果切得太大，检索不准；切得太小，模型拿不到完整上下文。所以我们用父子块：子块做向量匹配，命中后返回对应父块给模型生成。

**第三层：深挖细节**

chunk size 不是固定套模板，而是根据文档结构和离线效果调整。判断标准是：Top-K 能不能命中正确段落、返回给模型的上下文是否完整、是否超过模型上下文预算。检索粒度要小，生成上下文要完整，这是核心原则。

---

## 5. 怎么防止幻觉？

**第一层：一句话结论**

不能说完全防止，只能通过 RAG、工具调用、引用来源和兜底机制降低幻觉。

**第二层：项目做法**

我们让回答基于两类依据：一类是 Function Calling 查到的实时业务数据，一类是 RAG 检索到的内部文档。RAG 回答会附带来源，相关性不足时走兜底，不让模型硬答。

**第三层：深挖细节**

幻觉通常来自两个地方：检索上下文不相关，或者模型不按上下文回答。所以要控制检索阈值、rerank 质量、Prompt 约束和兜底逻辑。系统还要记录用户反馈，把“答错/没用”的问题回流到知识库优化。

---

## 6. 如果检索不到怎么办？

**第一层：一句话结论**

检索不到就兜底，不让模型硬编。

**第二层：项目做法**

如果检索结果相关性低，就返回“知识库未找到相关依据，请联系人工客服或运维人员”，同时记录这个问题，后续用于补充知识库。

**第三层：深挖细节**

兜底率本身也是一个指标。兜底太高说明知识库覆盖不足或阈值太严格；兜底太低可能说明不相关文档也被喂给模型，容易幻觉。要在召回率和准确性之间调平衡。

---

## 7. 怎么评估 RAG 效果？

**第一层：一句话结论**

不能只看模型回答顺不顺，要看检索是否命中、回答是否有依据、用户是否认可。

**第二层：项目做法**

可以用几类指标：Top-K 命中率、人工相关性评分、无答案兜底率、用户反馈有用率、重复问题缓存命中率。上线前用真实问题集离线评估，上线后看反馈和日志。

**第三层：深挖细节**

测试集要来自真实业务问题，比如操作步骤、设备故障、制度问答。每个问题要有标准参考文档或人工标注答案。评估时先看检索阶段有没有把正确文档召回，再看生成答案有没有引用正确来源。这样能区分是检索问题还是生成问题。

---

## 8. 为什么要 Reranker？

**第一层：一句话结论**

向量检索是粗召回，Reranker 是精排，用来把最相关的文档排到前面。

**第二层：项目做法**

Milvus 先召回一批候选文档，然后 BGE Reranker 把 query 和每个候选 doc 成对打分，重新排序。最后只把最相关的片段交给大模型。

**第三层：深挖细节**

Embedding 检索是把文本压成向量，速度快但理解比较粗。Reranker 是交叉编码器，会同时看 query 和 doc 的细粒度关系，效果更好但成本更高。所以通常是先召回 Top-N，再 rerank，不能全库 rerank。

---

## 9. 为什么稠密 + 稀疏混合检索？

**第一层：一句话结论**

稠密检索抓语义，稀疏检索抓关键词和专业术语，两者互补。

**第二层：项目做法**

比如“建商品”和“新增货物”语义接近，稠密向量更容易召回；但油站编码、制度编号、专有名词，稀疏关键词更稳。企业文档里专业词很多，所以用混合检索更保险。

**第三层：深挖细节**

单纯稠密检索容易漏掉精确编号和术语，单纯关键词检索又不理解同义表达。混合检索的关键是权重和融合策略，要结合业务问题调，不是固定稠密一定优于稀疏。

---

## 10. Milvus 为什么这么选？不用 Elasticsearch 或 pgvector？

**第一层：一句话结论**

Milvus 更适合向量检索场景，尤其是向量规模较大、需要 ANN 和混合检索时。

**第二层：项目做法**

项目里知识库文档量比较大，而且需要稠密向量和稀疏向量检索。Milvus 对向量索引、相似度检索、Top-K 查询支持更专业。

**第三层：深挖细节**

Elasticsearch 更强的是关键词和倒排索引，适合传统搜索；pgvector 适合轻量级向量检索，部署简单，但大规模向量检索和复杂索引能力不如专门向量库。选型要看规模、团队运维能力和检索复杂度。

---

## 11. AI 接口慢怎么办？

**第一层：一句话结论**

用缓存、流式输出、超时控制和降级来处理。

**第二层：项目做法**

高频问题先走 Redis 缓存；需要生成的回答用 SSE 流式输出，用户不用等完整结果；接口层设置超时，失败时走兜底提示。

**第三层：深挖细节**

延迟要拆开看：缓存查询、embedding、向量检索、rerank、模型生成分别耗时。优化也要分层：缓存减少调用次数，SSE 改善体感，Top-K 和上下文长度控制生成成本，超时和降级保证主链路不被拖死。

---

## 12. 大模型 API 失败怎么办？

**第一层：一句话结论**

不能让 AI 失败影响主业务，要有超时、重试、降级和日志。

**第二层：项目做法**

模型调用设置超时时间，短暂失败可以重试。多次失败就返回兜底提示，并记录日志用于后续排查。关键业务操作不依赖 AI 自动完成。

**第三层：深挖细节**

重试不能无限重试，否则会放大故障。要区分网络超时、限流、模型错误、内容安全拒答等错误类型。高峰期可以降级到低成本模型或只返回知识库检索片段。

---

## 13. 怎么控制成本？

**第一层：一句话结论**

核心是减少不必要的大模型调用和控制上下文长度。

**第二层：项目做法**

先查 Redis 缓存，再走 Function Calling，最后才 RAG 生成。RAG 结果也会对高频问题做短期缓存。Prompt 只放必要的 Top 文档片段，避免把大段无关内容塞给模型。

**第三层：深挖细节**

成本主要来自输入 token、输出 token、embedding、rerank 和模型调用次数。企业落地不能只看效果，还要看 ROI。高频问题缓存、模型分级、上下文裁剪和失败兜底都是成本控制手段。

---

## 14. Function Calling 和普通接口调用有什么区别？

**第一层：一句话结论**

普通接口是程序决定调哪个，Function Calling 是模型根据用户问题和工具描述决定是否调用工具。

**第二层：项目做法**

我们把设备查询、告警查询、客户查询封装成 Spring AI 的 `@Tool`。用户问“查一下某油站设备状态”，模型会识别需要调用设备查询工具，生成参数，后端执行后把结果交给模型组织回答。

**第三层：深挖细节**

Function Calling 不能等于让模型直接操作系统。模型只负责选择工具和生成参数，后端必须做参数校验、权限校验、异常处理和结果过滤。敏感写操作还要人工确认。

---

## 15. 工具调用会不会乱调？

**第一层：一句话结论**

有风险，所以工具描述、参数校验和权限控制必须放在后端。

**第二层：项目做法**

工具函数尽量做成只读查询，比如查设备、查告警、查客户信息。参数必须校验，用户权限也必须校验。模型传错参数时，后端不能盲目执行。

**第三层：深挖细节**

工具调用的安全边界在后端，不在 Prompt。Prompt 只能引导模型，不能保证安全。对于写操作、审批操作、资金相关操作，必须二次确认，不能让模型直接完成。

---

## 16. 怎么做权限控制？

**第一层：一句话结论**

权限必须在 Java 后端做，不能靠 Prompt。

**第二层：项目做法**

Function Calling 查询业务数据时，复用原系统的用户身份、角色、组织范围和油站范围。RAG 文档入库时带权限元数据，检索时按用户权限过滤。

**第三层：深挖细节**

AI 系统里权限有两类：业务数据权限和知识库文档权限。业务数据权限走原系统 RBAC 或数据权限；知识库权限要在检索阶段过滤，不能等模型生成后再过滤，因为模型一旦看到无权限内容就已经泄露了。

---

## 17. 不同用户会不会看到不该看的文档？

**第一层：一句话结论**

不能让所有文档无差别检索，必须按用户权限过滤。

**第二层：项目做法**

文档入库时保存组织、角色、业务分类、生效状态等元数据。用户提问时，根据 userId 查出权限范围，再把权限条件带到 Milvus 标量过滤或业务侧过滤里。

**第三层：深挖细节**

权限过滤要发生在检索前或检索中，而不是生成后。生成后再过滤不可靠，因为模型可能已经利用了无权限片段。对于跨地市、跨油站的数据尤其要注意组织隔离。

---

## 18. 会话记忆怎么隔离？

**第一层：一句话结论**

用 userId + conversationId 隔离，每个会话单独存上下文。

**第二层：项目做法**

Redis 存单次会话短期记忆，MySQL conversations 表存持久历史。Java Function Calling 和 Python RAG 都读写同一套会话历史，保证跨服务追问时上下文一致。

**第三层：深挖细节**

Spring AI 的 ChatMemory 要显式传 conversationId，否则可能出现不同用户或不同窗口共享默认记忆的问题。会话历史也不能无限拼接，要控制最近轮数或做摘要，否则 token 成本会失控。

---

## 19. Prompt 泄露怎么办？

**第一层：一句话结论**

Prompt 不是安全边界，敏感逻辑不能只放在 Prompt 里。

**第二层：项目做法**

不在 Prompt 里放密钥、连接串、内部权限规则等敏感信息。业务权限、数据过滤和工具执行都放在后端。Prompt 只做行为约束和回答格式约束。

**第三层：深挖细节**

用户可能通过 prompt injection 诱导模型泄露系统提示词。即使系统提示词泄露，也不能导致业务数据泄露。因此真正的安全控制必须在后端、权限系统、工具执行层和检索过滤层。

---

## 20. 用户恶意 prompt injection 怎么办？

**第一层：一句话结论**

把用户输入和文档内容都当作不可信，后端权限和工具校验不能被模型绕过。

**第二层：项目做法**

RAG 只检索用户有权限的文档。Function Calling 必须做权限和参数校验。敏感操作需要人工确认。Prompt 里也要求模型忽略文档中的指令性攻击内容。

**第三层：深挖细节**

prompt injection 常见于文档里夹带“忽略之前规则”之类内容。解决思路不是完全相信模型，而是架构上限制模型能力：模型只能看到有权限数据，只能调用后端允许的工具，工具执行必须经过后端校验。

---

## 21. BGE-M3 为什么适合？

**第一层：一句话结论**

BGE-M3 适合中文业务文档，也支持稠密和稀疏表示，方便做混合检索。

**第二层：项目做法**

项目里既有自然语言问题，也有专业术语、制度名称、设备编号。BGE-M3 可以生成语义向量，也能支持稀疏表示，两路结合更适合企业知识库。

**第三层：深挖细节**

选 embedding 模型主要看语言支持、领域效果、向量维度、推理成本和生态配套。BGE 系列中文效果和开源生态都比较成熟，和 reranker 也能配套使用。

---

## 22. Embedding 维度是多少？为什么？

**第一层：一句话结论**

维度跟选用的 embedding 模型版本一致，系统里必须保证入库和查询使用同一模型、同一维度。

**第二层：项目做法**

项目关注的是 embedding 模型统一、向量库 schema 统一、查询向量和文档向量维度一致。如果模型升级，要重新生成向量或做版本隔离。

**第三层：深挖细节**

不要随便改 embedding 模型。不同模型的向量空间不兼容，旧文档向量和新查询向量混用会导致检索效果异常。模型升级时要考虑全量重建索引、灰度切换和版本字段。

---

## 23. 向量库中文档更新怎么处理？

**第一层：一句话结论**

新版本文档重新切分、重新向量化，旧版本标记失效或软删除。

**第二层：项目做法**

文档入库时带 docId、version、生效状态、生效日期等元数据。新版本上传后生成新的 chunk 和向量，旧版本标记为 inactive，检索时只查生效版本。

**第三层：深挖细节**

不能只追加新文档不管旧文档，否则模型可能引用旧制度。文档更新要有版本管理、索引重建、缓存失效和引用来源更新。高频答案缓存也要在文档更新后清理。

---

## 24. 重复文档、脏文档怎么处理？

**第一层：一句话结论**

入库前要清洗、去重和抽检，知识库质量决定 RAG 上限。

**第二层：项目做法**

上传文档后做格式解析、OCR、标题识别、重复内容去重。扫描件质量差的文档要人工抽检，避免 OCR 错字影响检索和回答。

**第三层：深挖细节**

RAG 的问题很多不是模型问题，而是知识库问题。重复文档会让检索结果互相干扰，脏文本会污染 embedding，过期文档会导致回答错误。所以知识库治理和模型链路同样重要。

---

## 25. 知识库质量差怎么办？

**第一层：一句话结论**

要做知识库治理和问题回流，不能只靠模型补救。

**第二层：项目做法**

对低质量文档做清洗、拆分、标题补全和版本标记。用户反馈“没用”的问题记录下来，定期补充 FAQ 或优化文档切分。

**第三层：深挖细节**

知识库质量差时，模型再强也只能基于错误或缺失的上下文回答。优化顺序应该是：先补知识库覆盖，再调切分和检索参数，最后才考虑换模型或微调。

---

## 26. 为什么不用 Spring AI 全做？

**第一层：一句话结论**

Spring AI 适合 Java 业务集成，但复杂文档处理、embedding、rerank、混合检索在 Python 生态更成熟。

**第二层：项目做法**

Java 侧用 Spring AI 做 ChatClient、Function Calling 和知识库管理入口。Python 侧用 LangChain、pymilvus、BGE-M3 和 Reranker 处理 RAG 检索流水线。

**第三层：深挖细节**

Spring AI 的 VectorStore 更偏统一抽象和管理面，但生产级 RAG 还需要 OCR、切分、稠密/稀疏向量、rerank 和评估工具。BGE 和 Reranker 主要在 Python/PyTorch 生态里更成熟，所以两边分工更实际。

---

## 27. 为什么不用 LangChain4j？

**第一层：一句话结论**

LangChain4j 也可以，但项目当时选择 Spring AI，是因为和 Spring Boot/Spring Cloud 体系集成更自然。

**第二层：项目做法**

我们的业务函数、鉴权、配置、缓存和网关都在 Java Spring 体系里。Spring AI 的 `@Tool` 和 ChatClient 接入成本更低，也更符合团队技术栈。

**第三层：深挖细节**

框架选型不是谁更火，而是看团队栈、维护成本和业务边界。LangChain4j 更偏 Java 版 AI 编排，Spring AI 更贴近 Spring 生态。项目里复杂 NLP 流水线已经交给 Python，所以 Java 侧更需要的是稳定集成，而不是全功能编排。

---

## 28. MCP 你了解吗？项目用了没？

**第一层：一句话结论**

了解 MCP，但项目当前主要用 Spring AI Function Calling，没有把所有服务正式 MCP 化。

**第二层：项目做法**

当前做法是把 Java 业务函数通过 Spring AI `@Tool` 暴露给模型。后续如果要让多个 Agent 或多个客户端统一调用，可以把查询类 Java API 封装成 MCP Server。

**第三层：深挖细节**

MCP 的价值是把工具能力标准化暴露给 AI Agent，减少每个 Agent 单独适配接口。对于企业系统，适合先封装只读查询工具，比如设备查询、告警查询、知识库搜索。写操作、审批操作要谨慎，必须保留后端权限和人工确认。

---

## 29. AI 回答出错谁负责？

**第一层：一句话结论**

AI 是辅助建议，关键业务不能完全交给 AI 自动决策。

**第二层：项目做法**

AI 回答会附带来源，关键操作仍由人确认。系统保留日志、引用来源和用户反馈。涉及审批、库存、资金、设备处置的动作不能让模型直接自动执行。

**第三层：深挖细节**

企业 AI 落地要明确责任边界。AI 可以提高查询和咨询效率，但不能替代业务审批和责任人。系统设计上要区分“建议类回答”和“执行类动作”，执行类动作必须走原业务流程。

---

## 30. 这个项目最大的 AI 技术难点是什么？

**第一层：一句话结论**

最大难点不是接模型 API，而是把 AI 稳定接进已有 Java 企业业务系统。

**第二层：项目做法**

既要通过 Function Calling 查实时业务数据，又要通过 RAG 检索内部文档，还要处理权限、缓存、会话、成本、延迟和兜底。任何一块没处理好，AI 都很难真正上线使用。

**第三层：深挖细节**

Demo 阶段只要能回答就行，生产阶段要考虑更多：用户有没有权限看这个数据、知识库是不是最新、回答是否有来源、模型失败怎么办、成本是否可控、会话会不会串、缓存会不会返回旧答案。这些工程问题才是 AI 应用落地的核心。
