# Spring AI 面试追问三层回答

## 一、这份材料的定位

这份文档不是 Spring AI 教程笔记，而是面向 Java + AI 面试的表达材料。

你的核心定位应该是：

> 我是 Java 后端工程师，AI 方向不是做算法训练，而是把大模型能力工程化接入 Java 企业系统，包括模型调用、工具调用、RAG 对接、会话隔离、流式返回、权限控制和业务系统集成。

面试中不要把 Spring AI 讲成“我会一个新框架”，而要讲成：

> Spring AI 帮我把大模型能力以 Spring Boot 的方式接入业务系统，让 AI 能力进入原有的 Java 微服务体系。

---

## 二、Spring AI 是什么？

### 第一层：基础回答

Spring AI 是 Spring 生态里用于接入大模型能力的框架。它把模型调用、Prompt、ChatClient、Embedding、VectorStore、RAG、Function Calling、Advisor 等能力封装成 Java 后端熟悉的方式，方便在 Spring Boot 项目中集成 AI 能力。

### 第二层：结合项目回答

在我的项目里，Spring AI 更适合承担 Java 侧的 AI 接入层。比如用户在经营分析平台里发起自然语言查询，Java 后端可以通过 Spring AI 接入模型能力，并结合 Function Calling 调用业务查询工具，再把结果通过 SSE 流式返回给前端。

复杂 RAG 检索、Embedding、重排这类能力，我更倾向于放到 Python RAG 服务中处理。Java 侧负责业务编排、权限校验、缓存、会话和系统集成。

### 第三层：边界和追问

Spring AI 的价值不是替代所有 AI 框架，而是让 Java 企业应用更容易接入大模型。它适合做业务系统集成，但复杂 RAG、文档解析、Embedding 优化、模型评估等链路，Python 生态通常更成熟。

所以我会把 Spring AI 定位为 Java 业务系统里的 AI 接入层，而不是把所有 AI 能力都塞进 Java 服务里。

---

## 三、为什么不用直接 HTTP 调大模型 API？

### 第一层：基础回答

直接 HTTP 调 API 可以实现最简单的问答，但工程化能力弱。Spring AI 提供了统一模型抽象、Prompt 管理、流式调用、工具调用、Advisor、记忆、RAG 等能力，更适合在后端项目里长期维护。

### 第二层：结合项目回答

如果只是简单问答，HTTP 调用就够了。但在企业业务里，问题通常不是“调用模型”，而是要解决：

- 用户身份和权限怎么控制；
- 业务数据怎么查询；
- 哪些问题需要查数据库；
- 哪些问题需要走知识库；
- 多轮对话怎么隔离；
- 长回答怎么流式返回；
- 异常和超时怎么兜底。

这些更适合通过 Spring AI 结合 Spring Boot、Redis、SSE、业务服务一起做。

### 第三层：边界和追问

我理解直接调 API 是能力接入，Spring AI 是工程化封装。真正上线时，模型调用只是其中一环，后面还要考虑缓存、限流、审计、权限、参数校验、结果过滤、日志追踪和降级。

所以我不会把 AI 项目理解成“调通一个接口”，而是要把它接进原有企业系统的工程链路里。

---

## 四、ChatClient / ChatModel / Prompt 怎么理解？

### 第一层：基础回答

ChatModel 是底层模型能力抽象，ChatClient 是更上层、更方便使用的客户端封装。Prompt 是发送给模型的输入，里面可以包含用户问题、系统角色、上下文和工具信息。

### 第二层：结合项目回答

在项目里，我更关注的是 ChatClient 这一层，因为业务代码不应该到处直接拼模型请求。通过 ChatClient，可以把系统提示词、默认参数、Advisor、工具调用等统一配置，业务接口只负责传入用户问题和上下文。

### 第三层：边界和追问

Prompt 不能替代后端逻辑。比如权限判断、参数合法性、数据库查询、指标口径校验，不能只靠 Prompt 要求模型遵守，而要由后端服务强制执行。

我会把 Prompt 看成引导模型行为的方式，但不会把它当成安全边界。

---

## 五、Advisor 是什么？

### 第一层：基础回答

Advisor 可以理解为 Spring AI 里的对话增强组件，可以在模型调用前后插入逻辑，比如日志记录、会话记忆、RAG 上下文增强等。

常见的 Advisor 包括：

- SimpleLoggerAdvisor：记录请求和响应日志；
- MessageChatMemoryAdvisor：处理会话记忆；
- QuestionAnswerAdvisor：结合 VectorStore 做 RAG 问答。

### 第二层：结合项目回答

在业务系统中，Advisor 的价值是把一些通用 AI 能力从业务代码里抽出来。比如所有对话都需要日志追踪，所有多轮问答都需要会话 ID，知识库问答都需要先检索再生成，这些都适合通过 Advisor 统一处理。

### 第三层：边界和追问

Advisor 不是业务权限的替代品。比如 RAG Advisor 可以补充上下文，但检索前的数据权限、检索后的结果过滤，仍然应该在业务服务或知识库服务里做。

我会把 Advisor 用作增强链路，而不是安全控制的唯一入口。

---

## 六、会话记忆怎么做？conversationId 为什么重要？

### 第一层：基础回答

会话记忆是让模型在多轮对话中知道前面聊过什么。Spring AI 可以通过 ChatMemory 和 MessageChatMemoryAdvisor 管理会话上下文。

conversationId 很关键，因为它用来区分不同用户、不同会话的上下文。

### 第二层：结合项目回答

在项目里，如果用户打开多个对话窗口，或者多个用户同时使用 AI 助手，后端必须用 conversationId 隔离会话。这个 ID 通常由前端的 chatId 或后端生成的会话 ID 传入。

如果所有请求都用默认 conversationId，就可能出现串会话：A 用户的问题影响 B 用户的回答，或者当前窗口带上了另一个窗口的上下文。

### 第三层：边界和追问

会话记忆和会话历史不是一回事。

会话历史偏存储和展示，主要用于前端回显、审计、恢复对话；会话记忆偏模型上下文，决定哪些历史内容会被带入模型请求。

实际项目里还要考虑 token 长度限制，不能无限把历史消息塞给模型。通常要做截断、摘要、重要信息提取，或者只保留最近几轮关键上下文。

---

## 七、Function Calling 是什么？

### 第一层：基础回答

Function Calling 是让模型在需要时调用后端定义好的工具或函数。模型负责判断是否调用、调用哪个工具、传什么参数；后端负责真正执行函数，并把结果返回给模型生成最终回答。

在 Spring AI 中，可以通过 `@Tool`、`@ToolParam` 这类方式暴露工具能力。

### 第二层：结合项目回答

在经营分析平台里，很多问题不能只靠模型直接回答。比如用户问“某个单位本月销售额异常原因是什么”，模型本身没有实时业务数据，必须调用后端工具查询指标、知识库或业务接口。

所以我会把经营指标查询、知识库检索、业务规则查询等封装成工具，让模型通过 Function Calling 选择调用。Java 后端执行工具时，再做参数校验、权限判断、缓存处理和结果过滤。

### 第三层：边界和追问

Function Calling 不是让模型直接操作数据库，也不是模型说什么后端就执行什么。模型只是生成工具调用意图，真正能不能执行、参数是否合法、用户有没有权限，都必须由后端判断。

另外，Function Calling 也不能保证模型 100% 选对工具。实际项目里要做工具描述优化、参数 schema 约束、异常兜底、调用日志和人工可追踪。

---

## 八、为什么业务查询不能只靠 Prompt？

### 第一层：基础回答

Prompt 适合引导模型生成答案，但不适合保证业务数据准确性。涉及数据库、权限、实时状态、审批流程、指标口径的问题，必须走后端业务逻辑。

### 第二层：结合项目回答

比如经营分析场景里，销售额、库存、预警、设备状态这些数据都来自业务系统。模型不能凭空生成，也不能靠历史知识回答。正确做法是让模型识别用户意图，然后通过工具调用查询真实业务数据。

### 第三层：边界和追问

我会把模型当成自然语言理解和答案组织层，把业务服务当成事实来源和安全边界。

模型可以帮用户把问题表达成查询意图，也可以把结果解释成人话，但不能绕过后端直接决定业务事实。

---

## 九、Spring AI 里的 RAG 怎么理解？

### 第一层：基础回答

RAG 是检索增强生成。它先从知识库或向量库中检索相关文档，再把这些文档作为上下文交给模型回答，从而降低模型幻觉，提高回答和企业知识的相关性。

Spring AI 中可以通过 Document、Embedding、VectorStore、QuestionAnswerAdvisor 这类组件实现基础 RAG。

### 第二层：结合项目回答

在我的项目里，Java 侧可以通过 Spring AI 理解 RAG 的整体流程，并接入简单知识库问答。但复杂 RAG 能力，比如文档切分、向量召回、混合检索、重排、Embedding 模型优化，更适合由 Python RAG 服务承接。

所以我的表达是：Java 侧负责调用和整合 RAG 结果，Python 服务负责具体检索链路。

### 第三层：边界和追问

RAG 只能降低幻觉，不能彻底消除幻觉。

真正上线还要考虑：

- 文档切分是否合理；
- 向量召回是否准确；
- 是否需要关键词检索和向量检索结合；
- 是否需要重排；
- 检索结果是否带权限；
- 模型回答是否引用了真实上下文；
- 没有检索到资料时是否明确拒答。

如果面试官问为什么不用 Spring AI 全做 RAG，我会回答：

> Spring AI 做基础 RAG 很方便，但复杂 RAG 工程链路 Python 生态更成熟。我的项目更关注 Java 业务系统集成，所以 Java 侧负责业务编排和接口整合，Python 侧负责专业 RAG 检索。

## 十、Spring AI 和 LangChain 怎么分工？

### 第一层：基础回答

Spring AI 更适合 Java 企业应用集成，LangChain 更适合复杂 AI 应用链路和原型开发。两者不是完全替代关系，而是适合不同层面。

### 第二层：结合项目回答

我的理解是：

- Spring AI：负责 Java 侧模型接入、工具调用、会话、SSE、权限、业务服务集成；
- LangChain / Python：负责复杂 RAG、Embedding、文档处理、检索链路、Agent 原型。

在 Java 企业项目里，不能为了 AI 把原有业务系统全部迁移到 Python；但也没必要把所有 AI 数据处理都强行放到 Java。

### 第三层：边界和追问

技术选型的核心不是哪个框架更火，而是谁更适合当前职责。

如果是已有 Spring Cloud / Spring Boot 企业系统，Spring AI 的优势是融入现有 Java 工程体系；如果是做复杂 RAG、实验型 Agent、多模型编排，Python 生态迭代更快。

所以我会采用“Java 业务编排 + Python AI 能力服务”的分工。

---

## 十一、MCP 是什么？和 Function Calling 有什么区别？

### 第一层：基础回答

MCP 是 Model Context Protocol，可以理解为大模型连接外部工具和数据源的一套标准协议。它的目标是让不同模型、不同客户端可以用统一方式连接工具、资源和服务。

Function Calling 更像某个模型调用工具的能力，MCP 更强调工具接入的标准化。

### 第二层：结合项目回答

我项目当前主线不是 MCP，而是 Spring AI Function Calling。因为当前更重要的是把经营分析查询、知识库检索、业务接口封装成 Java 后端可控的工具。

但从后续演进看，可以把一些通用查询能力封装成 MCP Server，比如指标查询、设备查询、知识库检索、报表查询，让不同 AI 客户端都能复用。

### 第三层：边界和追问

我不会把 MCP 说成项目已经完整落地。如果面试官问，我会明确说：

> 当前项目主要是 Function Calling 形态，MCP 是我关注的后续标准化方向。

MCP 解决的是工具接入标准化问题，但不自动解决权限、审计、参数校验和数据安全。这些仍然要由服务端系统控制。

---

## 十二、流式返回怎么讲？

### 第一层：基础回答

流式返回是指模型生成答案时，不等完整结果生成完再返回，而是一边生成一边推送给前端。这样用户体验更好，尤其适合长文本回答。

### 第二层：结合项目回答

在 Java 后端里，可以通过 Spring AI 的 streaming 调用结合 SSE 返回给前端。前端收到一段展示一段，用户能更快看到响应。

在经营分析场景里，AI 回答通常比较长，比如原因分析、改进建议、报表解释，流式输出能降低用户等待感。

### 第三层：边界和追问

流式返回不只是前端体验问题，后端还要处理异常、超时、中断、日志追踪和工具调用结果。

如果涉及 Function Calling，不能简单认为所有内容都会按普通文本流回来。有些模型在工具调用流式输出时，参数可能是分片返回的，后端需要能拼接和解析完整工具参数。

## 十三、面试高频追问与推荐回答

### 1. 你项目里 Spring AI 到底做了什么？

推荐回答：

> 我主要把 Spring AI 放在 Java 侧 AI 接入层，用它承接模型调用、Function Calling、会话上下文和流式返回。复杂 RAG 检索链路由 Python 服务处理，Java 侧负责调用、权限、缓存和业务结果整合。

### 2. 你为什么不用纯 LangChain？

推荐回答：

> 因为我的项目主体是 Java 企业系统，权限、业务接口、用户体系、缓存、SSE、微服务治理都在 Java 侧。LangChain 更适合复杂 RAG 和 Agent 编排，所以我更倾向于 Java 负责业务编排，Python 负责 AI 数据链路。

### 3. Function Calling 安全吗？

推荐回答：

> Function Calling 本身只是让模型产生工具调用意图，不等于安全。安全要靠后端做，包括权限校验、参数白名单、工具范围控制、日志审计和异常兜底。

### 4. RAG 能不能解决幻觉？

推荐回答：

> RAG 可以降低幻觉，但不能彻底消除。它能把回答约束到检索到的资料上，但前提是检索质量、文档切分、权限过滤和提示词约束都做得足够好。

### 5. conversationId 不传会怎样？

推荐回答：

> 如果使用默认 conversationId，多个用户或多个会话可能共享上下文，导致串会话。企业系统里必须用用户维度、会话维度隔离上下文。

### 6. MCP 你们项目用了没有？

推荐回答：

> 当前项目没有把 MCP 作为主链路，主要还是 Spring AI Function Calling。MCP 我理解为后续工具标准化方向，可以把通用业务查询能力封装成 MCP Server，但权限和审计仍然要在服务端控制。

## 十四、面试时不要这样说

不要说：

> 我们用 Spring AI 完整替代了 LangChain。

更稳说法：

> Spring AI 负责 Java 业务侧集成，Python 服务负责复杂 RAG 检索。

不要说：

> Function Calling 可以保证模型一定调用正确工具。

更稳说法：

> Function Calling 让工具调用结构化，但后端仍要做校验、兜底和审计。

不要说：

> RAG 可以彻底解决幻觉。

更稳说法：

> RAG 可以降低幻觉，但还要依赖检索质量和回答约束。

不要说：

> 我们项目已经全面 MCP 化。

更稳说法：

> 当前是 Function Calling，MCP 是后续工具标准化方向。

不要说：

> 只要 Prompt 写好就能控制权限。

更稳说法：

> Prompt 只能引导模型，权限必须由后端服务强制控制。

## 十五、最终背诵版

如果面试官问你 Spring AI 相关问题，可以用这段作为总回答：

> 我理解 Spring AI 的核心价值不是让 Java 调一下大模型 API，而是把大模型能力接进 Spring Boot 企业系统。比如模型调用、Function Calling、会话记忆、RAG 增强、流式返回，都可以用 Java 后端熟悉的方式集成。
>
> 在我的项目里，Java 侧更适合负责业务编排、权限控制、工具调用、缓存和 SSE 返回；复杂 RAG 检索、Embedding、重排这些能力更适合由 Python 服务处理。这样既保留 Java 企业系统的稳定性，也能利用 Python AI 生态的能力。
>
> 我不会把 Function Calling、RAG、MCP 这些能力理解成万能方案。Function Calling 需要后端做权限和参数校验，RAG 只能降低幻觉不能彻底消除，MCP 更适合作为后续工具标准化方向。我的重点是把 AI 能力可靠地接进业务系统，而不是停留在 Demo 层面。
