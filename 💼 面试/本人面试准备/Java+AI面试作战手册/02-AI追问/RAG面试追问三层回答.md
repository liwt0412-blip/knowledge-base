## 一、RAG 总流程

### 1. RAG 是什么？为什么需要 RAG？

**第一层：业务化短答**

RAG 就是先从企业知识库检索相关资料，再让大模型基于这些资料回答。它解决的是大模型不知道企业内部知识、容易编造、答案不可追溯的问题。

**第二层：项目落点**

石化 AI 助手要回答操作手册、安全规范、故障案例这类内部知识，模型本身不一定知道，也不能让它自由发挥。所以先检索内部文档，再基于文档生成，并返回引用来源。

**第三层：技术深挖**

RAG 分两段：检索和生成。检索负责找到相关上下文，生成负责基于上下文组织答案。效果差时要拆开排查：是没召回正确文档，还是召回了但模型没按上下文回答。

---

### 2. RAG 完整链路怎么走？

**第一层：业务化短答**

文档先解析、切分、向量化、入库；用户提问时，检索相关片段，拼进 Prompt，再由大模型生成回答。

**第二层：项目落点**

石化知识库里的 PDF、Word、PPT、图片等文档先做文本抽取和 OCR，再切成块，生成向量写入 Milvus。用户提问时，通过向量检索和重排序找到相关片段，再生成答案。

**第三层：技术深挖**

典型流程：

```text
文档解析 -> 文本清洗 -> 父子块切分 -> Embedding -> Milvus 入库
用户问题 -> Query Embedding -> dense/sparse 混合检索 -> Reranker
Top 文档 -> Prompt -> LLM 生成 -> 引用来源 -> 反馈/日志
```

---

### 3. RAG 和微调怎么选？

**第一层：业务化短答**

业务知识频繁更新时，优先 RAG。微调成本高、周期长，也不方便追溯答案来源。

**第二层：项目落点**

石化业务文档会更新，比如新政策、新操作规范、新安全要求。RAG 更新文档就能生效，还能告诉用户答案来自哪份资料。

**第三层：技术深挖**

微调更适合学习稳定能力、风格或任务模式；RAG 更适合外部知识、频繁变化知识和可追溯场景。企业知识库问答通常优先 RAG。

## 二、文档解析

### 4. 文档入库前为什么要解析？

**第一层：业务化短答**

因为企业文档格式复杂，PDF、Word、PPT、图片里的文字不能直接被模型和向量库使用，必须先提取成文本。

**第二层：项目落点**

石化知识库里会有操作手册、培训课件、安全规范、故障截图。如果扫描件或图片文字不做 OCR，检索时就搜不到。

**第三层：技术深挖**

不同格式解析方式不同：

- PDF：提取原生文本，对大图或扫描页做 OCR
- Word：提取段落、表格、嵌入图片
- PPT：提取文本框、表格、图片
- 图片：直接 OCR

解析质量直接决定 RAG 上限。

---

### 5. 扫描版 PDF 怎么处理？

**第一层：业务化短答**

扫描版 PDF 本质是图片，需要 OCR 识别图片里的文字。

**第二层：项目落点**

很多企业文档可能是扫描件或截图，比如制度扫描件、设备故障截图。需要 OCR 提取文字后才能入库检索。

**第三层：技术深挖**

可以用 PyMuPDF 提取 PDF 页面图片，再用 RapidOCR / PaddleOCR 识别文字。非扫描 PDF 可直接提取原生文本；扫描件则走 OCR。为提升性能，可只对大图或页面主体区域做 OCR，避免小图标干扰。

---

### 6. 文档里的表格怎么处理？

**第一层：业务化短答**

表格不能直接丢掉，需要提取单元格文本，并尽量保留行列语义。

**第二层：项目落点**

石化业务文档里可能有盘点表、设备参数表、规范清单。如果只抽普通段落，表格信息会丢失。

**第三层：技术深挖**

Word/PPT 表格可以遍历行和单元格提取文本；PDF 表格更复杂，可能需要专门表格解析工具或 OCR 后结构化。面试时不要过度承诺“表格完美解析”，更稳的说法是：先保证文本提取，再对关键表格做规则化或人工校验。

---

### 7. OCR 识别错了怎么办？

**第一层：业务化短答**

OCR 错误会污染知识库，所以需要抽检、清洗和反馈修正。

**第二层：项目落点**

如果安全规范或操作步骤被 OCR 识别错，AI 回答就可能错误。关键文档入库后要做人工抽检或高频问题验证。

**第三层：技术深挖**

常见做法：

- OCR 结果清洗
- 低置信度内容标记
- 关键文档人工抽检
- 用户反馈回流
- 失败问题补充 FAQ

RAG 不是只调模型，知识库治理同样重要。

## 三、文本切分

### 8. 为什么不能整篇文档向量化？

**第一层：业务化短答**

整篇文档太长，检索粒度太粗，模型上下文也放不下。

**第二层：项目落点**

员工问“盘点怎么操作”，答案可能只在手册某一小节。如果整篇手册向量化，检索不精准；如果整篇喂给模型，token 成本也高。

**第三层：技术深挖**

切块解决两个问题：第一，向量检索需要细粒度匹配；第二，LLM 上下文窗口有限。chunk 太大，precision 差；chunk 太小，context 不完整。

---

### 9. 父子块有什么价值？

**第一层：业务化短答**

子块负责精确检索，父块负责提供完整上下文。

**第二层：项目落点**

在石化知识库里，用户问题可能命中某个具体步骤，但回答需要前置条件和注意事项。用子块匹配，再返回父块给模型，答案更完整。

**第三层：技术深挖**

父块按较大粒度切，比如段落或章节；子块在父块内细切。向量化和检索用子块，命中后根据 parent_id 找回父块。这样平衡 precision、recall 和上下文完整性。

---

### 10. chunk size 怎么定？

**第一层：业务化短答**

不是固定模板，要根据文档结构、检索效果和上下文预算调整。

**第二层：项目落点**

操作手册、制度文档、培训课件结构不同，切分粒度不能完全一样。原则是检索时能命中准确片段，生成时上下文足够完整。

**第三层：技术深挖**

调 chunk size 要看：

- Top-K 是否能召回正确文档
- 返回上下文是否完整
- 是否超出模型上下文窗口
- 是否产生大量重复片段

chunk_overlap 可以缓解边界信息丢失，但 overlap 太大也会增加冗余。

---

### 11. 中文文档切分有什么特别点？

**第一层：业务化短答**

中文没有天然空格，切分时要尽量按标题、段落、中文标点保持语义完整。

**第二层：项目落点**

石化操作手册里很多步骤和注意事项是中文连续文本，直接按固定长度硬切，可能把一句话或一个步骤切断。

**第三层：技术深挖**

可以用中文递归切分器，按优先级使用 `\n\n`、`\n`、句号、问号、分号、逗号等分隔符递归切分。目标是尽量保留完整句子和段落。

---

## 四、Embedding 和混合检索

### 12. Embedding 在 RAG 里起什么作用？

**第一层：业务化短答**

Embedding 把文本变成向量，让系统可以按语义相似度检索文档。

**第二层：项目落点**

用户说“新增商品怎么弄”，文档里可能写“创建非油商品档案”。关键词不完全一样，但语义相近，向量检索能召回。

**第三层：技术深挖**

文档向量可以预先生成并入库；查询向量在用户提问时实时生成。入库和查询必须使用同一个 embedding 模型和同一维度，否则向量空间不一致。

---

### 13. BGE-M3 为什么适合？

**第一层：业务化短答**

BGE-M3 支持中文和多语言，也支持稠密和稀疏表示，适合企业知识库混合检索。

**第二层：项目落点**

石化文档既有自然语言描述，也有专业术语、设备编号、制度名称。BGE-M3 的 dense 和 sparse 能分别覆盖语义和关键词。

**第三层：技术深挖**

BGE-M3 dense 向量用于语义匹配，sparse 向量用于词汇匹配。它支持较长文本和多语言，常见 dense 维度是 1024。实际使用时要统一模型版本、向量维度和索引配置。

---

### 14. 稠密向量和稀疏向量有什么区别？

**第一层：业务化短答**

稠密向量抓语义，稀疏向量抓关键词和专业词。

**第二层：项目落点**

比如“建商品”和“新增货物”适合稠密向量；“GIL”“430101 油站”“92 号汽油”这种精确词更适合稀疏检索。

**第三层：技术深挖**

稠密向量是固定维度浮点向量，适合同义改写和语义相似。稀疏向量更接近词项权重，适合精确词命中。企业文档里二者互补。

---

### 15. 为什么要混合检索？

**第一层：业务化短答**

单纯向量检索可能漏专业词，单纯关键词检索又不理解同义表达，所以要混合。

**第二层：项目落点**

石化员工提问方式不固定，但文档里又有大量专业词、编码和制度名。混合检索比单一路径更稳。

**第三层：技术深挖**

常见做法是 dense 和 sparse 各自召回 Top-K，再用加权融合排序。权重不是固定真理，要结合业务问题评估。稠密权重大说明更看重语义，稀疏权重大说明更看重关键词。

---

### 16. BM25 和向量检索有什么区别？

**第一层：业务化短答**

BM25 是关键词匹配，适合已知 FAQ 和精确词；向量检索是语义匹配，适合表达不完全一致的问题。

**第二层：项目落点**

高频固定问题可以用 BM25 或 FAQ 快速命中，降低成本；复杂或未知问题再走 RAG 向量检索。

**第三层：技术深挖**

BM25 是 TF-IDF 的改进，有词频饱和和文档长度惩罚。它不理解语义，但速度快、可解释。向量检索能处理同义表达，但对编号、术语不一定稳定。因此两者可互补。

---

## 五、Milvus

### 17. 为什么用 Milvus？

**第一层：业务化短答**

Milvus 是专门做向量检索的数据库，适合存储和查询大量 embedding 向量。

**第二层：项目落点**

石化知识库文档量大，切块后会产生大量向量。需要支持 Top-K 相似度检索、向量索引和元数据过滤，Milvus 比普通数据库更适合。

**第三层：技术深挖**

Milvus 中 Collection 类似表，Schema 定义字段，Index 是向量索引。检索前 Collection 需要 load 到内存。字段通常包括 id、文本、向量、source、parent_id、权限元数据等。

---

### 18. Collection、Schema、Index 是什么关系？

**第一层：业务化短答**

Collection 像数据库表，Schema 是表结构，Index 是建在向量字段上的检索加速结构。

**第二层：项目落点**

知识库每个 chunk 都是一条记录，里面有文本、dense_vector、sparse_vector、来源、父块信息、权限字段等。

**第三层：技术深挖**

流程是：定义 Schema -> 创建 Collection -> 给向量字段建 Index -> 插入向量和元数据 -> load Collection -> search/query。没有索引也能查，但会变成暴力检索，性能差。

---

### 19. IVF_FLAT 是什么？为什么用它？

**第一层：业务化短答**

IVF_FLAT 是先聚类再在部分簇里暴力搜索，能在精度和速度之间折中。

**第二层：项目落点**

知识库向量量级上来后，全量 FLAT 暴力搜索太慢；IVF_FLAT 能通过聚类减少扫描范围，同时保留较好精度。

**第三层：技术深挖**

IVF 类似先把向量分到多个簇，查询时只搜索最接近的几个簇。nlist 控制建多少簇，nprobe 控制查询时搜多少簇。nprobe 越大召回越好但越慢。

---

### 20. nprobe 是什么？

**第一层：业务化短答**

nprobe 控制 IVF 检索时扫描多少个聚类簇。

**第二层：项目落点**

如果检索漏答案，可以适当调大 nprobe；如果延迟高，可以适当调小。它是召回率和速度的平衡参数。

**第三层：技术深挖**

nprobe 太小，只搜很少簇，速度快但容易漏；nprobe 太大，接近更多簇，召回提高但查询慢。需要用评估集观察 Top-K 命中和响应时间。

---

### 21. IP、L2、Cosine 怎么选？

**第一层：业务化短答**

L2 看距离，越小越相似；IP 看内积，越大越相似；Cosine 看方向相似度。

**第二层：项目落点**

BGE-M3 这类 embedding 如果向量做过归一化，内积和余弦相似度效果接近，IP 计算更方便。

**第三层：技术深挖**

度量方式要和模型训练目标、向量是否归一化一致。不能随便换 metric，否则排序结果可能变化。

---

## 六、Reranker

### 22. Reranker 是什么？

**第一层：业务化短答**

Reranker 是重排序模型，用来对初步召回的候选文档重新打分，把最相关的排前面。

**第二层：项目落点**

石化知识库检索可能召回多个相似片段，Reranker 可以进一步判断哪个片段更适合回答用户问题。

**第三层：技术深挖**

Embedding 检索通常是双塔模型，query 和 doc 分别编码，速度快但交互弱。Reranker 是 CrossEncoder，把 query 和 doc 成对输入，相关性判断更细，但成本高，所以只对 Top-N 候选精排。

---

### 23. 为什么不能只用 Reranker？

**第一层：业务化短答**

Reranker 精度高但慢，不能对全库每条文档都跑一遍。

**第二层：项目落点**

石化知识库切块后文档量大，必须先用向量检索快速召回候选，再用 Reranker 精排。

**第三层：技术深挖**

合理链路是召回 + 精排：Embedding/Milvus 做粗召回，Reranker 做精排。召回阶段保证不漏，精排阶段保证顺序更准。

---

### 24. Reranker 效果怎么验证？

**第一层：业务化短答**

用真实问题集对比加不加 Reranker 的 Top-K 命中和人工相关性评分。

**第二层：项目落点**

不能只说“感觉更准”。要看石化高频问题中，正确文档是否排得更靠前，回答是否更有依据。

**第三层：技术深挖**

可以设计 A/B 或离线评估：

- 固定问题集
- 标注标准文档或标准答案
- 对比 Top-1 / Top-3 / Top-5 命中
- 观察 latency 增加是否可接受

---

## 七、Prompt 和幻觉控制

### 25. RAG Prompt 怎么设计？

**第一层：业务化短答**

Prompt 要约束模型基于检索上下文回答，并在上下文不足时兜底。

**第二层：项目落点**

石化业务问题不能让模型自由发挥。Prompt 里要告诉模型：优先依据内部文档，必要时给出引用来源，找不到依据就提示人工确认。

**第三层：技术深挖**

Prompt 通常包含：

- 角色和任务
- 检索上下文
- 用户问题
- 回答格式
- 不足时兜底规则
- 引用来源要求

Prompt 不是安全边界，权限仍要靠后端控制。

---

### 26. RAG 怎么降低幻觉？

**第一层：业务化短答**

通过检索可信上下文、引用来源、低相关性兜底来降低幻觉。

**第二层：项目落点**

AI 助手回答操作步骤时，必须基于内部手册。没有检索到相关资料，就不能硬编。

**第三层：技术深挖**

幻觉要分两类：

- 检索幻觉：检索结果不相关或漏召回
- 生成幻觉：有上下文但模型编了上下文没有的信息

前者靠检索评估，后者靠 Prompt、faithfulness 评估和引用来源控制。

---

### 27. 低相关性时怎么办？

**第一层：业务化短答**

直接兜底，不让模型硬答。

**第二层：项目落点**

如果石化知识库里没有对应制度或操作步骤，回答“未找到相关依据，请联系人工客服或运维人员”，比编一个答案更安全。

**第三层：技术深挖**

可以结合相似度分数、Reranker 分数、Top-K 差距和业务阈值判断。兜底率过高说明知识库覆盖不足或阈值过严；兜底率过低可能带来幻觉。

---

## 八、RAG 评估

### 28. RAG 效果怎么评估？

**第一层：业务化短答**

要分别评估检索和生成，不能只看答案像不像。

**第二层：项目落点**

石化 AI 助手可以用真实员工问题做测试集，看正确文档是否被召回、答案是否基于文档、是否切题。

**第三层：技术深挖**

RAGAS 常见指标：

- Context Relevancy：检索上下文是否相关
- Context Recall：必要信息是否召回
- Faithfulness：答案是否忠于上下文
- Answer Relevancy：答案是否回答问题

---

### 29. Context Relevancy 和 Context Recall 区别？

**第一层：业务化短答**

Relevancy 看“别多拿无关内容”，Recall 看“别漏掉必要信息”。

**第二层：项目落点**

如果 AI 助手搜回很多无关制度，Relevancy 低；如果漏掉关键操作步骤，Recall 低。

**第三层：技术深挖**

Relevancy 高但 Recall 低，说明检索很干净但漏信息；Recall 高但 Relevancy 低，说明召回覆盖广但冗余多。RAG 要在二者之间平衡。

---

### 30. Faithfulness 是什么？

**第一层：业务化短答**

Faithfulness 衡量答案是否忠于检索上下文，主要用于评估幻觉。

**第二层：项目落点**

如果上下文只说“盘点前需锁定库存”，模型回答“盘点后自动扣库存”，但文档没依据，这就是忠实度问题。

**第三层：技术深挖**

评估方法通常是从答案中抽取事实陈述，再判断每个陈述是否能被上下文支持。支持比例越高，faithfulness 越高。

---

### 31. Answer Relevancy 是什么？

**第一层：业务化短答**

Answer Relevancy 衡量答案是否真正回答了用户问题。

**第二层：项目落点**

用户问“怎么新增商品”，模型回答了一堆非油业务背景但没给操作步骤，就是答案相关性差。

**第三层：技术深挖**

常见评估方法是基于答案反推可能的问题，再和原始问题做语义相似度比较。它关注“是否切题”，不等同于“是否真实”。

---

### 32. 测试集怎么构造？

**第一层：业务化短答**

测试集要来自真实业务问题，并配标准答案或标准文档。

**第二层：项目落点**

可以收集油站员工高频问题、运维问题、培训题目，人工标注对应文档和参考答案。

**第三层：技术深挖**

测试集要覆盖：

- 高频 FAQ
- 操作步骤类
- 故障排查类
- 无答案问题
- 多轮追问
- 权限受限问题

否则评估结果会偏。

---

## 九、工程集成

### 33. Python RAG 服务怎么和 Java 系统集成？

**第一层：业务化短答**

Java 负责业务入口和权限，Python RAG 服务通过 REST/SSE 接口提供检索和生成能力。

**第二层：项目落点**

石化项目里 Java AI 助手入口先处理缓存、用户身份、Function Calling；需要知识库时再调用 Python RAG 服务。

**第三层：技术深挖**

接口需要传：

- query
- session_id / conversation_id
- userId 或权限上下文
- source_filter / 文档范围
- 是否流式

Python 返回答案、引用来源、错误状态和日志标识。

---

### 34. SSE 流式输出怎么实现？

**第一层：业务化短答**

模型生成时逐 token 返回，后端通过 SSE 一段一段推给前端。

**第二层：项目落点**

RAG 生成可能需要几秒，SSE 能让用户先看到输出，体感更好，不像接口卡住。

**第三层：技术深挖**

FastAPI 可用 `StreamingResponse`，每次 yield 一个 SSE 格式 chunk。Java 侧也可以用 `SseEmitter` 或响应流转发。要处理连接中断、错误事件和最终完整答案保存。

---

### 35. 对话历史怎么管理？

**第一层：业务化短答**

按 session_id 或 conversationId 保存最近几轮历史，生成时注入必要上下文。

**第二层：项目落点**

会话历史建议由 Java 侧统一管理并持久化。调用 Python RAG 时，由 Java 按需传递当前问题、最近几轮上下文和用户权限范围，避免两个服务直接共同写业务表，同时保证跨服务追问连贯。如果项目真实实现不同，以代码为准。

**第三层：技术深挖**

不能无限保留所有历史，否则 token 成本爆炸。常见做法是保留最近 N 轮，或做摘要。必须按 userId + conversationId 隔离，避免串会话。

---

### 36. 为什么只保留最近几轮？

**第一层：业务化短答**

为了控制 token 成本和减少无关历史干扰。

**第二层：项目落点**

石化 AI 助手通常只需要最近几轮上下文，比如用户先问某油站，再追问“它的设备状态呢”。太久以前的历史反而可能干扰回答。

**第三层：技术深挖**

历史管理要平衡上下文完整性和成本。复杂场景可做摘要记忆，或基于相关性选择历史，而不是简单全量拼接。

---

### 37. 文档更新后缓存怎么办？

**第一层：业务化短答**

文档更新后要重建向量，并清理相关缓存，避免回答旧知识。

**第二层：项目落点**

如果操作手册更新了，但 AI 高频回答缓存没清，用户可能拿到旧操作步骤。

**第三层：技术深挖**

需要处理：

- 文档版本字段
- 旧向量失效
- 新向量入库
- Redis 答案缓存清理
- 引用来源更新
- 生效日期过滤

---

### 38. 文档权限怎么做？

**第一层：业务化短答**

文档入库时带权限元数据，检索时按用户权限过滤。

**第二层：项目落点**

不同地市、油站、角色能看的制度和数据不同，AI 不能把无权限文档检索出来。

**第三层：技术深挖**

权限过滤必须在检索前或检索中完成，不能等模型生成后再过滤。Milvus 可用标量字段过滤，或者业务侧先计算可访问文档范围再传给 RAG 服务。

---

## 十、排查和优化

### 39. 检索结果不相关，怎么排查？

**第一层：业务化短答**

先看文档质量和切分，再看 embedding、检索参数和 rerank。

**第二层：项目落点**

如果员工问操作步骤却搜到无关制度，可能是文档切分不合理、关键词缺失、embedding 效果差或权限过滤范围错。

**第三层：技术深挖**

排查顺序：

1. 原文是否正确解析
2. chunk 是否切断语义
3. query 是否表达清楚
4. embedding 模型是否适合
5. Top-K / nprobe 是否过小
6. sparse/dense 权重是否合理
7. rerank 是否把正确文档排上来

---

### 40. 答案不完整和回答不相关分别是什么问题？

**第一层：业务化短答**

答案不完整多半是召回信息不全；回答不相关多半是检索错了或模型没按上下文答。

**第二层：项目落点**

如果回答漏掉盘点注意事项，可能是父块上下文不够；如果回答成了客户核算内容，可能是检索结果跑偏。

**第三层：技术深挖**

答案不完整重点查 Context Recall 和 chunk 上下文；回答不相关重点查 Context Relevancy、Prompt 和 Answer Relevancy。不同问题优化方向不同。

---

### 41. RAG 系统上线后怎么持续优化？

**第一层：业务化短答**

通过日志、用户反馈、失败问题回流和评估集持续优化。

**第二层：项目落点**

石化 AI 助手可以记录“未找到答案”“用户点没用”“人工客服高频问题”，定期补知识库和 FAQ。

**第三层：技术深挖**

持续优化闭环：

```text
线上问题日志 -> 人工标注 -> 更新知识库/FAQ -> 调整检索参数
-> 离线评估 -> 灰度上线 -> 继续收集反馈
```

企业 RAG 是长期运营系统，不是一次开发完就结束。
