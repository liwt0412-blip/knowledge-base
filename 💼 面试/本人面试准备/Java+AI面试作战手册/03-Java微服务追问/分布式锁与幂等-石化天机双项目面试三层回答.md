# 分布式锁与幂等-石化天机双项目面试三层回答

## 一、这份文档的定位

分布式锁、幂等、MQ 可靠性是微服务面试的高频追问。回答策略是**一个模式，两处呼应**：

- **石化场景当主例**：真实工作项目，按深挖底稿的事实分级讲（IoT 是我直接负责，非油盘点是了解架构）；
- **天机实现当佐证**：学习项目里有具体代码，能讲到类名和设计细节，证明不是背八股。

主项目讲不清的细节用天机补，天机撑不起的"真实业务感"用石化补。两个项目指向同一套方法论，面试官感受到的是"这人真做过"，而不是"这人背过两个项目"。

## 二、分布式锁怎么用的？

### 第一层：基础回答（石化主例）

石化项目里两个典型场景：

1. **非油盘点隔离**（了解架构，同事模块）：盘点期间不能长持数据库行锁，用 Redis 分布式锁按门店维度隔离，盘点进行中阻塞并发调拨写入；并发调拨本身用 version 乐观锁防超扣。
2. **定时任务多实例互斥**：对账、离线扫描这类 XXL-Job 在多实例部署下要防重复执行，拿不到锁的实例直接跳过本轮。

### 第二层：结合面试回答（天机佐证）

天机的 `@Lock` 是我在 tj-common 里深入研究过的声明式分布式锁（Redisson + AOP），能讲清四个工程细节：

1. **锁粒度用 SPEL 拼出来**：`@Lock(name = "pay:order:#{#dto.bizOrderNo}")`，按业务单号加锁而不是全局一把锁；
2. **拿不到锁怎么办是策略，不是写死的**：枚举内置 5 种策略——`SKIP_FAST`（静默跳过，适合定时任务互斥）、`FAIL_FAST`（抛"请求太频繁"，适合防用户连点）、`SKIP_AFTER_RETRY_TIMEOUT` / `FAIL_AFTER_RETRY_TIMEOUT`（等待一段时间后跳过或抛错，默认）、`KEEP_RETRY`（死等，适合必须执行的补偿）；
3. **锁时长默认交给 watchdog**：leaseTime 不传时 Redisson 默认 30s 加看门狗自动续期，业务没执行完锁不会提前释放；只有手动关 autoUnlock 时才强制显式给 leaseTime；
4. **真实用例能对上号**：支付单申请用短窗口锁防同一业务单 3 秒内重复发起；积分月结清理步骤用锁防多实例并发清表。

一句话主线：

> 分布式锁的关键不是"用了 Redisson"，而是锁粒度按业务键收敛、拿不到锁的行为按场景选策略、锁时长靠看门狗兜底。

### 第三层：边界和追问

- **锁不是首选方案**：能不用锁就不用。判重优先用 Redis 原子操作（SETNX/SETBIT 返回值）或 DB 唯一键；防超扣优先乐观锁；分布式锁只留给"必须互斥执行一段逻辑"的场景（盘点隔离、任务互斥）。被问"为什么这里不用锁"时这个顺序就是答案；
- **Redisson watchdog 的边界**：续期依赖锁服务存活，Redis 主从切换时锁可能丢失（RedLock 争议场景），金融级强一致不能靠分布式锁，要靠唯一键/状态机兜底——石化口径同样如此：锁隔离盘点，最终不超扣靠 version 乐观锁；
- **深挖底稿红线**：不说"分布式锁完全不会出问题"；非油盘点口径永远是"了解关键设计"，不说主导开发。

## 三、MQ 重复消费怎么保证幂等？

### 第一层：基础回答（石化主例）

RabbitMQ 是 at-least-once 投递，重复消息必然存在，消费端必须幂等。石化 IoT 链路的做法：

- 消费端基于 messageId，或 `deviceId + reportTime` 业务键做幂等，重复上报直接丢弃；
- 设备状态变更用**条件更新**——`UPDATE device SET status='OFFLINE' WHERE id=? AND status='ONLINE' AND last_active_time < 阈值`，旧事实覆盖不了新事实；
- 离线告警用业务唯一键防重，MQ 重投或 XXL-Job 重复扫描不会产生重复告警。

### 第二层：结合面试回答（天机佐证 + 通用模式）

幂等的通用做法是**双防线**：

1. **拦截层（Redis 原子操作先行）**：积分链路用 SETNX 幂等键（`userId+type+bizId+action`）先挡重复投递；签到用 SETBIT 返回旧值原子判重，连幂等键都不用建；
2. **裁决层（DB 唯一键最终裁决）**：积分流水表 `uk_user_type_biz_action` 唯一键。拦截层可能失效（Redis 抖动、key 过期），唯一键是最后一道。

顺序有个刻意设计：**流水 INSERT 成功才去 ZINCRBY 更新 Redis 榜**。如果先加榜再插流水，撞唯一键回滚时榜已经双加了。一句话：

> 幂等不是一道锁，是"原子操作拦截 + 唯一键裁决"两道防线，且操作顺序要保证任一防线生效时都不留半完成状态。

### 第三层：边界和追问

- **为什么两道都要？** 只有 Redis 拦截：key 过期或 Redis 故障后重放会双加；只有 DB 唯一键：高并发下大量重复请求打到 DB 才知道重复，浪费且慢。拦截层解决效率，裁决层解决正确性；
- **计数类更新另有一套**：点赞计数用"版本号 + 绝对值覆盖"防乱序——旧版本消息到达直接丢弃，不靠幂等键（详见 [[../01-石化经营分析平台/石化AI反馈通道-点赞模式平移面试三层回答|AI 反馈通道设计]]）；
- **石化侧边界**：设备幂等键的存储介质和 TTL 以项目代码为准，不编细节；能讲的是"业务键幂等 + 条件更新 + 告警唯一键"这套思路。

## 四、MQ 消息丢了怎么办？（可靠性闭环）

### 第一层：基础回答

分三段保证：发送端 confirm 确认到达 broker；broker 侧队列 durable、消息持久化；消费端本地重试耗尽后不丢消息，进错误队列等人工介入。

### 第二层：结合面试回答（天机佐证）

天机 tj-common 的 MQ 体系能讲全这个闭环：

1. **发送侧**：`RabbitMqHelper` 每条消息带 UUID CorrelationData + requestId（MDC 全链路传递），confirm/nack 可追踪；
2. **消费失败兜底**：`RepublishMessageRecoverer`——本地重试耗尽后把消息 Republish 到 `error.topic` → `error.{服务名}.queue`，失败消息不丢、可人工重放；积分链路的幂等键异常回滚设计就是配合它的（重放安全）；
3. **最严的一条链（点赞）**：事件先写 Redis Stream 当 Outbox，定时任务批量读出来发 MQ，**broker confirm 成功后才 ACK 并删除 Stream 记录**——进程在"Redis 已写、MQ 未发"的窗口宕机，事件还在 Stream 里可重投。

石化 IoT 链路可以直接呼应：

> 设备上报走 RabbitMQ 削峰，消费端幂等保证重放安全，失败消息进错误队列人工兜底——削峰只是 MQ 的一半价值，另一半是"消息不丢、可重放"。

### 第三层：边界和追问

- **为什么不用 Seata 保证"业务+发消息"一致？** 弱一致场景最终一致即可，本地消息表/Outbox 就够；为几分积分或一条设备状态上分布式事务不值。天机全仓库只有一处 `@GlobalTransactional`（课程下架跨服务拷题），印证"事务靠最终一致为主"的整体取舍；
- **事务内发消息的幻影窗口**：事务里发 MQ、事务后回滚，消息已经出去了。收敛方案是 afterCommit 发送或 Outbox 表——主动讲这个比被问出来强；
- **石化侧边界**：IoT 链路的 confirm/错误队列细节以项目实现为准，不编"我们就是这么配的"，口径是"链路上识别了这些风险，采用的是这套标准做法"。

## 五、延迟消息阶梯重查

### 第一层：基础回答

对第三方支付结果这种"结果不确定、不能干等"的场景，用 RabbitMQ 延迟消息做主动重查：消费时查一次，没结果就按递增间隔再发一条延迟消息，直到拿到终态或达到次数上限。

### 第二层：结合面试回答（天机佐证）

天机 tj-trade 的支付结果查询就是完整实现：内置 3s/5s/15s/30s/60s/2min 递增间隔序列，最多重查 6 次，支付成功或订单状态已变即终止。相比固定间隔轮询，前密后疏既快收敛又省调用。实现上靠 `rabbitmq-delayed-message-exchange` 插件给消息加 `x-delay` 头，`RabbitMqHelper.sendDelayMessage` 统一入口。

**石化映射**（设计口径，非既有事实）：

- 设备离线告警：心跳 TTL 过期后不立即告警，延迟 1~2 分钟重查一次数据库 `lastActiveTime` 二次确认，防网络抖动误报；
- 审批/工单超时：提交时发一条延迟消息，到期检查状态，未处理自动升级或关闭。

### 第三层：边界和追问

- **延迟插件 vs 死信队列 TTL**：插件方案消息粒度灵活、支持任意时长；死信方案（TTL + DLX）有队头阻塞问题，同队列不同延迟会互相堵。被问选型要能答出这个差异；
- **阶梯序列是业务参数**：间隔和次数按第三方响应分布调，不是拍脑袋；石化侧如果讲这个设计，口径是"建议方案/设计过"，不说已上线。

## 六、面试官可能的深挖与应对清单

| 追问 | 一句话应对 |
|------|-----------|
| 分布式锁锁不住怎么办？ | 锁只解决互斥效率，正确性靠唯一键/乐观锁/状态机兜底；Redisson 主从切换丢锁是已知边界，强一致场景不能只靠锁 |
| 你为什么不用锁防重复消费？ | 锁是互斥语义不是幂等语义；判重用 SETNX/SETBIT 原子返回值或唯一键，成本更低且无锁开销 |
| 幂等键 TTL 设多久？ | 覆盖消息最大重放窗口即可（天机积分 45 天对齐流水生命周期）；过期后靠 DB 唯一键继续兜底，两道防线不同时失效 |
| 消息进 error.queue 之后呢？ | 保留完整消息体和失败原因，人工判断后重放；重放安全由消费端幂等保证，所以 error.queue 才敢开放人工操作 |
| 延迟消息会不会雪崩？ | 阶梯间隔本身就是削峰；重查调用走限流，且每次只查一个单，量级可控 |
| 石化项目 QPS 多少用得上这些？ | 不编数字：设备上报峰值约 500-1000 条/s（已确认口径），这些设计防的是"早晚高峰 + 重试风暴"的叠加，不是日常均值 |

- 非油盘点、支付、订单相关口径：了解架构/参与联调，不说主导开发；
