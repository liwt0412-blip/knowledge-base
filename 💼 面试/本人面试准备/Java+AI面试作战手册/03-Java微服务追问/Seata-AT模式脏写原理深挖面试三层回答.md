## 二、第一层：60 秒短答

AT 模式一阶段业务 SQL 在本地直接提交，靠 undo log 记录这行的前镜像（before image）和后镜像（after image）。两个都走 Seata 的全局事务改同一行时，一阶段提交前要向 TC 申请这一行的全局锁，拿不到就回滚重试，所以**全局事务之间不会脏写**。

真正的脏写风险来自**绕过 Seata 代理的普通本地事务**直接改同一行：全局事务二阶段回滚时，用 before image 和当前数据比对，发现已经被改过了，自动回滚失败，只能记录异常、人工介入。

一句话主线：

> AT 的全局锁防的是"自己人"，防不住绕过代理的"外人"——所以防脏写的关键不在 Seata 本身，而在规范上要求所有写路径统一走全局事务。

## 三、第二层：原理展开

### 1. 一阶段流程

RM 代理数据源拦截业务 SQL，按顺序做四件事：

1. 解析 SQL，查出 **before image**（改动前的行数据）；
2. 执行业务 SQL；
3. 查出 **after image**（改动后的行数据）；
4. 向 TC 注册分支事务，并**申请这行的全局锁**，拿到锁才本地提交，把 undo log（前后镜像）一并写入本地库，最后向 TC 上报分支状态。

关键点：全局锁在一阶段就拿到，本地提交后才释放时机由二阶段决定——这就是"一阶段提交、二阶段决议"的结构。

### 2. 二阶段流程

- **全局提交**：各分支一阶段已经本地提交了，二阶段只需**异步删除 undo log**，所以 AT 的全局提交很快；
- **全局回滚**：用 undo log 里的 before image 还原数据。还原前先拿 **after image 与当前数据比对**：一致才还原；不一致，说明一阶段提交后这行又被改过——这就是脏写，无法安全自动回滚。

### 3. 全局锁的边界

全局锁只对**走 Seata 代理数据源的事务**有效。绕过代理的普通本地事务（直连数据源、不走 @GlobalTransactional 的写路径）改同一行时，不参与全局锁竞争，Seata 拦不住它。所以全局锁是"防自己人不防外人"。

### 4. 关联读隔离

AT 一阶段本地已提交，默认相当于**读未提交**——其他事务能读到尚未全局决议的中间状态。需要一致性读时两个手段：

- `SELECT FOR UPDATE`：会被代理识别，去 TC 申请该行全局锁，等持锁的全局事务决议完才返回；
- `@GlobalLock` 注解：让**非全局事务**也参与全局锁竞争，读/写前先去抢全局锁。

## 四、第三层：追问应对

### "脏写之后怎么办？"

before image 校验失败后，Seata 无法安全自动回滚——强行还原会把别人的合法写入覆盖掉。所以只能记录异常、人工核对补偿。正因为事后没有自动解法，**重点在事前防**，不在事后救。

### "那怎么防？"

规范上要求所有写路径都纳入全局事务，不留绕过代理的写入口子；确实不能纳入全局事务的读写路径，加 `@GlobalLock` 让它也参与全局锁竞争。**基线组件的价值就在于这种统一约束**——集团基线定死写路径规范，比每个团队自己注意要可靠。

### "全局锁会不会成为瓶颈或死锁？"

锁冲突时，后到的分支一阶段提交失败，按配置重试（lock retry）。热点行会放大冲突——大家都抢同一行的全局锁，重试和回滚成本都上来。所以 **AT 不适合高频热点行场景**，这是模式选型问题，不是配置问题。

### "你们项目遇到过脏写吗？"（诚实口径）

没有实际遇到过。我理解它的发生条件是存在绕过 Seata 代理的写入路径，集团基线规范要求写路径统一走全局事务，防的就是这个。

被继续追问"那你怎么确定没发生过"时答：

> 没有相关告警和人工补偿记录；而且我直接负责的两个模块——设备 IoT 和 AI 助手——都是模块内本地事务，不涉及跨服务写，这条风险路径本来就不在我的模块里。

### "AT 适合你们哪些场景？"

跨服务、低频的写，比如调拨类业务。设备上报这种高频链路不适合——设备链路用 MQ 削峰 + 本地事务 + 幂等就能保证一致性，不需要分布式事务，也就不该为引入 AT 付出全局锁的成本。
