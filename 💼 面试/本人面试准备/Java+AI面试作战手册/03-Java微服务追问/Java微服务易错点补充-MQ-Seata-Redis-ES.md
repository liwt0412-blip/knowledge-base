## 二、MQ 容易说错的点

### 1. Exchange 存不存消息？

不存。

Exchange 只负责路由，消息真正存在 Queue 里。

面试话术：

> RabbitMQ 里 Exchange 不存消息，只根据 routingKey 和绑定关系把消息路由到 Queue。真正存消息的是 Queue。如果 Exchange 没有绑定任何队列，消息可能直接丢失。

### 2. Fanout 是不是轮询？

不是。

Fanout 是广播，一条消息复制给所有绑定队列。

轮询发生在多个消费者消费同一个 Queue 的场景，是消费者侧分发，不是交换机行为。

### 3. 为什么不让生产者直接发 Queue？

直接发 Queue 会让生产者依赖具体队列名，耦合太强。

Exchange 是路由抽象层，生产者只关心消息类型，不关心谁消费。

### 4. Work Queue 默认有什么问题？

默认可能平均分配，不考虑消费者处理能力。慢消费者堆积，快消费者空闲。

解决：

```properties
spring.rabbitmq.listener.simple.prefetch=1
```

让消费者 ack 后再拿下一条，实现能者多劳。

### 5. Spring AMQP 默认序列化有什么坑？

默认 JDK 序列化有几个问题：

- 体积大；
- 控制台不可读；
- 跨语言差；
- 排查困难。

更常见做法是改成 Jackson JSON 序列化。

### 6. MQ 最大代价是什么？

异步没有同步返回值。

生产者发完消息，不天然知道消费者是否处理成功，所以必须考虑：

- 消息可靠性；
- 重试；
- 死信队列；
- 幂等消费；
- 业务补偿。

## 三、Seata 只讲到什么程度最安全

你的主项目里 Seata 不要讲成自己深度负责。

安全口径：

> 项目基线里了解 Seata，自己主要负责模块内本地事务和联调排查。跨服务分布式事务由团队按集团规范处理。我能讲清基本原理，但不包装成主负责人。

### 1. TC / TM / RM 怎么讲

| 角色 | 作用 |
|---|---|
| TC | 事务协调者，维护全局事务和分支事务状态 |
| TM | 事务管理器，全局事务入口，通常对应 `@GlobalTransactional` |
| RM | 资源管理器，每个服务里管理本地分支事务 |

口诀：

> TC 是裁判，TM 是发令员，RM 是运动员。

### 2. `@Transactional` 和 `@GlobalTransactional`

`@Transactional` 管本地事务。

`@GlobalTransactional` 标记全局事务入口。

下游服务仍然需要本地事务管理自己的分支事务。

### 3. AT 和 XA 区别

XA：

- 一阶段不提交；
- 锁住数据库资源；
- 二阶段统一提交或回滚；
- 强一致，但性能差。

AT：

- 一阶段提交本地事务；
- 记录 before-image / after-image 到 `undo_log`；
- 二阶段成功删除 undo_log；
- 失败时用快照反向恢复；
- 性能更好，但属于最终一致。

### 4. 面试收口

> Seata 我会讲基本原理，但不会把它说成我主负责的模块。真实项目里，我更熟的是本地事务、MQ 幂等、缓存一致性和接口联调。

## 四、Redis 面试补充

### 1. Redis 为什么快？

常见原因：

- 基于内存；
- 单线程执行命令，避免线程切换和锁竞争；
- I/O 多路复用；
- 数据结构高效。

不要只说“因为内存快”。

### 2. Redis 单线程为什么还能高并发？

Redis 的命令执行是单线程，但网络 I/O 可以通过多路复用处理大量连接。命令本身大多很短，内存操作很快。

### 3. Redis 用在你项目哪里？

结合你的项目：

- AI 高频问答缓存；
- 会话短期记忆；
- 设备心跳 TTL；
- 分布式锁；
- 热点基础数据缓存。

### 4. 哪些 key 不能依赖淘汰策略？

分布式锁、关键状态 key、幂等 key 不能靠淘汰策略保证安全。

要设置合理 TTL、容量规划和监控。

## 五、ES 和 RAG 怎么区分

### ES 适合什么？

ES 核心是倒排索引，适合：

- 关键词搜索；
- 过滤；
- 聚合；
- 日志检索；
- 商品搜索。

### Milvus / 向量库适合什么？

向量库适合：

- 语义相似度；
- 表达不完全一致的问题；
- RAG 语义召回；
- embedding 检索。

### 面试话术

> ES 和向量库不是简单替代关系。ES 强在关键词、过滤、聚合和可解释搜索；向量库强在语义相似度。企业 RAG 里专业词、编号、制度名很多，关键词检索和向量检索经常要结合。

## 六、最终背诵版

> MQ 里 Exchange 不存消息，只做路由，Queue 才存消息。Fanout 是广播，不是轮询；轮询是多个消费者消费同一个队列时的分发问题。异步 MQ 最大代价是没有同步返回值，所以要考虑可靠性、重试、死信和幂等。
>
> Seata 我保持了解口径，不包装成主负责。基本原理是 TC 协调全局事务，TM 开启事务，RM 管理分支事务。AT 模式一阶段提交并写 undo_log，二阶段失败再用快照回滚。
>
> Redis 不只是缓存，还能做设备心跳 TTL、分布式锁、幂等 key 和 AI 高频问答缓存。ES 和向量库也不是替代关系，ES 适合关键词和聚合，向量库适合语义检索。
