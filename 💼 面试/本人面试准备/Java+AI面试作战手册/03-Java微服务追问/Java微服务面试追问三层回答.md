## 0. 总原则

所有微服务问题都按这个结构回答：

```text
一句话结论：
项目里怎么用：
机制或风险怎么解释：
```

不要只背课程概念。每个问题都必须能落到石化经营分析平台：

- Gateway：统一入口、JWT 鉴权、userId 透传
- Nacos：服务注册发现、配置中心
- OpenFeign：服务间同步调用
- RabbitMQ：设备上报削峰、告警通知异步解耦
- Redis：设备心跳 TTL、热点缓存、分布式锁
- XXL-Job：离线扫描、报表任务、保质期预警
- Sentinel：限流、熔断、降级、防止雪崩
- Seata：平台基线了解，不要说成自己主负责

## 一、微服务拆分与 Nacos

### 1. 为什么要拆微服务？

**第一层：短答**

单体项目简单，但业务复杂后会出现代码边界模糊、团队协作困难、构建部署慢、一个模块故障影响整体。微服务通过按业务拆分，让服务职责更清晰，可以独立开发、部署和扩展。

**第二层：项目落点**

石化经营分析平台拆成设备管理、账册表单、客户核算、AI 助手、网关鉴权、通知等服务。设备模块和 AI 模块的业务边界差异很大，拆开后各自迭代更清晰。

**第三层：深挖机制**

微服务拆分原则是高内聚、低耦合。按业务纵向拆分核心服务，按公共能力横向抽取网关、通知、鉴权等公共服务。拆分不是越细越好，服务过多会增加远程调用、事务一致性和运维复杂度。

---

### 2. 微服务拆分原则是什么？

**第一层：短答**

核心原则是高内聚、低耦合，优先按业务边界拆，不按数据库表机械拆。

**第二层：项目落点**

设备管理服务围绕油站、加油机、加油枪和设备状态；AI 助手服务围绕知识库问答和业务工具调用；网关服务只做统一入口和鉴权。这些边界比较清晰。

**第三层：深挖机制**

拆分要考虑数据归属、团队职责、调用频率和一致性要求。强一致、高频互相调用的能力不一定适合拆太开，否则会把本地调用变成远程调用，复杂度上升。

---

### 3. Nacos 注册发现流程是什么？

**第一层：短答**

服务提供者启动后把服务名、IP、端口注册到 Nacos；消费者调用时根据服务名从 Nacos 获取实例列表，再通过负载均衡选择一个实例调用。

**第二层：项目落点**

石化项目里的设备服务、AI 助手服务、通知服务等都注册到 Nacos。Gateway 和 OpenFeign 都可以通过服务名找到具体实例，不需要写死 IP。

**第三层：深挖机制**

服务实例会通过心跳维持健康状态。实例异常时，Nacos 会剔除或标记异常，并把变更推送给客户端。消费者本地也会缓存服务列表，避免每次调用都请求注册中心。

---

### 4. Nacos 注册中心和配置中心分别解决什么？

**第一层：短答**

注册中心解决“服务在哪”；配置中心解决“配置怎么集中管理和动态更新”。

**第二层：项目落点**

项目里服务注册发现靠 Nacos，公共配置、业务开关、网关动态路由这类配置也可以放到配置中心统一管理。

**第三层：深挖机制**

配置中心适合放共享配置和业务配置，例如限流开关、业务参数、路由规则。不是所有配置都适合热更新，数据库连接、底层依赖这类基础配置通常仍需要重启或谨慎变更。

---

### 5. Nacos AP 还是 CP？

**第一层：短答**

不能绝对说 Nacos 只有 AP 或 CP。注册中心场景更偏 AP，优先保证服务发现可用；配置中心更强调一致性。

**第二层：项目落点**

服务发现阶段，短时间实例列表不完全一致通常比注册中心不可用更容易接受。但配置变更，尤其是业务开关和路由规则，需要更谨慎。

**第三层：深挖机制**

注册中心关注可用性和健康检查，配置中心关注配置一致和变更通知。面试时不要背“AP/CP”标签，要结合具体功能说。

## 二、OpenFeign

### 6. OpenFeign 是什么？和 RestTemplate 有什么区别？

**第一层：短答**

OpenFeign 是声明式 HTTP 客户端，用接口和注解描述远程调用，比 RestTemplate 手写 URL 和参数更清晰。

**第二层：项目落点**

石化项目里服务间同步调用可以用 OpenFeign，比如 AI 助手调用设备服务查询设备状态，或者业务服务调用通知服务。

**第三层：深挖机制**

OpenFeign 底层通过动态代理生成接口实现，调用接口方法时拼接 HTTP 请求。它通常结合 Nacos 和负载均衡，通过服务名调用实例。

---

### 7. Feign 调用时用户信息怎么透传？

**第一层：短答**

网关鉴权后把 userId 写入请求头，下游服务用拦截器取出放到 ThreadLocal，Feign 调用时再用 RequestInterceptor 把 userId 加到新的请求头里。

**第二层：项目落点**

AI 助手调用设备或客户服务时，必须带上当前用户身份，否则下游无法做数据权限过滤。

**第三层：深挖机制**

ThreadLocal 用完要清理，避免线程复用导致用户信息串号。Feign 拦截器只负责透传，不负责鉴权，真正的数据权限仍在下游服务校验。

---

### 8. Feign 调用失败怎么办？

**第一层：短答**

要设置超时、重试、降级和日志，避免一个下游服务慢把调用方拖死。

**第二层：项目落点**

AI 助手查设备状态时，如果设备服务异常，不能让整个 AI 接口一直卡住，应该返回降级提示或兜底结果。

**第三层：深挖机制**

Feign 可以结合 Sentinel 做 fallback。降级逻辑要区分查询类和写入类：查询失败可以返回空结果或提示稍后再试；写入失败不能随便吞异常，避免数据不一致。

---

## 三、Gateway 路由与鉴权

### 9. Gateway 怎么路由到后端微服务？

**第一层：短答**

Gateway 根据 Predicate 匹配请求，比如 Path 路径匹配；匹配后通过 `lb://service-name` 从 Nacos 获取服务实例，再负载均衡转发。

**第二层：项目落点**

项目里前端请求先到 Gateway，比如 `/api/device/**` 路由到设备服务，`/api/ai/**` 路由到 AI 助手服务。

**第三层：深挖机制**

Gateway 内部会把匹配到的路由和过滤器组成过滤器链，先执行 pre 逻辑，最后由 NettyRoutingFilter 转发到目标服务，再倒序执行 post 逻辑。

---

### 10. Gateway 鉴权流程怎么做？

**第一层：短答**

GlobalFilter 对所有请求生效：先判断路径是否放行，再取 Token，校验签名和过期时间。失败返回 401，成功后把 userId 写入请求头转发给下游。

**第二层：项目落点**

石化平台统一入口在 Gateway，登录、静态资源、健康检查可放行，业务接口必须鉴权。AI 助手和设备查询都不能绕过网关权限。

**第三层：深挖机制**

鉴权失败要在网关层拦截，不能让请求打到业务服务。下游服务仍要做数据权限校验，因为网关只解决“是谁”，业务服务还要判断“他能看什么”。

---

### 11. GatewayFilter 和 GlobalFilter 区别？

**第一层：短答**

GatewayFilter 作用于某个路由，GlobalFilter 作用于所有路由。

**第二层：项目落点**

全局鉴权、日志追踪适合 GlobalFilter；某个服务特有的请求头处理或限流策略可以放 GatewayFilter。

**第三层：深挖机制**

两类过滤器最终都会合并到同一条过滤器链里执行，顺序由 Ordered 或 `@Order` 控制。值越小，优先级越高。

---

### 12. 为什么鉴权放网关，不放每个服务？

**第一层：短答**

网关是统一入口，把通用鉴权放网关可以避免每个服务重复写校验逻辑，也能更早拦截非法请求。

**第二层：项目落点**

石化平台有多个微服务，如果每个服务都单独校验 Token，会重复且容易不一致。网关统一校验后透传 userId，下游做业务权限即可。

**第三层：深挖机制**

网关鉴权不是全部安全。认证放网关，授权和数据权限仍要在业务服务做，尤其是不同地市、不同油站的数据隔离。

## 四、RabbitMQ 基础

### 13. 为什么用 MQ？不用同步调用？

**第一层：短答**

MQ 用来异步解耦、削峰填谷、提升系统抗压能力。同步调用会阻塞、耦合高，容易级联失败。

**第二层：项目落点**

设备上报高峰时如果同步写库和同步通知，接口容易被打满。用 RabbitMQ 先接住流量，消费者再异步写 MySQL、刷新 Redis 心跳、触发告警。

**第三层：深挖机制**

MQ 的代价是没有同步返回值，生产者不知道消费者是否处理成功，所以必须考虑可靠性、重试、死信和幂等。

---

### 14. RabbitMQ 有哪些角色？

**第一层：短答**

Producer 发送消息，Exchange 负责路由，Queue 存消息，Consumer 消费消息，Broker 是 RabbitMQ 服务本身。

**第二层：项目落点**

设备服务作为生产者发送设备上报或告警消息，通知服务监听队列消费告警并推送给负责人。

**第三层：深挖机制**

Exchange 不存消息，只做路由。消息真正存放在 Queue。生产者不应该直接依赖具体队列，而是发到交换机，由交换机按 routingKey 路由。

---

### 15. 三种交换机怎么选？

**第一层：短答**

Fanout 广播，Direct 精确匹配 routingKey，Topic 支持通配符匹配。

**第二层：项目落点**

如果一条告警要多个服务都收到，可以用 Fanout；如果不同告警类型发给不同队列，可以用 Direct 或 Topic。

**第三层：深挖机制**

Fanout 是复制给所有绑定队列，不是轮询。轮询发生在多个消费者消费同一个队列时，是消费者侧的负载分配。

---

### 16. Work Queue 默认分发有什么问题？

**第一层：短答**

默认轮询分发不考虑消费者处理速度，可能慢消费者堆积、快消费者空闲。

**第二层：项目落点**

设备消息处理如果有的消费者处理慢，默认分发可能导致某个消费者堆积。可以设置 prefetch，让消费者处理完再拿下一条。

**第三层：深挖机制**

`prefetch=1` 表示消费者一次只拿一条，ack 后再拿下一条，实现能者多劳。但吞吐量和公平性要平衡，不是所有场景都设 1。

---

### 17. 为什么要改 JSON 序列化？

**第一层：短答**

JDK 默认序列化体积大、控制台乱码、不跨语言。JSON 更易排查，也更适合跨语言消费。

**第二层：项目落点**

石化项目里 Java 和 Python 服务都存在，消息和接口数据尽量用 JSON，排查和跨语言处理更方便。

**第三层：深挖机制**

生产者和消费者两端的消息转换器要一致，否则一端发 JSON，另一端按 JDK 反序列化会失败。消息里最好带 messageId，方便幂等处理。

---

## 五、RabbitMQ 可靠性与幂等

### 18. MQ 消息可能在哪些环节丢？

**第一层：短答**

三段：生产者到 Broker、Broker 自身、Broker 到消费者。

**第二层：项目落点**

设备告警消息如果丢了，负责人可能收不到告警。所以要考虑生产者确认、消息持久化、消费者 ack 和失败重试。

**第三层：深挖机制**

生产者侧用 Confirm/Return；Broker 侧交换机、队列、消息持久化；消费者侧手动 ACK、本地重试、死信队列。三段串联，任何一段断了都可能丢消息。

---

### 19. Publisher Confirm 和 Return 区别？

**第一层：短答**

Confirm 关心消息是否到达 Broker；Return 关心消息到达 Broker 后是否路由到队列。

**第二层：项目落点**

告警消息如果 exchange 写错，会收到 Confirm NACK；如果 routingKey 没有匹配队列，会触发 Return，方便排查路由配置。

**第三层：深挖机制**

路由失败时可能 Confirm 是 ACK，因为消息确实到达了 Broker，只是没有路由到队列。所以 Confirm 和 Return 解决的问题不同。

---

### 20. 消费者 ACK/NACK/Reject 区别？

**第一层：短答**

ACK 表示处理成功，消息删除；NACK 表示处理失败可重新入队；Reject 表示拒绝消息，通常丢弃或进死信。

**第二层：项目落点**

告警通知消费失败时，如果是临时异常可以重试；如果是消息格式错误，重试也没用，应进死信或记录人工处理。

**第三层：深挖机制**

不能笼统说“异常就重试”。业务异常可能重试成功，格式异常重试再多也没用。重试耗尽后要进入死信队列，不要无限重试造成死循环。

---

### 21. 消息重复消费怎么办？

**第一层：短答**

消费端必须做幂等，常见方法是唯一消息 ID 去重或业务状态机原子更新。

**第二层：项目落点**

设备告警消息重复消费时，不能重复生成告警或重复通知。可以用告警唯一键、设备 ID + 告警类型 + 时间窗口做幂等。

**第三层：深挖机制**

先查再改有并发风险，最好用一条 SQL 原子更新，比如 `where id=? and status=?`。幂等是异步系统必须有的，因为重试天然可能导致重复消费。

---

### 22. 死信队列有什么用？

**第一层：短答**

死信队列用来接收多次重试仍失败的消息，避免消息直接丢失。

**第二层：项目落点**

告警通知如果多次推送失败，可以进入死信队列，由运维或定时任务后续补偿处理。

**第三层：深挖机制**

重试不是目的，最终处理掉消息才是目的。死信队列是异常消息的兜底通道，配合告警和人工排查使用。

---

### 23. 为什么 MQ 还需要定时任务兜底？

**第一层：短答**

没有 100% 可靠的单通道。MQ 是实时 push，定时任务是最终一致的 pull 兜底。

**第二层：项目落点**

设备离线告警不能只依赖实时消息。即使某次消息漏了，XXL-Job 定时扫描 Redis 心跳，也能发现设备离线。

**第三层：深挖机制**

push 实时性好但可能失败，pull 延迟高但能兜底。关键业务常用 push + pull 双通道保证最终一致性。

---

## 六、Sentinel 服务保护

### 24. 什么是服务雪崩？

**第一层：短答**

一个下游服务变慢或故障，调用方线程被阻塞，线程池耗尽，故障继续向上游扩散，最终整条链路崩溃。

**第二层：项目落点**

如果设备服务或客户服务变慢，AI 助手通过 Feign 查询时也会被拖慢，最终影响 AI 接口响应。

**第三层：深挖机制**

雪崩的本质是资源被慢调用占满。解决思路是限流、线程隔离、熔断和降级，防止故障扩散。

---

### 25. Sentinel 限流、熔断、降级区别？

**第一层：短答**

限流控制入口流量，熔断在下游异常或慢调用比例过高时快速失败，降级是失败后的兜底逻辑。

**第二层：项目落点**

设备查询接口高峰时可以限流；下游设备服务慢时可以熔断；AI 助手拿不到设备状态时返回降级提示。

**第三层：深挖机制**

限流保护服务提供者，熔断和降级保护调用者。熔断器有关闭、打开、半开三个状态，半开时放少量请求试探恢复。

---

### 26. 线程隔离和信号量隔离区别？

**第一层：短答**

线程池隔离用独立线程池执行，可以隔离资源；信号量隔离只是限制并发数，调用仍在原线程执行。

**第二层：项目落点**

如果 AI 助手调用多个下游服务，可以给慢服务调用做隔离，避免拖垮主业务线程。

**第三层：深挖机制**

线程池隔离资源成本高，但隔离更彻底；信号量隔离成本低，但不能中断正在执行的慢调用。选型看调用类型和性能要求。

---

### 27. Feign fallback 怎么设计？

**第一层：短答**

Feign 调用失败时走 fallback，返回兜底结果或抛业务异常，避免异常扩散。

**第二层：项目落点**

AI 助手查询设备状态失败时，可以返回“设备服务暂不可用，请稍后再试”，而不是让接口一直阻塞。

**第三层：深挖机制**

查询类接口可以降级为空或默认提示；写操作不能随便降级成功，否则会造成数据不一致。FallbackFactory 比普通 fallback 更好，因为能拿到异常原因。

---

## 七、Seata 分布式事务

### 28. 为什么 `@Transactional` 管不了分布式事务？

**第一层：短答**

`@Transactional` 只能管理当前服务、当前数据源的本地事务，管不了其他微服务的数据库。

**第二层：项目落点**

跨服务业务如果涉及多个数据库，例如订单、库存、购物车，就需要全局事务或最终一致性方案。石化项目里我更多接触的是模块内事务和 MQ 最终一致性。

**第三层：深挖机制**

每个服务的本地事务叫分支事务，整个业务叫全局事务。只靠本地事务，一个分支失败时其他分支可能已经提交，原子性就没了。

---

### 29. Seata TC、TM、RM 分别是什么？

**第一层：短答**

TC 是事务协调者，TM 是事务管理器，RM 是资源管理器。

**第二层：项目落点**

如果平台某个跨服务业务用了 Seata，入口服务里的 `@GlobalTransactional` 对应 TM，各个微服务的数据源代理对应 RM，TC 统一协调提交或回滚。

**第三层：深挖机制**

TM 开启全局事务，RM 注册分支事务并汇报状态，TC 维护全局事务状态并决定最终提交还是回滚。

---

### 30. XA 和 AT 区别？

**第一层：短答**

XA 一阶段不提交、锁资源，强一致但性能差；AT 一阶段提交并记录 undo_log，性能好，但属于最终一致。

**第二层：项目落点**

面试里如果问到 Seata，可以说平台基线了解 AT/XA 原理，但自己项目主负责场景更多是本地事务、乐观锁、MQ 兜底。

**第三层：深挖机制**

AT 一阶段记录 before-image 和 after-image 到 undo_log，然后提交释放锁。二阶段成功删 undo_log，失败用 before-image 反向恢复数据。

---

### 31. `@GlobalTransactional` 加在哪里？

**第一层：短答**

加在全局事务的入口方法上，也就是最先发起整个跨服务业务的方法。

**第二层：项目落点**

如果是下单链路，就加在订单服务创建订单的入口方法上。石化项目里不要主动说自己深度负责 Seata，保持了解口径。

**第三层：深挖机制**

入口方法开启全局事务，下游服务仍需要本地事务管理自己的分支。全局事务和本地事务不是替代关系，而是协作关系。

---

## 八、Redis 缓存与集群

### 32. Redis 在你项目里怎么用？

**第一层：短答**

主要用于热点缓存、设备心跳状态、分布式锁和 AI 高频问答缓存。

**第二层：项目落点**

设备在线状态用 `device:heartbeat:{deviceId}`，每次上报刷新 TTL；AI 高频问题缓存回答；盘点期间用 Redis 分布式锁隔离并发操作。

**第三层：深挖机制**

Redis 适合读多写少、短生命周期状态和高频访问数据。但要注意缓存一致性、key 过期、内存淘汰和故障恢复。

---

### 33. Redis TTL 为什么适合设备心跳？

**第一层：短答**

设备在线状态天然是短生命周期数据，设备持续上报就续期，不上报 TTL 到期就视为离线。

**第二层：项目落点**

每次收到设备上报就刷新心跳 key，XXL-Job 定时扫描离线设备生成告警。比频繁查 MySQL 判断最后上报时间更轻。

**第三层：深挖机制**

Redis 过期不是到点立刻删除，而是惰性删除 + 周期删除。但业务判断可以通过 key 是否存在或 TTL 判断。对离线告警来说，允许秒级误差。

---

### 34. 缓存穿透、击穿、雪崩怎么防？

**第一层：短答**

穿透查不存在数据，用布隆过滤器或空值缓存；击穿是热点 key 过期，用互斥锁或逻辑过期；雪崩是大量 key 同时过期，用 TTL 随机化和限流。

**第二层：项目落点**

AI 高频问答、油站基础数据、商品信息都可能成为热点。设备心跳 key 大量设置相同 TTL 时，也要注意同时过期带来的抖动。

**第三层：深挖机制**

击穿重点是“一个热点 key”，雪崩重点是“大量 key”。两者不要混。空值缓存要设置短 TTL，防止长期缓存无效数据。

---

### 35. Redis 分布式锁怎么安全释放？

**第一层：短答**

加锁时设置唯一 value 和 TTL，释放时先校验 value，确认是自己的锁再删除。

**第二层：项目落点**

盘点期间用 Redis 锁防止销售出库和盘点同时改库存。锁必须有 TTL，避免业务异常导致死锁。

**第三层：深挖机制**

释放锁的“判断 value + 删除 key”要保证原子性，通常用 Lua 脚本或 Redisson。复杂场景可用 Redisson 看门狗自动续期。

---

### 36. Redis 主从、哨兵、分片集群分别解决什么？

**第一层：短答**

主从解决读扩展和数据备份，哨兵解决主节点故障自动切换，分片集群解决单机容量和写入瓶颈。

**第二层：项目落点**

项目里 Redis 如果只是缓存和心跳，普通主从或哨兵就能满足高可用；如果数据量和写入压力很大，才需要分片集群。

**第三层：深挖机制**

主从同步有全量和增量；哨兵负责监控、选主、故障转移和通知客户端；分片集群通过 16384 个 hash slot 分布数据。

---

### 37. Redis 过期 key 是立刻删除吗？

**第一层：短答**

不是。Redis 通过惰性删除和周期删除结合清理过期 key。

**第二层：项目落点**

设备心跳 TTL 到期后，不要求毫秒级立刻删除。XXL-Job 扫描离线时允许短暂误差。

**第三层：深挖机制**

惰性删除是在访问 key 时检查是否过期；周期删除是定时抽样部分 key 删除。Redis 不会给每个 key 设置独立定时器，那样成本太高。

---

### 38. Redis 内存淘汰策略怎么讲？

**第一层：短答**

内存达到上限时，Redis 会按配置淘汰 key，常见策略有 noeviction、LRU、LFU、random、ttl 等。

**第二层：项目落点**

AI 问答缓存和热点基础数据可以被淘汰，但分布式锁和关键状态 key 不能依赖淘汰策略保证安全，要设置合理 TTL 和容量规划。

**第三层：深挖机制**

缓存场景常用 allkeys-lru 或 allkeys-lfu。noeviction 是默认策略，内存满后写入会失败。生产环境要结合业务选择。

---

## 九、Elasticsearch 与 RAG 辅助问题

### 39. ES 和 Milvus 有什么区别？

**第一层：短答**

ES 更适合关键词检索和倒排索引，Milvus 更适合向量相似度检索。

**第二层：项目落点**

RAG 场景下，企业文档既有语义问题，也有制度名称、编号、油站编码等关键词。单纯向量不一定够，所以混合检索更稳。

**第三层：深挖机制**

ES 的核心是倒排索引，适合精确词项、过滤、聚合；Milvus 的核心是向量索引，适合语义相似度。二者不是绝对替代关系。

---

### 40. 倒排索引是什么？

**第一层：短答**

倒排索引就是从“词”到“文档”的映射，方便根据关键词快速找到包含该词的文档。

**第二层：项目落点**

知识库里如果用户搜制度名、设备编号、油站编码，关键词检索会比纯向量更稳定。

**第三层：深挖机制**

传统数据库索引通常是从文档或行到字段值，倒排索引反过来，从词项定位文档集合。搜索引擎会结合分词、相关性评分和过滤条件返回结果。

## 十一、红线提醒

不要这样说：

- “Seata 我很熟，项目跨服务事务都是我做的。”如果不是你主负责，不要这么说。
- “MQ 能保证 100% 不丢。”没有单通道 100% 可靠。
- “Redis key 到期就立刻删除。”Redis 不是这样工作的。
- “网关鉴权后下游就不用管权限了。”下游仍要做数据权限。
- “Fanout 是轮询。”Fanout 是广播复制，轮询是消费者侧。

更稳的说法：

- “Seata 是平台基线，我了解原理；我直接负责更多是本地事务、MQ 最终一致性和 Redis 锁。”
- “MQ 通过多段机制尽量保证可靠，关键业务还需要定时任务兜底。”
- “Redis 过期是惰性删除和周期删除结合。”
- “网关负责认证，下游负责授权和数据权限。”
