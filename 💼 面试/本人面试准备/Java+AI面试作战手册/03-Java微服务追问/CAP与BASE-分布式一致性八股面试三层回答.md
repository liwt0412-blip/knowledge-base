# CAP与BASE-分布式一致性八股面试三层回答

## 二、第一层：60 秒短答

**CAP**：一致性 C（Consistency）、可用性 A（Availability）、分区容错 P（Partition tolerance）。分布式系统里网络分区必然发生，**P 不能放弃**，所以真实取舍只有分区那一刻：保 C——拒绝写入或阻塞读到一致为止（CP）；还是保 A——继续服务、允许暂时不一致（AP）。

**BASE**：Basically Available（基本可用）、Soft state（软状态）、Eventually consistent（最终一致）。它不是第三种理论，而是**选了 AP 之后的工程化补偿思路**：允许中间态存在，靠异步和补偿机制让数据最终收敛到一致。

一句话主线：

> CAP 告诉你分区时只能二选一，BASE 告诉你选了 AP 之后怎么把一致性"追回来"。

## 三、第二层：原理展开

### 1. 为什么 P 不能放弃

跨机房、跨服务通信的网络不可控，丢包、延迟、断连是必然事件，分区不是"会不会发生"而是"什么时候发生"。放弃 P 等于假设网络永远可靠，那只能退回单机——单机系统才谈得上 CA。所以分布式语境下，CAP 的真实命题是：**分区发生时，你选 CP 还是 AP**。

### 2. CP 典型场景

ZooKeeper、etcd、Nacos 配置中心——宁可不可用，也不给错数据。因为**配置错了比读不到更危险**：读到旧配置可能把流量打到错误的环境或错误的参数上，宁可让客户端等一等、拿不到，也不能让它拿到错的。

### 3. AP 典型场景

Eureka、Nacos 注册中心默认——服务发现宁可拿到旧列表也不能全挂。旧列表里大部分实例还是好的，客户端照常能调通；如果注册中心自己先不可用，整个集群的服务发现就瘫了，损失远大于"个别已下线实例调用失败后重试"。

### 4. 注册中心偏 AP、配置中心偏 CP 的第一性原理

同一个 Nacos 两种取舍，依据是同一条：

- **注册中心：读多写少且容忍旧值**——服务列表变化不频繁，读到几秒前的列表代价很小（个别实例调不通可以重试），可用性优先 → AP。
- **配置中心：错了代价高**——配置是全局生效的，一份错配置影响所有客户端，宁可短暂读不到 → CP。

一句话：**取舍不看组件，看"读到旧值/错值的代价"和"不可用的代价"谁更大。**

### 5. BASE 的工程手段

BASE 不是口号，落地靠一组具体手段拼起来：

- **MQ 异步解耦**：写操作先入队，下游慢慢消费，削峰的同时容忍中间态；
- **本地消息表**：业务写库和消息记录在一个本地事务里，保证"业务成功了消息一定有"；
- **失败重试**：消费失败按策略重投，允许暂时不成功；
- **死信补偿**：重试耗尽进死信队列，告警 + 人工/定时任务补偿；
- **消费幂等**：同一消息重复消费不产生副作用，这是重试敢放开的前提。

这些拼起来，就是"最终一致"的实现：**不保证每时每刻一致，保证在有限时间内收敛到一致，且过程可追踪、可恢复**。

## 四、第三层：追问应对

### 1. "你们项目哪里体现 CAP 取舍？"

只讲两个真锚点，不展开虚构场景：

- **Nacos 的 AP/CP 分工**：我们微服务体系里 Nacos 同时做注册中心和配置中心，注册中心偏 AP 保服务发现可用，配置中心偏 CP 保配置正确，取舍依据就是"读到旧值的代价"不同（对应 [[面试追问准备作战手册]] 题 29）。
- **设备上报链路选 AP**：石化设备 IoT 上报是我直接负责的模块。上报先进 MQ，断连、重试、乱序都允许中间态，最终落库一致。选 AP 的依据：设备状态晚几秒一致业务上可以接受，但**上报通道不能拒收**——拒收就是丢数据，丢数据比晚一致严重得多。

### 2. "最终一致怎么保证不出错？"

四件套，缺一个都不稳：

- **幂等**：重复消费不产生副作用（如消费前去重/状态判断/唯一约束），这是敢重试的前提；
- **可重试**：失败消息按策略重投，而不是直接丢；
- **可补偿**：重试耗尽进死信，有告警和补偿通道兜底；
- **可观测**：失败可追踪，能定位到是哪条消息、哪一步失败。

一句话：最终一致不是"保证每一步都对"，而是"**每一步错了都能被发现、被重放、被补偿**"。

### 3. "Seata AT 是强一致吗？"

不是。AT 一阶段业务 SQL 就在本地提交了，全局锁防的是并发全局事务之间的脏写，不保证跨服务的强一致——它属于**柔性事务、BASE 一侧**的方案；XA（两阶段全程持锁）才偏强一致。AT 选性能换一致性窗口，这正是 BASE 思路在事务层的落地。详见 [[Seata-AT模式脏写原理深挖面试三层回答]]。

### 4. "BASE 和 ACID 矛盾吗？"

不矛盾，**层次不同**：ACID 管单库本地事务的强一致，BASE 管跨服务长链路的最终一致。同一个系统里两者并存——比如调拨业务，单库内的扣减和写入走 ACID 本地事务，跨服务、跨模块的链路靠 MQ + 幂等 + 补偿走 BASE。ACID 是点，BASE 是线。

### 5. "CAP 里 CA 系统存在吗？"

存在，但只在单机语境：单机数据库没有网络分区问题，天然 CA。分布式语境下讨论 CA 没有意义——这是面试官常挖的坑，直接点破"放弃 P 等于退回单机"即可。
