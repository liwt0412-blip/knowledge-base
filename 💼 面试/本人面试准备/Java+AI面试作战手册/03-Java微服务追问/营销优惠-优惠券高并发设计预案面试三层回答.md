# 营销优惠-优惠券高并发设计预案面试三层回答

## 一、这份文档的定位与事实边界（先读）

**这是纯方案储备，不是项目实绩。** 两条事实边界：

1. **天机学堂仓库里优惠券没有实现**：`tj-trade` 的 `OrderServiceImpl.placeOrder` 里只有 `// TODO 2.2.计算优惠金额` 占位，discount 字段恒为 0。口径是"订单金额模型预留了优惠字段，券服务的完整方案我设计过、未编码"；
2. **石化平台的营销活动在客户核算服务里，是同事的模块**，我了解架构。且石化是内部平台，真实营销量级大概率用不上高并发方案——正确口径是"**量级驱动选型**：量小就是直写库 + 唯一键，量上来才启用这套预扣方案"（与底稿 4.11 反馈入口的第一性原理一致）。

使用场景：面试官主动问优惠券、秒杀、抢购、营销类高并发场景时拿出来，并主动交代边界：

> 这个方案我在学习项目里设计过完整链路（订单模块预留了优惠入口），工作中营销模块是同事负责、我了解架构。方案本身我可以完整展开。

## 二、优惠券系统整体怎么设计？

### 第一层：基础回答

三张核心表：**券模板**（批次、总量、每人限领、有效期、门槛与折扣规则）、**用户券**（券实例，状态机：未使用/已冻结/已使用/已过期）、**核销记录**（哪张券在哪个订单上用的）。三条链路：发券（运营建模板）、领券（高并发入口）、核销（下单用券 + 退单回券）。

### 第二层：结合面试回答

设计的关键是把三条链路的并发等级分开看：**发券是低频管理操作，直写库；核销和订单同频，走条件更新就够；只有领券是真正的高并发入口**——"总量有限 + 先到先得 + 每人限量"三个约束叠在一起，才是 Redis 预扣方案存在的理由。

### 第三层：边界和追问

- 被问"你们量级有多大"：不编数字，按设计讲——方案按"峰值几百到几千 QPS 抢同一批券"设计；低于这个量级直写库 + 唯一约束就是正确答案，过度设计反而增加一致性问题；
- 规则（门槛、折扣、叠加）是另一套子问题（规则引擎/责任链），面试聚焦高并发领取，规则计算只讲"下单时按模板快照计算，券实例冗余当时规则，防模板改价影响已领券"。

## 三、领券怎么防超发？（核心卖点）

### 第一层：基础回答

Redis 预扣库存 + 一条 Lua 原子完成"校验 + 扣减 + 已领标记"，成功后发 MQ 异步落库，DB 唯一键做最终裁决。

### 第二层：结合面试回答

完整链路：

```text
预热：券模板库存加载进 Redis（SETNX 初始化，防并发重复加载）
  ↓
领取请求 → 一条 Lua 原子执行：
  ① 已领标记判重（SISMEMBER / SETNX）→ 已领过直接返回
  ② 库存 > 0 校验 → DECR 库存
  ③ 写已领标记 + 写待发事件
  ↓
MQ 异步落库：消费端 INSERT 用户券（唯一键 uk_user_template 裁决）
  撞唯一键 = 重复 → 补偿：INCR 回库存、删已领标记
```

三个设计要点，每个都能展开：

1. **为什么必须一条 Lua**："查库存 → 扣库存 → 记标记"拆成多条命令就是 check-then-act，并发下超发；Lua 单线程执行天然原子。这也是"用 Lua 而不是分布式锁"的选型——锁是互斥语义有开销，Lua 无锁原子（选型框架见 [[☕ Java笔记/分布式锁策略选型-锁非首选判断框架与Redisson实践]]）；
2. **为什么 DB 还要唯一键**：Redis 是拦截层不是裁决层。Redis 抖动、key 被逐出、标记丢失时，唯一键保证一人一券不超发；需要"每人限领 N 张"时唯一键带序号，或 user_coupon 计数 + 条件判断；
3. **库存的最终事实**：DB 侧用条件扣减 `UPDATE template SET stock = stock - 1 WHERE id = ? AND stock > 0`，影响行数为 0 即失败——即使 Redis 预扣全部失效，DB 这层也扣不出负数。

### 第三层：边界和追问

- **Redis 库存和 DB 不一致怎么办**：设计口径是 Redis 只多不少地拦截（宁可少卖不可超卖）；XXL-Job 对账：Redis 剩余库存 + 已发券数 = 模板总量，不平则以 DB 为准修复；
- **MQ 落库失败**：消息进 error.queue 人工重放，消费端幂等保证重放安全（同一套 MQ 可靠性方法论，见 [[分布式锁与幂等-石化天机双项目面试三层回答]] 第四节）；
- **热点 key**：单批券一个 stock key 是单点热点，峰值再高就拆桶（stock 拆 N 个桶，Lua 按用户哈希取桶扣减）——追问深化用，不主动讲；
- **入口保护**：网关/Sentinel 限流 + 前端倒计时 + 答题/排队是体验层手段，不替代服务端原子扣减。

## 四、核销与退券怎么处理？

### 第一层：基础回答

核销用条件更新：`UPDATE user_coupon SET status='已使用' WHERE id=? AND user_id=? AND status='未使用' AND 有效期覆盖当前`，影响行数为 0 就是并发重复核销或券不可用，直接失败。

### 第二层：结合面试回答

- **与订单的一致性**：锁券（条件更新置"已冻结"）→ 建订单 → 确认核销，同事务或本地消息表保证最终一致；订单支付超时用**延迟消息阶梯重查**关单并回券（这套设施天机 tj-trade 支付链路已有实现，3s/5s/15s/30s/60s/2min 递增，最多 6 次）；
- **退单回券**：订单取消事件走 MQ，消费端条件更新把券置回"未使用"，幂等键防重复回券；
- **过期处理**：领券时按有效期发一条延迟消息到期置"已过期"，或 XXL-Job 每日扫描兜底——延迟消息是精确触发，定时扫描是兜底，两者互补。

### 第三层：边界和追问

- 被问"为什么不用 Seata 保证订单和券一致"：券状态机 + 条件更新 + MQ 最终一致就够，为一张券上分布式事务不值（与天机全仓库只用一处 `@GlobalTransactional` 的取舍一致）；
- 券冻结状态是防"下单未支付期间券被二次使用"的关键，讲状态机时主动提。

## 五、石化映射与面试讲法

石化客户核算服务有营销活动模块（了解架构，同事负责）。结合内部平台量级，诚实讲法分两层：

> 我们平台的营销场景量级不大，正确做法是直写库 + 唯一约束，简单可靠。但高并发抢券的标准方案我系统设计过：Redis 预扣 + Lua 原子扣减防超发、DB 唯一键做最终裁决、MQ 异步落库、延迟消息处理过期和关单回券。量级到了可以直接启用。

这个讲法把"为什么没用上高并发"变成加分项——量级驱动选型，和 AI 反馈入口（4.11）是同一个第一性原理。

## 六、面试官可能的深挖与应对清单

| 追问 | 一句话应对 |
|------|-----------|
| 超发怎么防的？ | 三层：Lua 原子"判重+扣减+标记"无 check-then-act 窗口；DB 唯一键一人一券；条件扣减 `stock>0` 扣不出负数 |
| Redis 挂了券多发吗？ | 不多发：Redis 只是拦截层，DB 唯一键和条件扣减是裁决层；宁可少卖不超卖，对账任务以 DB 为准修复 |
| 一人一券怎么保证？ | `uk_user_template` 唯一键；限领 N 张带序号或计数条件判断 |
| 为什么 Lua 不用分布式锁？ | Lua 无锁原子、单次往返；锁有开销且是互斥语义不是原子语义（见分布式锁选型框架） |
| 并发重复核销同一张券？ | 条件更新带 `status='未使用'`，影响行数 0 即失败，状态机保证只有未使用能核销 |
| 支付超时券怎么办？ | 锁券置冻结 → 延迟消息阶梯重查支付结果 → 关单回券，设施与支付链路复用 |
| 在哪真实做过？ | 如实答：学习项目订单模块预留了优惠入口、方案完整设计未编码；工作营销模块同事负责、我了解架构 |
