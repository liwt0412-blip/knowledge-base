# Java基础八股P0清单

## 使用原则

这份不是百科笔记，是面试前一个月必须打稳的底盘。

你的主线是：

> Java 后端工程师 + 石化经营分析平台 + AI 助手本地化落地。

所以 Java 八股不能背成孤立概念，要能落到项目：

- 讲集合，要能落到缓存、DTO、权限集合、去重。
- 讲并发，要能落到线程池、异步任务、ThreadLocal、接口并发安全。
- 讲 JVM，要能落到线上排查、内存、类加载、日志定位。
- 讲 Spring，要能落到 Controller、Service、事务、AOP、自动配置。
- 讲 MySQL/Redis，要能落到经营分析查询、缓存、权限、登录状态。

面试回答结构固定为三段：

1. 先说结论。
2. 再说原理。
3. 最后落项目或踩坑。

## 一、Java基础与集合

### 1. HashMap 底层结构是什么？

一层回答：

> HashMap 底层是数组 + 链表 + 红黑树，JDK 8 之后当链表过长并且数组容量达到条件时，会把链表转成红黑树，提高查询效率。

二层回答：

> put 时会根据 key 的 hash 计算数组下标。如果发生 hash 冲突，就放到同一个桶里。桶里元素少时用链表，元素多时转红黑树。扩容时容量通常变为原来的 2 倍，并重新分布元素。

三层落项目：

> 项目里经常用 Map 做字典转换、权限映射、统计结果组装。我的理解是 HashMap 适合单线程局部变量场景，如果是多线程共享场景，不能直接用 HashMap，要考虑 ConcurrentHashMap 或者把共享状态收敛到 Redis、数据库、线程安全容器里。

警惕：

- 不要说 HashMap 线程安全。
- 不要只背“数组链表红黑树”，要能说冲突、扩容、使用场景。

### 2. HashMap 为什么线程不安全？

一层回答：

> 因为 HashMap 的 put、resize 等操作没有并发控制，多线程同时修改可能导致数据覆盖、丢失或者结构异常。

二层回答：

> 多线程同时 put 同一个桶，可能覆盖链表节点；扩容时如果多个线程同时迁移元素，也可能出现不可预期结果。JDK 8 虽然改善了早期死循环问题，但 HashMap 本身仍然不是线程安全容器。

三层落项目：

> 我在项目里会把 HashMap 用在方法内部的临时数据组装，比如把数据库查询结果转成 Map。如果要跨线程共享，比如缓存、统计状态、并发计数，就不会直接用普通 HashMap。

### 3. ConcurrentHashMap 怎么保证线程安全？

一层回答：

> ConcurrentHashMap 通过更细粒度的并发控制保证线程安全。JDK 8 主要使用 CAS + synchronized 锁桶头节点，减少全表加锁。

二层回答：

> get 基本无锁，put 时先定位桶。如果桶为空，用 CAS 放入；如果桶不为空，对桶头节点加 synchronized。这样锁粒度是桶级别，不是整张表。

三层落项目：

> 如果本地内存里需要维护并发访问的映射关系，比如短生命周期的任务状态，可以考虑 ConcurrentHashMap。但如果涉及多实例共享状态，就应该放 Redis 或数据库，不能只靠单 JVM 内存。

### 4. equals 和 hashCode 的关系？

一层回答：

> 如果两个对象 equals 相等，它们的 hashCode 必须相等；hashCode 相等，不代表 equals 一定相等。

二层回答：

> HashMap、HashSet 这类集合会先用 hashCode 定位桶，再用 equals 判断是否是同一个 key。如果只重写 equals 不重写 hashCode，集合判断可能出错。

三层落项目：

> DTO、实体类如果作为 Map key 或 Set 元素，要特别注意 equals/hashCode。一般业务里我更倾向于用明确的业务 id、编码字符串作为 key，而不是直接用复杂对象当 key。

### 5. String、StringBuilder、StringBuffer 区别？

一层回答：

> String 不可变；StringBuilder 可变、非线程安全、性能较好；StringBuffer 可变、线程安全，但同步开销更大。

二层回答：

> 字符串频繁拼接时，如果在循环里直接用 String，可能产生很多临时对象。单线程拼接用 StringBuilder 更合适。StringBuffer 现在业务代码里用得相对少，除非确实需要同步。

三层落项目：

> 项目里组装日志、SQL 片段、提示词时，如果拼接较多，我会用 StringBuilder，普通少量字符串拼接用 `+` 也可以，因为编译器会做优化。

### 6. BigDecimal 为什么常用于金额？

一层回答：

> 因为 double/float 是二进制浮点数，可能有精度误差；BigDecimal 能更精确地表达十进制数。

二层回答：

> 使用 BigDecimal 时要注意不要直接用 `new BigDecimal(double)`，更推荐用字符串或 `BigDecimal.valueOf`。除法时要指定精度和舍入模式，否则可能抛异常。

三层落项目：

> 石化经营分析里如果涉及金额、单价、利润率、同比环比等指标，不能随便用 double 做最终业务金额计算，至少在关键结果上要用 BigDecimal 保证精度可控。

## 二、并发与线程

### 7. volatile 解决什么问题？

一层回答：

> volatile 主要解决可见性和禁止指令重排序，不保证复合操作的原子性。

二层回答：

> 一个线程修改 volatile 变量，其他线程能更快看到最新值。它适合做状态标记，但像 `i++` 这种读改写操作，volatile 不能保证线程安全。

三层落项目：

> 如果只是控制任务是否停止，可以用 volatile 标记。但如果是并发计数、库存扣减、状态流转，就要用锁、Atomic 类、数据库事务或者 Redis 原子操作。

### 8. synchronized 和 Lock 有什么区别？

一层回答：

> synchronized 是 JVM 内置锁，使用简单，会自动释放；Lock 是 JUC 提供的显式锁，功能更灵活，但需要手动释放。

二层回答：

> Lock 支持尝试加锁、可中断锁、公平锁等能力。synchronized 在现代 JVM 中经过优化，普通场景性能并不差。

三层落项目：

> 业务代码里我优先避免共享可变状态。如果确实需要同步，简单临界区可以用 synchronized；复杂并发控制才考虑 Lock，并且必须放在 finally 里释放。

### 9. ThreadLocal 是什么？有什么坑？

一层回答：

> ThreadLocal 是线程本地变量，每个线程都有自己的一份数据，常用于保存当前用户、请求上下文等。

二层回答：

> 它的 key 是弱引用，但 value 仍然可能被线程持有。在线程池场景下线程会复用，如果不 remove，可能造成内存泄漏或用户上下文串号。

三层落项目：

> 我项目里遇到过当前用户上下文依赖 ThreadLocal 的问题，核心经验是：入口拦截器 set，请求结束 finally remove；业务层只读取，不应该随意写入上下文。

### 10. 线程池核心参数有哪些？

一层回答：

> 核心线程数、最大线程数、空闲存活时间、阻塞队列、线程工厂、拒绝策略。

二层回答：

> 提交任务时，先用核心线程；核心线程满了进队列；队列满了再创建非核心线程；达到最大线程数后触发拒绝策略。

三层落项目：

> AI 或报表类任务如果耗时较长，不能无脑开线程。要区分 CPU 密集、IO 密集，并设置队列和拒绝策略，防止把服务拖垮。面试里我会强调先评估任务类型和下游承载能力。

### 11. 线程池为什么不建议用 Executors 直接创建？

一层回答：

> 因为 Executors 的一些快捷方法可能使用无界队列或过大的线程上限，容易导致 OOM 或资源耗尽。

二层回答：

> 比如 fixedThreadPool 使用无界队列，cachedThreadPool 最大线程数很大。生产中更推荐显式创建 ThreadPoolExecutor，把线程数、队列长度、拒绝策略写清楚。

三层落项目：

> 如果经营分析平台里有异步导出、AI 调用、消息消费，我会显式配置线程池参数，并通过日志、监控观察队列堆积和耗时。

### 12. CompletableFuture 适合什么场景？

一层回答：

> CompletableFuture 适合异步编排，比如多个接口并行查询、结果合并、异步回调。

二层回答：

> 它能把串行 IO 改成并行 IO，但底层仍然依赖线程池。关键是要指定业务线程池，处理异常和超时，不能让异常静默丢失。

三层落项目：

> 如果一个经营看板需要同时查销量、库存、价格、告警，可以考虑异步并行查询再聚合。但如果这些查询有强事务依赖，就不适合简单并行。

## 三、JVM

### 13. JVM 运行时内存区域有哪些？

一层回答：

> 主要包括堆、虚拟机栈、本地方法栈、程序计数器、方法区。

二层回答：

> 堆主要放对象实例，是 GC 重点区域；栈保存方法调用的局部变量、操作数栈等；程序计数器记录线程执行位置；方法区保存类元信息、常量、静态变量等相关信息。

三层落项目：

> 面试里如果问线上 OOM，我会先区分是堆内存、元空间、线程栈还是直接内存，再结合日志、dump、监控去定位，而不是笼统说“内存不够”。

### 14. 什么对象可以作为 GC Roots？

一层回答：

> 栈帧中的局部变量、静态变量、常量、JNI 引用等可以作为 GC Roots。

二层回答：

> JVM 通过可达性分析判断对象是否存活。从 GC Roots 出发能到达的对象，一般认为是存活对象；不可达对象才可能被回收。

三层落项目：

> 如果出现内存泄漏，常见原因是对象仍然被集合、静态变量、ThreadLocal、缓存引用着。排查时要看引用链，而不是只看对象数量。

### 15. Minor GC 和 Full GC 区别？

一层回答：

> Minor GC 主要回收新生代；Full GC 通常涉及老年代、方法区等更大范围，停顿影响更大。

二层回答：

> 大量短生命周期对象会增加 Minor GC 频率；老年代空间不足、元空间压力等可能触发 Full GC。Full GC 频繁通常是需要重点排查的问题。

三层落项目：

> 报表导出、批量查询、AI 大文本拼接都可能制造大量对象。实际开发中要控制一次性加载数据量，分页处理，避免把大结果全集塞进内存。

### 16. 类加载过程是什么？

一层回答：

> 类加载大致包括加载、验证、准备、解析、初始化。

二层回答：

> 加载是读取 class；验证保证字节码安全；准备给静态变量分配默认值；解析把符号引用转直接引用；初始化执行静态变量赋值和静态代码块。

三层落项目：

> Spring 项目启动时大量 Bean、配置类、代理类都会涉及类加载。遇到类冲突、依赖版本问题、`ClassNotFoundException`，需要看依赖树和实际加载的类。

### 17. 线上 OOM 你怎么排查？

一层回答：

> 先看错误类型和日志，再看监控指标，必要时导出 heap dump，用工具分析大对象和引用链。

二层回答：

> 如果是 Java heap space，重点看堆对象；如果是 Metaspace，重点看类加载；如果是 unable to create new native thread，重点看线程数；如果是 direct buffer memory，重点看 NIO/Netty/大缓冲区。

三层落项目：

> 我会先保护现场，确认影响范围和触发操作，再定位最近上线、慢接口、大查询、缓存增长。修复上通常是分页、限流、释放引用、控制缓存容量，而不是简单把内存调大。

## 四、Spring 与 Spring Boot

### 18. Spring Bean 生命周期？

一层回答：

> Bean 生命周期包括实例化、属性填充、初始化、使用、销毁。

二层回答：

> 中间会穿插 BeanPostProcessor、Aware 回调、初始化方法、销毁方法。AOP 代理也通常在后置处理器阶段生成。

三层落项目：

> 面试里我会把它和项目里的依赖注入、代理、事务联系起来。很多 Spring 问题不是业务代码没执行，而是 Bean 初始化、代理、扫描路径或生命周期时机不对。

### 19. Spring AOP 原理是什么？

一层回答：

> Spring AOP 主要通过动态代理实现，在方法调用前后织入增强逻辑。

二层回答：

> 如果目标类实现接口，通常可以用 JDK 动态代理；没有接口时常用 CGLIB 生成子类代理。事务、日志、权限校验都可以基于 AOP 做横切增强。

三层落项目：

> 我会注意一个坑：同一个类内部方法自调用不会经过代理，所以事务、AOP 可能不生效。需要通过拆分 Service 或从代理对象调用来解决。

### 20. Spring 事务什么时候会失效？

一层回答：

> 常见失效场景包括方法不是 public、自调用、异常被吞掉、抛出非默认回滚异常、数据库引擎不支持事务、多数据源配置不正确。

二层回答：

> `@Transactional` 依赖代理生效。默认回滚 RuntimeException 和 Error，如果 catch 了异常不抛出，事务管理器就不知道需要回滚。

三层落项目：

> 在经营分析或数据写入场景，我会把事务边界放在 Service 层，避免事务里调用长耗时外部接口。AI 调用、远程接口、MQ 发送一般要谨慎放进本地数据库事务。

### 21. Spring Boot 自动配置原理？

一层回答：

> Spring Boot 通过自动配置类和条件注解，根据 classpath、配置属性、已有 Bean 自动装配常用组件。

二层回答：

> 核心是自动配置导入机制，加上 `@ConditionalOnClass`、`@ConditionalOnMissingBean`、`@ConditionalOnProperty` 等条件判断。它不是魔法，而是按条件加载配置类。

三层落项目：

> 我理解自动配置的价值是降低重复配置，但排查问题时要知道某个 Bean 为什么生效、为什么没生效。看启动日志、配置属性、依赖和条件匹配报告，是比较实际的排查路径。

### 22. Controller 参数绑定常见方式？

一层回答：

> 常见有 `@RequestParam`、`@PathVariable`、`@RequestBody`，以及请求头、Cookie 等。

二层回答：

> GET 查询参数一般用 `@RequestParam` 或对象接收；路径变量用 `@PathVariable`；JSON 请求体用 `@RequestBody`。不能把这些场景混用。

三层落项目：

> 我之前整理过接口联调经验，参数绑定问题要先看 URL、请求方式、Header、Content-Type、Controller 注解和前端实际传参，不能只盯 Service。

## 五、MySQL

### 23. MySQL 索引为什么常用 B+ 树？

一层回答：

> B+ 树高度低、范围查询友好，适合磁盘 IO 场景。

二层回答：

> B+ 树非叶子节点主要存索引，叶子节点存数据或主键，并且叶子节点有序连接，适合范围扫描。相比二叉树，它能减少磁盘 IO 次数。

三层落项目：

> 经营分析平台里经常有按时间、组织、油站、商品维度查询的场景。建索引时要结合查询条件、排序、分组，而不是看到字段就建索引。

### 24. 最左前缀原则是什么？

一层回答：

> 联合索引要从最左边的字段开始连续使用，才能更好利用索引。

二层回答：

> 比如索引 `(org_id, stat_date, product_id)`，如果查询条件没有 `org_id`，后面的字段很可能无法充分利用这个联合索引。

三层落项目：

> 设计经营分析查询索引时，我会把过滤性强、必传、常用等值查询的字段放前面，再考虑范围查询和排序字段。

### 25. 覆盖索引和回表是什么？

一层回答：

> 覆盖索引是查询字段都在索引里，不需要再回主键索引查完整数据；回表是二级索引查到主键后，再去主键索引查数据。

二层回答：

> 覆盖索引可以减少 IO，但不能为了覆盖所有查询建过宽索引。索引越多，写入成本和维护成本也越高。

三层落项目：

> 报表列表页如果只展示少数字段，可以考虑覆盖索引优化。但如果字段很多、更新频繁，就要权衡索引体积和写入性能。

### 26. 事务隔离级别和 MVCC？

一层回答：

> MySQL 常见隔离级别有读未提交、读已提交、可重复读、串行化。InnoDB 通过 MVCC 提供一致性读。

二层回答：

> MVCC 通过版本链和 ReadView 让读操作看到一个相对一致的数据快照，减少读写冲突。MySQL InnoDB 默认可重复读，并通过 next-key lock 等机制处理部分幻读问题。

三层落项目：

> 如果经营分析读的是统计结果，通常更关注一致性口径和统计时点；如果是交易写入，就要更关注事务边界、并发更新和锁冲突。

### 27. 慢 SQL 你怎么排查？

一层回答：

> 先看慢日志和执行计划，再看索引、扫描行数、回表、排序、分组、锁等待和返回数据量。

二层回答：

> `explain` 里重点看 type、key、rows、Extra。常见问题包括没走索引、索引失效、函数包裹字段、like 前缀通配、大 offset 分页、一次查太多字段。

三层落项目：

> 分析类系统里慢 SQL 很常见。我会优先确认查询条件是否符合索引、是否需要分页、是否能预聚合、是否能把复杂计算前置到离线或定时任务。

## 六、Redis

### 28. 缓存穿透、击穿、雪崩是什么？

一层回答：

> 穿透是查不存在的数据绕过缓存打到数据库；击穿是热点 key 失效瞬间大量请求打到数据库；雪崩是大量 key 同时失效或 Redis 故障导致数据库压力暴增。

二层回答：

> 穿透可以用空值缓存、布隆过滤器；击穿可以用互斥锁、逻辑过期；雪崩可以给过期时间加随机值、多级缓存、限流降级。

三层落项目：

> 石化 AI 助手链路里 Redis 可以缓存会话、热点知识结果或业务工具结果。但缓存不是越多越好，要区分数据时效性、权限隔离和失效策略。

### 29. Redis 分布式锁要注意什么？

一层回答：

> 要保证加锁原子性、设置过期时间、释放锁校验持有者，避免误删别人的锁。

二层回答：

> 常见做法是 `SET key value NX EX` 加锁，value 放唯一标识，释放时用 Lua 脚本判断 value 一致再删除。还要考虑锁超时、业务执行过长、可重入等问题。

三层落项目：

> 如果只是防止重复提交、重复任务触发，可以用 Redis 锁。但如果涉及强一致资金类业务，不能只靠 Redis 锁，还要结合数据库唯一约束、事务和幂等设计。

### 30. 缓存和数据库一致性怎么处理？

一层回答：

> 常见策略是先更新数据库，再删除缓存，必要时配合延迟双删、消息补偿或 Canal 订阅。

二层回答：

> 不能简单说“更新数据库同时更新缓存”就一定安全，因为并发下可能出现旧值覆盖新值。删除缓存让下一次读取回源加载，通常更稳。

三层落项目：

> 经营分析类数据如果是统计结果，可以接受短暂延迟，就要明确缓存 TTL 和刷新策略；权限、登录态、强时效数据则要更谨慎，不能让缓存污染越权结果。

## 七、项目高频口径

### 31. JWT + Redis 登录状态怎么讲？

一层回答：

> JWT 负责携带身份信息，Redis 可以保存登录态、权限或 token 黑名单，实现服务端可控。

二层回答：

> 纯 JWT 的问题是签发后不容易主动失效。结合 Redis 后，可以支持退出登录、续期、踢下线、权限变更后刷新。

三层落项目：

> 中州养老和若依体系里这类登录权限链路很常见。面试里我会重点讲清楚：认证解决“你是谁”，授权解决“你能访问什么”。

### 32. 全局异常处理有什么价值？

一层回答：

> 统一异常返回格式，避免 Controller 到处 try-catch，提高接口一致性。

二层回答：

> 可以用 `@RestControllerAdvice` + `@ExceptionHandler` 处理业务异常、参数异常、系统异常，并配合日志记录。

三层落项目：

> 面试里我会强调异常不能全吞。业务异常给用户明确提示，系统异常记录日志并返回统一错误码，避免把堆栈直接暴露给前端。

### 33. 接口联调排查顺序？

一层回答：

> 先看请求是否到 Controller，再看参数绑定、权限、Service、SQL、返回结构。

二层回答：

> 具体看 URL、method、Header、token、Content-Type、请求体、后端日志、断点、SQL 日志、前端实际读取字段。

三层落项目：

> 我之前准备了真实团队开发流程文档。我的口径是：接口问题不要一上来就改业务逻辑，先把请求链路拆开，定位到底是入口、权限、参数、查询还是返回映射问题。
