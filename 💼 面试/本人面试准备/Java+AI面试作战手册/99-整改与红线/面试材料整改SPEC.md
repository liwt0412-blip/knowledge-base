## 1. 背景

当前面试材料集中在石化经营分析平台，主打方向是 Java 后端 + AI 应用落地。材料整体有竞争力，但存在口径不一致、技术表述过满、部分数字缺少可验证依据、职责边界容易被质疑等问题。

本次整改目标不是把材料写得更炫，而是让材料更可信、更抗追问、更适合一个月后的 Java + AI 面试。

## 2. 整改目标

1. 统一 AI 模块架构口径，避免前后矛盾。
2. 明确本人职责边界，降低“过度包装”风险。
3. 保留高价值亮点，删除或降级容易被打穿的表述。
4. 把项目表达从“技术名词堆叠”调整为“业务问题 → 技术方案 → 工程取舍 → 结果”。
5. 准备面试官高概率追问的标准答案。

## 3. 目标人设

统一包装为：

> Java 后端工程师，做过复杂企业业务系统，最近参与 Java + AI 应用落地。能把 RAG、Function Calling、知识库问答、缓存、权限、会话记忆接进真实业务系统，懂成本、延迟、稳定性和工程取舍。

避免包装为：

- AI 算法工程师
- 大模型训练工程师
- 全链路从零自研负责人
- 什么都主导、什么都精通的全能架构师

## 4. 核心问题清单

### 4.1 AI 架构口径冲突

当前冲突：

- 逐字稿：当前生产架构是 Redis 缓存 + Function Calling 自路由 + Python RAG，不使用独立路由层。
- AI 专项素材：描述了三层递进检索、BM25、BERT 意图分类，并像是当前主链路。

整改要求：

- 当前主链路统一为：Redis 缓存 → Spring AI Function Calling → Python RAG。
- BERT 分类器统一降级为“探索阶段做过的技术储备，最终未上线主链路”。
- BM25 统一降级为“RAG 内部可选前置候选召回，不作为独立主路由层”。

标准表述：

> 当前生产链路没有独立意图分类器。用户问题先查 Redis 缓存，未命中后进入 Spring AI。模型如果触发 Function Calling，就调用 Java 业务函数查实时数据；如果没有触发工具调用，就转发到 Python RAG 服务检索知识库。BERT 分类器只作为早期探索，因缺少完整、可复现的生产评估证据，不讲具体准确率，也不作为当前主链路。

### 4.2 “100% 确定 / 零误判”表述过强

当前风险：

- “Function Calling API 返回 100% 确定”
- “零误判”
- “RAG 解决幻觉”

整改要求：

改成更专业、更稳的表达。

推荐替换：

- “工具是否被调用这个事件是确定的”
- “比文本分类路由更可观测”
- “降低误判风险”
- “RAG 能显著降低幻觉，但不能彻底消灭幻觉”

标准表述：

> Function Calling 的优势不是模型一定不会选错工具，而是工具调用结果是结构化、可观测、可记录的。相比让模型输出一段文字再由程序解析，工具调用是否发生、参数是什么、返回结果是什么，都能被系统明确捕获。

### 4.3 职责边界不够可信

当前冲突：

- 文档写集团供应商输出 RAG、会话管理、向量检索等标准化底座。
- 同时又写本人主导 AI 模块全链路设计和实现。

整改要求：

将“全链路主导”调整为“省公司本地化落地和关键链路实现”。

标准表述：

> 集团侧提供统一 AI 底座和技术规范，我主要负责省公司经营分析平台的本地化落地：业务工具封装、知识库接入、缓存策略、RAG 参数调优、会话历史打通，以及 AI 助手和现有 Java 微服务的数据对接。

### 4.4 数字过多且部分缺少证据链

当前高风险数字：

- AI 日均调用 4 万多
- 月末峰值 7.5 万
- DAU 1100-1300
- Redis 命中 60%+
- Reranker Top-5 提升 35%
- 文档 1.2 万份

整改要求：

保留少量最有业务价值的数字。所有保留数字必须准备来源说明。

只有补齐来源后才建议保留：

- 14 个地市、近 2000 座加油站
- 设备上报峰值（必须区分生产监控、压测结果和容量估算）
- 知识库一万多份文档
- Redis 缓存效果（有入口计数和统计窗口才讲比例）

建议降级：

- “Top-5 提升 35%”改为“离线评估中相关性有明显提升”
- “日均 4 万调用”除非能讲清日志统计口径，否则改为“高峰期调用量较大”

标准表述：

> 如果有 Redis 命中计数和 RAG 入口请求日志，可以说明统计窗口、总请求数、命中数和排除规则；如果没有，只讲“针对高频重复问题设计了缓存”，不讲命中比例。

### 4.5 技术栈过满

当前风险：

技术栈覆盖太广，容易被问“这些都是你做的吗？”

整改要求：

每项技术标明关系：

- 我直接负责
- 我参与联调
- 平台已有，我了解调用方式
- 同事负责，我了解架构

建议在项目表述中增加一句：

> 其中 AI 助手和设备 IoT 接入是我直接负责的，非油、客户核算、网关通知是团队其他成员负责，我了解整体链路和关键设计，方便联调和排障。

## 6. 面试标准回答模板

### 6.1 你 AI 模块到底做了什么？

> 集团侧提供统一 AI 底座和规范，我主要做的是省公司经营分析平台的本地化落地。具体包括三块：第一，把设备查询、告警查询、客户查询这些 Java 业务能力封装成 Spring AI 的工具函数；第二，把省公司操作手册、安全规范、培训课件接入知识库，联调 Python 侧 RAG 检索链路；第三，做缓存、会话历史和流式输出这些工程化能力，保证实际使用时延迟和成本可控。

### 6.2 为什么不用 BERT 做意图分类？

> BERT 分类器我们探索过，小规模验证中有一定效果，但没有形成完整、可复现的生产评估证据。这个场景是垂类业务助手，前置分类本身还会引入误判。最后没有把它放进生产主链路，而是用 Function Calling 做结构化工具调用，没触发工具调用的问题再走 RAG。

### 6.3 怎么控制 AI 成本？

> 第一层是 Redis 缓存，命中后可以直接避免模型调用。未命中时由模型判断是否调用业务 Tool；Function Calling 通常仍会产生模型调用，但结构化业务查询不必进入 RAG 长上下文链路。知识类问题才进入 RAG。这样主要减少重复生成、无效检索和长上下文成本。

### 6.4 怎么减少幻觉？

> 我们不让模型自由发挥，而是让回答基于检索到的内部文档或工具函数返回的数据。RAG 回答会附带来源，相关性低于阈值时走兜底，不强行回答。Function Calling 查到的是系统实时数据，这部分比纯文本生成更可靠。当然幻觉不能说完全消灭，只能通过检索质量、引用来源、阈值和反馈机制持续降低。

### 6.5 你和 Python 服务边界怎么分？

> Java 侧负责企业系统集成，包括鉴权、业务函数、配置、缓存、会话、知识库管理入口。Python 侧负责 NLP 流水线，包括 OCR、切分、embedding、Milvus 混合检索和 Reranker。因为 BGE-M3 和 Reranker 都是 PyTorch 生态，放 Python 更合适；而业务数据和权限在 Java 微服务体系里，所以 Java 负责业务入口。

## 7. 一个月内整改优先级

### P0：必须完成

1. 统一 AI 架构口径。
2. 删除或降级 BERT/BM25 主链路描述。
3. 修正“全链路主导”职责边界。
4. 替换“100% / 零误判 / 彻底解决幻觉”等绝对化表达。

### P1：高优先级

1. 给关键数字补统计口径。
2. 准备 Java 基础追问：Spring、Redis、MySQL、RabbitMQ、线程池、事务。
3. 准备 AI 工程追问：chunk size、向量库参数、Reranker、权限过滤、缓存一致性。

### P2：可优化

1. 把逐字稿压缩到 2 分钟内。
2. 给每个模块准备一个“问题-方案-结果”的 30 秒版本。
3. 准备 2 个反向提问，避免显得刻意炫技。

## 8. 验收标准

整改完成后，材料必须满足：

1. 任意文件对 AI 架构的描述不冲突。
2. 面试官问“这是你做的吗？”时，能清楚分出本人负责、团队负责、平台已有。
3. 每个关键数字都能说出来源；说不出来源的数字不主动讲。
4. 不再出现“100% 零误判”“彻底解决幻觉”等绝对化表达。
5. 2 分钟内能讲完项目，5 分钟内能讲清 AI 模块，15 分钟追问不自相矛盾。

## 9. 最终推荐版本

主线表达统一为：

> 这个项目不是从零训练模型，而是把大模型能力接进已有 Java 业务系统。集团提供统一 AI 底座，我负责省公司侧的本地化落地：把业务查询封装成 Function Calling 工具，把知识库接入 RAG 检索链路，用 Redis 缓存和 SSE 流式输出控制体验和成本。我的优势是懂 Java 企业系统，也理解 RAG、工具调用、缓存、会话这些 AI 应用落地时真正会遇到的问题。
