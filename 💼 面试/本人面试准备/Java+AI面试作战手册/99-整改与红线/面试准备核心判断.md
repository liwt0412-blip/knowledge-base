# 面试准备核心判断

## 一句话结论

你的材料比普通 Java 求职者强，但也更容易被追问打穿。

你的优势不是“会 AI”，而是能把自己定位成：

> Java 后端工程师，做过复杂企业业务系统，最近把 RAG、Function Calling、知识库问答接进真实业务里，懂成本、延迟、稳定性和工程取舍。

这个定位适合 Java + AI 应用岗，比包装成 AI 算法工程师更可信。

## 面试官真正看什么

从第一性原理看，面试官主要判断三件事：

1. 你能不能立刻干活。
2. 你说的项目是不是真做过。
3. 你进来后会不会变成团队风险。

所以材料不是越炫越好，而是要让面试官相信：你真实参与过复杂业务，有工程判断，知道 trade-off，不是背了一套 RAG 术语。

## 你的亮点

### 1. 项目故事完整

你不是只讲用户、角色、菜单、订单这种普通 CRUD 项目，而是能讲清楚一个省级经营分析平台：

- 14 个地市
- 近 2000 座加油站
- 设备 IoT
- 非油库存
- 客户核算
- 经营报表
- AI 智能助手

这比大多数 Java 候选人的项目有业务厚度。

### 2. Java + AI 定位准确

你没有必要包装成算法工程师。你的最佳定位是 Java 后端 + AI 应用落地。

合理分工是：

- Java 做企业系统集成、鉴权、业务函数、缓存、会话、Function Calling。
- Python 做 OCR、切分、Embedding、Milvus 检索、Reranker。

这个口径符合真实企业落地方式。

### 3. 有工程化意识

你材料里有这些高价值点：

- Redis 缓存降低重复调用
- Function Calling 查询实时业务数据
- RAG 处理知识库问题
- SSE 流式输出改善体验
- 会话记忆隔离
- 引用来源和兜底降低幻觉
- Java + Python 分工

这些比“我会调 DeepSeek API”强很多。

### 4. 会讲取舍

你能讲：

- 为什么 RAG 不微调
- 为什么不用 BERT 做主链路分类器
- 为什么 Java + Python 双语言
- 为什么设备心跳用 Redis TTL
- 为什么调拨用事务和乐观锁

这类回答更像真实工程师，而不是只背技术名词。

## 你的不足

### 1. 最大风险是口径不一致

原材料里有过冲突：

- 一个版本说当前主链路是 Redis + Function Calling + Python RAG。
- 另一个版本说有三层递进检索、BM25、BERT 意图分类。

这两个不能同时作为当前生产架构。

统一口径必须是：

> 当前生产主链路：Redis 缓存 -> Spring AI Function Calling -> Python RAG。  
> BERT 分类器是探索阶段技术储备，最终没进入主链路。  
> BM25 是 RAG 内部可选候选召回，不是独立主路由层。

### 2. 绝对化表达容易被打穿

不要说：

- 100% 确定
- 零误判
- 彻底解决幻觉
- 全链路都是我主导

应该说：

- 工具调用结果是结构化、可观测的
- 能降低误判风险
- RAG 能显著降低幻觉，但不能彻底消灭
- 我负责省公司本地化落地和关键链路实现

### 3. 数字太多会反噬

这些数字如果没有统计口径，不要主动讲：

- AI 日均调用 4 万多
- 月末峰值 7.5 万
- DAU 1100-1300
- Reranker 提升 35%

建议主动讲这些更稳：

- 14 个地市、近 2000 座加油站
- 加油机约 8000 台
- 设备存在短时突发上报（有监控或压测证据后再讲精确 QPS）
- 知识库一万多份内部文档
- 高频重复问题适合缓存或候选问答拦截（有入口计数后再讲命中比例）

### 4. “主导”边界要收住

如果说集团有统一 AI 底座，就不要再说整套 AI 架构完全由你主导。

可信表达是：

> 集团侧提供统一 AI 底座和规范，我主要负责省公司经营分析平台的本地化落地：业务工具封装、知识库接入、缓存策略、RAG 检索链路联调、会话历史打通，以及 AI 助手和现有 Java 微服务的数据对接。

## 相对其他求职者的优势

### 1. 比普通 Java 候选人项目更复杂

普通候选人常见项目：

- 后台管理系统
- 权限管理
- 商品订单
- 简单 CRM
- 单纯 CRUD

你的项目能讲：

- 省级业务系统
- IoT 设备接入
- RabbitMQ 削峰
- Redis 心跳
- ClickHouse 分析
- RAG 知识库
- Function Calling

复杂度和业务可信度更强。

### 2. AI 不是兴趣爱好

普通候选人可能只会说：

> 我用过 ChatGPT，对 AI 感兴趣。

你的材料能讲：

- RAG 架构
- 父子块切分
- 稠密 + 稀疏混合检索
- Reranker
- SSE 流式输出
- Function Calling
- 会话记忆
- 缓存降本

这在 Java 候选人里是明显加分项。

### 3. 懂企业落地

你不是只讲模型能力，而是能讲：

- 业务知识库怎么更新
- 成本怎么控制
- 幻觉怎么降低
- 权限和会话怎么处理
- Java 业务系统怎么接大模型

这更符合企业真实招聘需求。

## 相对其他求职者的劣势

### 1. Java 基础会被重点考

如果面试官绕开 AI，直接打 Java 基础，你必须扛住：

- JVM
- 线程池
- 并发
- MySQL 索引
- 事务隔离
- Redis 数据结构
- MQ 幂等
- Spring Bean 生命周期
- Spring 事务失效场景

AI 亮点救不了 Java 基础短板。

### 2. AI 应用面试官会追评估体系

可能被问：

- 召回率怎么测？
- 测试集怎么构造？
- Reranker 提升怎么验证？
- chunk size 为什么这么设？
- Milvus 参数怎么调？
- embedding 失败怎么补偿？
- 权限过滤怎么做？
- 文档更新后旧向量怎么处理？

这部分要准备标准答案，不能只背概念。

### 3. 材料容易显得过满

技术栈太多时，面试官会问：

> 这些都是你做的吗？

所以必须主动区分：

- 我直接负责
- 我参与联调
- 团队其他人负责，我了解架构
- 平台已有，我调用或对接

## 一个月内最该补

### P0：材料口径

1. 统一 AI 主链路。
2. BERT/BM25 降级为探索或可选方案。
3. 所有“我主导全链路”改成“我负责本地化落地和关键链路实现”。
4. 删除绝对化表达。

### P1：Java 基础

重点补：

- Spring Boot 自动配置
- Spring 事务失效
- MySQL 索引和慢查询
- Redis 缓存穿透、击穿、雪崩
- RabbitMQ 消息可靠性、重复消费、幂等
- 线程池参数和拒绝策略
- JVM 内存和 GC 基础

### P1：AI 应用追问

重点补：

- RAG 为什么不微调
- chunk size 怎么定
- 向量检索和关键词检索区别
- Reranker 作用
- 幻觉怎么降低
- 成本怎么控制
- 权限过滤怎么做
- 会话记忆怎么隔离
- Function Calling 和普通 API 调用区别

### P2：表达训练

准备三种版本：

1. 30 秒自我介绍。
2. 2 分钟项目介绍。
3. 5 分钟 AI 模块展开。

原则：

> 主动讲 70% 可信内容，保留 30% 深度细节给追问。

## 最终推荐人设

不要说自己是 AI 专家。

更好的表达是：

> 我是 Java 后端工程师，做过复杂企业业务系统。最近参与了 Java + AI 应用落地，能把大模型能力接进现有业务系统，处理 RAG、Function Calling、缓存、会话、成本和稳定性这些工程问题。

这个人设最适合你当前材料，也最容易通过 Java + AI 应用岗面试。
