## 四、AI 助手模块

### 4.1 业务场景

AI 助手面向三个场景：

1. **业务操作顾问**
   一线员工不知道系统功能怎么操作时，直接问 AI，AI 基于操作手册给步骤指引。

2. **系统运维顾问**
   运维人员输入错误信息或异常现象，AI 基于故障案例和运维文档给排查建议。

3. **智能答题助手**
   基于安全规范、培训课件生成题目、答案和解析，辅助内部培训。

### 4.2 当前生产主链路

当前主链路统一为：

```text
用户问题
  -> Java AI 助手入口
  -> Redis 缓存命中则直接返回
  -> 未命中进入 Spring AI
  -> 模型触发 Function Calling：调用 Java 业务函数查实时数据
  -> 未触发工具调用：转发 Python RAG 服务检索知识库
  -> 生成回答，SSE 流式返回
  -> 高频结果回写 Redis
```

这里没有独立的 BERT 意图分类器作为主路由。工具是否被调用、参数是什么、返回结果是什么，都能被系统结构化记录，比解析模型输出文字更可观测。

### 4.3 为什么不用 BERT 分类器做主链路

BERT 分类器在探索阶段做过，小规模验证中有一定效果，但没有形成足够完整、可复现的生产评估证据。最终没有放入主链路，原因有三个：

1. 这是垂类业务助手，用户大多数问题本来就是业务问题，通用闲聊比例很低。
2. 前置分类器会引入额外误差，误判后可能让专业问题绕过知识库。
3. Spring AI 的 Function Calling 已经能完成业务工具调用判断，链路更短，也更容易观测。

所以当前口径是：

> BERT 是探索和技术储备，不是生产主链路。

### 4.4 Java + Python 分工

| 侧 | 负责内容 | 原因 |
|---|---|---|
| Java / Spring AI | 鉴权、业务函数、Function Calling、缓存、会话、知识库管理入口 | 与现有企业 Java 微服务体系集成最自然 |
| Python / LangChain | OCR、文档切分、Embedding、Milvus 混合检索、Reranker | BGE-M3、BGE Reranker 等模型在 Python/PyTorch 生态更成熟 |

这个分工不是重复造轮子，而是让 Java 做企业集成，Python 做 NLP 流水线。

### 4.5 RAG 检索链路

RAG 侧核心流程：

```text
文档上传
  -> OCR/文本抽取
  -> 父子块切分
  -> BGE-M3 生成稠密向量和稀疏向量
  -> 写入 Milvus

用户提问
  -> BGE-M3 编码
  -> 稠密语义检索 + 稀疏关键词检索
  -> WeightedRanker 融合
  -> BGE Reranker 重排序
  -> Top 文档片段注入 Prompt
  -> LLM 生成回答并返回引用来源
```

父子块策略：

- 子块用于精确匹配，粒度更小。
- 父块用于喂给模型，保证上下文完整。

不要主动说固定数值，除非面试官追问。可以说：

> 具体 chunk 大小是根据文档结构和离线效果调出来的，不是固定套模板。原则是检索粒度要小，生成上下文要完整。

### 4.6 幻觉控制

RAG 能降低幻觉，但不能说彻底消灭幻觉。当前做法：

1. 回答基于检索到的内部文档或 Function Calling 返回的真实业务数据。
2. 回答附带引用来源，方便用户确认依据。
3. 检索相关性不足时走兜底，不强行回答。
4. 用户反馈“有用/没用”用于后续优化检索参数和知识库质量。

标准说法：

> 我们不是让模型自由发挥，而是尽量让它基于内部资料和业务系统数据回答。幻觉不能说完全杜绝，只能通过检索质量、阈值、引用来源和兜底机制持续降低。

### 4.7 成本和延迟控制

主要靠三层：

1. **Redis 缓存**
   高频重复问题直接返回，避免每次都调模型。

2. **Function Calling**
   能查业务数据的问题直接调用 Java 函数，不走 RAG。

3. **RAG 结果缓存**
   高频 RAG 问题的最终回答设置 TTL，短期内复用。

标准说法：

> AI 成本不是理论问题，实际落地时必须控制调用次数和上下文长度。缓存命中可以直接避免模型调用；Function Calling 通常仍需模型选择工具和组织答案，但可以避免无关问题进入高成本的 RAG 长上下文链路。

### 4.8 会话记忆

会话分两层：

| 层 | 存储 | 作用 |
|---|---|---|
| 短期记忆 | Redis / ChatMemory | 单次会话内保留最近若干轮上下文 |
| 持久历史 | MySQL conversations 表 | Java 统一管理持久历史，调用 Python RAG 时按需传递必要上下文，避免跨服务共同写业务表 |

关键坑：

> Spring AI 使用 ChatMemory 时必须显式传 conversationId，否则不同用户、不同窗口的记忆可能串到一起。入口处要为每个会话生成唯一 conversationId，并在 Advisor 上下文里传递。

## 五、设备 IoT 模块

### 5.1 业务场景

设备管理服务负责全省加油站加油机、加油枪、液位仪、IoT 网关等设备档案和状态监控。

核心链路：

```text
加油机/MQTT
  -> 油站管控机
  -> 市公司前置机房
  -> 省中心
  -> RabbitMQ
  -> MySQL/ClickHouse/Redis
```

平台侧主要消费省中心汇聚后的数据，不单独维护底层设备传输链路。

### 5.2 技术实现

1. **RabbitMQ 削峰**
   早晚高峰设备集中上报，通过消息队列缓冲，避免接口被瞬时流量打满。

2. **Redis TTL 心跳**
   每次设备上报时刷新 `device:heartbeat:{deviceId}`，设备持续在线就续期，断开后 TTL 过期。

3. **XXL-Job 离线扫描**
   定时扫描设备状态，发现离线后生成告警，通过 RabbitMQ 推送到通知服务。

### 5.3 面试讲法

> 设备模块的核心不是 CRUD，而是数据上报、异步处理和状态判断。RabbitMQ 解决流量峰值，Redis TTL 解决心跳超时，XXL-Job 做离线告警补偿。相比轮询数据库，Redis TTL 更适合设备在线状态这种短生命周期数据。

## 六、常见追问标准答案

### 6.1 你 AI 模块到底做了什么？

集团侧提供统一 AI 底座和规范，我主要做省公司经营分析平台的本地化落地。具体包括三块：第一，把设备查询、告警查询、客户查询这些 Java 业务能力封装成 Spring AI 工具函数；第二，把省公司操作手册、安全规范、培训课件接入知识库，联调 Python 侧 RAG 检索链路；第三，做缓存、会话历史和流式输出这些工程化能力，保证实际使用时延迟和成本可控。

### 6.2 为什么不用 BERT 做意图分类？

BERT 分类器探索过，小规模验证中有一定效果，但没有形成完整评估证据。这个场景是垂类业务助手，前置分类器本身会引入误判。最后选择 Function Calling 做结构化工具调用，没触发工具的问题再走 RAG，链路更简单，也更可观测。

### 6.3 怎么控制 AI 成本？

第一层是 Redis 缓存，命中后直接返回并避免模型调用。未命中时由模型判断是否调用业务 Tool；结构化业务查询无需进入 RAG，但通常仍会产生工具选择和答案组织的模型调用。知识类问题才进入 RAG。这样主要减少无效检索、长上下文和重复生成成本。

### 6.4 怎么减少幻觉？

回答基于两类依据：一类是 Function Calling 查到的实时业务数据，一类是 RAG 检索到的内部文档。RAG 回答会附带来源，相关性不足时走兜底，不强行回答。幻觉不能说完全消灭，只能通过检索质量、引用来源、阈值和反馈机制持续降低。

### 6.5 为什么 Java + Python 双语言？

Java 侧负责企业系统集成，包括鉴权、业务函数、配置、缓存、会话、知识库管理入口。Python 侧负责 NLP 流水线，包括 OCR、切分、Embedding、Milvus 混合检索和 Reranker。BGE-M3 和 Reranker 属于 PyTorch 生态，放 Python 更合适；业务权限和数据在 Java 微服务体系里，所以 Java 做入口。

### 6.8 项目中有哪些跨存储一致性问题？

先说判断原则：只有同一份业务事实需要写入两个不能共享本地事务的存储或中间件时，才构成跨存储一致性问题。石化项目中可以从三个场景回答。IoT 设备状态是最贴近本人职责、最适合优先讲的场景；AI 会话和知识库更新用于继续追问。

#### 场景一：设备心跳、持久状态与离线告警

设备消息经过 RabbitMQ 消费后，业务数据或最后活跃时间需要持久化，Redis TTL 心跳用于快速判断在线状态，离线后还会落告警并发送通知。风险包括：

- MySQL 已记录最新上报，但 Redis 心跳刷新失败，设备可能被误判离线。
- Redis 心跳已刷新，但持久化失败，页面短期显示在线，数据库却缺少可追溯记录。
- 心跳恢复和 XXL-Job 离线检测并发，旧的离线任务可能覆盖新的在线事实。
- MQ 重投或定时任务重复扫描，可能重复生成告警、重复通知。

处理原则：

1. MySQL 保存上报记录和最后活跃时间，承担可追溯事实；Redis TTL 只承担实时在线判断。
2. 消费端基于 messageId，或者 `deviceId + reportTime`，做幂等处理。
3. 心跳 Key 过期后不立即认定离线，XXL-Job 再用数据库 `lastActiveTime` 与离线阈值二次确认。
4. 更新离线状态时带旧状态和时间条件；只有条件更新成功，才生成告警。
5. 告警通过业务唯一键防重，通知消费端同样幂等；关键失败记录 deviceId、messageId、traceId 和重试结果。

```sql
UPDATE device
SET status = 'OFFLINE'
WHERE id = #{deviceId}
  AND status = 'ONLINE'
  AND last_active_time < #{offlineThreshold};
```

#### 场景二：AI 会话的 Redis 短期记忆与 MySQL 持久历史

同一条问答既可能进入 Redis / ChatMemory，供下一轮快速读取，也需要进入 MySQL，供历史查询和审计。可能出现 MySQL 成功而 Redis 失败，导致下一轮缺少上下文；也可能 Redis 成功而 MySQL 失败，TTL 到期后历史永久丢失。SSE 中断和用户重试还可能产生重复消息。

生产化处理原则：

1. MySQL 是持久会话的最终事实来源，Redis 只保存生成中的临时内容和最近若干轮上下文。
2. 每次生成使用唯一 requestId / messageId，数据库建立唯一约束或使用幂等更新。
3. 流式过程中可缓存已输出内容；结束、失败或取消时统一更新消息状态并落库。
4. MySQL 保存成功前不删除临时内容；保存失败可以按同一 messageId 重试，成功后再更新或失效 Redis。
5. Redis 丢失时可以从 MySQL 重建最近几轮上下文；定时修复长时间停留在 `generating` 的异常消息。

#### 场景三：知识库元数据、Milvus 向量与 Redis 答案缓存

文档版本和生效状态在 MySQL，文档向量在 Milvus，基于文档生成的答案可能缓存在 Redis。若先启用新版本再完成向量索引，会出现知识空窗；若新版本已经生效但旧答案缓存仍命中，用户会看到过期知识。

生产化处理采用版本切换，而不是同时强行修改三个系统：

```text
v1 active
v2 uploaded -> parsing -> chunking -> embedding -> indexing
v2 indexed 并校验成功
  -> MySQL 切换 activeVersion = v2
  -> 新请求使用带 kbVersion 的缓存 Key
  -> v1 标记 inactive，暂时保留用于回滚
```

检索必须过滤 `enabled + activeVersion`。缓存 Key 绑定 `kbVersion + userScopeHash + questionHash`，切换版本后自然不命中旧答案，旧缓存等待 TTL 过期。新版本失败时继续使用 v1；需要回滚时把 activeVersion 切回 v1，不在同步更新流程中直接物理删除旧向量。

#### 面试标准说法

> 我在项目里重点关注过三类跨存储一致性。最典型的是设备状态：MySQL 保存可追溯的最后活跃时间，Redis TTL 负责实时在线判断，离线任务会根据数据库时间二次确认，并通过条件更新和告警唯一键避免并发覆盖与重复告警。AI 会话以 MySQL 为持久事实，通过 messageId 保证落库幂等，Redis 只保存临时内容和最近上下文。知识库更新则采用版本化索引，新版本 indexed 后才切换 activeVersion，缓存 Key 绑定 kbVersion，从设计上避开 MySQL、Milvus、Redis 必须瞬间同时成功的问题。
