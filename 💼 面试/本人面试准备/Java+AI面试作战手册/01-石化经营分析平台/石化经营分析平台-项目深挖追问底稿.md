## 1. 项目业务全景

### 1.1 这个系统到底解决什么问题？

一句话：

> 把分散在油站、地市、省公司的经营数据和设备状态集中起来，让省公司能看经营、看设备、看异常，也让一线员工能查操作、查业务、查系统问题。

业务背景：

- 油站每天有设备状态、油品销售、便利店非油商品、客户交易、安全巡检、台账填报等数据。
- 过去很多数据靠纸质表单、Excel 或分散系统，汇总慢，口径不统一。
- 经营分析平台的价值是把这些数据沉淀成统一后台，支持填报、查询、汇总、分析和告警。
- 后期引入 AI 助手，是为了解决一线员工“不会操作、找不到制度、问运维成本高”的问题。

### 1.2 省公司、地市、油站分别怎么用？

| 使用方 | 主要关注 | 典型动作 |
|---|---|---|
| 省公司 | 全省经营概览、设备异常、指标趋势、制度执行 | 看大盘、看地市排名、下发任务、查看告警 |
| 地市公司 | 本地市油站经营情况、异常处理、报表汇总 | 审核数据、跟进异常、组织培训 |
| 油站 | 日常填报、设备状态、非油库存、安全巡检 | 填台账、查设备、处理告警、问 AI 操作步骤 |
| 运维人员 | 系统异常、设备告警、接口联调 | 查日志、查告警、定位接口问题 |

### 1.3 数据从哪里来？

| 数据类型 | 来源 | 进入平台方式 | 用途 |
|---|---|---|---|
| 设备状态 | 加油机、液位仪、IoT 网关等 | 省中心汇聚后平台消费 | 在线状态、异常告警、设备分析 |
| 经营填报 | 油站/地市人工录入 | Web 后台表单 | 台账、审核、统计 |
| 客户交易 | 原交易/客户系统或数据同步 | 同步/汇总表 | 客户核算、客户分析 |
| 非油库存 | 便利店进销存链路 | 表单/业务服务 | 库存、调拨、盘点 |
| 知识文档 | 操作手册、安全规范、培训课件 | 文档上传入库 | AI 问答、培训答题 |

### 1.4 面试时的业务闭环讲法

> 比如一个油站日常运营时，设备会持续上报状态，一线人员会填报经营台账和处理非油库存，地市会审核和汇总，省公司看整体经营分析和异常趋势。如果员工不知道某个功能怎么操作，可以问 AI 助手，AI 基于内部操作手册或实时业务工具返回答案。这个平台不是单个 CRUD 系统，而是把设备、经营、报表、知识库和业务问答放在一起。

## 3. 设备 IoT 接入深挖

### 3.1 设备模块业务链路

一句话：

> 设备模块核心不是维护设备档案，而是接住设备上报数据，判断在线离线，发现异常后生成告警。

典型链路：

```text
设备/管控机/省中心汇聚
  -> 平台消费设备上报消息
  -> RabbitMQ 削峰
  -> 消费者落库 / 刷新 Redis 心跳
  -> XXL-Job 定时扫描离线设备
  -> 生成或更新告警
  -> RabbitMQ 通知服务
```

### 3.2 上报消息大概包含什么？

可以按“不要报具体内部字段，只讲通用结构”回答：

```json
{
  "stationCode": "油站编码",
  "deviceId": "设备ID",
  "deviceType": "设备类型",
  "eventType": "heartbeat/status/alarm",
  "status": "online/error",
  "reportTime": "上报时间",
  "payload": {}
}
```

说明：

- `stationCode` 用于组织和油站维度过滤。
- `deviceId` 用于幂等和心跳 key。
- `eventType` 区分心跳、状态、告警。
- `reportTime` 用于乱序判断和延迟分析。
- `payload` 存不同设备类型的扩展字段。

### 3.3 为什么用 RabbitMQ？

第一层：

> 设备上报有波峰波谷，RabbitMQ 用来削峰和异步解耦，避免上报高峰直接打满业务接口。

追问展开：

- 高峰集中上报时，请求先进队列，消费者按能力处理。
- 上报接收和后续落库、告警判断解耦。
- 消费失败可以重试或进死信队列，便于补偿。
- 不需要每条上报都同步阻塞等待数据库写入。

### 3.4 设备消息怎么做幂等？

稳妥回答：

> 设备上报可能重复，所以不能简单收到就插入告警。幂等一般靠业务唯一键，比如 deviceId + eventType + reportTime，或者设备侧 messageId。如果是状态类消息，可以用最后上报时间判断，只处理比当前状态更新的消息；如果是告警类消息，用唯一告警键避免重复生成。

可讲实现：

- 心跳类：刷新 Redis TTL，本身天然幂等。
- 状态类：比较 `reportTime`，旧消息不覆盖新状态。
- 告警类：用 `deviceId + alarmType + alarmStartTime` 或业务流水号做唯一约束。
- 消费端捕获重复键异常，视为已处理成功，避免重复重试。

### 3.5 Redis TTL 心跳怎么做？

回答：

> 每次收到设备上报，就刷新 `device:heartbeat:{deviceId}`，设置一个过期时间。设备持续上报就持续续期；如果超过一段时间没有上报，key 过期，就认为可能离线。XXL-Job 定时扫描设备清单或状态表，把 Redis 中不存在心跳 key 的设备标记为离线。

注意不要说：

- “TTL 一过期就一定离线。”  

更稳说法：

> TTL 过期只能说明超过阈值没有收到心跳，实际还要结合设备类型、网络抖动和最近上报时间做判断。

### 3.6 离线告警怎么避免重复生成？

回答：

> 设备离线不是每次扫描都新建一条告警，而是维护告警状态。如果设备已经有未恢复的离线告警，下一次扫描只更新最后检测时间，不重复插入。等设备恢复上报后，再把告警状态改成已恢复，并记录恢复时间。

可讲字段：

- `device_id`
- `alarm_type`
- `alarm_status`: active / recovered
- `first_alarm_time`
- `last_check_time`
- `recover_time`

### 3.7 Redis 过期后怎么同步数据库状态？

回答：

> Redis 负责快速判断心跳状态，数据库保存最终设备状态和告警记录。XXL-Job 扫描时发现 Redis 心跳不存在，会更新设备状态表为离线，并生成或更新告警。设备再次上报时，消费者刷新 Redis，同时把数据库状态改回在线，并恢复离线告警。

### 3.8 消费失败怎么办？

回答：

> 消费失败不能丢。一般会手动 ack，处理成功才 ack；处理失败按错误类型重试。短暂数据库异常可以重试，多次失败进入死信队列或失败表，后续人工或定时补偿。关键是消息要可追踪，有 messageId、失败原因和重试次数。

### 3.9 设备模块最容易被问的坑

| 问题 | 稳妥回答 |
|---|---|
| RabbitMQ 会不会丢消息？ | 生产者确认、消息持久化、消费者手动 ack、失败重试和死信队列。 |
| Redis 挂了怎么办？ | Redis 是在线状态加速层，数据库仍保存最后状态；Redis 异常时降级查数据库最近上报时间。 |
| 乱序消息怎么办？ | 根据 reportTime 或版本号判断，新状态不被旧消息覆盖。 |
| 告警重复怎么办？ | 未恢复告警只更新，不重复插入。 |
| 为什么不用数据库轮询？ | 全量轮询成本高，Redis TTL 更适合短生命周期状态。 |

### 4.1 AI 助手真实解决什么问题？

三类场景：

- 业务操作顾问：员工不知道系统功能怎么操作，问 AI 获取步骤。
- 系统运维顾问：报错、异常、设备问题，问 AI 获取排查建议。
- 智能答题助手：基于培训课件和安全规范生成题目、答案、解析。

面试收口：

> AI 助手不是替代业务流程，而是降低查询文档、咨询运维和培训答题的成本。

### 4.2 当前主链路

```text
用户问题
  -> Java AI 入口
  -> 鉴权和用户上下文
  -> Redis 查缓存
  -> Spring AI 判断是否触发工具调用
  -> 触发工具：调用 Java 业务函数，返回实时数据
  -> 未触发工具：调用 Python RAG 服务
  -> SSE 流式输出
  -> 保存会话历史和日志
  -> 高频结果写缓存
```

### 4.3 我封装过哪些 Tool？

不要说太多，准备 3 个就够：

| Tool | 用途 | 参数 | 权限点 |
|---|---|---|---|
| `getDeviceStatus` | 查询油站设备在线/离线和异常状态 | `stationCode` / `deviceId` | 用户是否有该油站权限 |
| `queryAlarmList` | 查询设备或系统告警 | `stationCode`, `alarmType`, `timeRange` | 组织范围、角色 |
| `queryCustomerSummary` | 查询客户核算概览 | `customerId` / `customerName` | 客户数据权限、脱敏 |

示例讲法：

> 比如用户问“查一下 430101 油站设备状态”，模型会根据工具描述生成 `stationCode=430101`，后端真正执行 `getDeviceStatus`。但模型传来的参数不能直接信，要先校验油站编码格式，再校验当前用户有没有访问这个油站的权限。

### 4.4 Tool 参数怎么校验？

回答：

> 参数校验放在后端，不靠 Prompt。先做格式校验，比如 stationCode、deviceId、时间范围；再做业务校验，比如油站是否存在；最后做权限校验，比如当前用户是否属于对应地市或油站范围。

常见校验：

- 必填字段不能为空。
- 油站编码、设备 ID 格式合法。
- 时间范围不能过大，避免查爆数据库。
- 当前用户组织范围必须覆盖查询对象。
- 返回结果要按权限过滤和必要脱敏。

> 如果被追问模型选错工具、重复调用、Tool schema 怎么设计，看 [[石化AI助手-生产化追问补充#1.6.3 Function Calling 工具调用怎么避免乱调？]]。

### 4.5 用户权限怎么传给 Tool？

回答：

> Java 入口从登录态拿到 userId、角色、组织范围和油站范围，放到请求上下文里。Tool 执行时不是只看模型传参，而是用当前用户上下文拼接查询条件。模型不能指定“我要查全省”，除非用户本身有全省权限。

标准句：

> AI 只负责理解问题和选择工具，权限边界永远在后端。

### 4.6 RAG 权限过滤怎么做？

回答：

> 文档入库时带权限元数据，比如组织、角色、文档分类、生效状态。用户提问时，Java 侧根据 userId 算出可访问范围，调用 Python RAG 时传入 source_filter 或权限条件。检索阶段只召回有权限的文档，不能等模型生成后再过滤。

继续深挖：

> 如果被追问权限元数据怎么入库、filter 怎么传、权限变化后缓存和向量怎么失效，看 [[石化AI助手-生产化追问补充#1.6.1 RAG 权限过滤怎么真正落地？]]。

### 4.7 Redis 缓存 key 怎么设计？

不要讲死实现，可以讲原则：

> 缓存 key 要包含问题归一化结果、用户权限范围、业务场景和知识库版本，避免不同权限用户共享到不该看的答案。

示例：

```text
ai:answer:{scene}:{userScopeHash}:{kbVersion}:{questionHash}
ai:tool:{toolName}:{userScopeHash}:{paramHash}
```

注意点：

- 不能只用问题 MD5，否则不同权限用户可能命中同一答案。
- 文档更新后要通过 `kbVersion` 或主动删除让旧缓存失效。
- 实时业务数据 TTL 要短，制度类问答 TTL 可以相对长。

继续深挖：

> 如果被追问用户权限变了、文档更新了、实时数据变化了旧答案怎么办，看 [[石化AI助手-生产化追问补充#1.6.4 缓存失效和缓存污染怎么处理？]]。

### 4.8 SSE 流式输出中断怎么办？

回答：

> SSE 是长连接，用户关闭页面、网络中断、模型异常都可能导致中断。服务端要捕获中断事件，记录已生成内容和错误状态。对会话历史来说，最好不要只依赖框架自动保存，而是在流式过程中缓存已输出文本，结束或取消时统一落库。

可以说：

- 前端收到 token 逐步渲染。
- 服务端记录 traceId，方便排查。
- 中断时保存状态为 interrupted / failed。
- 用户重试时可以基于上一轮问题重新生成。

继续深挖：

> 如果被追问前端断开怎么感知、模型任务是否取消、已输出 token 怎么落库、重试怎么避免重复消息，看 [[石化AI助手-生产化追问补充#1.6.2 SSE 流式中断怎么处理？]]。

### 4.9 会话历史表怎么设计？

不要背完整 DDL，讲核心字段：

| 表 | 作用 | 关键字段 |
|---|---|---|
| `conversation` | 会话元信息 | conversationId、userId、title、createdAt、updatedAt |
| `message` | 消息明细 | conversationId、role、content、source、createdAt |
| `answer_source` | 引用来源 | messageId、docId、chunkId、score、sourceName |

回答：

> 会话必须按 userId + conversationId 隔离，避免串会话。生成回答时只取最近几轮或摘要，不能无限拼历史，否则 token 成本和干扰都会上升。

### 4.10 AI 模块异常怎么兜底？

| 异常 | 处理 |
|---|---|
| 模型超时 | 有限重试，失败后兜底提示 |
| Embedding 失败 | 入库阶段进重试队列；查询阶段降级关键词/FAQ 或兜底 |
| RAG 无结果 | 返回未找到依据，并记录问题 |
| Tool 参数错误 | 返回参数不足或格式错误，引导用户补充 |
| 权限不足 | 明确提示无权限，不暴露数据 |
| SSE 中断 | 保存中断状态，允许重试 |

### 4.11 回答反馈入口怎么设计？（设计标准形态，待代码核实）

> 状态：**设计标准形态（2026-07-22 从第一性原理推导），项目代码尚未核实**。核实前口径：“反馈入口的方案是我设计的”；核实后按代码事实修正本节和台账。

第一性原理推导（为什么是“直写库”而不是“高并发架构”）：

1. **最小业务事实**：一个用户对一条 AI 回答判定有用/没用，最小单元是“用户 × messageId”；
2. **真实负载**：内部平台，AI 调用量有限，反馈量远小于对话量（多数回答不会被评价），写 QPS 峰值个位数——这个量级下 MySQL 直写就是正确解，Redis 异步计数是过度设计；
3. **分析诉求**：反馈必须和 `message`、`answer_source`、知识库版本 JOIN 才能指导 RAG 优化，放同一个 MySQL 最直接；
4. **结论**：现状标准形态 = 简单直写库 + 唯一键幂等；高并发方案（Redis Set + Lua + 异步计数）只作演进储备，见 [[石化AI反馈通道-点赞模式平移面试三层回答]]。

表设计（对齐 4.9 的会话表族）：

| 表 | 作用 | 关键字段 |
|---|---|---|
| `answer_feedback` | 回答反馈 | messageId、userId、feedback（1 有用 / -1 没用）、reasonTag（答案错误/没有依据/答非所问/其他，选填）、reasonText（选填）、createTime、updateTime；唯一键 `uk_user_message(user_id, message_id)` |

接口：`POST /ai/feedback {messageId, feedback, reasonTag?, reasonText?}`，upsert 语义；`feedback=0` 或 DELETE 取消。

五条设计约束（每条都要能讲为什么）：

1. **一人一判**：唯一键 + upsert，改判更新同一行，双击/重试不产生重复数据——幂等靠 DB 唯一键裁决，这个量级不需要 Redis 拦截层；
2. **只能反馈已完成回答**：message 必须 role=assistant 且状态 finished（4.8 的中断/失败态不接受反馈），否则“有用”的语义不成立；
3. **只能反馈自己的会话**：按 conversation.userId 校验归属，和平台权限主线一致；
4. **负反馈带结构化原因**：reasonTag 选填但强推——纯 polarity 是弱信号，带原因才能定位是检索、文档还是生成问题，成本几乎为零；
5. **反馈失败不影响主链路**：独立接口、回答完成后才发起，天然解耦；失败前端提示重试即可，不做重保证。

回答：

> 反馈入口我做了方案设计：用户 × messageId 一人一判，MySQL 直写加唯一键 upsert，改判取消都是同一行。这个量级不需要上 Redis 异步计数——反馈量远小于对话量，而且反馈要和引用来源表 JOIN 做 RAG 优化分析，放同一个库最直接。负反馈设计了结构化原因标签，因为单纯一个“踩”没法指导优化。如果未来反馈要实时展示计数或量级上来，演进方案是按高并发点赞的标准做法做异步计数，方案我有储备。

继续深挖：

> 高并发演进方案（Redis Set + Lua 原子 toggle + 版本号绝对计数）见 [[石化AI反馈通道-点赞模式平移面试三层回答]]；反馈如何反哺 RAG 优化见该文档第五节。

## 5. RAG 工程追问底稿

### 5.1 召回率怎么测？

> 用固定测试集，看正确文档是否出现在 Top-K 结果里。比如每个问题人工标注标准文档或标准段落，再统计 Top-1、Top-3、Top-5 命中率。RAG 不能只看最终答案像不像，必须先看检索有没有把正确依据找回来。

### 5.2 测试集怎么构造？

> 测试集要来自真实业务问题，覆盖操作步骤、故障排查、制度问答、无答案问题、权限受限问题和多轮追问。每个问题配标准答案或标准文档。否则评估会偏，比如只测 FAQ，无法反映复杂文档检索能力。

### 5.3 Reranker 提升怎么验证？

> 固定问题集，对比加 Reranker 前后的 Top-K 命中、正确文档排名和人工相关性评分。同时看延迟增加是否可接受。不能只说“感觉更准”，也不要主动讲百分比，除非能说明测试集和统计口径。

继续深挖：

> 如果被追问怎么判断是召回问题、重排问题还是模型生成问题，看 [[石化AI助手-生产化追问补充#1.6.6 RAG 评估怎么定位问题在哪一层？]]。

### 5.4 chunk size 为什么这么设？

> chunk size 不是固定模板。判断标准是：Top-K 能不能命中正确段落、返回给模型的上下文是否完整、是否超出上下文预算、是否产生大量重复。父子块的思路是子块负责精确匹配，父块负责补上下文。

### 5.5 Milvus 参数怎么调？

> 参数调优主要围绕召回率和延迟平衡。比如 IVF_FLAT 里 nlist 影响聚类粒度，nprobe 影响查询时扫描多少簇。nprobe 调大召回更高但延迟更高，调小更快但可能漏召回。实际要用评估集看 Top-K 命中和接口耗时。

### 5.6 Embedding 失败怎么补偿？

> 入库阶段 embedding 失败要记录 docId、chunkId、错误原因和重试次数，进入重试队列；多次失败标记 failed，后台人工处理。查询阶段 embedding 失败可以降级到 FAQ/BM25/关键词检索，或者直接兜底提示。核心是失败可观测、可重试，不要静默丢文档。

### 5.7 文档更新后旧向量怎么处理？

> 文档要有 docId、version、生效状态和生效日期。新版本上传后重新切分、重新向量化、写入新版本向量；旧版本标记 inactive 或软删除，检索时只查生效版本。相关答案缓存也要清理，否则会回答旧知识。

继续深挖：

> 如果被追问新版本未 indexed 前旧版本能不能用、旧向量软删还是物理删、知识库怎么回滚，看 [[石化AI助手-生产化追问补充#1.6.5 文档更新后旧向量怎么处理更稳？]]。
