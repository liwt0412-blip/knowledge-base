# 面试追问准备作战手册

> 目标：把知识库里的追问材料从“资料仓库”收敛成“面试作战手册”。  
> 使用方式：先背 P0，再练 P1，P2 只用于查漏补缺。

---

## 0. 统一口径

所有回答都要围绕这条主线：

> 我是 Java 后端工程师，做过复杂企业业务系统。最近参与 Java + AI 应用落地，负责把集团 AI 底座接进省公司经营分析平台，处理业务工具封装、知识库接入、RAG 检索链路联调、缓存、会话、流式输出和稳定性问题。

不要说：

- AI 模块全是我从零搭起来的
- 100% 确定
- 零误判
- 彻底解决幻觉
- AI 让开发效率提升 50%
- 所有技术栈都是我一个人负责

建议说：

- 集团提供 AI 底座和规范，我负责本地化落地和关键链路实现
- 工具调用结果结构化、可观测
- RAG 能降低幻觉，但不能彻底消灭
- 明显减少重复性工作
- 我直接负责 AI 助手和设备 IoT，其他模块了解架构用于联调排障

## 1. P0 必背 20 题

P0 是面试前必须背熟的题。每题控制在 30-60 秒，先讲项目落点，再讲技术原理。

### 1. 自我介绍怎么说？

**短答：**

我叫李文韬，主要做 Java 后端开发。最近参与的是某能源央企省公司的经营分析平台，技术栈是 Spring Boot + Spring Cloud Alibaba，项目后期引入了 AI 智能助手。我直接负责 AI 助手本地化落地和设备 IoT 接入，重点是把大模型能力接进已有 Java 业务系统。

**项目落点：**

AI 侧负责业务工具封装、知识库接入、RAG 联调、缓存、会话、SSE 流式输出；设备侧负责 RabbitMQ 削峰、Redis TTL 心跳、XXL-Job 离线告警。

---

### 2. 这个项目整体是做什么的？

**短答：**

这是省公司经营分析平台，覆盖 14 个地市、近 2000 座加油站，处理设备管理、非油商品、客户核算、经营报表和 AI 智能问答。

**项目落点：**

我直接负责 AI 智能助手和设备 IoT 接入。非油、客户核算、网关通知由团队其他成员负责，我了解整体链路用于联调和排障。

---

### 3. 你在项目里到底负责什么？

**短答：**

我直接负责两个方向：AI 智能助手省公司本地化落地，以及设备 IoT 接入和状态监控。

**项目落点：**

AI 不是我一个人从零造完整底座。集团侧提供 AI 底座和规范，我负责省公司侧接入：业务工具封装、知识库接入、RAG 链路联调、缓存、会话历史和流式输出。

---

### 4. AI 模块主链路怎么走？

**短答：**

用户问题先查 Redis 缓存；未命中进入 Spring AI；如果触发 Function Calling，就调用 Java 业务函数查实时数据；如果没触发工具调用，就转发 Python RAG 服务检索知识库；最后 SSE 流式返回。

**项目落点：**

当前生产主链路是 `Redis -> Function Calling -> Python RAG`。BERT 分类器是探索方案，不是当前主链路。

---

### 5. 为什么用 RAG，不用微调？

**短答：**

业务文档经常更新，RAG 更新知识库就能生效，微调成本高、周期长，而且不方便追溯答案来源。

**项目落点：**

石化项目里操作手册、安全规范、培训资料会更新。RAG 能附带引用来源，知道回答来自哪份文档。

---

### 6. RAG 完整流程是什么？

**短答：**

文档解析和 OCR -> 切分 -> embedding -> 写入 Milvus。用户提问时向量化 -> 混合检索 -> rerank -> 拼 Prompt -> LLM 生成 -> 返回引用来源。

**项目落点：**

Python 侧负责文档处理、BGE-M3、Milvus、Reranker；Java 侧负责业务入口、权限、缓存和会话。

---

### 7. 怎么降低幻觉？

**短答：**

不能说完全防止，只能降低。我们让回答基于内部文档或 Function Calling 查到的业务数据，附带引用来源，低相关性时兜底，不让模型硬答。

**项目落点：**

RAG 回答来自知识库文档；实时数据来自 Java 业务函数。模型不能绕过后端权限和数据校验。

---

### 8. 怎么控制 AI 成本？

**短答：**

先缓存，再工具调用，最后才 RAG 生成。高频问题走 Redis，能查业务数据的走 Function Calling，RAG 结果也可以短期缓存。

**项目落点：**

不是所有问题都调大模型。缓存和 Function Calling 能拦截大量重复或结构化查询。

---

### 9. 为什么 Java + Python 双语言？

**短答：**

Java 负责企业系统集成，Python 负责 NLP/RAG 流水线。两边各做自己擅长的事。

**项目落点：**

Java 做鉴权、业务函数、缓存、会话和知识库管理入口；Python 做 OCR、切分、Embedding、Milvus 混合检索和 Reranker。

---

### 10. Function Calling 是什么？

**短答：**

模型不直接操作数据库，而是根据工具描述生成工具名和参数，后端真正执行工具函数并做权限和参数校验。

**项目落点：**

设备查询、告警查询、客户查询等业务函数通过 Spring AI `@Tool` 暴露给模型。模型只“点菜”，后端才“做菜”。

---

### 11. 为什么不用 BERT 分类器做主链路？

**短答：**

BERT 探索过，但没有放进主链路。垂类业务助手里大多数问题本来就是业务问题，前置分类器会引入额外误判。

**项目落点：**

我们选择 Function Calling 做结构化工具调用，没触发工具调用的问题再走 RAG。这条链路更短、更可观测。

---

### 12. chunk 和父子块怎么讲？

**短答：**

子块负责精确检索，父块负责保证上下文完整。

**项目落点：**

业务手册切太大检索不准，切太小上下文不完整。命中子块后返回父块给模型生成。

---

### 13. 为什么要 Reranker？

**短答：**

向量检索是粗召回，Reranker 做精排，把最相关的文档排到前面。

**项目落点：**

Milvus 先召回候选片段，BGE Reranker 再对 query-doc 成对打分，最后只把最相关的内容喂给模型。

---

### 14. 权限怎么控制？

**短答：**

权限必须在 Java 后端控制，不能靠 Prompt。

**项目落点：**

Function Calling 查业务数据时复用原系统用户权限；RAG 文档入库时带权限元数据，检索时按用户组织、角色、油站范围过滤。

---

### 15. 会话记忆怎么隔离？

**短答：**

用 userId + conversationId 隔离，每个用户、每个窗口单独存上下文。

**项目落点：**

Redis 存短期记忆，MySQL conversations 表存持久历史。Spring AI 的 ChatMemory 必须显式传 conversationId，避免串会话。

---

### 16. 设备 IoT 模块难点是什么？

**短答：**

设备模块核心不是 CRUD，而是数据上报、异步处理和在线状态判断。

**项目落点：**

高峰 500-1000 条/s 用 RabbitMQ 削峰；设备在线状态用 Redis TTL；XXL-Job 定时扫描离线设备并触发告警。

---

### 17. 为什么 XXL-Job 不用 `@Scheduled`？

**短答：**

`@Scheduled` 多实例部署会重复执行，XXL-Job 支持分布式调度、可视化管理、失败重试和日志。

**项目落点：**

设备离线检测、保质期预警、报表任务都适合由 XXL-Job 管理。Cron 修改不用重新发版。

---

### 18. Gateway 怎么做路由和鉴权？

**短答：**

Gateway 根据 Path Predicate 匹配路由，通过 `lb://service-name` 从 Nacos 选择实例转发。GlobalFilter 做 Token 校验，失败直接返回 401，成功后把 userId 透传给下游。

**项目落点：**

鉴权失败在网关层拦截，不让请求穿透到业务服务。

---

### 19. Redis 缓存三连击怎么答？

**短答：**

穿透是查不存在数据，防法是布隆过滤器和空值缓存；击穿是热点 key 过期，防法是互斥锁或逻辑过期；雪崩是大量 key 同时失效，防法是 TTL 随机化、多级缓存和限流。

**项目落点：**

商品、油站基础数据适合缓存；设备心跳用 TTL，但要避免大量 key 同时过期带来抖动。

---

### 20. 离职原因怎么说？

**短答：**

项目交付后进入运维期，实施团队合约到期。我个人还是想继续做开发，尤其是 Java + AI 应用落地方向，所以没有续约。

**项目落点：**

不要抱怨甲方或公司。强调项目阶段变化和个人技术方向。

## 2. P1 追问展开 30 题

P1 用于面试官继续深挖。每题先用一句话接住，再补项目落点。

### AI 方向

1. **检索不到怎么办？**  
   低相关性直接兜底，不让模型硬编；记录问题，后续补知识库。

2. **RAG 效果怎么评估？**  
   看 Top-K 命中率、人工相关性评分、兜底率、用户反馈和缓存命中率。

3. **为什么稠密 + 稀疏混合检索？**  
   稠密抓语义，稀疏抓关键词、编号和专业术语。

4. **Milvus 为什么不用 ES 或 pgvector？**  
   Milvus 更适合大规模向量检索和混合检索；ES 偏关键词；pgvector 适合轻量场景。

5. **AI 接口慢怎么办？**  
   缓存、SSE、超时、降级、控制 Top-K 和上下文长度。

6. **大模型 API 失败怎么办？**  
   超时、有限重试、降级兜底、记录日志，不影响主业务。

7. **工具调用会不会乱调？**  
   会有风险，所以工具描述要清晰，后端必须做参数校验和权限校验。

8. **不同用户会不会看到不该看的文档？**  
   文档入库带权限元数据，检索阶段按用户权限过滤。

9. **Prompt 泄露怎么办？**  
   Prompt 不是安全边界，不放密钥和敏感规则，安全控制放后端。

10. **Prompt injection 怎么办？**  
    用户输入和文档内容都不可信，工具执行必须经过后端权限和参数校验。

11. **BGE-M3 为什么适合？**  
    支持中文和多语言，能做稠密和稀疏表示，适合企业知识库混合检索。

12. **Embedding 维度怎么处理？**  
    维度跟模型版本一致，入库和查询必须用同一模型和同一维度。

13. **文档更新怎么处理？**  
    新版本重新切分向量化，旧版本标记失效，缓存同步清理。

14. **脏文档怎么办？**  
    入库前清洗、去重、OCR 抽检；知识库质量决定 RAG 上限。

15. **为什么不用 Spring AI 全做？**  
    Spring AI 适合 Java 业务集成，复杂 OCR、embedding、rerank、混合检索在 Python 生态更成熟。

16. **为什么不用 LangChain4j？**  
    可以用，但项目当时 Spring AI 和 Spring 体系集成更自然，复杂 NLP 已放 Python。

17. **MCP 了解吗？项目用了没？**  
    了解。项目当前主要用 Function Calling，没有正式 MCP 化。后续可把只读查询类 API 封装成 MCP Server。

18. **AI 回答出错谁负责？**  
    AI 是辅助建议，关键业务仍由人确认，系统保留日志、来源和反馈。

19. **最大 AI 难点是什么？**  
    不是接 API，而是把 AI 稳定接进企业系统，处理权限、缓存、会话、成本、延迟和兜底。

20. **怎么防止旧知识回答？**  
    文档版本管理、生效状态过滤、旧向量失效、相关缓存清理。

### Java / 微服务方向

21. **MQ 消息重复消费怎么办？**  
    消费端做幂等，使用业务唯一键、状态表或去重表，确保重复消息不重复写业务结果。

22. **RabbitMQ 如何保证消息可靠？**  
    生产者 confirm、交换机/队列持久化、消息持久化、消费者手动 ack、失败重试和死信队列。

23. **Spring 事务什么时候失效？**  
    同类内部方法调用、非 public 方法、异常被吞、数据库不支持事务、没有被 Spring 代理管理。

24. **乐观锁怎么防止库存超扣？**  
    更新时带 version 条件，`where id=? and version=?`，更新成功才说明抢到库存。

25. **Redis 分布式锁要注意什么？**  
    加唯一 value，释放时校验 value，设置 TTL，避免误删别人的锁。复杂场景可用 Redisson 看门狗。

26. **MySQL 慢查询怎么排查？**  
    看慢 SQL 日志、执行计划、索引命中、扫描行数、回表、排序和临时表。

27. **线程池参数怎么讲？**  
    核心线程数、最大线程数、队列长度、存活时间、拒绝策略。IO 密集型可以适当多线程，CPU 密集型接近 CPU 核心数。

28. **Sentinel 熔断怎么讲？**  
    关闭、打开、半开三个状态；慢调用或错误比例达阈值后打开，走降级；冷却后半开试探。

29. **Nacos AP 还是 CP？**  
    注册中心偏 AP，保证服务发现可用；配置中心更偏一致性。回答时不要绝对化，结合模式和场景说。

30. **Seata 在项目里怎么用？**  
    如果不是你直接负责，不要硬讲细节。可以说平台基线包含 Seata，自己主要接触本地事务和模块内一致性，跨服务事务由团队按集团规范处理。

## 6. 红线提醒

这些话不要说：

- “这个 AI 模块都是我从零做的。”
- “Function Calling 100% 不会错。”
- “RAG 可以彻底解决幻觉。”
- “我用 AI 提效 50%。”
- “BGE-M3 / Milvus / Reranker 全部参数我都调过。”
- “Seata 跨服务事务我很熟。”如果不是你直接负责，不要这么说。

更稳的说法：

- “集团提供底座，我负责省公司本地化落地和关键链路实现。”
- “工具调用结果结构化、可观测，但仍要后端校验。”
- “RAG 能降低幻觉，但不能完全消灭。”
- “AI 明显减少重复工作，但所有代码都要 review。”
- “参数调优主要围绕检索效果、延迟和成本做平衡。”
