## 二、项目概述（1-2 分钟）

这个项目是某能源央企省公司的经营分析平台，覆盖 14 个地市、近 2000 座加油站。业务包括设备管理、非油商品进销存、客户核算、经营报表、安全巡检和 AI 智能问答。

平台 2023 年 2 月上线，后面持续迭代。早期重点是核心业务跑稳，中期做报表和经营分析，最近一年引入 AI 助手，主要用来降低一线员工培训和咨询成本。

我所在的是项目实施团队，驻场在省中心机房，和甲方信息部一起协作。项目是 Spring Cloud Alibaba 微服务架构，服务之间通过 Nacos 注册发现，OpenFeign 同步调用，RabbitMQ 异步解耦，Gateway 统一入口。

我直接负责两个方向：一个是 AI 智能助手在省公司的本地化落地和关键链路实现，另一个是设备 IoT 接入和状态监控。非油、客户核算、网关通知这些模块由团队其他成员负责，我了解整体链路，主要用于联调和排障。

AI 这块不是我一个人从零造完整底座。集团侧有统一 AI 底座和规范，我做的是省公司经营分析平台的本地化落地：把 Java 业务查询封装成工具函数，把省公司知识库接入 RAG 检索链路，再补齐缓存、会话、流式输出和业务系统对接。

## 三、AI 模块展开（面试官追问时讲）

### 3.1 AI 助手解决什么问题？

AI 助手主要有三个场景。

第一个是业务操作顾问。一线员工不知道系统功能怎么操作时，可以直接问 AI，AI 基于内部操作手册给步骤指引。

第二个是系统运维顾问。运维人员输入错误信息或异常现象，AI 基于故障案例和运维文档给排查建议。

第三个是智能答题助手。省公司做安全规范培训时，AI 可以根据内部文档生成题目、答案和解析。

### 3.2 当前主链路怎么走？

当前主链路是：

用户问题先进入 Java AI 助手入口，先查 Redis 缓存。缓存命中就直接返回，未命中就进入 Spring AI。

如果模型触发 Function Calling，就调用 Java 侧业务函数，比如查设备状态、查告警、查客户信息，拿到真实业务数据后再回答。

如果没有触发工具调用，就转发到 Python RAG 服务，检索知识库文档，再把相关片段交给大模型生成回答。生成过程用 SSE 流式输出，高频回答会回写 Redis 缓存。

这里没有把 BERT 分类器放在生产主链路里。工具调用是否发生、参数是什么、返回结果是什么，都是结构化可记录的，比让模型输出文字再解析更可观测。

### 3.3 为什么不用 BERT 做意图分类？

这个方案我们探索过。BERT 二分类准确率能做到 90% 以上，但最后没有放进主链路。

原因是这个场景是垂类业务助手，用户大多数问题本来就是业务问题。如果前面加一个 90% 准确率的分类器，反而可能把专业问题误判出去。最后我们选择用 Function Calling 做结构化工具调用，没触发工具调用的问题再走 RAG，这条链路更简单，也更容易观测。

所以我的口径是：BERT 是探索阶段的技术储备，不是当前生产主链路。

### 3.4 为什么 Java + Python 双语言？

Java 负责企业系统集成，比如鉴权、业务函数、配置、缓存、会话和知识库管理入口。Python 负责 NLP 流水线，比如 OCR、文档切分、Embedding、Milvus 混合检索和 Reranker。

BGE-M3 和 BGE Reranker 都是 PyTorch 生态，放在 Python 侧更合适；业务权限和数据在 Java 微服务体系里，所以 Java 侧做入口和业务集成。

### 3.5 RAG 怎么做？

文档上传后，先做文本抽取和 OCR，再按父子块切分。子块用于精确检索，父块用于给大模型提供完整上下文。

检索时用 BGE-M3 同时做稠密向量和稀疏向量。稠密检索负责语义相近，稀疏检索负责关键词命中。两路结果融合后，再用 Reranker 重排序，最后把最相关的文档片段注入 Prompt，让模型基于内部文档回答。

*如果面试官追问 chunk 大小：*

chunk 大小不是固定套模板，而是根据文档结构和离线效果调整。原则是检索粒度要小，生成上下文要完整。

### 3.6 怎么减少幻觉？

我不会说 RAG 能彻底消灭幻觉，只能说能显著降低。

我们主要做了几层控制：第一，回答基于内部文档或 Function Calling 查到的真实业务数据；第二，RAG 回答附带引用来源；第三，相关性不足时走兜底，不强行回答；第四，用户反馈“有用/没用”，后续用来优化知识库和检索参数。

### 3.7 怎么控制成本？

AI 成本控制主要靠三层。

第一层是 Redis 缓存，高频重复问题直接返回。第二层是 Function Calling，能查业务数据的问题直接查数据库，不走 RAG。第三层才是 RAG 检索和大模型生成。RAG 生成后的高频回答也会回写缓存，设置 TTL。

所以不是所有问题都调大模型，而是先缓存，再工具调用，最后才 RAG。

### 3.8 会话记忆怎么做？

会话记忆分两层。

单次对话内的短期记忆用 Redis 和 Spring AI 的 ChatMemory。跨服务、跨会话的历史记录放在 MySQL conversations 表里，Java 侧 Function Calling 和 Python 侧 RAG 都读写这张表，生成前取最近几轮历史。

这里有个坑：Spring AI 如果不显式传 conversationId，不同用户、不同窗口的记忆可能串到一起。所以入口处必须为每个会话生成唯一 conversationId，并传给 ChatMemory。

## 四、设备模块展开

### 4.1 设备数据流怎么走？

设备链路大致是：加油机通过 MQTT 上报到油站管控机，再到市公司前置机房，最后汇聚到省中心。平台侧主要消费省中心汇聚后的数据。

省中心的数据进入平台后，通过 RabbitMQ 削峰，部分写入 MySQL 做业务查询，部分进入 ClickHouse 做经营分析，设备在线状态则用 Redis TTL 维护。

### 4.2 设备模块技术难点？

设备模块的核心不是 CRUD，而是数据上报、异步处理和状态判断。

早晚高峰时设备会上报得比较集中，高峰大概 500 到 1000 条每秒，所以用 RabbitMQ 做削峰缓冲。

设备在线状态用 Redis TTL 来做。每次收到设备上报，就刷新 `device:heartbeat:{deviceId}`。设备一直在线就持续续期，断开后 TTL 自动过期。XXL-Job 每隔一段时间扫描离线设备，生成告警，再通过 RabbitMQ 推送给通知服务。

相比轮询数据库，Redis TTL 更适合设备心跳这种短生命周期状态。

## 五、常见追问预备

### 5.1 你在团队里是什么角色？

后端三人。我直接负责 AI 智能助手本地化落地和设备 IoT 接入。其余两位分别负责账册表单、客户核算、清洁打卡、网关和通知。我对其他模块了解整体链路，主要用于联调和排障。

### 5.2 离职原因？

项目交付完成后进入运维期，实施团队合约到期。我个人还是想继续做开发，尤其想往 Java + AI 应用落地方向深入，所以没有续约，开始准备新的机会。

### 5.3 非油模块有什么技术难点？

非油模块我不是主负责，但了解关键链路。调拨审批涉及跨油站库存变更，必须保证扣 A 站库存和加 B 站库存在同一个事务里。并发调拨用 version 乐观锁防止超扣。盘点期间用 Redis 分布式锁隔离销售出库，不用数据库行锁，因为盘点可能持续几十分钟，长事务持有数据库锁风险太高。

### 5.4 客户核算有什么技术难点？

客户核算主要问题是交易量大，不能每次查报表都实时聚合。做法是定时任务按数仓分层预聚合到应用层汇总表，查询直接走汇总表。详情页再查当日实时流水，补足时效性。

### 5.5 Spring AI 在项目里起什么作用？

Spring AI 不是简单调 API。它主要做三件事：第一，ChatClient 封装模型调用；第二，`@Tool` 把 Java 业务函数暴露给模型做 Function Calling；第三，VectorStore 抽象用于知识库管理入口。真正复杂的文档处理和混合检索放在 Python 侧。

可以直接说：

这个项目不是我一个人从零做完全部系统。我的直接职责是 AI 助手本地化落地和设备 IoT 接入。集团底座、基础设施、非油和核算模块都有团队分工。我能讲其他模块，是因为联调和排障需要理解整体链路，但不会把不是我负责的部分说成我写的。
