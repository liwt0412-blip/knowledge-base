# 石化AI反馈通道-点赞模式平移面试三层回答

> 上级入口：[[../00-Java+AI面试作战MOC|Java+AI面试作战MOC]]
> 关联：[[重构版-李文韬-石化经营分析平台-面试表述]] 4.6 节（幻觉控制中的用户反馈）；方案来源为天机学堂 tj-remark 点赞服务实现（仓库代码与 `docs/specs/tj-remark-like-service.md` 为准）。

## 一、这份文档的定位与事实边界（先读）

AI 助手回答页有"有用/没用"反馈，是面试表述 4.6 里已有的叙事（反馈用于优化检索参数和知识库质量）。但注意两条事实边界：

1. **反馈入口在深挖底稿中已有设计标准形态（见 [[石化经营分析平台-项目深挖追问底稿]] 4.11，2026-07-22 从第一性原理推导），但项目代码尚未核实**。核实之前，面试口径是：**"反馈入口的方案是我设计的"**，不说"反馈系统是我上线运行的"；核实后按代码事实同步修正本文与底稿 4.11；
2. 本文的高并发统计方案来自天机学堂 tj-remark 点赞服务的真实实现（学习项目，补强材料），**平移到石化 AI 场景是设计预案**。被问"这套方案在哪验证过"时如实说："在学习项目的点赞服务里完整实现过，石化 AI 反馈通道按同一套思路设计"。

这个设计的面试价值在于：它落在**我直接负责的 AI 模块**内，能把"高并发设计能力"和"AI 工程落地"两条线拧在一起讲，而且反馈数据天然闭环到 RAG 优化——这是大多数只讲接 API 的候选人讲不出来的。

## 二、反馈通道整体怎么设计的？

### 第一层：基础回答

每条 AI 回答有唯一 messageId，用户可点"有用"或"没用"，可取消、可改判。反馈走"Redis 实时读写 + 异步计数 + 定时落库"的链路：用户操作实时写 Redis 保证响应速度，计数变更异步汇总到 MySQL，反馈统计供知识库质量分析使用。

### 第二层：结合面试回答

为什么不用"直接写库"这种最朴素的方案？三个原因：

1. **反馈是高频低价值写**：每条回答都可能带一次写操作，直接写库会让 AI 主链路的延迟被反馈拖住，DB 写入量也随对话量线性涨；
2. **改判/取消是常态**：用户先点"有用"再改"没用"，直接写库会产生反复 UPDATE，而 Redis 集合操作天然覆盖语义；
3. **反馈数据允许最终一致**：它用于离线质量分析，晚几秒汇总没有任何业务影响——这是"展示/统计类数据接受最终一致"的典型场景。

### 第三层：边界和追问

- 反馈入口的**现状以项目代码为准**：如果底稿确认后只是简单落库，口径就是"现状是直写库，并发不高够用；我设计了高并发演进方案，流量上来后可以平滑切换"——"现状 + 演进方向"比硬撑"已上高并发方案"可信得多；
- 反馈和回答是**弱绑定**：反馈写失败不影响回答返回，降级时记录日志异步补偿，绝不让非核心功能拖垮主链路（呼应 AI 助手的稳定性主线）。

## 三、点赞/点踩状态与计数怎么存？

### 第一层：基础回答

三类 Redis 结构：`feedback:user:{userId}` 存用户反馈状态（Set，member 是 messageId，区分有用/踩）；`feedback:count:{messageId}` 存每条回答的聚合计数；另有一个"脏标记"记录哪些 messageId 的计数变了、需要同步。

### 第二层：结合面试回答

核心设计是**一条 Lua 脚本完成整个 toggle**（来自 tj-remark 的实现）：

```text
SISMEMBER 判重（重复点/重复取消直接返回，按幂等成功处理）
→ SADD/SREM 更新用户状态
→ INCR/DECR 更新计数（下限 0，防负数）
→ HINCRBY 版本号 +1
→ 写脏标记 + XADD 写事件流
```

为什么必须一条 Lua？如果拆成多条 Redis 命令，服务在两条命令之间宕机就留下半完成状态——用户状态改了计数没改，或者反过来。Lua 保证"判重 + 改状态 + 改计数 + 记事件"是一个原子动作。

另一个要点是**Redis 数据可自愈**：用户状态 Set 丢了，可以从 MySQL 关系表重建（SETNX 加载锁 + 双检查防并发重建）；计数 key 用 SETNX 初始化，不覆盖并发增减。Redis 在这个方案里是加速器，不是唯一事实。

### 第三层：边界和追问

- **有用/踩是两个互斥状态**：改判 = 取消旧状态 + 写入新状态，同一条 Lua 里完成两个集合的切换和两个计数的一减一加；
- **为什么不直接用 Hash 存所有字段**：key 粒度按 messageId 拆分是为了热点分散和单 key 体积可控；
- **计数下限 0 保护**：历史数据补录或补偿重放时 DECR 不会把计数扣成负数——防御性设计的细节，被问能讲。

## 四、异步计数与落库怎么保证不丢、不乱？

### 第一层：基础回答

计数变更不实时逐条同步，而是**定时聚合脏标记，批量发送"当前绝对计数 + 版本号"**，下游以版本号裁决，旧版本直接丢弃。关系表（谁点了什么）批量同步到 MySQL，以 Redis 当前状态为准。

### 第二层：结合面试回答

三个可靠性设计（均来自 tj-remark 实现）：

1. **事件不丢**：用户操作在 Lua 里同时写一条 Redis Stream 当 Outbox；后台任务以 Stream 消费者组批量读出，**必须在 MQ broker confirm 成功后才 ACK 并删除 Stream 记录**——进程在"Redis 已写、MQ 未发"的窗口宕机，事件还在 Stream 里，重启后继续投；
2. **计数不乱**：发的是**绝对值 + 版本号**而不是增量。增量在乱序、重投下会双加或丢加；绝对值配合"已处理版本号"记录，旧版本消息到达直接丢弃，新计数永远不会被旧计数覆盖；
3. **落库不重复**：同步队列用 `x-single-active-consumer` 保证单消费者串行；批内去重只留每个 user+biz 的最后一条；落库以 Redis 当前状态为准，不盲信历史动作——取消又点赞反复操作后，库里的终态和 Redis 一致。

外加一个 XXL-Job 补偿任务兜底核对，防整个异步链路静默失败。

### 第三层：边界和追问

- **为什么计数和关系走两条链路**：计数是高频读（回答卡片上的数字/质量分），要绝对值+版本号防乱序；关系是低频批量落库，追求终态一致。混在一起会互相拖累；
- **版本号存哪**：下游（消费计数的一方）在 Redis 记录每个 messageId 已处理的版本，加 Redisson 锁防多实例并发判断——多实例收到乱序不同版本时旧版本直接丢弃；
- **这套方案的真实出处**：天机学堂 tj-remark 点赞服务，类名可讲——`LikeServiceImpl`（Lua toggle）、`LikeEventPublisher`（Stream outbox + confirm 后删）、`LikePersistenceListener`（批量落库）、`LikeCompensationJob`（XXL-Job 兜底）。石化侧是按同一思路为 AI 反馈设计，边界见第一节。

## 五、反馈数据怎么反哺 RAG 优化？

### 第一层：基础回答

反馈计数按 messageId 聚合后，关联回答用到的知识库文档和版本，低分回答聚类分析：是检索没召回对的文档，还是文档本身质量差，还是模型组织答案的问题。

### 第二层：结合面试回答

这是反馈通道真正的业务价值，让 AI 优化从"凭感觉调"变成"有数据调"：

1. **反馈关联 kbVersion**：知识库是版本化管理的（见面试表述 6.8 场景三），反馈统计必须带上版本维度，否则新版本上线后旧版本的差评会误导判断；
2. **负反馈触发复查**：踩计数超过阈值的回答进入人工复查队列，定位到具体文档片段，决定是补文档、调切分还是调检索参数；
3. **正反馈反哺缓存**：高频正反馈的问题组合可以进入答案缓存白名单，和 Redis 缓存策略联动。

### 第三层：边界和追问

- 口径：**"反馈数据用于优化"是方向性叙事，具体分析流程以项目实际为准**；不说"我们通过反馈把准确率提升了 X%"（无统计依据，撞红线）；
- 被问"反馈样本偏差"（不满意的用户更倾向点踩）要能接：反馈是相对信号不是绝对指标，看趋势和聚类不看单点，这也是为什么不拿它直接当模型评分。

## 六、面试官可能的深挖与应对清单

| 追问 | 一句话应对 |
|------|-----------|
| 这个反馈功能你们线上在跑吗？ | 以底稿确认后的事实为准；确认前口径："入口和方案我参与设计，高并发统计这套是我按点赞场景标准做法做的设计预案" |
| 为什么用 Lua 不用分布式锁？ | 锁是互斥语义，这里要的是"判重+改状态+改计数"的原子性；Lua 单线程执行天然原子，无锁开销，还能把版本号和事件一起写了（选型框架见 [[☕ Java笔记/分布式锁策略选型-锁非首选判断框架与Redisson实践]]） |
| Redis 挂了反馈丢不丢？ | 关系数据可从 MySQL 重建（SETNX 加载锁 + 双检查）；计数是统计类数据，丢了从关系表重算，业务可接受 |
| 消息乱序怎么办？ | 计数发绝对值+版本号，旧版本直接丢弃；关系落库以 Redis 当前状态为准，不按消息顺序回放 |
| 刷反馈怎么办？ | 用户维度的状态 Set 天然一人一条；恶意刷需要多账号，那是风控层的题，反馈通道本身只保证单用户幂等 |
| 反馈数据量级？ | 不编数字；按设计讲——反馈量 ≤ 对话量，Redis 结构是 messageId 粒度小 key，瓶颈不在存储在写放大，所以才有批量聚合 |

## 七、红线

- 石化 AI 反馈的**落地状态以深挖底稿和项目代码为准**；未确认前只说"设计过方案"，不说"上线运行"；
- tj-remark 是天机学堂学习项目的实现，口径补强材料，可讲类名细节，不包装成石化项目代码；
- 反馈优化效果不编百分比、不编数据量；
- 方案讲"为什么这样设计"（原子性、不丢、不乱、可自愈），不堆 Redis 命令清单。
