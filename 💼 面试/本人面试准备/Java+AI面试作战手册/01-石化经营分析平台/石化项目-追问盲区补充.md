## 1. 经营分析平台到底分析什么（业务指标层）

> 事实层级：**行业通用常识**（已联网核实，以下分析主题均为石化零售行业通行内容）。可讲主题和管理价值，**不编具体指标数值**。

被问"这个平台到底分析什么"时，按五类组织回答：

### 1.1 油品经营

- 分油号销量趋势：92# / 95# / 0# 分号销量、同比环比、站日均销量。
- 油品进销存：进货、销售、库存的日/月平衡。
- 损耗率异常监控：运输损耗、零售损耗超阈值预警。

### 1.2 非油经营

- 品类销售分析：便利店各品类销售结构与趋势。
- TOP 商品排行、库存周转。
- 临期商品监控预警、盘点损益分析。

### 1.3 客户分析

- 会员新增、活跃度、客单价、消费频次分层。
- 客群结构：私家车 / 车队 / 货车等不同客群的消费特征。

### 1.4 设备与安全

- 设备在线率、告警量与告警处理及时率。
- 安全巡检完成率。

### 1.5 管理驾驶舱

- 地市排名、周通报、月考核——省公司抓地市执行的主要抓手。

### 1.6 讲法口径

> 我能讲清平台支撑哪几类分析主题、每类解决什么管理问题，但不编具体指标数值。被问"你做过哪张报表"时按职责边界回答：报表模块不是我主责，我用这些口径讲清平台价值；我直接负责的是设备 IoT 接入和 AI 助手两个方向。

## 5. 开放题预案（可直接背）

### 5.1 "重做一遍会改什么？"

1. **缓存 key 的权限隔离设计应该更早做**（教训导向）——初版缓存 key 只有问题哈希，权限维度是后面补的；如果一期就把 userScopeHash 设计进去，不会留串权限的隐患窗口。
2. **反馈入口应该随 AI 助手一期就上线**——效果闭环会早很多，RAG 优化不用靠人工抽检起步。
3. **traceId 全链路应该在双语言联调初期就统一**——后期补 traceId 透传，排查跨语言问题走过弯路。

### 5.2 "8000 台涨到 8 万台，哪里先扛不住？"

先给结论：

> 先扛不住的是扫描和告警写入，不是 MQ。

展开：

| 环节 | 瓶颈 | 扩容方向 |
|---|---|---|
| MQ | 单队列消费能力到顶 | 队列分区 / 多消费者扩容 |
| XXL-Job | 全量扫描 8 万台跑不完一轮 | 分片广播，按地市或设备号段分片 |
| Redis | 心跳 key 数量级上升，扫描方式失效 | 扫描改 namespace 分批，或换时间轮/ZSET 方案 |
| 告警表 | 告警风暴时写入打满 | 批量写入 + 分表 + 告警聚合 |

### 5.3 "AI 答题助手的题目质量怎么保证？"

> 三层保证：① 生成必须带依据引用——每道题标注来自哪份文档、哪个段落，没有依据的题目不出；② 题目入题库前人工审核；③ 错误题目有反馈下线通道。这三层和 RAG 评估体系是闭环的——答题质量差最终都能归到检索层、文档层或生成层去定位。

### 5.4 "等保 / 安全合规怎么做的？"

> 平台部署在甲方内网，按等保基本要求做访问控制和审计日志；客户数据在查询层做脱敏——比如 queryCustomerSummary 这个工具，返回前按权限过滤并脱敏敏感字段（呼应主底稿 4.3 的脱敏权限点）。等保测评本身是甲方组织的，实施团队按整改要求配合。

## 6. 排错故事（现象→根因→修复→验证）

> ⚠️ 使用边界：三个故事基于项目真实架构写成，技术细节已按 RabbitMQ/Redis/MySQL 真实行为校准。使用前必须本人对照记忆过一遍：哪个故事最接近真实经历就主讲哪个，细节（时间、量级、先后顺序）按真实记忆微调；三个都不要讲成"同一天发生的"。

### 6.1 故事 A：设备离线告警风暴（TTL 集中过期）

**① 面试 60 秒版**

> 有一次省中心侧网络抖了一下，我们告警服务几分钟内涌进来几百条设备离线告警，地市运维的群直接刷屏了，但奇怪的是这批设备几分钟之后又陆续自己恢复在线了。我查下来发现不是设备真坏了：这批设备是同一批接入的，心跳起始时间天然对齐，Redis 心跳 key 的 TTL 阈值又完全一样，网络一抖，几百个 key 在同一两个扫描周期里集体超阈值，XXL-Job 一扫就批量生成了离线告警。修复做了三件事：TTL 加随机抖动打散过期时间点、离线判定加二次确认不一刀切、告警按油站聚合——同站多设备同时离线合并成一条油站级告警。改完之后同规模的抖动，告警量从几百条降到聚合后的十几条，观察期内没有再出现误报风暴。

**② 完整经过**

- **现象**：省中心侧一次网络抖动后，告警服务短时间内涌入几百条设备离线告警，通知通过 MQ 一路推下去，地市运维的告警群被刷屏。反常的是，这批设备几分钟后又陆续自动恢复在线——真正的设备故障不会这样成片出现又成片消失。
- **排查**：先看告警时间分布，发现这几百条告警集中在一两个 XXL-Job 扫描周期内生成，时间高度对齐。抽查这批设备的 Redis 心跳 key（`device:heartbeat:{deviceId}`），发现它们的过期时间几乎一致。回溯接入记录，这批设备是同一批接入上线的，首次心跳时间天然对齐，而 TTL 阈值又是写死的同一个值，相当于几百个 key 被埋了同一个雷，网络一抖就同时引爆。
- **根因**：三个因素叠加。一是 TTL 一刀切、没有任何抖动，同批设备过期时间天然集中；二是离线判定把"超阈值未上报"直接等同于离线，逐台生成告警，网络抖动这种场景被误判；三是告警没有聚合，一个站几十台设备就刷几十条。
- **修复**：第一，TTL 改为"基础阈值 ± 随机偏移"，把过期时间点打散，消除同时过期的雷。第二，离线判定加二次确认——结合设备类型和最近上报时间，网络抖动场景下第一次超阈值不直接告警，确认后再定。第三，告警按油站聚合，同站多台设备在同一时间窗内离线，合并为一条油站级告警，附上设备清单。
- **验证**：修复后遇到同规模网络抖动，告警量从几百条降到按站聚合后的十几条，且全部是真实需要处理的告警；持续观察期内没有再出现误报风暴，运维侧反馈告警可处理性明显改善。

**③ 追问预案**

- **为什么不直接把 TTL 设长一点？** TTL 越长，发现真实离线就越慢，灵敏度和误报率是取舍关系。我们的思路不是牺牲灵敏度，而是保持基础阈值不变、用随机抖动解决"同时过期"这个结构性问题，再用二次确认兜住误报。
- **二次确认的窗口怎么定？** 结合设备上报周期来定：窗口至少覆盖一到两个正常上报周期，保证正常设备在网络恢复后有机会上报一次心跳；窗口太短兜不住抖动，太长又会拖慢真实离线的发现，按设备类型分别取。
- **告警聚合 key 怎么设计的？** 用 `stationCode + alarmType + 时间窗` 做聚合维度：同一油站、同一告警类型、落在同一时间窗内的设备告警合并为一条，设备明细挂在聚合告警下面，运维看到的是"某站 N 台设备离线"而不是 N 条独立告警。
- **为什么网络抖动会导致心跳中断？** 设备心跳上报走 RabbitMQ，网络抖动时连接断开、上报失败，心跳 key 无法续期 TTL，到了扫描周期就被判为超阈值。抖动恢复后设备重连上报，key 续期，设备又自动回到在线——这正好解释了"成片出现又成片消失"的现象。

### 6.2 故事 B：早高峰 RabbitMQ 消费积压（消费端慢 SQL）

**① 面试 60 秒版**

> 早高峰监控发现设备上报队列的 Ready 数持续上涨，设备状态更新延迟从秒级拉到了分钟级。我在 RabbitMQ 管理后台看了五个指标：Consumers 在线、Unacked 不高、但 Deliver rate 明显低于 Publish rate——说明消费者活着但吃得慢，问题在消费逻辑不在 MQ 本身。顺着消费端日志和慢查询日志查下去，定位到设备状态更新的 SQL 缺联合索引，逐条 INSERT/UPDATE 在峰值几百到一千条每秒的量级下被数据库拖死了。修复是消费端改批量处理减少数据库往返，再补上 stationCode 加 reportTime 的联合索引，手动 ack、重试、死信这套可靠性语义保持不变。改完积压曲线回落，高峰期延迟回到秒级。这个故障最大的教训是：MQ 积压不一定是 MQ 的问题，瓶颈很可能在下游数据库。

**② 完整经过**

- **现象**：早高峰监控发现设备上报队列的 Ready 消息数持续上涨，设备状态更新延迟从秒级拉到分钟级，下游看板上的设备状态明显滞后。
- **排查**：在 RabbitMQ 管理后台看五个关键指标：Ready 持续上涨、Consumers 在线数量正常、Unacked 不高、Publish rate 处于高峰正常水位、但 Deliver rate 明显低于 Publish rate。Consumers 在线加 Unacked 不高，说明消费者没有卡死也没有大量堆积未确认消息；Deliver 追不上 Publish，说明消费者在跑但单条吃得慢，瓶颈在消费逻辑本身。查消费端日志没有异常报错，但单条消息处理耗时明显偏长；再查数据库慢查询日志，定位到设备状态更新的 SQL——按 stationCode + reportTime 条件查询更新，缺联合索引。峰值 500-1000 条/s 的逐条 INSERT/UPDATE 全压在没索引的表上，数据库先扛不住。
- **根因**：瓶颈不在 MQ 而在数据库——消费端逐条写入，叠加状态更新语句缺联合索引，单条耗时被拉长，消费速度追不上生产速度，积压在队列里越堆越多。
- **修复**：第一，消费端改批量处理，攒批后一次性 flush，大幅减少数据库往返次数；第二，给设备状态表补上 stationCode + reportTime 的联合索引；第三，可靠性语义不动——仍然是手动 ack、失败重试、进死信队列，批量只改变写入方式，不改变消息确认模型。
- **验证**：修复后队列积压曲线回落并稳定，早高峰 Ready 数不再持续上涨；设备状态更新延迟回到秒级；慢查询日志中该语句消失。消费能力相比修复前有明显提升（定性口径，不讲具体倍数）。

**③ 追问预案**

- **为什么不直接加消费者实例扩容？** 瓶颈在数据库不在消费者数量——消费者已经在线且在跑，只是每条都慢。这时加消费者实例只会让更多连接并发去打同一张缺索引的表，加剧行锁争用，可能适得其反。正确顺序是先治数据库（索引 + 批量），如果治完还不够，再考虑水平扩容。
- **积压期间会不会丢消息？** 不会。消息是持久化的，消费端手动 ack，处理失败会重试、最终进死信队列，可靠性语义保证消息不丢，积压带来的只是延迟不是丢失。这也是我们敢先定位根因再修复、而不是急着扩容的底气。
- **批量大小怎么定的？** 按单条消息大小和 flush 间隔两个维度平衡：批量太小起不到减少往返的效果，太大会拉长单批处理时间、影响时效，还会增大单批失败时的重试成本。按高峰期的实际上报量定了一个攒批阈值加最大等待间隔的组合，吞吐和时效都能接受（定性口径）。
- **Unacked 为什么不高？** 因为消费者没有卡死，每条消息处理完都正常 ack，只是处理得慢；Unacked 高通常意味着消费者 hang 住或大量消息长时间不确认，和"慢但在动"是两种不同的故障形态，管理后台这五个指标的组合正好能把两者区分开。

### 6.3 故事 C：AI 答案缓存串权限（内测发现）

**① 面试 60 秒版**

> AI 助手内测的时候发现一个权限问题：A 地市的员工问了一个涉及本油站数据的问题，之后 B 地市的员工问了个相似的问题，直接拿到了包含 A 地市数据的回答。我复现后确认第二个请求是缓存命中直接返回的，再查缓存 key 设计，发现 key 里只有问题归一化后的哈希，完全没有用户维度——不同权限的人命中同一份答案。这是设计阶段埋的雷，不是偶发 bug：当初设计 key 只考虑了命中率，漏了权限边界。修复是把权限范围维度加进 key，改成 ai:answer 冒号 scene 冒号 userScopeHash 冒号 kbVersion 冒号 questionHash，同时清掉历史缓存，并补了权限变化时的失效策略。验证用两个不同权限的账号回归相同问题，各看各的数据；同一权限范围内仍然共享缓存，命中率影响可以接受。这个事给我的教训是：缓存设计里，权限边界和命中率是同等重要的第一性需求。

**② 完整经过**

- **现象**：AI 助手内测阶段，A 地市员工提问涉及本油站经营数据的问题后，B 地市员工问了一个相似问题，回答里直接出现了 A 地市的数据——B 地市员工本无权限看到这些数据。
- **排查**：用两个不同地市的测试账号复现，确认第二个请求没有走 Function Calling / RAG 链路，而是缓存命中直接返回的。顺着查缓存 key 的设计，发现 key 只由问题归一化后的哈希构成，没有任何用户或权限维度。
- **根因**：缓存 key 设计时只考虑了命中率——相同问题命中同一份答案，命中率高、响应快。但漏了权限边界：不同数据权限的用户问相同问题，应该看到不同答案，却被同一份缓存无差别地服务了。这是设计阶段的缺陷，不是偶发 bug，只要两个权限不同的用户问相似问题就必然复现。
- **修复**：第一，缓存 key 加入权限范围维度，改为 `ai:answer:{scene}:{userScopeHash}:{kbVersion}:{questionHash}`——userScopeHash 按用户的数据权限范围（而非个人）计算，scene 区分应用场景；第二，清理存量历史缓存，杜绝旧 key 继续串权限；第三，补权限变化时的失效策略，用户权限范围调整后旧 scope 的缓存不再被其命中。
- **验证**：用两个不同权限账号回归相同问题，各自只能看到本权限范围内的数据；同一权限范围内的多用户仍共享缓存，命中率相比纯 questionHash 方案的下降在可接受范围内；内测后续阶段未再出现串权限反馈。

**③ 追问预案**

- **命中率会不会崩掉？** 不会崩。userScopeHash 按组织权限范围聚合而不是按个人粒度——同一个地市的员工共享同一个 scope，相同问题仍然命中同一份缓存。牺牲的只是跨地市的重复缓存，份数从"全公司一份"变成"每个权限范围一份"，数量级可控，换来的缓存响应加速依然成立。
- **为什么不在生成答案后做权限过滤，而非要动缓存 key？** 权限过滤本来就在检索阶段做——Function Calling 查数、RAG 检索文档时都按用户权限过滤，缓存是最后一道防线而不是唯一防线。但防线必须是真的：key 里不带权限维度，缓存就直接绕过了前面所有过滤，所以权限必须进 key，这不是冗余是闭环。
- **知识库文档更新了，旧答案怎么办？** kbVersion 进 key 就是解决这个的：知识库版本升级后 kbVersion 变化，旧版本答案的 key 天然失效，新请求落到新版本上重新生成，不需要手动清缓存。
- **为什么是内测才发现，设计上怎么漏的？** 坦率讲这是设计评审的教训：初版设计聚焦在"缓存能不能降延迟、提命中率"，权限维度是在内测真实多角色使用后才暴露的。后续流程上补了一条：凡涉及缓存的 key 设计，评审时必须过一遍"这份缓存谁能看"的检查项。
