## Q1：XXL-Job 定时任务

> 石化项目里保质期预警和设备离线检测都用了 XXL-Job。**为什么选 XXL-Job 而不用 Spring 自带的 @Scheduled？XXL-Job 的调度模型是怎么工作的？**

### 标准答案（60秒）

石化项目里保质期预警、设备离线检测这些定时任务，用的都是 XXL-Job，没用 Spring 的 @Scheduled。

选型原因就两点：

1. **@Scheduled 是多实例的定时炸弹**：多实例部署时每个节点各跑一次，同一批告警发两遍。XXL-Job 有分布式调度能力，只让一个 Executor 执行。
2. **运维方便**：@Scheduled 改 Cron 要改代码→打包→重新部署。XXL-Job 在管理界面点一下实时生效，不用重启。失败有自动重试和告警。

XXL-Job 调度模型：执行器启动时注册到调度中心 → 每隔 30 秒发心跳汇报存活 → Cron 触发时调度中心通过 HTTP 调用执行器的任务方法 → 执行完返回结果 → 调度中心记录日志和状态。

海量数据用**分片模式**：比如 3 台 Executor 各处理 id%3 对应的数据段，互不干扰，效率翻 3 倍。

## Q2：Redis 缓存三连击

> 石化项目里大量用 Redis 做缓存。**缓存穿透、缓存击穿、缓存雪崩分别是什么？怎么防？**

### 标准答案（90秒）

| 现象 | 触发 | 手段 |
|------|------|------|
| **穿透** | 查数据库和 Redis 都不存在的 Key，比如恶意用户查不存在的商品 ID | 布隆过滤器（不存在就一定不存在，存在可能不存在）+ 空值缓存 |
| **击穿** | 热点 Key 过期，大量请求打穿 Redis 直击 MySQL | 互斥锁（只让一个线程查 MySQL 重建缓存）/ 逻辑过期（不设 TTL 自动过期，value 带时间戳，读到过期后先返回旧值异步更新） |
| **雪崩** | 大量 Key 同时过期，请求全砸 MySQL | TTL + 随机偏移（如基础 30 分钟 ±5 分钟）/ 多级缓存 + 限流 |

石化项目里：商品信息缓存容易触发击穿——爆款商品热点 Key 到期，几十个请求一起穿透 Redis。设备状态缓存（3000台加油机心跳）容易雪崩——所有设备心跳 Key 设相同 TTL，集体过期后 3000 请求同时打 MySQL。

## Q3：Spring AI Function Calling

> AI 助手模块要让大模型能查询业务数据。**大模型本身不会写 SQL，项目是怎么让 AI 助手"学会"调用业务接口的？**

### 标准答案（60秒）

AI 助手模块使用的是 Spring AI 的 Function Calling 机制。

具体做法分四步：

1. 在业务方法上加 `@Tool` 注解，参数用 `@ToolParam` 标注。Spring AI 自动生成 JSON Schema 附在请求里发给大模型。
2. 大模型看到工具定义后，判断要不要调工具。如果要调，它**不在自己体内执行**——只返回一个 JSON，包含工具名和参数值。
3. 后端收到这个 JSON，通过 ToolCallback 找到对应的业务方法，真正去执行——比如调 inventoryMapper 查数据库。
4. 查到的结果拼回对话上下文，发给大模型继续对话。

**关键原则：大模型只"点菜"——告诉后端调哪个工具、带什么参数；后端才是真正"做菜"的人。**

石化项目里，业务顾问助手调了库存查询接口——用户问"430101站的92号汽油库存多少"，大模型识别意图后返回调用指令，后端查 MySQL 把数据返给它组织回复。
