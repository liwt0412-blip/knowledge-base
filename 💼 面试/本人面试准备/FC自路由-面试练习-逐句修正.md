# FC 自路由 — 面试练习逐句修正

> Hermes 逐句拷问，4 轮渐进修正。攻克：Function Calling 返回值做路由判断。

## 问题

**石化AI助手里，Function Calling 和 RAG 是怎么分工的？为什么不是所有问题都走 RAG？中间的判断逻辑是什么？**

---

## 第一轮

### 我的原始回答
> 凡克健康里主要是对接的我们业务的接口……Spring端分康适合做健全以及函数调用，Python的rug适合做切块向量检索……

### 逐句修正
| 我说的 | 正确说法 |
|--------|---------|
| 凡克健康里 / 分康 | Function Calling（翻克-深 靠-令） |
| rug / rock的long | RAG（瑞-盖斯）/ LangChain |
| 没提路由判断逻辑 | 没答到核心——谁决定走哪条路？ |

### 缺了什么
- **完全没答路由判断机制**——面试官问的是"中间判断逻辑是什么"，回答偏到了 Java vs Python 分层理由

### 得分：40/100
| 维度 | 评价 |
|------|:--:|
| 核心原理 | ❌ 没答到点上 |
| 表述完整度 | ❌ 偏题 |
| 术语准确 | ❌ |

---

## 第二轮

### 我的原始回答
> 我们是没有单独的路由层的，我们会根据大模型的那个提示词来判断是否需要调用方克新康里……

### 逐句修正
| 我说的 | 正确说法 |
|--------|---------|
| 根据提示词判断 | **致命错误。** 判断信号是 Function Calling API 返回值，不是提示词 |
| 通用知识块 | 意图分类器 / 垂类领域不需要额外分类 |

### 核心纠正
> Prompt 可以引导模型行为，但不能作为路由判断信号。LLM 看到 @Tool 的 Schema 后自己决定调不调——调了 → API 响应带 `tool_calls` 字段（100% 确定性信号）；没调 → 没有 tool_call → 转发 Python RAG。

### 得分：55/100
| 维度 | 评价 |
|------|:--:|
| 核心原理 | ⚠️ 方向对，机制错 |
| 表述完整度 | ⚠️ FC 调用链路说对了 |
| 术语准确 | ❌ |

## 第三轮

### 我的原始回答
> 它的信号不是提示词，而是function calling的那个API返回大模型……如果掉了的话，它就走API返回里面有脱扣字段……

### 逐句修正
| 我说的 | 正确说法 |
|--------|---------|
| API返回大模型 | LLM 返回的响应里带 `tool_calls` 字段 |
| 脱扣字段 | **tool_calls**（兔-靠）——不是断路器脱扣 |
| to calling | 没有 tool_call（不是 to calling） |
| 掉不掉 | 调不调（diào） |

### 进步
- **逻辑全通了**——FC返回值做路由 + LLM天生自信 + 垂类领域防幻觉，主线清晰
- "不依赖模型说实话"是金句

### 得分：72/100
| 维度 | 评价 |
|------|:--:|
| 核心原理 | ✅ |
| 表述完整度 | ✅ |
| 术语准确 | ⚠️ tool_calls 发音偏 |

## 第四轮

### 得分：65/100

逻辑没有问题，术语发音退步（手机输入因素，非能力问题）。

## ✅ 正确版本（直接背）

> 没有独立路由层。判断逻辑就是 Function Calling 的返回值：LLM 看到 @Tool 函数的 Schema，自己决定调不调。调了 → 响应里带 `tool_calls` 字段，Java 侧执行。没调 → 转发 Python 侧走 RAG。
>
> 为什么不用文本输出判断？因为 LLM 天生自信，不会说"不知道"，会硬编。垂直领域最怕幻觉，所以我用 `tool_calls` 这个确定性信号做路由——不依赖模型说实话。
