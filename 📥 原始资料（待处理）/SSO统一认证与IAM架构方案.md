# SSO 统一认证与 IAM 架构方案

> 结论：搭建统一 SSO 完全没问题，是企业多业务系统标准最优解，不存在设计缺陷；但要区分 SSO 解决什么、不能解决什么，同时避开落地常见坑。

---

## 一、多系统重复用户/登录的弊端

- **用户体验极差**：多套账号密码，反复登录；离职漏删权限。
- **维护成本爆炸**：每个系统独立开发登录、验证码、密码加密、找回密码、锁定策略、会话管理，大量重复 CRUD。
- **数据不一致**：多套用户表，同步延迟、漏同步、数据错乱。
- **权限/审计无法统一**：无法全局登录日志、风控（异地登录、异常设备）、组织架构权限。

统一 SSO + 中心化 IAM 一次性解决以上全部痛点。

---

## 二、标准配套架构

SSO 不能单独裸跑，必须搭配用户中心。

### 2.1 用户中心（IAM）

- 唯一存储全量用户、组织、岗位、角色、密码、员工状态
- 业务系统不存完整用户表，只存 `userId`
- 基础信息实时从 IAM 拉取
- 员工新增/离职/调岗只操作一处

### 2.2 SSO 认证服务

- 统一登录页、验证码、密码校验
- 生成全局登录票据（Token/JWT/Session）
- 票据校验、登出、自动跳转
- 业务系统不再自建登录页

### 2.3 业务系统对接

- 无登录态 → 自动跳转 SSO 登录页
- 登录成功 → 下发全局凭证
- 拦截器/过滤器校验凭证合法性
- 通过 `userId` 去 IAM 拉取基础信息
- 全局登出一次性销毁所有会话

---

## 三、方案优势

1. 一套账号通行所有系统，一次登录免重复认证
2. 一套用户数据，员工生命周期统一管控
3. 登录逻辑只开发一次，不用每个服务重复写
4. 统一安全策略：密码复杂度、定期改密、MFA、登录风控、操作审计
5. 微服务天然适配：网关统一拦截校验 token，下游无需处理登录

---

## 四、落地关键坑点

### 4.1 只做 SSO，不做独立用户中心

业务库仍保留完整用户字段 → 数据还是多份，同步问题依旧。

**正确做法**：业务库仅保留 `userId` 外键，姓名、部门等基础信息实时调用 IAM 获取，不落地存储。

### 4.2 SSO 和业务服务强耦合

业务系统不能直接依赖 IAM 的 DO 实体包。

**正确做法**：IAM 对外只提供轻量 DTO 接口，业务系统只依赖独立 `iam-api` 契约包（DTO + 枚举 + 接口定义），身份服务可独立迭代发布。

### 4.3 会话/Token 设计不合理

多标签冲突、Token 无刷新、登出不同步。

**正确做法**：采用 CAS / OAuth 2.0 / OIDC 标准协议，支持全局单点登出（SLO）、Refresh Token。

### 4.4 未区分「认证」和「授权」

SSO 只管你是谁（认证）；细粒度菜单、接口、数据权限由各业务系统自己维护。

IAM 只提供全局角色（管理员/普通员工），不把业务权限堆在身份中心。

### 4.5 内外网不隔离

**正确做法**：内网 IP 白名单、员工工号绑定、二次验证、异常登录告警，避免 SSO 漏洞导致全系统账号泄露。

## 五、技术选型（Java 微服务）

| 层次 | 选型 |
|------|------|
| 认证框架 | Spring Security OAuth2 / Spring Authorization Server、CAS、Keycloak |
| 网关拦截 | Spring Cloud Gateway 统一校验 JWT |
| 协议 | **OIDC 1.0**（标准化对接，前端/微服务/第三方统一适配） |

---

## 六、总结

1. **多内部业务系统绝不重复搭建独立用户+登录体系**，统一 SSO + IAM 是行业标准规范
2. **核心边界**：SSO 解决登录单点通行，IAM 解决统一用户数据生命周期，二者缺一不可
3. **微服务规范**：身份服务与业务服务通过轻量 api 契约通信，禁止共享数据库实体 DO，保证解耦独立迭代
