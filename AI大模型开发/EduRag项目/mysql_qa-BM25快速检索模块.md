# mysql_qa — BM25 快速检索模块

## 1. 模块定位

mysql_qa 是 EduRAG 项目中的 **快速路由**，不依赖大模型，纯靠关键词匹配回答已知问题。

```
用户提问
    │
    ▼
mysql_qa (BM25 + MySQL + Redis)     ← 快路由：已知问题秒回
    ├─ 命中 → 直接返回答案
    └─ 未命中 → "SQL未找到答案" → 交给 rag_qa
                                        │
                                        ▼
                              rag_qa (Milvus向量检索 + BERT分类 + LLM生成)
                              ← 慢路由：未知问题走语义检索 + 推理
```

- **mysql_qa**：查 2048 条预置 Q&A，关键词匹配，毫秒级
- **rag_qa**：向量检索 + 大模型生成，针对库里没有的问题

## 2. 核心逻辑步骤

```
启动
 │
 ├─1. MySQLClient 连 MySQL（数据库 subjects_kg，表 jpkb_temp）
 ├─2. RedisClient 连 Redis
 │
 ├─3. BM25Search._load_data()
 │     ├─ 先查 Redis 有没有缓存的分词问题列表
 │     ├─ Redis 没有 → 从 MySQL 拉全部 2048 个问题
 │     ├─ jieba 逐条分词
 │     ├─ 用分词结果初始化 BM25Okapi 模型（算 IDF）
 │     └─ 原始问题 + 分词结果写回 Redis（下次启动免查库）
 │
 └─4. 进入交互循环，等待用户输入
       │
       ├─ ① 查 Redis 答案缓存
       │     key = "answer:{用户原始输入}"
       │     └─ 命中 → 直接返回，BM25 都不跑
       │
       ├─ ② jieba 分词用户输入
       │     "springboot原理是什么" → ["springboot", "原理", "是什么"]
       │
       ├─ ③ BM25 对 2048 个候选问题逐个打分
       │
       ├─ ④ Softmax 归一化 → 概率分布（总和=1）
       │
       ├─ ⑤ 取最高分，判断是否 ≥ 0.85
       │     ├─ 是 → 拿匹配到的问题原文，查 MySQL 取答案
       │     │       答案写入 Redis 缓存（下次同样问题秒回）
       │     └─ 否 → 返回 "SQL未找到答案"（该走 RAG 了）
       └─ 循环
```

## 3. BM25 算法

### 一句话

**BM25 是 TF-IDF 的升级版**，用来算一个查询跟一堆文档里每个文档的匹配度。

### 跟 TF-IDF 的区别

| | TF-IDF | BM25 |
|---|---|---|
| **词频饱和** | 线性增长（出现10次=1次的10倍） | 对数饱和，超过一定次数不再加分 |
| **文档长度惩罚** | 无 | 长文档自动降权 |

### 公式核心

```
score(D, Q) = Σ IDF(qi) × [ tf(qi,D)×(k1+1) / (tf(qi,D) + k1×(1-b+b×|D|/avgDL)) ]
```

三个关键参数：
- **IDF(qi)**：这个词在整个语料库中稀不稀有？越稀有权重越大
- **k1**（默认 1.5）：词频饱和度控制，越大越接近线性
- **b**（默认 0.75）：文档长度惩罚，0=不管长度，1=完全按比例惩罚

代码用的是 `rank_bm25.BM25Okapi`，k1=1.5, b=0.75（Okapi 默认）。

### 在这个项目里的应用

用户输入 = "查询"，库里 2048 条问题 = "文档集合"。BM25 对每个候选问题打分，分越高 = 跟用户输入越像。

## 4. 为什么用 Softmax 归一化 + 0.85 阈值

BM25 原始分数是绝对值，没有上下限，没法设阈值：

```
"springboot原理" vs 2048个问题  → 原始分: [15.2, 3.1, 0.8, ...]
"今天天气怎么样" vs 2048个问题  → 原始分: [2.3, 2.1, 1.9, ...]
```

两个场景的最高分分别是 15.2 和 2.3，但 15.2 算高还是低？没有参照。

**Softmax 之后**，所有分数变成概率分布（总和=1）：

```
"springboot原理" → Softmax: [0.96, 0.02, 0.01, ...]  → 最高 0.96 ≥ 0.85 ✓ 靠谱
"今天天气怎么样" → Softmax: [0.28, 0.26, 0.24, ...]  → 最高 0.28 < 0.85 ✗ 不靠谱
```

**0.85 的含义**：必须在 2048 个候选问题中有一个"鹤立鸡群"（占比 ≥ 85%），才认为匹配足够靠谱。

## 5. MySQL 和 Redis 的分工

```
┌──────────────────────────────────────────────────┐
│                    MySQL                          │
│  数据库: subjects_kg  表: jpkb_temp               │
│  ┌───────────────────────────────────────────┐   │
│  │ id │ subject_name │ question │ answer      │   │
│  │  1 │ Python学科   │ 用上下文… │ import…     │   │
│  │  2 │ Python学科   │ 什么是GIL │ GIL是…      │   │
│  │ ... │     ...      │   ...    │   ...       │   │
│  └───────────────────────────────────────────┘   │
│  数据来源: JP学科知识问答.csv (2048行)，         │
│           启动时一次性灌入，之后只读不写           │
└──────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────┐
│                    Redis                          │
│  三层缓存：                                       │
│                                                   │
│  ① 启动预热（_load_data 时写入）：                │
│     qa_original_questions  → 所有问题原文列表      │
│     qa_tokenized_questions → 所有问题分词后列表    │
│     用途：下次启动免查 MySQL + 免 jieba 分词      │
│                                                   │
│  ② 答案缓存（BM25 命中后才写入）：                │
│     answer:{用户输入} → 答案文本                   │
│     用途：相同问题第二次来，跳过 BM25 直接返回     │
└──────────────────────────────────────────────────┘
```

**关键区分**：
- MySQL 的数据是启动时一次性灌入的，运行中只读不写
- Redis 的问题列表在启动预热时写，答案缓存在查询命中后才写
- 问题列表全程不参与查询匹配——它只在启动时用一次，之后 BM25 在内存里跑

## 6. jieba 分词

### 为什么要分词

BM25 不理解句子，只能比较词 vs 词。必须把连续的中文字符串切成有意义的词。

```
输入: "springboot的原理是什么"
  ↓ jieba.lcut()
输出: ["springboot", "的", "原理", "是", "什么"]
```

### 两端都要分词

启动时对 2048 个候选问题逐条分词，查询时对用户输入分词。两端都是词列表后，BM25 才能做词级匹配。

### 为什么不用 `str.split()`

中文没有空格，"springboot的原理是什么" 按空格切得到的是一个完整字符串，等于没切。jieba 内置词典，知道 "springboot"、"原理" 各是一个词。

## 7. 完整流程（一图到底）

```
用户输入: "springboot原理是什么"
    │
    ▼
① Redis 查答案缓存
    key = "answer:springboot原理是什么"
    ├─ 命中 → return，结束
    └─ 未命中 → 继续
         │
         ▼
② jieba 分词
    ["springboot", "原理", "是什么"]
         │
         ▼
③ 内存 BM25 对 2048 个分词问题逐个打分
    问题0: 0.02
    问题1: 0.01
    问题2: 0.91  ← 最高
    ...
         │
         ▼
④ Softmax 归一化 → 最高分 0.91
         │
         ├─ < 0.85 → return "SQL未找到答案"
         └─ ≥ 0.85 → 继续
              │
              ▼
⑤ MySQL 查答案
    SELECT answer FROM jpkb_temp WHERE question = 'springboot原理详解'
              │
              ▼
⑥ 答案写入 Redis: "answer:springboot原理是什么" = "SpringBoot是..."
              │
              ▼
          返回答案
```
