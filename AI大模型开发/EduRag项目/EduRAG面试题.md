## 模块一：Milvus 向量数据库（5 题）

### Q1: Milvus 中有哪几种索引类型？项目中用了哪个？

**答**：五种——

| 索引 | 原理 | 场景 |
|------|------|------|
| **FLAT** | 暴力搜索，100% 精确 | 百万级以下小数据 |
| **IVF_FLAT** | K-means 聚类 + 簇内暴力 | 精度速度平衡，**项目用的** |
| **IVF_SQ8** | IVF + 标量量化（4字节→1字节） | 省内存，略有精度损失 |
| **IVF_PQ** | IVF + 乘积量化（拆子向量） | 十亿级超大规模 |
| **HNSW** | 图索引，多层跳转 | 高并发、极致速度 |

EduRAG 中稠密向量字段用 **IVF_FLAT**，稀疏向量字段用 **SPARSE_INVERTED_INDEX**，度量都是内积（IP）。

---

### Q2: nprobe 参数是什么？调大调小各有什么影响？

**答**：nprobe 控制 IVF_FLAT 搜索时扫描的聚类簇数量。

- **调大**：搜更多簇，召回更多候选，精度提高，但查询变慢
- **调小**：只搜最近几个簇，速度快，但可能漏掉答案

本质是**精度 vs 速度的权衡**。

---

### Q3: L2、内积、余弦相似度有什么区别？项目中用哪个？

**答**：

| 度量 | 公式思想 | 值含义 |
|------|----------|--------|
| **L2 欧几里得** | 两点直线距离 | 越小越相似 |
| **IP 内积** | 方向+长度乘积和 | 越大越相似（归一化后） |
| **COSINE 余弦** | 只看方向不管长度 | 越接近 1 越相似 |

EduRAG 项目用 **IP（内积）**，BGE-M3 输出的向量已归一化，内积等价于余弦相似度且计算更高效。

---

### Q4: Collection、Schema、Index 的关系？

**答**：

- **Collection** = 传统数据库的表，存向量+元数据
- **Schema** = 表结构定义，指定有哪些字段（id、text、dense_vector、sparse_vector、parent_id、source 等）
- **Index** = 建在向量字段上的加速结构（IVF_FLAT 等），不建索引 = 暴力搜索

创建流程：`建 Collection → 定义 Schema/字段 → 为向量字段建 Index → load Collection 到内存 → 可操作`

---

### Q5: 为什么 Milvus 需要手动 load Collection？

**答**：Milvus 默认不把数据加载到内存。不 load 的话数据在磁盘上，search/query 都不可用。load 后才从磁盘读到内存，后续检索直接走内存。

## 模块三：MySQL + BM25 + Redis 快速路由（6 题）

### Q1: mysql_qa 在整个系统中的定位是什么？

**答**：**快速路由 / 一级缓存**。不依赖大模型，纯关键词匹配已知的 2048 条 Q&A。

```
用户提问 → Redis 查答案缓存
         → 未命中 → BM25 匹配 MySQL 问题表
                  → 命中(≥0.85) → 返回 + 写 Redis
                  → 未命中 → 交给 rag_qa 走语义检索
```

设计目的：高频问题毫秒级响应，减轻 RAG 管道压力。

---

### Q2: BM25 和 TF-IDF 有什么区别？

**答**：

| | TF-IDF | BM25 |
|---|---|---|
| **词频饱和** | 线性增长（出现 10 次 = 1 次×10） | 对数饱和，超过阈值不再加分 |
| **文档长度惩罚** | 无 | 长文档自动降权 |

BM25 核心公式：`score = Σ IDF(qᵢ) × [tf × (k1+1) / (tf + k1 × (1-b+b×|D|/avgDL))]`

两个关键参数：**k1=1.5**（词频饱和度）、**b=0.75**（长度惩罚强度）。

---

### Q3: 为什么 BM25 分数要做 Softmax 归一化，再加 0.85 阈值？

**答**：BM25 原始分是绝对值，不同问题之间没有可比性——

```
"springboot原理" → 最高分 15.2
"今天天气怎么样" → 最高分 2.3
```

15.2 算高还是低？没法设统一阈值。

**Softmax 后变成概率分布**（所有分加起来=1），0.85 的含义：在 2048 个候选问题中，必须有一个"鹤立鸡群"（占比 ≥85%），才认为匹配靠谱。

---

### Q4: MySQL 和 Redis 在这个模块里的分工？

**答**：

| | MySQL | Redis |
|---|---|---|
| **存什么** | 2048 条 Q&A 原始数据 | ① 分词后的问题列表 ② 答案缓存 |
| **什么时候写** | 启动时一次性灌入 | ① 启动预热时写问题列表 ② 查询命中后写答案缓存 |
| **运行时** | 只读不写 | 读多写少 |
| **作用** | 持久化存储 | 启动加速 + 查询加速 |

关键：运行中 **BM25 全在内存算**，不查 MySQL 也不查 Redis（除非 Redis 命中了答案缓存）。

---

### Q5: jieba 分词在这里起什么作用？

**答**：BM25 不理解句子，只能做**词级别的匹配**。中文没有天然空格，必须用 jieba 把连续字符串切成有意义的词——

```
"springboot原理是什么" → ["springboot", "原理", "是", "什么"]
```

启动时对 2048 个候选问题逐条分词，查询时对用户输入分词。两端都是词列表后，BM25 才能做词级比对。

---

### Q6: 为什么不用 `str.split()` 而用 jieba？

**答**：中文没有空格，`str.split()` 等于没切——"springboot原理是什么" 按空格切还是一个完整字符串。jieba 内置词典，知道 "springboot"、"原理" 各是一个独立的词。

## 模块四：RAG 语义检索 — 双路召回 + 重排序（6 题）

### Q1: 混合检索的完整流程是怎样的？

**答**（**在线过程**）：

```
用户问题（专业知识类）
   │
   ▼
① BGE-M3 向量化 → 稠密向量 + 稀疏向量
   │
   ▼
② Milvus 双路加权混合检索
   稠密向量权重 1.0  +  稀疏向量权重 0.7  → 加权融合 → Top K
   │
   ▼
③ 召回结果 → 转为子块 Document → 按 parent_id 按父块内容去重 → 父文档列表
   │
   ▼
④ BGE Reranker Large 重排序 → 对每个 (问题, 父块) 对打分 → 取 Top M
   │
   ▼
⑤ Top M 上下文 → 组装 Prompt → Qwen LLM 生成
```

---

### Q2: 为什么用稠密 + 稀疏双路检索？各负责什么？

**答**：

| | 稠密向量 (dense) | 稀疏向量 (sparse) |
|---|---|---|
| **原理** | 语义相似度（1024 维浮点） | 词频匹配（词→权重） |
| **擅长** | 同义改写、语义相近但用词不同 | 专业术语、精确关键词 |
| **例子** | "怎么学编程" ↔ "Python 入门方法" | "GIL" ↔ "GIL 全局解释器锁" |

互补：稠密抓语义，稀疏抓关键词。单独用任何一个都会漏。

权重稠密 1.0 > 稀疏 0.7，因为语义匹配比纯词频更重要。

---

### Q3: 为什么要去重 + 重排序？直接返回混合检索结果不行吗？

**答**：

**去重原因**：父子块机制下，同一父块可能拆出多个子块，混合检索可能召回同一父块的多个子块。不去重会返回大量重复上下文，浪费 LLM 的 token 窗口。

**重排序原因**：混合检索的排序是粗排（向量相似度），没有真正理解 "问题和文档的语义匹配程度"。BGE Reranker 是一个**交叉编码器**（Cross-Encoder），把 (问题, 文档) 作为一对输入，输出精确的相关性分数，精度远高于双塔模型的余弦相似度。

---

### Q4: BGE-M3 和 BGE Reranker 的关系？为什么不能只用一个？

**答**：

| | BGE-M3 | BGE Reranker |
|---|---|---|
| **角色** | 召回（粗排） | 重排序（精排） |
| **架构** | 双塔模型（问题/文档分别编码） | 交叉编码器（问题+文档联合编码） |
| **速度** | 快（向量可预计算） | 慢（每对都要重新算） |
| **精度** | 中等 | 高 |

**不能只用 Reranker**：不可能对全库每条数据都跑一遍 Reranker（太慢）。必须先用 BGE-M3 粗召回 Top K，再用 Reranker 在 K 里精排。

---

### Q5: BERT 微调做意图分类的流程？

**答**：

1. 加载预训练 `bert-base-chinese` 模型
2. 读取 1000 条标注数据（通用知识 / 专业咨询）
3. 7:1.5:1.5 划分训练/验证/测试集
4. 文本编码（vocab.txt 将字符→数字）
5. 配置训练参数：epoch=3, batch_size=8, warmup=50, max_length=128
6. 训练 + 每 epoch 验证 → 保存最佳模型
7. 测试集评估 → 准确率从微调前 48% → 微调后 90%+

两种意图：
- **通用知识** → 直接调 LLM 回答
- **专业咨询** → 走 RAG 检索流程

---

### Q6: 四种检索策略分别是什么？

**答**：问题改写（Search Strategy），不同策略应对不同复杂度的查询：

| 策略 | 做法 | 适用场景 |
|------|------|----------|
| **不改写** | 原问题直接向量化检索 | 简单明确的问题 |
| **HyDE** | LLM 先生成假设答案，用假设答案向量化去检索 | 抽象/模糊问题 |
| **子问题** | 将复杂问题拆成多个子问题，分别检索后合并 | 多方面综合问题 |
| **回溯** | 将复杂问题简化为更基础的问题再检索 | 过于具体难匹配的问题 |

## 模块五：RAGAS 评估框架（5 题）

### Q1: RAGAS 评估哪四个指标？各自测什么？

**答**：

| 指标 | 评估对象 | 测什么 | 不达标表现 |
|------|----------|--------|------------|
| **Context Relevancy** | 检索 | 上下文是否精准、无冗余 | 检索回来一堆无关内容 |
| **Context Recall** | 检索 | 上下文是否覆盖所有必要信息 | 漏了关键文档 |
| **Faithfulness** | 生成 | 答案是否基于上下文（不编造） | 幻觉，说上下文里没有的东西 |
| **Answer Relevancy** | 生成 | 答案是否切题 | 答非所问 |

---

### Q2: Context Relevancy 和 Context Recall 的区别？

**答**：一对互补指标——

- **Relevancy（相关性）**：检索回来的内容是不是都跟问题有关？= **别多拿**
  - 公式：`相关句子数 / 上下文总句子数`
- **Recall（召回率）**：该拿的内容是不是都拿回来了？= **别漏**
  - 依赖人工标注的 ground_truth，公式：`被上下文支持的声明数 / 总声明数`

两者需要平衡：recall 高但 relevancy 低 = 搜回一堆东西但很多无关；relevancy 高但 recall 低 = 搜得很精准但漏了信息。

---

### Q3: Faithfulness 怎么测？

**答**：

1. 从生成答案中提取所有事实陈述
2. 逐条检查每个陈述是否能从上下文中推断出来
3. `忠实度 = 被上下文支持的陈述数 / 总陈述数`

例：上下文说"苹果 1976 年由乔布斯、沃兹尼亚克、韦恩创立"，答案说"苹果由乔布斯和沃兹尼亚克创立，总部在库比蒂诺" → 韦恩漏了(不算全错但减分) + 总部信息是编的 → 得分 2/3 ≈ 0.67

---

### Q4: Answer Relevancy 用什么方法测？

**答**：**反向问题生成法**——不直接算问题和答案的相似度，而是：

1. 基于生成答案，用 LLM 反推可能的问题
2. 计算生成的问题与原始问题的语义相似度
3. 平均得分 = 答案相关性

好处：避免了"问题短、答案长"导致的相似度失真。

---

### Q5: 评估脚本需要准备哪些数据？

**答**：四字段的评估数据集（JSON 格式）——

```json
{
  "question": "用户原始问题",
  "answer": "RAG 系统生成的答案",
  "contexts": ["检索到的上下文1", "检索到的上下文2"],
  "ground_truth": "人工标注的标准答案"
}
```

注意：**contexts 必须是列表**（即使只有一个上下文）；ground_truth 是唯一需要人工标注的字段。

## 模块六：FastAPI 集成 + 流式输出（4 题）

### Q1: EduRAG 的整体请求链路是怎样的？

**答**：

```
用户请求（query + source_filter + session_id）
   │
   ▼
① 查 Redis 缓存 → 命中 → 返回
   │ 未命中
   ▼
② BM25 查 MySQL（阈值 0.85）→ 命中 → 返回 + 写 Redis
   │ 未命中
   ▼
③ BERT 意图分类 → 通用知识 → LLM 直接回答
   │ 专业咨询
   ▼
④ RAG 管道：混合检索 + 重排序 → 组装 Prompt → Qwen 流式生成
   │
   ▼
⑤ 写入对话历史（MySQL conversations 表，保留最近 5 轮）
```

---

### Q2: 对话历史怎么管理？

**答**：

- **存储**：MySQL `conversations` 表（session_id, question, answer, timestamp）
- **获取**：按 session_id 查最近 5 轮，时间倒序取后再反转为正序
- **更新**：插入新记录后，删除超出 5 轮的旧记录（子查询 "保留最近 5 条 id"）
- **清除**：按 session_id 全删
- **前端**：若无 session_id，自动生成 UUID 作为新会话

---

### Q3: 流式输出是怎么实现的？

**答**：两种方式：

**CLI 模式**：用 Python 生成器逐 token yield，主循环里 `print(token, end="", flush=True)` 逐个打印，不换行。

**FastAPI 模式**：用 **SSE（Server-Sent Events）**，每次 yield 一个 JSON chunk：
```json
{"type": "token", "content": "Spring"}
{"type": "token", "content": "Boot"}
...
{"type": "end"}
```

LLM 调用时设置 `stream=True`，遍历 `completion` 的每个 chunk，提取 `delta.content` 逐 token 产出。

---

### Q4: BM25 直返和 RAG 流式返回的区别？

**答**：

| | BM25 命中 | RAG 流式 |
|---|---|---|
| **返回方式** | 一次性返回完整答案 | 逐 token 流式发送 |
| **用户体验** | 瞬间出结果 | 像打字一样逐字出 |
| **适用** | 高频已知问题 | 未知/复杂问题 |

在 FastAPI 中，BM25 命中走普通 HTTP POST 返回 JSON；RAG 走 `/api/stream` SSE 接口。

## 面试练习记录（2026-06-26）

### 题1：Milvus 索引类型

**标准答案**：FLAT（暴力）/ IVF_FLAT（聚类+倒排）/ IVF_SQ8（标量量化）/ IVF_PQ（乘积量化）/ HNSW（图索引）

**纠正**：IVF_FLAT 读法是"I-V-F Flat"，不是"env flat"；倒排索引不是"倒叙"；nprobe 控制搜簇数不是磁盘；度量用 IP 不是"层级"

**过关比喻**（图书馆分类）：先把书按主题分成 N 个书架（聚类）→ 每个书架贴标签（中心点）→ 查最近标签的几个书架（nprobe）→ 在这几个书架逐本找（暴力搜索）

---

### 题2：BM25 + Softmax + 0.85 阈值

**标准答案**：BM25 打绝对值（15.2 vs 2.3 不可比）→ Softmax 统一成概率（0~1）→ 0.85 = 在 2048 个候选里必须有一个"鹤立鸡群"

**纠正**：读法 B-M-25 不是 BN；阈值 0.85 不是 80%

---

### 题3：稠密 + 稀疏双路检索

**标准答案**：稠密=语义（同义改写能匹配），稀疏=词频（精确关键词命中），互补

**最佳举例**：搜"GIL" — 稠密拉回一堆多线程文章（80%不解释GIL），稀疏精准命中"GIL 全局解释器锁"；搜"零基础怎么入门" — 稀疏不认识，稠密匹配"Python 初学者指南"

**纠正**："持平"这个词不要用
