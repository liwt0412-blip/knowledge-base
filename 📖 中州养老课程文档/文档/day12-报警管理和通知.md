01- 课程内容介绍

	1- 目前各种智能设备 数据已经上报至我们自己的系统，现在要基于这些数据，实现异常数据的【监控】 和 【通知】

		1- 要识别出什么样的数据是异常数据 - 【设定报警规则（报警规则管理）】

		2- 要持续根据这个规则识别我们上报的数据 - 【定时任务】

		3- 识别出来是异常数据后要通知对应的管理人员 - 【Websocket】

		

	3- 实现今天的需求，需要准备一套新的环境，原因：

04- 报警规则-介绍

	1- 作用：要识别出什么样的数据是异常数据

	2- 具体包括：

		1- 监控范围：指定监控 哪个【产品】下哪个【服务】（模块）下的哪些【设备】 下的 哪个【功能】

		2- 【报警数据类型】：老人异常数据 或者 设备异常数据（决定将来是通知 护理人员 还是 维修人员）

		3- 【报警规则名称】

		4- 触发时机：当设备数据【运算符】【阈值】时 ，但是也不是立马触发规则，还要看是否符合【持续周期】

		5-【持续周期】（将来是通过定时任务监控，1分钟跑一次，1分钟就是1个周期）：指定 异常数据上报持续几个周期时 触发该规则

		6-【报警生效时间】：指定该报警规则生效时间段，比如有些设备晚上就不需要报警规则的监控了

		7-【沉默周期】：报警触发后，多久才会继续 触发，有些设备报警后，由于人员到场需要时间，一定时间范围内无需报警了

	3- 当触发报警规则时，保存报警数据至 报警数据表 和 站内通知表

07- 报警业务-执行流程分析

	0- 提前维护好 报警规则：报警规则= 功能名称+运算符号+阈值，持续x个周期就真正报警

	1- 通过定时任务- 每分钟根据 后台设定的有效规则集 检测一次目前所有智能设备最新上报数据

	2- 查询库内的所有有效规则数，判断数量是否大于0，如果小于0，【定时任务直接结束】

	3- 如果存在规则，从Redis中获取所有设备的最新上报数据（Hash结构的values - 存的所有设备最新上报数据）

	4- 判断从values获取的最新上报数据集合是否为空，为空，【定时任务直接结束】

	5- 对values数据做解析，将所有的上报数据（DeviceData）存入至一个大的待检测集合中（List<DeviceData>）

	6- 遍历待检测集合，对每一个上报数据（DeviceData）进行检测（检测方法），形参：DeviceData

	

    7- 检测该条上报数据 上报时间是否是一分钟之内的，如果不是，【结束当前方法，遍历下一条数据】

    8- 根据上报数据中的 产品ID、设备ID、功能名称 获取该数据 对应的报警规则集合

    9- 判断报警规则集合是否为空，如果为空，【结束当前方法，遍历下一条数据】

    10- 如果报警规则集合不为空，针对该条上报数据，对报警规则进行循环遍历检测，形参：DeviceData AlertRule

	  

    11- 判断该条上报数据 上报时间是否在 该条报警规则的有效时段内，如果不在，【结束当前方法，遍历下一条规则】

    12- 如果在有效时段内，检测上报数据是否触发该条报警规则的阈值，如果未触发，重置报警数（持续周期） - 删除Redis中的报警数

    13- 如果触发了该条报警规则的阈值，开始判断目前是否在该条规则+当前数据的 沉默周期范围，如果在，结束当前方法，遍历下一条规则】

    14- 如果不在沉默周期范围，Redis中报警数+1，拿到+1后的数值（最新的报警数）

    15- 判断最新的报警数 是否 大于等于 了报警规则中设定的 持续周期，如果没有，结束当前方法，遍历下一条规则

    16- 如果有，开始真正报警：

		1、删除Redis报警数（持续周期）

		2、添加新的沉默周期

		3、通知对应的人

			1）、判断该条规则如果触发，是通知护理人员还是后勤人员

			2）、如果是通知护理人员，需要根据设备数据中的bind_location字段 找到 对应的护理人员ID

			3）、如果是后勤人员，需要根据角色名称找到对应的 维修工 ID

			4）、拿到所有管理员ID

			5）、合并所有的人员ID，开始发站内消息通知

			6）、发websocket消息，通过页面弹框形式通知

	

08- 报警数据表结构介绍

	1- alert_data：一旦某条上报数据真正触发报警，就在这张表新增报警数据，需要通知给几个人，就保存几条数据

	2- 个别字段说明

		- access_location  接入位置（老人、床位、房间ID）

		- alert_reason 报警原因（功能名称+运算符+阈值+持续周期 拼接的字符串）

		- data_value 触发报警的那条上报数据值

		- processing开头的字段：报警数据后续是需要人点击处理的，该类字段存储 处理信息，如谁在什么时候处理的，新增时为空即可

		- status：标注该条报警数据是否已处理，新增时 默认 待处理 0

		- user_id：被通知人ID

	3- 代码生成
