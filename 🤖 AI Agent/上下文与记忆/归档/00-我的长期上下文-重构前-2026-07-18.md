# 00-我的长期上下文

> 用途：这是给所有 Agent 共用的长期上下文入口，不绑定 Codex、Claude、ChatGPT 或任何单一工具。  
> 原则：只写长期稳定、会反复影响决策的信息；不写临时任务、短期情绪、一次性细节、未经确认的猜测。  
> 使用方式：任何 Agent 开始处理我的长期任务前，应先读取本文件，再读取本文件引用的专题文档。

---

## 0. 多 Agent 使用协议

### 0.1 Agent 开始工作前

任何 Agent 如果要处理以下任务，应先读取本文件：

- 整理知识库
- 修改面试材料
- 生成长期规划
- 重构项目文档
- 管理个人上下文
- 延续上一次 Agent 的工作

读取顺序：

1. 先读本文件。
2. 再读本文件指向的专题入口。
3. 最后再读任务相关的具体文件。

### 0.2 Agent 写入本文件前

写入前必须判断信息是否满足至少两条：

- 未来一个月以上大概率仍然有效。
- 会反复影响后续 Agent 的判断。
- 和我的身份、目标、项目、知识库结构、表达红线、协作偏好有关。
- 如果不写，下次 Agent 容易误判或重复沟通。

不满足这些条件的内容，不应写入本文件。

### 0.3 修改规则

- 不要把本文件变成流水账。
- 不要大段粘贴临时聊天记录。
- 不要写未经我确认的事实。
- 如果是推断，必须标注“推断”。
- 如果是过期信息，移动到“历史记录”或删除，不要和当前事实混在一起。
- 大改本文件前，应说明改动意图。

### 0.4 冲突处理

如果本文件和其他笔记冲突：

1. 以最近更新、且更具体的专题文档为准。
2. 如果冲突影响面试口径、项目真实性或文件移动，先向我确认。
3. 不要自行把高风险口径写成事实。

## 7. Agent 协作偏好

- Agent 应先读文件再判断，不要凭空发挥。
- 重要或批量修改前，先从第一性原理说明判断逻辑和改动方案。
- 大规模改动前先确认；用户明确授权后再执行。
- 修改知识库时优先归档，不要直接删除。
- 面试材料先质疑真实性和抗追问性，再做表达润色。
- 对高风险项目经历要标明“直接负责 / 参与联调 / 了解架构 / 平台已有”。
- 面试练习中，如果技术术语或英文发音明显错误，应直接纠正。
- 使用技能或工具时，应遵守该技能/工具的完整工作流，不把技能当普通参考文档。
- 制作 AI/技术科普视频时，默认使用 `D:\workspece\GitHup\remotion` 项目，并遵循“确认脚本 → IndexTTS2 克隆音色 → 以最终音频确定时长并对齐字幕 → 生成场景 → 预览确认 → 完整渲染”的流程；具体执行规范以该项目的 `AGENTS.md` 和 [[remotion科普视频]] 为准。
- 每次长对话结束或阶段性任务完成后，Agent 应主动判断本轮是否产生需要写入长期上下文的信息；只给出候选项，不默认写入，等待用户确认。
- 编写 Java 项目时默认重视面向对象封装：保护可变状态，校验对象不变量，按业务语义组织方法，并区分公开接口与内部实现；不机械生成全部 getter/setter，也不把所有校验都塞进 setter。完整规范见 [[☕ Java笔记/Java编码规范与面向对象封装]]。
- Java 开发默认按“正确性与安全优先、可维护性默认、抽象按真实变化、性能按证据优化”的顺序决策；避免机械行数阈值、过度抽象和无证据微优化。完整规范见 [[☕ Java笔记/Java全套代码质量与优化规范]]。
- 编写 Java 对外接口时，遵循项目既有的 API 文档体系（项目使用 Swagger 时补齐 Swagger 注解）；复杂 Service 逻辑使用简短步骤注释说明关键约束，避免逐行重复代码含义的注释。
- 实现 Java 服务接口时，默认遵循数据归属、用户隔离、对外契约、历史数据确定性和异步幂等原则；详细规则见 [[☕ Java笔记/Java全套代码质量与优化规范]] 与相关专题笔记。
- 处理涉及数据库读写、关联查询、统计、删除、事务或幂等的任务时，Agent 应先从项目中的 DDL、迁移文件、实体和 Mapper 核对主键、唯一约束、索引、空值、默认值、外键及时间字段语义；仓库中找不到或代码与表结构冲突时，再向用户索取相关表的建表语句，不得仅凭实体类猜测数据库约束。
- Java 聚合查询默认使用专用 DTO/Projection 承接结果，不为一次性统计随意向持久化实体添加非表字段；只有该派生字段稳定属于领域模型且被多处复用时才考虑放入实体。
- 时间范围统计默认使用左闭右开的半开区间（如 `time >= 本周开始 && time < 下周开始`），避免相邻周期在边界处重复统计。
- 异步链路必须具备可观测性：关键状态变化、消费失败、重试、幂等命中和最终结果应记录带业务主键的结构化日志；高频成功消息按采样、汇总指标或 debug 日志处理，避免逐条 info 输出。
- 对高频且只关心最终最新值的数据，优先采用“Redis合并写 + 延迟刷盘”；对首次完成、库存、支付等关键状态，必须以 MySQL 条件更新和事务作为最终事实。缓存、ZSet、MQ 只能承担加速、调度和异步解耦，不能替代数据库最终一致性；设计方案必须区分“已实现、已验证”和“仅设计”。
- 长期上下文只保留会长期、反复影响 Agent 决策的稳定信息；项目接口、表字段、队列名等具体事实写入项目专题笔记，不写入长期上下文。

- 不为了使用工具类而强行封装简单逻辑；如果 Java 原生写法更清晰且不违背项目风格，可以使用原生写法。

## Codex 协作规则：企业代码评估（2026-07-17）

- 评估企业代码时，不能只看功能是否能运行，还要同时考虑安全、隐私、空值边界、性能、N+1、远程调用和可维护性。
- 架构选择应根据业务场景、复杂度、性能和维护成本进行权衡，不把某一种实现模式绝对化。

## 性能设计：避免循环外部 I/O

- 对数据库、Redis、消息队列、Feign/HTTP 等外部 I/O：若循环次数随数据规模增长，默认避免在循环中逐条调用；优先收集 ID 后批量查询/批量处理，再用 Map 回填。
- 允许例外仅限循环上限明确且很小，或顺序/事务语义不可替代；需要写明原因、上限和失败策略。
- 评审分页代码时，重点排查 N+1 数据库查询和 N+1 远程调用。
- 代码评审时，显式检查 `for`/`stream` 循环内是否逐条操作数据库、Redis/缓存、MQ 或 Feign/HTTP 等中间件与远程依赖；若循环随数据量增长，优先改为批量调用，避免 N+1。

## Java Web 路径参数规范

- Controller 中使用 `@PathVariable("参数名")` 显式绑定路径参数，例如 `@PathVariable("id") Long id`。
- 这样不依赖编译器是否保留方法参数名，接口契约更明确，便于 Swagger 文档生成和团队维护。
- 这是通用编码原则；具体项目仍应优先遵循项目既有风格。
## Swagger/Knife4j 模型注解规范

- 如果项目使用 Swagger 或 Knife4j，新增 DTO、VO 应按照项目既有风格补充 `@ApiModel` 和 `@ApiModelProperty`。
- 这不是业务运行的硬性要求，但能保持接口文档完整、字段含义清晰，并减少前后端沟通成本。
- 是否添加及注解粒度要遵循当前项目规范，避免跨项目强行统一。
## 缓存设计：基础数据归属与两级缓存取舍

- 遇到课程分类、地区、字典等“读多写少、体量小、允许短暂最终一致”的基础数据，优先评估缓存，而不是让每个业务请求重复查库或远程调用。
- 缓存应围绕数据归属设计：数据拥有服务负责数据正确性与失效事件；消费服务可以使用本地缓存降低远程调用，但不要复制一套无失效机制的业务缓存。
- 微服务常用取舍：单实例或可容忍短暂不一致时使用 Caffeine 本地缓存；跨实例一致性要求更高时使用 Redis 作为共享缓存，并通过消息/事件失效各实例 L1 缓存。
- 先确认项目已有缓存组件和约定，优先复用。例如天机项目的 `CategoryCache` 缓存全量分类映射；启用后业务代码只消费缓存 API，不直接重复调用分类 Feign 接口。
- 缓存不是默认答案：写频繁、数据敏感、强一致要求高、命中率低或数据量不可控时，先考虑索引、批量查询、读模型等方案。

## Java 依赖注入取舍

- 默认采用构造器注入，依赖字段声明为 `private final`，使依赖关系显式且不可变，测试时也便于替换。
- 项目已使用 Lombok 且构造器没有自定义逻辑时，优先 `@RequiredArgsConstructor` 消除纯赋值样板代码；不要再手写等价构造器。
- 需要构造期校验、初始化、多个构造器或团队禁止 Lombok 时，手写构造器更合适。
- 新代码先遵循当前项目已有注入风格，避免同一模块混用字段注入、手写构造器和 Lombok 构造器。

## 2026-07-17：互动问答文本内容审核（阿里云）

- 审核范围：提问标题/描述、回答、评论；所有用户可见文本都必须经过同一审核入口。
- 状态机：`PASS`（直接展示）、`PENDING`（疑似违规或阿里云调用失败，落库但 `hidden=true`，等待人工审核）、`REJECT`（明确违规，抛业务异常且不落库）。这是一种 fail-closed（故障保守）策略。
- 领域数据中单独记录 `auditStatus`、`auditReason`、`auditTime`，不要只复用 `hidden`：`hidden` 表示展示状态，审核状态表示为什么不可展示，二者职责不同。
- 管理员将待审核内容取消隐藏时，同步写为人工审核通过；隐藏一级回答时，级联隐藏其评论。
- 阿里云凭据放在 Nacos `shared-aliyun.yaml`，由需要审核能力的服务显式导入。共享凭据应配合 Nacos 权限控制，并使用最小权限 RAM 用户，禁止把 AK/SK 写入代码、日志或知识库。
- 历史数据迁移默认审核通过，避免上线后存量内容被误隐藏；新增字段必须通过 SQL 迁移脚本部署。
### 内容安全集成原则（长期）

- 用户生成文本（标题、正文、回答、评论）应走统一内容审核入口；审核结论与展示状态分离建模。
- 默认采用 fail-closed：第三方审核超时、失败或未知风险时进入人工审核并暂不展示；明确违规直接拒绝。
- 密钥只存受控配置中心；使用最小权限 RAM 身份，禁止提交、打印或写入知识库。共享配置由服务显式按需导入，并通过配置中心 ACL 约束。
