# 护理任务 & 报警数据接口开发问题记录

## 护理任务模块（5 个接口）

### 1. 实体字段与数据库不一致
- **问题**：SQL 文件 `ry-zzyl.sql` 中 `nursing_task` 表没有 `task_type` 列，但实际数据库有，INSERT 时报 `Field 'task_type' doesn't have a default value`
- **解决**：实体 `NursingTask.java` 补上 `taskType` 字段

### 2. 接口 URL 不匹配
- **问题**：文档写 `/nursing/task/list`，前端实际请求 `/nursing/nursingTask/list`，404
- **解决**：改后端 URL 为 `/nursing/nursingTask`，与前端一致

### 3. Service 接口 import 被误删
- **问题**：编辑 `INursingTaskService.java` 时把 `IService` 的 import 覆盖掉了，编译报错
- **解决**：补回 `import com.baomidou.mybatisplus.extension.service.IService;`

### 4. OSS 图片 URL 双 https:// 导致图片无法显示

#### 现象
前端在执行任务时上传图片，上传接口返回成功，但页面上的图片不显示（回显失败）。

#### 排查过程
查看上传接口 `POST /common/upload` 的响应：
```json
{
  "code": 200,
  "url": "https://web-qk-xxt.https://oss-cn-beijing.aliyuncs.com/32e7ee13-xxx.png",
  "fileName": "https://web-qk-xxt.https://oss-cn-beijing.aliyuncs.com/32e7ee13-xxx.png"
}
```
URL 中出现了双 `https://`，这个 URL 作为 `<img>` 的 `src` 时浏览器无法解析，图片自然不显示。

#### 根因分析
OSS 上传流程分两步：

**第一步（上传文件）**：`OSSAliyunFileStorageService.store()` 调用阿里云 SDK 的 `putObject()`，把文件字节流推送到 OSS 桶。这一步只涉及 SDK 与 OSS 服务的 TCP 通信，不依赖 URL 拼接，所以**上传成功**，文件确实在桶里。

**第二步（返回 URL）**：同一方法在文件上传后拼接访问 URL：
```
OSSAliyunFileStorageService.java 第 62-68 行：

代码:    "https://" + bucketName + "." + endpoint + "/" + objectName
              ↓
实际:   https://web-qk-xxt.https://oss-cn-beijing.aliyuncs.com/xxx.png
```

代码硬编码了 `https://` 前缀，而 `application-dev.yml` 中 OSS 配置的 endpoint 也带了 `https://`：
```yaml
oss:
  endpoint: https://oss-cn-beijing.aliyuncs.com   # ← 多了 https://
  bucketName: web-qk-xxt
```

两者叠加导致 URL 变成双 `https://`。test/prod 环境的 endpoint 配置是 `oss-cn-beijing.aliyuncs.com`（无 `https://`），所以那两个环境没有此问题。

#### 解决
修改 `application-dev.yml` 第 191 行，去掉 endpoint 的 `https://` 前缀：
```yaml
# 改前
endpoint: https://oss-cn-beijing.aliyuncs.com
# 改后
endpoint: oss-cn-beijing.aliyuncs.com
```
拼出的 URL 变为 `https://web-qk-xxt.oss-cn-beijing.aliyuncs.com/xxx.png`，正常访问。

#### 教训
遇到"功能正常但数据不对"的 bug（上传成功、URL 看起来也对、但图片不显示），先看**实际数据**：打开浏览器 F12 Network，检查接口响应的 URL 是否真正可用，不要对着代码推演"理论上应该没问题"。

### 5. @PreAuthorize 导致 403
- **问题**：护士账号登录后调用接口返回 `没有权限，请联系管理员授权`
- **解决**：移除 `NursingTaskController` 和 `AlertDataController` 中所有 `@PreAuthorize` 注解，数据隔离由 Service 层 `SecurityUtils.getUserId()` 保证

## 报警数据模块（2 个接口）

### 6. nickname 列不存在
- **问题**：SQL 文件有 `nickname` 列，但实际数据库没有，查询报 `Unknown column 'nickname' in 'field list'`
- **根因**：SQL 文件与实际数据库不同步
- **解决**：从 XML 和实体中移除 `nickname` 字段

### 7. ISO 时间格式解析失败
- **问题**：前端传 `2026-06-29T16:00:00.000Z`，后端用 `DateTimeFormatter.ISO_LOCAL_DATE_TIME` 解析报错
- **根因**：`ISO_LOCAL_DATE_TIME` 不支持 `Z` 时区后缀和毫秒
- **解决**：改用 `Instant.parse()` 转 `LocalDateTime`

## 通用教训

| 教训 | 场景 |
|------|------|
| 先看数据再看代码 | OSS URL 双 https |
| 不要只信 SQL 文件 | nickname 列、task_type 列 |
| 小心 import 覆盖 | IService import 丢失 |
| URL 以实际前端调用为准 | /nursingTask vs /task |
| 前端时间格式注意时区后缀 | .000Z 格式 |
