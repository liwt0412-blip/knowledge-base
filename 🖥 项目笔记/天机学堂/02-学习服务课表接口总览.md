# 学习服务课表接口总览

> 范围：本对话完成或分析过的课表查询、校验、删除接口，以及支付/退款后的 MQ 联动。

## 一、HTTP 接口总表

| 编号 | 接口 | 职责 | 数据来源 | 关键规则 |
|---|---|---|---|---|
| 1 | `GET /lessons/page` | 分页查询我的课表 | `learning_lesson` + 课程服务 | 当前用户隔离；课表分页后批量远程补课程名、封面、总小节数，避免 N+1 调用 |
| 2 | `GET /lessons/now` | 查询最近正在学习的课程 | `learning_lesson` + 课程/目录服务 | 取最近学习时间最新的“学习中”记录；补课程信息和最近学习小节信息 |
| 3 | `GET /lessons/{courseId}/valid` | 校验用户是否有有效学习权益 | `learning_lesson` | 当前用户 + 指定课程；排除 `EXPIRED`；永久课程 `expireTime == null`，限时课程要求 `expireTime > now`；有效时返回课表 ID，否则返回 `null` |
| 4 | `GET /lessons/{courseId}` | 查询指定课程的课表状态 | `learning_lesson` | 当前用户 + 课程 ID；重复购买记录时按购买时间取最新一条；未加入课表返回 `null` |
| 5 | `DELETE /lessons/{courseId}` | 用户删除已失效课程 | `learning_lesson` | 只能删除当前用户自己的课程；仅允许状态已失效或到期时间已过的课程；有效课程删除应报业务异常 |

## 二、接口 4：指定课程状态

```http
GET /lessons/{courseId}
```

返回的是课表领域状态，而不是课程详情：

```json
{
  "id": 10,
  "courseId": 1002101,
  "status": 1,
  "learnedSections": 5,
  "createTime": "2026-07-13T10:00:00",
  "expireTime": "2026-12-13T10:00:00",
  "planStatus": 1
}
```

这些字段都在本地课表表中，因此不需要远程调用课程服务。只有还要展示课程名、封面、课程总小节数等课程领域字段时，才应批量远程查询课程服务。

## 三、MQ 消费链路

| 事件 | Routing Key | 学习服务动作 | 注意事项 |
|---|---|---|---|
| 订单支付成功 | `order.pay` | 将订单课程加入用户课表 | 消息字段校验失败后必须 `return`；不能仅记录日志后继续访问空对象 |
| 退款成功 | `order.refund` | 删除退款课程的课表记录 | 只能在支付渠道确认退款成功后发送；删除必须幂等，重复消费不能失败 |

```text
订单支付成功 → order.pay → 学习服务新增课表
退款成功     → order.refund → 学习服务删除课表
```

## 四、数据模型与一致性边界

当前兼容实现可按 `userId + courseId` 删除退款课程，但它有前提：同一用户不会同时保留同一门课程的多次购买权益。

若业务允许重复购买，必须升级为：

1. 课表记录保存 `orderDetailId`。
2. 支付、退款消息都携带 `orderDetailId`。
3. 新增、校验、退款删除按订单明细精确关联。

否则退款旧订单时，可能误删用户重新购买后产生的新课表记录。

## 五、实现检查清单

- [ ] Controller 有正确的 Spring MVC 映射和 Swagger 注解。
- [ ] 所有用户接口均从 `UserContext` 获取用户，不能信任前端传入的用户 ID。
- [ ] 单条课表查询使用 `null` 判断；批量集合使用集合判空。
- [ ] 课程列表补充信息采用批量远程调用，不在循环内逐条调用。
- [ ] 课程有效性同时考虑状态和过期时间。
- [ ] MQ 消费者校验消息后立即返回；业务删除具备幂等性。
- [ ] 重复购买场景有订单明细关联，或明确禁止重复权益并建立约束。
