# 新增互动问题接口实现

## 接口契约

- 请求方式：`POST`
- 请求路径：`/questions`
- 返回值：空响应体
- 业务场景：课程详情页或视频学习页提交互动问题。

请求示例：

```json
{
  "courseId": 1578558664933920770,
  "chapterId": 1578558664933920770,
  "sectionId": 1578558664933920356,
  "title": "JDK 哪里下载",
  "description": "找不到下载地址",
  "anonymity": false
}
```

## 实现约束

1. `QuestionFormDTO` 只接收课程、章节、小节、标题、描述和匿名字段，不接收 `userId`。
2. 服务层从 `UserContext` 获取当前登录用户，未登录时抛出 401，避免前端伪造提问人。
3. 新问题保存时初始化：`answerTimes = 0`、`hidden = false`、`status = UN_CHECK`。
4. DTO 校验 ID 必须大于 0，标题不为空且不超过 255 字符，描述不为空且不超过 2048 字符。

## 企业项目视角下的查询架构取舍（2026-07-17）

企业实现不能只判断“能否查询成功”，还要检查以下边界：

- 隐私：匿名数据不仅不能返回名称和头像，也不能泄露真实 `userId`。
- 空值：批量 ID 集合、远程调用结果、可选 Boolean 条件都必须安全处理。
- 远程调用：集合为空时不调用用户服务；用户信息必须批量查询，避免 N+1。
- 查询范围：课程 ID 和小节 ID不能同时为空，避免无范围全表查询。
- 分层：Mapper 直接返回查询投影 VO 在复杂列表、报表和搜索场景中是可接受的；如果查询对象与接口 VO 差异较大，或需要更严格的分层，可以采用 `Mapper → Row/DTO → Service 组装 → VO`。

两种方案都属于企业中真实存在的做法，取舍依据是：

1. 查询复杂度和性能要求；
2. 数据库字段与接口字段的差异；
3. 后续复用和维护成本；
4. 是否需要严格隔离持久化模型与接口模型。

当前互动问题分页查询采用查询投影方式，并在 Service 层完成匿名清理、批量用户补全和参数边界校验；后续如果 VO 与数据库查询结果继续变复杂，再拆分内部查询 Row/DTO。

## 根据 ID 查询问题详情（2026-07-17）

- 请求方式：`GET`

## 路径参数注解规范（2026-07-17）

天机项目 Controller 的路径参数统一显式声明路径名称，例如：

```java
@GetMapping("/{id}")
public QuestionVO queryById(@PathVariable("id") Long id) {
    return questionService.queryById(id);
}
```

这样可以明确绑定 URL 中的 `{id}`，不依赖 Java 编译时是否保留方法参数名，也方便接口文档和团队协作维护。当前学习服务已有的 `courseId` 路径参数也保持显式绑定。
## 回答/评论分页查询（2026-07-17）

- 请求方式：`GET`
- 请求路径：`/replies/page`
- 请求参数：`pageNo`、`pageSize`、`questionId`、`answerId`；问题 ID 和回答 ID 至少传入一个。

- 点赞状态：当前项目尚未实现点赞关系表和点赞业务，因此本接口只返回 `likedTimes`，暂不返回当前用户是否点赞。

## 管理端问题分页查询（2026-07-17）

- 请求方式：`GET`
- 请求路径：`/admin/questions/page`
- 请求参数：`pageNo`、`pageSize`、`courseName`、`status`、`beginTime`、`endTime`。
- 管理端与用户端边界：管理端可以查询隐藏问题，因此不复用用户端的 `hidden = false` 过滤；管理端也需要展示提问人昵称。
- 课程名称筛选：学习服务不直接跨库查询课程表，先通过 `CourseClient` 调用课程服务按名称获取课程 ID，再将 ID 集合传给问题 Mapper，避免跨服务直接访问数据库。
- 展示信息：问题分页查询后按当前页的不同课程补充课程目录和分类名称，并批量补充提问人信息。
- 校验：状态只能为 0/1；开始时间不能晚于结束时间；课程名称查不到课程时直接返回空分页结果。
- 性能取舍：用户信息使用批量调用；课程详情按当前页不同课程去重调用。后续若管理端数据量或调用频率明显增加，再扩展课程服务批量目录接口或缓存。

## 管理端修改问题隐藏状态（2026-07-17）

- 请求方式：`PUT`
- 请求路径：`/admin/questions/{id}/hidden/{hidden}`
- 路径参数：`id` 为问题 ID；`hidden=true` 表示隐藏问题，`hidden=false` 表示取消隐藏。
- Controller 使用显式 `@PathVariable("id")`、`@PathVariable("hidden")`，只负责接收参数并委托 Service。
- Service 校验问题 ID 必须大于 0、隐藏状态不能为空、问题必须存在；校验通过后在事务中更新 `interaction_question.hidden`。
- 管理端接口不做普通用户的提问者归属校验，管理员身份与权限由认证授权链路负责。
- 用户端查询仍需过滤 `hidden=true` 的问题；管理端分页查询可以查看隐藏问题。

## 新增回答或评论（2026-07-17）

- 请求方式：`POST`
- 请求路径：`/replies`
- 请求字段：`questionId`、`content`、`anonymity`、可选 `answerId`、可选 `targetReplyId`。
- `answerId` 为空时表示直接回答问题：新增回复后原子递增问题 `answer_times`，并更新 `latest_answer_id`。
- `answerId` 非空时表示评论某条顶级回答：校验该回答属于当前问题，新增后原子递增顶级回答的 `reply_times`。
- `targetReplyId` 用于回复一条具体评论；服务端校验其归属，并从该回复推导 `target_user_id`，不接收客户端自报的目标用户 ID，避免伪造回复对象。
- 当前回答人统一从 `UserContext` 获取；接口文档中的 `isStudent` 没有对应数据库字段，且身份不能由客户端声明，因此不作为入参落库。
- 问题、上级回答或目标回复不存在、已隐藏或归属不匹配时，拒绝创建。

## 管理端修改回答或评论隐藏状态（2026-07-17）

- 请求方式：`PUT`
- 请求路径：`/admin/replies/{id}/hidden/{hidden}`
- `id` 是回答或评论 ID；`hidden=true` 表示隐藏，`false` 表示取消隐藏。
- 接口校验 ID、隐藏状态和回复是否存在，然后更新当前回复的 `hidden` 字段。
- 当目标是一级回答（`answerId` 为 `null` 或 `0`）且执行隐藏时，使用批量更新将其下全部评论同步隐藏。
- 取消隐藏一级回答时，不自动恢复其评论，避免将此前被单独审核隐藏的评论误公开。
- 一级回答判断需同时兼容 `answer_id = 0` 的数据库默认值和可能出现的 `null`。

## 课程分类本地缓存优化（2026-07-17）

- 场景：管理端问题分页需要拼接课程三级分类名称；原实现每次请求都通过 Feign 调用分类服务获取全量分类。
- 方案：复用 `tj-api` 的 `CategoryCache`，在 `tj-learning` 启动类添加 `@EnableCategoryCache`；分页组装分类名称时改为读取 `categoryCache.getCategoryMap()`。
- 缓存实现：Caffeine 本地缓存，键为固定的 `CATEGORY`，值为 `分类ID -> 分类信息` 映射，写入后 30 分钟过期。
- 收益：同一学习服务实例中，缓存有效期内全量分类只需一次远程调用；后续分页请求直接读取内存映射。
- 边界：这是本地缓存，各实例会在 TTL 内最终一致；若分类需要实时生效，应由课程服务在分类变更后发布失效通知，再清理各实例本地缓存。

## 2026-07-17：互动问答文本内容审核（阿里云）

- 审核范围：提问标题/描述、回答、评论；所有用户可见文本都必须经过同一审核入口。
- 状态机：`PASS`（直接展示）、`PENDING`（疑似违规或阿里云调用失败，落库但 `hidden=true`，等待人工审核）、`REJECT`（明确违规，抛业务异常且不落库）。这是一种 fail-closed（故障保守）策略。
- 领域数据中单独记录 `auditStatus`、`auditReason`、`auditTime`，不要只复用 `hidden`：`hidden` 表示展示状态，审核状态表示为什么不可展示，二者职责不同。
- 管理员将待审核内容取消隐藏时，同步写为人工审核通过；隐藏一级回答时，级联隐藏其评论。
- 阿里云凭据放在 Nacos `shared-aliyun.yaml`，由需要审核能力的服务显式导入。共享凭据应配合 Nacos 权限控制，并使用最小权限 RAM 用户，禁止把 AK/SK 写入代码、日志或知识库。
- 历史数据迁移默认审核通过，避免上线后存量内容被误隐藏；新增字段必须通过 SQL 迁移脚本部署。
## 阿里云文本内容审核接入流程（互动问答）

### 1. 需求与策略

审核范围：问题标题、问题描述、回答、评论。标题与正文要合并审核，避免只审核正文导致标题成为绕过入口。

| 阿里云结果 | 系统动作 | 数据状态 | 前台展示 |
| --- | --- | --- | --- |
| 正常/无风险 | 正常保存 | `PASS` | 立即展示 |
| 疑似风险、普通风险 | 正常保存 | `PENDING` | `hidden=true`，等待人工审核 |
| 明确违规 | 拒绝请求，不保存 | `REJECT`（不落库） | 返回“内容包含违规信息，无法发布” |
| 调用超时、网络异常、响应异常、未配置 | 正常保存 | `PENDING` | `hidden=true`，等待人工审核 |

这里采用 **fail-closed（故障保守）**：审核能力失效时，宁可暂不展示，也不能让未审核内容直接暴露。

### 2. 阿里云控制台与权限

1. 在阿里云内容安全开通文本审核，并选择适合 UGC 场景的审核服务。
2. 创建 RAM 用户，例如 `tj-learning-content-moderation`。
3. 为该 RAM 用户授权内容安全所需权限；遵循最小权限，避免使用主账号 AccessKey 或 `AdministratorAccess`。
4. 创建 RAM 用户的 AccessKey。`AccessKey Secret` 只在创建时展示一次，不能放进 Git、日志、截图或知识库。
5. 当前项目使用上海地域：`region-id: cn-shanghai`、`endpoint: green-cip.cn-shanghai.aliyuncs.com`。服务编码以阿里云控制台实际开通项为准；本项目配置为 `ugc_moderation_byllm_pro`。

### 3. Nacos 共享配置

在 Nacos 的 `public` 命名空间建立 `shared-aliyun.yaml`，由需要阿里云能力的服务显式导入：

```yaml
aliyun:
  content-moderation:
    endpoint: green-cip.cn-shanghai.aliyuncs.com
    region-id: cn-shanghai
    service-code: ugc_moderation_byllm_pro
    access-key-id: ${你的RAM用户AccessKeyId}
    access-key-secret: ${你的RAM用户AccessKeySecret}
    connect-timeout-ms: 3000
    read-timeout-ms: 5000
```

`tj-learning/src/main/resources/bootstrap.yml` 已导入该共享配置。共享配置方便其他微服务复用，但应通过 Nacos 权限控制限制读取范围；不要让无关服务导入含凭据的配置。

### 4. 代码结构

- `AliyunContentModerationProperties`：绑定 Nacos 配置。
- `AliyunContentModerationConfiguration`：创建阿里云 SDK `Client`。
- `IContentAuditService`：审核抽象，业务代码不直接依赖 SDK。
- `AliyunContentAuditServiceImpl`：调用阿里云并把标签映射为 `PASS/PENDING/REJECT`。
- `ContentAuditStatus`：审核领域状态。
- 问题、回答表：`audit_status`、`audit_reason`、`audit_time` 保存审核痕迹。

`hidden` 与 `auditStatus` 不应混为一谈：前者表示是否展示；后者说明审核结论及原因。管理员取消隐藏待审核内容时，系统记录“人工审核通过”。

## 2026-07-17 联调修复：阿里云 UGC 大模型审核接口与成功标签

### 现象

发布一条正常问题后，数据库记录为：

```text
hidden = 1
audit_status = 0
audit_reason = 审核服务响应异常
```

第一次问题来自接口类型不匹配：项目配置的服务是 `ugc_moderation_byllm_pro`（UGC 文本审核大模型专业版），但代码错误地调用了旧版 `TextModeration` 接口并按 `labels/reason` 结构解析。

### 根因与修复

| 项目 | 错误实现 | 正确实现 |
| --- | --- | --- |
| SDK 操作 | `TextModeration` | `TextModerationPlus` |
| 请求类型 | `TextModerationRequest` | `TextModerationPlusRequest` |
| SDK 调用 | `textModerationWithOptions()` | `textModerationPlusWithOptions()` |
| PLUS 风险结果 | 误按 `data.labels` 解析 | 按 `data.result[]` 解析每个 `label` |
| 无风险结果 | 未处理 | `nonlabel` 表示未命中风险，应映射为 `PASS` |

阿里云真实返回 `labels=nonlabel` 时，说明审核调用已经成功且内容无风险。修复后将 `nonlabel`（并兼容旧版 `normal`）映射为：

```text
hidden = 0
audit_status = 1
audit_reason = ''
```

### 排障方法

1. 不要只根据“审核服务响应异常”猜权限问题，先记录并查看 SDK 的 HTTP 状态、阿里云业务 `code`、`message`、`requestId`。
2. 第三方服务的“服务编码、API 操作、响应模型”必须作为同一套契约核对；服务编码正确，不代表可以套用旧 API。
3. 接入后一定做真实的正常文本联调。编译通过只能验证 Java 类型，不代表服务端协议、标签语义正确。
4. 已在错误版本下保存的待审核记录不会自动变为通过；应由管理端人工审核取消隐藏，或在确认无风险后进行受控数据修复。
