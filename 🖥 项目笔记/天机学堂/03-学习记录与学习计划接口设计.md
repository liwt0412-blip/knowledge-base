# 学习记录与学习计划接口设计

## 一、接口总览

| 接口 | 作用 | 关键约束 |
|---|---|---|
| `GET /learning-records/course/{courseId}` | 查询当前用户指定课程的学习进度 | 先查当前用户课表；未加入返回 `null`；记录按更新时间倒序 |
| `POST /learning-records` | 提交视频或考试学习记录 | 校验课表归属和有效期；仅第一次完成时累计进度 |
| `POST /lessons/plans` | 创建或更新学习计划 | 当前用户 + 有效课程；写入 `week_freq` 和计划进行中状态 |
| `GET /lessons/plans` | 分页查询学习计划 | 返回本周汇总、每门课程计划进度，并批量补充课程信息 |

## 二、核心数据关系

```text
courseId
   ↓
learning_lesson.id（lessonId）
   ↓
learning_record.lesson_id
```

`learning_record` 没有 `course_id`，因此学习记录按课表 ID 归属。每门计划本周完成数应按 `lesson_id` 分组，并用 `LearningLesson.id` 匹配。

## 三、提交学习记录的关键规则

1. `lessonId` 必须属于当前登录用户。
2. 课程状态不能是失效，有限期课程必须尚未到期。
3. 考试提交即完成；视频达到约定观看比例后完成。
4. 只有小节从“未完成”变成“已完成”时，`learned_sections` 才能增加。
5. 重复提交已完成小节不能重复累计。
6. 视频播放位置只允许前进，乱序请求不能让进度回退。
7. 全部小节完成后把课程状态更新为已学完。

## 四、查询学习计划的字段来源

| 返回字段 | 来源 |
|---|---|
| `weekFreq` | `learning_lesson.week_freq` |
| `learnedSections` | `learning_lesson.learned_sections` |
| `latestLearnTime` | `learning_lesson.latest_learn_time` |
| `weekLearnedSections` | 本周完成记录按 `lesson_id` 分组统计 |
| `weekFinished` | 当前用户本周完成的小节总数 |
| `weekTotalPlan` | 进行中计划的 `week_freq` 求和 |
| `courseName` | 批量调用课程服务补充 |
| `sections` | 批量调用课程服务补充 |
| `weekPoints` | 当前工程没有积分数据源，明确返回 0，不能编造规则 |

## 五、为什么课程信息需要远程调用

学习服务只保存课程 ID 和学习领域数据，不保存课程名称、封面、课程总小节数。查询当前页计划后，提取全部 `courseId`，复用批量课程查询方法：

```text
当前页课表
  → courseId集合
  → CourseClient批量查询一次
  → 转换为Map
  → 补充courseName和sections
```

## 六、周统计的边界

本周统一采用：

```text
[本周一 00:00, 下周一 00:00)
```

SQL条件：

```sql
finish_time >= #{weekBegin}
AND finish_time < #{nextWeekBegin}
```

避免使用相邻周期都会包含边界的时间范围。

## 七、当前实现与课堂演示版本的取舍

- 当前实现使用 `IdAndNumDTO` 承接聚合结果，不向 PO 添加一次性 `num`、`totalWeekfreq` 字段。
- 当前实现校验课表归属和有效期，防止伪造 `lessonId` 修改其他用户数据。
- 当前实现对小节完成做幂等控制；课堂演示版的考试重复提交可能重复新增和累计。
- 课堂演示版可以复用一次分组结果计算总数，SQL 次数更少；当前实现更适合把全局统计与当前页统计范围分开。

## 八、已知问题与后续优化

- `duration=0` 的前后端契约需要统一：前端应取得视频总时长后再上报。
- 每页计划的分组统计可以传入当前页 `lessonIds`，减少无关聚合。
- `addLessons()` 遇到课程服务返回空时不能只记日志后继续，应明确抛出业务异常。
- 课程学习人数是否排除失效课表，需要产品统一统计口径。
- 并发完成同一小节时，最终应配合数据库唯一约束或条件更新防止重复累计。
