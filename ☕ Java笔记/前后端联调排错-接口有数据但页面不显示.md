# 前后端联调排错：接口有数据但页面不显示

> 来源：天机学堂积分页"流水有、接口响应有、页面显示 0"故障（[[🖥 项目笔记/天机学堂/10-tj-promotion积分服务实现]]，2026-07-20）。

## 排查顺序（确定后端无 bug 前不要动后端）

1. **先确认写入链路真的成功**：日志里的 INSERT/消费记录只是单点证据，要交叉验证——例如幂等键 SETNX 在落库之前执行，落库成功即可反推缓存层可达，无需单独猜"是不是 MQ/Redis 的问题"。
2. **直接看接口响应**：绕过页面调一次接口（或看浏览器 Network 响应体）。响应数据正确 ⇒ 后端无责，问题在前端渲染。
3. **看页面的匹配逻辑**：页面不是直接渲染接口数组时，前端往往写死一张配置表再按某个字段（名称/编码）匹配填数——匹配字段不一致就落默认值，且**恰好同名的行会正常显示**，极具迷惑性。

## 两条契约教训

- **按名称匹配是脆弱契约**：后端展示名（desc）是给人看的，不应承担匹配键职责。长期解法是让前端按数组顺序或稳定编码取值；短期对齐时改 desc 比改编码安全——编码一旦被数据库/缓存数据引用，改动即数据事故。
- **接入旧版/参考代码先核对枚举编码映射**：同一业务两套枚举（如旧版 签到=2/问答=3/笔记=4 vs 新版 1/4/5）会让同一张表的数据被张冠李戴。对照旧代码、前端写死值、SQL 注释任何一处之前，先列出两边"编码 ↔ 含义"全量映射表。

## 适用条件与失效边界

- 适用：前后端分离项目中"数据链路各环节都正常、唯独展示不对"的联调问题。
- 失效边界：若接口响应本身错误，回到后端排查；本清单不替代后端自证。

## 验证方法

修复后必须端到端验证：页面刷新后展示值与接口响应、数据库流水三者一致，而不仅是"页面有数了"。
