# 课表删除、退款 MQ 与数据一致性

## 一、先按业务触发方式拆分入口

删除课表课程通常有两类触发方式，入口不同，但应复用同一领域删除能力：

| 场景 | 触发入口 | 核心约束 |
|---|---|---|
| 用户手动删除 | HTTP `DELETE` 接口 | 只能删除当前用户自己的已失效课程 |
| 退款自动删除 | 退款成功后的 MQ 消息 | 只删除该退款订单对应的课程权益 |

不要让 Controller 直接操作 Mapper；Controller 负责接收请求，Service 负责权限、有效性和删除规则，监听器只负责消费消息和调用 Service。

## 二、手动删除：权限与失效状态缺一不可

删除条件至少应包含：

```java
userId = 当前登录用户
courseId = 路径参数课程 ID
expired = status == EXPIRED
       || (expireTime != null && expireTime <= now)
```

为什么同时判断 `status` 和 `expireTime`：有些系统通过定时任务把到期课程更新为 `EXPIRED`，但定时任务可能尚未执行。只依赖状态会让“已经到期但状态未刷新”的课程无法删除。

删除不到记录时，应明确返回“课程不存在或仍在有效期内”，不能把仍有效课程当作删除成功。

## 三、接口查询：本地字段不远程调用

若接口只返回课表本地字段，例如：

```text
id、courseId、学习状态、已学小节数、购买时间、过期时间、学习计划状态
```

直接查询 `learning_lesson` 即可。为了这些字段远程调用课程服务，只会增加网络延迟、服务耦合和故障点。

只有接口需要课程领域数据时，才批量远程补充，例如课程名称、封面、总小节数、上下架状态、讲师信息。

## 四、退款自动删除：以“退款成功”作为事件边界

退款申请、审批通过、调用支付渠道，都不等于退款成功。只有支付渠道确认退款成功后，交易服务才应发布退款事件；学习服务消费事件后删除课表。

```text
退款成功
  → 交易服务更新退款状态
  → 发布 order.refund 事件
  → 学习服务消费消息
  → 删除对应课表权益
```

这样避免退款失败或仍在处理中时，用户先失去学习权限。

## 五、MQ 消费者的最低要求

- 队列使用持久化声明，交换机、路由键、队列名使用统一常量。
- 先校验消息中的用户 ID、课程集合等必要字段；校验失败后必须 `return`，不能记录日志后继续处理。
- 删除操作必须幂等：消息重复投递、人工补发或重试时，记录已经不存在也不能报错。
- 为成功、空删和异常情况记录带业务主键的日志，便于对账。
- 监听器只做消息校验和服务调用，不承载复杂业务规则。

## 六、最重要的边界：按订单明细关联，不要只按用户和课程删除

如果系统允许同一用户重复购买同一门课程，仅以 `userId + courseId` 删除会有误删风险：退款旧订单时，可能删掉新订单产生的有效课表。

可靠设计：

1. `learning_lesson` 保存 `orderDetailId`（必要时同时保存 `orderId`）。
2. 支付成功事件和退款成功事件均携带 `orderDetailId`。
3. 新增课表、查询权益、退款删除都以 `orderDetailId` 精确关联。

若现有表和消息都没有订单明细 ID，只能实现“按用户 + 课程”的兼容方案；必须在设计评审或代码注释中明确其重复购买限制，不能误以为它在所有场景都正确。

## 七、接口与代码表达规范

- 对外 HTTP 接口使用语义正确的注解，例如删除使用 `@DeleteMapping`。
- Swagger 至少补齐 `@ApiOperation`，路径或请求参数需要说明时使用 `@ApiParam`。
- Service 的复杂流程使用简短编号注释，说明业务步骤、权限边界和为何这样判断；不要为显而易见的赋值逐行写废话注释。
- 返回对象使用专用 VO/DTO，不直接暴露持久化对象，避免泄露内部字段。

## 八、完成前检查清单

- [ ] 用户无法删除他人的课表。
- [ ] 有效课程无法被手动删除。
- [ ] 已到期但尚未更新状态的课程可以删除。
- [ ] 退款失败、退款处理中不会删除课表。
- [ ] 退款成功消息重复投递不会失败。
- [ ] 消息无效时记录日志后立即结束处理。
- [ ] 若允许重复购买，删除条件能精确定位订单明细。
- [ ] 接口、Service、MQ 消费路径均有编译或集成测试验证。
