# 分库分表选型：演进阶梯与 ShardingSphere-JDBC

> 来源：天机学堂积分/学习模块的存储规划讨论（2026-07-21）。通用结论，不依赖具体项目。

## 第一原则：分库分表是止痛药，不是保健品

单表未到瓶颈（经验线：1000 万+ 行 或 写入 QPS 成为瓶颈）不要提前分片。按阶梯逐级演进，每一级都比下一级便宜：

```text
索引优化 → 归档/定期清理 → MySQL 原生分区表 → 读写分离 → 分库分表
```

- **归档/清理**：周期性数据（月度流水、日志表）按周期归档到历史表或直接清理，主表永远攒不大；
- **分区表**：按时间 RANGE 分区的表，周期清理用 `DROP PARTITION`（秒级元数据操作）代替 `DELETE 全表`（大量 undo log、锁表风险）——"按月清零"类需求的正解；
- **读写分离**：读多写少先加从库，成本远低于分片；
- 以上都压不住，才进入分库分表。

## 选型结论：Java 体系优先 ShardingSphere-JDBC

| 方案 | 评价 |
|---|---|
| **ShardingSphere-JDBC** | Apache 顶级项目，国内 Java 分库分表事实标准；客户端模式（jar 增强 DataSource），无额外部署、性能损耗小；MyBatis-Plus/MyBatis 无感；可渐进接入（先单库分表再分库） |
| MyCAT / Cobar（Proxy 模式） | 需独立部署代理层，运维重、社区式微，不推荐新建项目使用 |
| ShardingSphere-Proxy | 多语言接入或 DBA 统一管理场景才考虑，纯 Java 服务没必要多一层网络跳转 |
| TiDB / PolarDB-X（NewSQL） | "换数据库底座"思路：透明扩缩容、免分片设计；适合从零选型或大体量重构，不适合在既有 MySQL 体系里渐进改造 |

## 落地的三个核心设计题（比框架选择更重要）

1. **分片键选择**：按"绝大多数查询携带的过滤条件"选。用户中心数据按 user_id 顺理成章；当一个实体同时按两个维度高频查询（如问答表按 course_id 查列表、按 user_id 查"我的"）——用**基因键**（在 course_id/订单号里嵌入 user_id 片段，如雪花 id 低位替换）让单键路由两类查询，或接受一类查询走广播表/绑定表。分片键选错，后面全是跨库笛卡尔查询。
2. **中间件与事务框架的叠加**：ShardingSphere 和 Seata AT 都会包装 DataSource，裸装会互相冲突——必须用 ShardingSphere 5.x 的 Seata 集成模式，接入前写 Spec 验证事务链路。
3. **接口设计约束前置**：跨分片事务、深翻页分页（`LIMIT 1000000, 20` 要逐分片归并）都是分片后的昂贵操作——"查询必须携带分片键"要在接口契约阶段就写死，而不是上线后补救。

## 适用边界

- 适用：Java/Spring 技术栈、MySQL 底座、单表数据量或写入量确定性超标的业务；
- 失效边界：数据量其实没到瓶颈（回阶梯上一级）；团队没有中间件运维能力且可从零选型（直接评估 NewSQL）；强全局一致 + 高频跨分片事务的业务（分片本身就不是正确答案）。

## 面试表述建议

被问"怎么做分库分表"时，阶梯式回答优于单点回答：先讲"没到量级不分"的判断标准，再讲逐级手段（重点提 `DROP PARTITION` 这个细节，区分度高），最后讲 ShardingSphere-JDBC 的分片键/基因键/Seata 集成三个设计点——体现的是选型权衡能力而非背框架名。
