# 高频最新状态的 Redis 合并写模式

## 一、第一性原理

高并发优化不是先选择 Redis 或 MQ，而是先回答三个问题：

1. 哪个分支真正高频？
2. 每次状态变化都必须保留，还是只需要最终最新值？
3. 哪些状态允许延迟落库，哪些状态必须同步成为最终事实？

如果业务只关心最新状态，例如播放位置从 `20 → 35 → 50 → 65`，中间值没有审计价值，就可以把多次数据库写合并为一次 `65`。这种模式称为“最新状态合并写”。

核心原则：

```text
高频且可覆盖的数据进入缓存合并
关键状态变化同步写入数据库
数据库保存最终事实
缓存同时匹配写入和读取访问方式
```

## 二、适用边界

| 适合 | 原因 |
|---|---|
| 视频/音频播放位置 | 只需要最新播放点 |
| 电子书阅读位置 | 中间翻页通常不需要逐次落库 |
| 草稿自动保存 | 新版本可以覆盖旧版本 |
| 用户最后在线时间 | 只关心最新时间 |
| 设备最新状态、车辆最新位置 | 查询通常关心当前状态 |
| 任务进度百分比 | 较大的新进度可覆盖旧进度 |

| 不适合          | 原因           |
| ------------ | ------------ |
| 账户余额变化       | 每次变动都必须可追溯   |
| 库存扣减         | 不能覆盖中间扣减事件   |
| 支付、退款、订单状态流水 | 需要审计和幂等事件    |
| 操作日志、审计日志    | 中间事件本身就是业务事实 |

判断标准不是“更新是否频繁”，而是“丢掉中间状态会不会破坏业务事实”。

## 三、先定位热点分支

同一个接口可能包含多个频率完全不同的分支：

```text
首次创建记录：低频
普通状态更新：高频
首次完成：低频但关键
查询：读请求
```

只优化高频普通更新，不把首次创建、首次完成等关键分支机械异步化。优化范围越精确，一致性成本越低。

## 四、让数据结构同时匹配读写模式

如果写入按“单个子项”发生，读取按“整个聚合”发生，可以使用 Redis Hash：

```text
Key：业务聚合ID
HashKey：子项ID
HashValue：子项最新状态
```

示例：

```text
Key：learning:record:{lessonId}

42 -> {id:1, moment:242, finished:true, version:8}
43 -> {id:2, moment:20,  finished:false, version:3}
```

对应访问：

```text
更新单项：HGET/HSET sectionId
读取整体：HGETALL lessonId
```

一个数据结构同时优化写入前读取和整体查询，比为读、写分别维护两份缓存更简单。

## 五、部分缓存不能冒充完整缓存

写请求可能只把一个子项放进 Hash。此时 `HGETALL` 得到的是“热数据”，不一定是数据库中的全部记录。

需要增加完整加载标记：

```text
__loaded -> 1
```

读取规则：

```text
存在 __loaded
  → HGETALL 直接返回

不存在 __loaded
  → 查询数据库全部记录
  → 与Redis中可能存在的更新值合并
  → 批量回填Hash
  → 写入__loaded=1
```

数据库空结果也要写完整标记，避免缓存穿透。回填时不能用数据库旧值覆盖 Redis 中尚未落库的新值，播放位置等单调字段应取较大值。

## 六、合并写与刷盘状态分离

读缓存和“是否尚未落库”是两个概念：

```text
Hash：保存可供读取的最新状态
ZSet/消息：标记需要刷盘的记录
```

例如：

```text
Hash
learning:record:{lessonId}

ZSet
learning:record:dirty
lessonId:sectionId -> flushAt
```

刷盘成功后清除 dirty 标记，但保留 Hash 作为读缓存。若刷盘成功就删除缓存，下一次查询仍会回源数据库，读优化会失效。

## 七、并发更新必须保证状态不回退

“先读取、Java 判断、再写入”不是原子操作。两个并发请求可能让旧进度覆盖新进度。

对于单调递增字段，应通过 Lua、CAS 或数据库条件更新保证：

```text
newMoment > oldMoment 才更新
```

刷盘数据库时同样防止回退：

```sql
UPDATE learning_record
SET moment = GREATEST(COALESCE(moment, 0), #{moment})
WHERE lesson_id = #{lessonId}
  AND section_id = #{sectionId};
```

缓存中可以保存 `version`。刷盘线程读取版本 `v` 后，只有当前版本仍等于 `v` 才能清除 dirty 标记；版本已变化说明刷盘期间又收到了新状态，需要再次安排刷盘。

## 八、关键状态不能只写缓存

播放位置可以延迟落库，但“首次完成”“扣减”“结算”等状态必须同步写数据库。

推荐流程：

```text
普通状态更新
  → Redis合并
  → 延迟刷盘

首次关键状态变化
  → 数据库条件更新
  → 根据影响行数判断是否首次
  → 同一事务更新关联状态
  → 事务成功后更新或失效缓存
```

数据库条件更新是最终幂等边界，Redis 锁不能替代唯一约束和条件更新。

## 九、刷盘调度方案选择

| 方案 | 优点 | 风险与适用范围 |
|---|---|---|
| JVM `DelayQueue` | 简单、延迟低 | 重启丢任务，多实例协调困难，只适合单实例学习或原型 |
| Redis ZSet | 最新状态和调度都在 Redis，容易恢复 | 需要租约、失败重试和监控，适合状态合并写 |
| MQ逐条消费并写库 | 可靠传输、削峰 | 不减少数据库写次数 |
| MQ消费后写Redis再批量刷盘 | 可靠且能合并 | 链路复杂，适合确有规模和可靠性要求的系统 |
| Redis写状态，MQ发刷盘信号 | 能合并且可异步 | Redis成功、MQ失败存在窗口，需要ZSet或扫描兜底 |

对于“只关心最新值”的状态，Redis Hash + dirty ZSet通常比保存每一个事件更贴合数据语义；关键完成事件可以在数据库事务成功后发布 MQ，供积分、通知、排行榜等下游使用。

## 十、缓存生命周期与故障恢复

- TTL 必须大于最大刷盘延迟、重试时间和故障恢复窗口。
- 每次有效更新刷新 TTL。
- dirty 记录没有对应 Hash 时要能清理。
- 多实例刷盘使用短期租约，实例宕机后其他实例可以接管。
- Redis 不可用时要明确降级为直接写库、限制上报或返回失败，不能静默丢数据。
- 删除主业务数据、退款或失效时同步清理缓存与 dirty 任务。
- Redis 持久化策略不能替代数据库最终事实。

## 十一、验证收益，不编造数字

实施前后使用相同口径观察：

- 播放进度接口 QPS 与延迟；
- 数据库 `UPDATE` 次数和写 QPS；
- Redis 命中率；
- dirty 数量与最老任务等待时间；
- 批量刷盘大小、耗时和失败次数；
- 数据库与缓存进度差异；
- 重复完成和进度回退次数。

没有监控或压测证据时，只能表述“设计上减少重复写”，不能宣称具体性能收益。

## 十二、实现检查清单

- [ ] 是否只优化了真正高频且可覆盖的分支？
- [ ] Hash 的聚合边界是否与主要读取方式一致？
- [ ] 是否能区分部分缓存与完整缓存？
- [ ] 并发请求是否可能让状态回退？
- [ ] 普通状态和关键状态是否采用不同持久化策略？
- [ ] 刷盘失败、实例重启和多实例竞争是否可恢复？
- [ ] 刷盘后是否错误删除了仍需用于读取的缓存？
- [ ] 删除、失效和退款是否同步清理缓存？
- [ ] 是否有数据库唯一约束或条件更新作为最终防线？
- [ ] 是否用真实指标验证了收益？

## 十三、可复用的工程判断

### 13.1 Redis中“看不到Key”不等于数据没有写入

Redis客户端通常会按冒号前缀分组显示。例如：

```text
learning (2)
```

只是表示当前存在两个以 `learning:` 开头的真实Key，不是一个永久目录。需要用真实Key检查：

```redis
TTL learning:record:{aggregateId}
TYPE learning:record:{aggregateId}
ZRANGE learning:record:flush 0 -1 WITHSCORES
```

Hash只剩最后一个field时，删除该field会导致整个Hash自动消失；ZSet的成员被刷盘成功后移除，ZSet为空时也会消失。这是Redis数据结构的正常行为，不应把命名空间是否显示当成业务数据是否存在的判断依据。

### 13.2 观察缓存必须同时看三件事

合并写方案排错时，不能只看Hash：

```text
Hash：当前可读取的最新状态
ZSet：是否存在待刷库任务
MySQL：最终持久化事实
```

如果Hash不存在但MySQL已有最新数据，可能是TTL过期或完成后主动失效；如果ZSet仍有member，说明还有待刷库任务；如果两者都不存在，则应回到接口日志、事务提交结果和数据库记录判断，而不是直接认定缓存写入失败。

### 13.3 延迟刷库Key必须有可观测性

调度member至少包含聚合ID和子项ID，例如 `lessonId:sectionId`；刷盘日志要带member、recordId、moment、任务score、处理结果和重试次数。这样才能回答“数据在哪个Key、什么时候应该刷库、为什么没有被消费”。

该排错方法可复用于草稿自动保存、设备最新状态、最后在线时间、任务进度等所有“Hash保存最新值 + ZSet调度刷盘”的场景。

## 十四、同一高频业务要按数据语义拆分链路

同一个操作可能同时产生三类数据，不能因为它们都来自一次请求就使用相同的合并策略：

| 数据语义 | 示例 | 能否合并 | 推荐处理 |
|---|---|---|---|
| 当前关系或状态 | 用户是否收藏、是否订阅、是否点赞 | 同一主体的重复目标状态可幂等 | Redis 原子维护，数据库保存可恢复的最终关系 |
| 聚合最新值 | 收藏数、在线人数、浏览热度、设备最新指标 | 可以，只保留最新绝对值 | dirty 标记 + 短窗口合并发布/批量刷盘 |
| 行为事实 | 谁在何时收藏、取消、关注、点击 | 通常不能，通知、审计和分析需要每次真实变化 | 逐事件写可靠 outbox，再异步投递 MQ |

例如用户快速执行“关注 → 取消 → 再关注”，最终关系只需要“已关注”，聚合数只需要最新绝对值，但行为分析可能需要三次真实变化。正确设计不是在“全部合并”和“全部逐条处理”之间二选一，而是按数据语义拆成不同通道。

### 14.1 原子边界要覆盖状态、计数和待投递标记

如果状态变化、计数增减、版本推进、dirty 标记和 outbox 写入由多条独立 Redis 命令完成，进程可能在中间宕机，留下“状态已变但没有待投递任务”等半完成数据。可以通过 Lua 将同一业务变化的 Redis 操作合成一个原子边界：

```text
校验当前状态
→ 幂等改变关系
→ 更新绝对计数
→ version + 1
→ 写 dirty
→ 追加 Stream/outbox 事件
```

Lua 只能保证 Redis 内部原子性，不能跨 Redis、RabbitMQ、MySQL 提供分布式事务，因此仍需要可靠转发和补偿。

### 14.2 Broker confirm 不等于目标队列收到消息

RabbitMQ publisher confirm 只能证明 broker/交换机接收了消息；如果 routing key 没有匹配任何队列，消息仍可能无法到达消费者。必达消息不能只依赖 confirm：

- 启用 mandatory 与 returns callback，识别不可路由消息；
- 将可选广播事件与必达计数/持久化事件采用不同发送策略；
- 必达事件在确认“已接收且可路由”前不得删除 dirty/outbox；
- 修改消费者绑定后必须重启消费者，并在管理端核对实际 exchange、routing key、queue 绑定；
- 使用对账或补偿任务作为最终安全网。

页面乐观更新、关系表成功写入、某个队列无积压，都只能证明局部链路，不能替代端到端验证。

### 14.3 实时处理与运维补偿应分工

秒级状态同步适合常驻监听或短周期任务；分钟级重试、对账、死信巡检适合调度平台。不要为了统一技术而让实时消息泵完全依赖低频调度：

```text
常驻监听/短周期任务：实时转发、最新值聚合
XXL-Job等调度平台：Pending重试、对账、死信和人工补偿
```

多实例环境下，调度频率本身不能保证一致性，仍需消费组、租约/分布式锁、幂等键和版本判断。
