# 兑换码设计：签名防伪与三层防爆破

> 来源：天机学堂 tj-promotion 优惠券兑换码设计（[[🖥 项目笔记/天机学堂/11-tj-promotion优惠券服务实现]]）。
> 配套：[[Bitmap标志位存储模式-连签统计与验重]]（验重位图的选型框架）。本文讲**生成与防伪**，防重复兑换用位图。

## 业务六约束（先背下来，方案都是倒推出来的）

可读性（≤10 字符、限定 32 字符集）、10 亿级容量、唯一、不可重兑、防猜测爆刷、生成验证不压 DB。

## 生成方案：自增 id + 签名 + Base32（MAC 思想，与 JWT 同构）

```
自增id(32位) → 按4位分8组 → 密钥S盒非线性替换 → 加权求和 → 签名(14位)
payload = (id << 14) | signature（46位） → Base32 → 定长10位码
```

- **唯一性**：自增 id（DB 号段/Redis 计数器分配）+ 一一对应的变换，数学保证，不依赖算法运气。
- **防伪**：签名 = f(id, 密钥)，密钥不泄露则无法伪造；篡改后验签必然不符。验签在**进 Redis/DB 之前**拒绝非法码，零存储成本。
- **为什么不用 MD5/RSA**：签名 128 位起步，塞进 10 位码不可能。按位加权签名可任意截短（14 位够用——防的是伪造不是碰撞）。
- **容量**：32¹⁰ ≈ 10¹⁵，远大于 10 亿需求；32⁶ ≈ 10.7 亿是下限，位数由"id+签名"的位预算决定。

### 纯加权签名的短板与 S 盒加固（重要取舍）

纯加权求和是**线性函数**：8 组权重 = 8 个未知数，收集 8 组码→id 对应关系列方程组可解出权重。加固：**加权之前每组先查密钥 S 盒（16 项置换表）做非线性替换**，方程组解法失效，签名长度不变。强度定位：混淆+实用级，非密码学级——课程/中小业务够用，高价值场景升级 Speck32 分组密码或 HMAC 截断签名。

## 三层防爆破（生产级答案，编码层只是第一层）

核心认知：**任何码都能被"试"，防爆破主力在请求侧——让试错成本高到不划算**。

| 层 | 手段 | 作用 |
|---|---|---|
| ① 编码层 | 签名防伪（本方案）/ 真随机码+大空间稀释（32¹⁰ 空间发几千个码，枚举命中率≈0）/ HMAC 截断签名 | 猜不到、伪造的码零成本拒绝 |
| ② 请求侧（主力） | Sentinel 按 userId+IP 限流；Redis INCR+TTL 失败 N 次锁定兑换 X 分钟（Lua 原子）；滑块/人机校验；每用户每日兑换上限 | 暴力枚举迅速自我锁死，机器成本陡增 |
| ③ 架构监控 | 布隆过滤器前置挡"肯定不存在"的码；兑换失败率突增告警；高价值券二次验证 | 兜底与态势感知 |

## 不可重兑：位图原子占位（承接验重模式）

- 键 `coupon:code:status:{couponId}`，offset = 码 id − 号段起始 id（号段稠密是前提，由分配策略保证）。
- `SETBIT` 返回旧值即判重：一次调用完成"查+占"，并发安全，无需 GETBIT+SET 两步、无需 DB 行锁。
- 位图是加速层，DB 仍是最终事实；Redis 故障回退 DB 乐观锁；重启后可从 DB 重建；TTL = 券有效期 + 缓冲期。
- 管理端"分页看码列表"仍走 MySQL——位图干不了条件筛选的活（见四问框架）。

## 适用边界

- S 盒/权重必须外置配置或 KMS，硬编码只是教学实现。
- int 号段上限 21 亿，超大规模换 long 并重新做位预算。
- 签名防伪 ≠ 防重兑 ≠ 防爆破，三件事各由一层负责，不能互相替代。
