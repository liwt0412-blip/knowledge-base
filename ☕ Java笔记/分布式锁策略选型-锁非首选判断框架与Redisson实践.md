# 分布式锁策略选型-锁非首选判断框架与Redisson实践

> 来源：天机学堂 tj-common `@Lock` 声明式锁实现与多模块用例评审（支付、对账、月结）；面试应用见 [[💼 面试/本人面试准备/Java+AI面试作战手册/03-Java微服务追问/分布式锁与幂等-石化天机双项目面试三层回答]]（双向链接）。
> 适用条件：任何"多实例/多线程需要对同一业务键互斥执行一段逻辑"的场景。不满足"互斥执行一段逻辑"这个前提的，先按第一节阶梯判断，大概率不需要锁。

## 一、锁非首选：判重与互斥的选型阶梯

按代价从低到高判断，能用低级手段就不用高级手段：

1. **单资源判重** → Redis 原子操作返回值（SETNX、SETBIT 返回旧值）或 DB 唯一键。无锁、单次往返；
2. **数值防超扣** → 乐观锁 / 条件更新（version、`WHERE status=旧值`），数据库自己保证；
3. **一段逻辑必须互斥** → 分布式锁（盘点隔离、定时任务多实例互斥、短窗口防重提交）；
4. **资金级强一致** → 不依赖锁：唯一键 + 状态机 + 对账兜底。锁在 Redis 主从切换时可能丢失（RedLock 争议场景），锁只解决互斥效率，正确性永远要有下层兜底。

- 判断口诀：如果这一级失效结果仍然正确，就用这一级；不正确，说明它只是优化，下一级才是裁决者。
- 验证：故意绕过本级（删掉 Redis key、制造并发重试），断言结果仍不双加、不超扣。

## 二、锁粒度按业务键收敛

锁名必须覆盖且只覆盖真实的冲突维度。

- 做法：SPEL 按方法实参拼 key（如 `pay:order:#{#dto.bizOrderNo}`），不同业务单互不阻塞；
- 反例一：全局一把锁——把并行度打回串行，锁住了不该互斥的请求；
- 反例二：粒度细到等于请求 id——等于没锁，冲突双方拿的不是同一把；
- 验证：写出"哪两个请求同时执行会出错"，这两个请求的锁 key 必须相同；不相干的两个请求 key 必须不同。

## 三、拿不到锁的行为是策略，不是写死的

选型依据是"拿不到锁时，这次执行还有没有价值、后果谁承担"：

| 策略 | 语义 | 典型场景 |
|---|---|---|
| 静默跳过 | 不等待，当没发生过 | 定时任务/对账多实例互斥（本就该只跑一份） |
| 快速失败 | 不等待，抛"请求太频繁" | 用户连点、防重提交（要给调用方明确信号） |
| 等待后跳过 | waitTime 内重试，超时跳过 | 可容忍短等但可放弃的常规业务 |
| 等待后失败 | waitTime 内重试，超时抛异常（常用默认） | 调用方需要明确失败结果的常规业务 |
| 死等 | 直到拿到锁 | 必须执行的补偿、结算步骤 |

- 取舍：策略应该收敛成枚举/注解参数统一收口，散在各处的 `tryLock` 手写分支迟早写出"忘了释放"或"该失败时静默跳过"的 bug；
- 验证：双实例并发打同一业务键，断言行为符合所选策略（只一个进入 / 另一个收到明确失败）。

## 四、锁时长：看门狗续期与显式 leaseTime 的分工

- 默认不传 leaseTime → Redisson 30s 初始 + watchdog 每 10s 续期，业务没执行完锁不提前释放；进程死亡后看门狗停，锁自然到期释放；
- 关闭自动释放（autoUnlock=false、跨方法持锁）时**必须显式给 leaseTime**，否则持锁进程死亡即死锁——这类约束应在注解/框架层强校验，靠约定一定会被违反；
- 失效边界：watchdog 续期依赖锁服务存活，网络分区 + 主从切换下锁可能双持；所以第四节和第一节第 4 级是同一个结论——强一致不能只靠锁；
- 验证：kill 持锁进程，断言锁在 leaseTime/看门狗周期后释放，另一实例能继续；业务超时执行时断言锁被续期未提前释放。

## 配套经验

- 临界区里不做远程调用、不放大持锁时间；锁内只放"必须互斥的最小一段"。
- 声明式注解（AOP + SPEL + 策略枚举）优于手写 try/finally tryLock：语义统一收口，减少散点错误。
- 锁与幂等是正交的两件事：锁防"同时"，幂等防"重复"——MQ 重复消费用幂等键/唯一键，不要用锁，见 [[事件驱动积分发放的幂等、防冤扣与重放清单]]。
- 判重优先级同样适用于"能不能用原子操作就不用锁"：签到 SETBIT、计数 Lua、状态条件更新，都是无锁解法。
