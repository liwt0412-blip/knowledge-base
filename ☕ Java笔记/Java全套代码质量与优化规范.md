# Java 全套代码质量与优化规范

## 第一性原则

> Java 代码优化不是堆叠设计模式或微优化，而是在保证正确性和安全性的前提下，用尽可能简单的结构实现业务，使代码容易理解、修改、测试和运行；性能问题必须根据证据优化。

判断优先级：

```text
正确性与安全
  -> 业务边界与可测试性
  -> 可读性与可维护性
  -> 复用与扩展
  -> 有证据的性能优化
```

当规则冲突时，靠前目标优先。禁止为了少几次 `new`、少几行代码或使用某个设计模式，破坏正确性和可读性。

---

## 一、四级执行体系

### 1. 必须遵守：正确性与安全

- 外部输入、客户端参数和模型生成参数默认不可信，必须在后端校验。
- 权限校验不能只依赖前端、Prompt、DTO字段或模型参数。
- 不吞异常，不写空 `catch`，失败必须有明确语义。
- IO、连接、流等资源必须可靠关闭，优先使用 try-with-resources。
- 事务边界清晰，不在长事务中执行不可控的远程调用。
- 消息消费者按照可能重复投递设计，关键写操作必须幂等。
- 共享可变状态必须有明确并发策略。
- 日志不得泄露密码、Token、密钥和完整敏感业务数据。
- SQL、文件路径、反序列化和远程调用必须考虑注入与越权风险。
- 修改必须进行与风险相匹配的测试或验证。

### 2. 默认采用：可维护性

- 字段默认 `private`，不可变字段优先 `final`。
- 使用构造器注入，避免业务代码直接 `new` 基础设施依赖。
- 保持高内聚、低耦合，组合优先于继承。
- 用有业务含义的方法保护对象状态，不机械生成全部 setter。
- 使用提前返回减少无意义嵌套。
- 方法保持单一、清晰的目标，不混合多个抽象层级。
- 对外只暴露完成用例所需的最小接口。
- 集合返回空集合而不是 `null`，不直接暴露内部可变集合。
- Controller负责协议适配，Service负责用例协调，Repository/Mapper负责持久化。
- 优先沿用项目已有、合理且一致的架构与命名。

### 3. 代码异味：发现后判断是否重构

- 方法过长，无法快速说清主流程。
- 参数过多，且一组参数经常一起出现。
- 多层 `if-else` 或大量类型分支。
- 相同业务语义重复实现。
- 一个类依赖过多组件或承担多个变化原因。
- 一个方法同时处理校验、SQL、远程调用、转换和通知。
- 到处传递 `Map<String, Object>` 或字符串状态。
- 公共工具类持续膨胀，开始包含业务逻辑和外部依赖。
- 为一个实现创建没有隔离价值的接口、工厂和抽象层。

代码异味是调查信号，不是机械判罚标准。不要只根据“50行、4个参数、重复2次”等数字决定重构。

### 4. 测量后优化：性能

以下优化必须有慢SQL、APM、Profiler、GC日志、线程Dump、监控或压测依据：

- 集合初始容量。
- 对象创建和复用。
- 字符串临时对象。
- 缓存及其TTL。
- 锁实现和锁粒度。
- 并行流、异步化和线程池参数。
- SQL索引和批量大小。
- JVM和GC参数。

优化前记录基线，优化后用同一口径复测；不能验证收益的优化不得宣称有效。

## 二、面向对象与依赖设计

### 职责与内聚

- 一个类围绕一组高度相关的职责组织，并尽量只有一个主要变化原因。
- 一个方法完成一个可命名的目标。
- 复杂逻辑按业务语义拆分，避免拆出 `doStep1()`、`handleData()` 这类无意义方法。

### 抽象与接口

- 对外依赖抽象，内部隐藏实现，但只在存在真实变化点时创建抽象。
- 多实现、第三方隔离、测试替换或跨模块契约是创建接口的合理理由。
- 只有一个稳定实现且没有隔离价值时，不机械创建 `XxxService` + `XxxServiceImpl`。
- 抽象类适合共享状态和模板流程；接口适合表达能力与契约。

### 继承与组合

- 组合优先于继承。
- 继承必须满足明确的“is-a”关系和可替换性。
- 避免多层继承树，防止父类变化影响全部子类。
- 横切能力优先使用组合、装饰器、策略或事件，而不是堆入万能父类。

### 访问权限与常量

- 使用满足需求的最小访问权限：`private` 优先，其次包级、`protected`、`public`。
- 工具类不允许实例化时使用私有构造器，或直接改为更合适的组件。
- 有稳定业务含义、跨位置复用的值抽为常量或枚举。
- 不把只出现一次且含义清楚的局部字符串机械提升为全局常量。

完整封装规则见 [[☕ Java笔记/Java编码规范与面向对象封装]]。

## 三、方法与控制流

### 方法拆分

- 主方法应能清楚呈现业务流程。
- 将解析、校验、计算、持久化、通知等不同抽象层级分开。
- 是否拆分取决于职责和可读性，不以固定行数作为唯一标准。

### 参数设计

- 参数较多且属于同一业务概念时，使用有明确语义的参数对象。
- 不为了减少参数数量创建没有含义的 `XxxParamDTO`。
- 参数对象进入业务层后应尽量稳定，避免在多层之间不断修改。
- 不使用 `boolean` 参数隐藏两种完全不同的行为，优先拆方法或使用枚举。

### 提前返回

优先处理异常和边界情况，使主流程保持扁平：

```java
public Result process(Order order) {
    if (order == null) {
        return Result.invalid("订单不存在");
    }
    if (!order.canProcess()) {
        return Result.rejected("订单状态不允许处理");
    }
    return execute(order);
}
```

### 重复代码

- 先判断重复逻辑的业务语义和变化原因是否相同。
- 同一业务概念的重复应抽取；偶然长得相似但会独立变化的代码不要强行合并。
- 不以“出现两次/三次”作为硬性重构门槛。

## 四、集合、字符串与对象创建

### 集合选择

- 通用顺序集合默认优先 `ArrayList`。
- 去重使用 `HashSet`；保持插入顺序使用 `LinkedHashSet`；排序使用 `TreeSet`。
- 通用键值存储使用 `HashMap`；保持插入顺序使用 `LinkedHashMap`；按键排序使用 `TreeMap`。
- 不因“频繁增删”就默认选择 `LinkedList`，必须结合定位成本、遍历方式和基准测试。
- 已知数据规模且处于热点路径时，可设置合理初始容量，但不机械预估所有集合。

### 循环

- 禁止在循环中无控制地逐条查询数据库或调用远程接口，优先批量查询并建立内存索引。
- 业务结果对象可以正常在循环中创建，不得为了避免 `new` 复用可变对象造成数据污染。
- 删除遍历中的元素使用 `Iterator.remove()`、`removeIf()` 或生成新集合。
- 并发修改问题取决于迭代和修改方式，不是简单禁止所有循环内修改。
- `ArrayList.size()` 通常是常量时间，不做没有证据的微优化。

### 字符串

- 少量固定拼接可以使用 `+`，编译器会处理简单场景。
- 循环或大量动态拼接使用 `StringBuilder`。
- `StringBuffer` 只用于确实需要共享可变缓冲区的并发场景。
- `split`、正则和临时字符串只有在热点路径或大数据处理中才需要专项优化。
- 字符串判空优先使用项目已有规范，不为判空引入不必要依赖。

### 包装类与内存

- 高频数值计算优先基本类型，避免无意义装箱拆箱和包装类空指针。
- 不根据 `Integer` 缓存范围改变业务建模。
- 局部变量离开作用域后通常无需手动置 `null`。
- 长生命周期对象不应长期持有已无用的大集合或大数组。
- 单例用于表达生命周期和资源共享，不作为普通的“少new对象”优化手段；Spring Bean默认已是单例作用域。

## 五、异常、日志与资源

### 异常分层

- 业务异常表达可预期的规则失败，例如库存不足、状态不允许。
- 系统异常表达数据库、网络、序列化等非业务故障。
- 深层代码捕获能够处理的具体异常。
- Controller Advice、任务调度器、消息消费者和线程入口等系统边界可以捕获 `Exception` 兜底，但必须记录上下文并产生明确失败结果。
- 不使用异常代替正常分支判断。

### 日志

- 日志记录能够定位问题的业务键、`traceId`、阶段和结果。
- 异常日志保留堆栈，不只打印 `e.getMessage()`。
- 避免同一异常在多层重复打印。
- 使用参数化日志，不在禁用日志级别时提前进行昂贵字符串拼接。
- 敏感字段脱敏。

### 资源管理

- IO流、JDBC资源等使用 try-with-resources。
- 使用连接池时仍应按时关闭连接，`close()` 通常表示归还连接池。
- 禁止将连接、流或事务跨越不受控的异步边界。

## 六、数据库与远程IO

- 查询只返回业务需要的字段，尤其避免宽表和大字段无条件 `select *`。
- 是否允许 `select *` 结合迁移脚本、映射稳定性和查询场景判断，不作脱离场景的形式主义检查。
- 使用分页、索引和执行计划处理大数据查询。
- 避免N+1查询：先批量查询，再按ID建立Map聚合。
- 批量写入控制批次大小、事务范围、失败重试和幂等。
- 远程调用设置超时，并根据幂等性决定是否重试。
- 不在数据库事务中长时间等待第三方接口。
- 缓存必须明确一致性、失效、穿透、热点和权限维度。
- 性能问题先看SQL与IO，再考虑集合和对象级微优化。

---

## 七、并发与异步

- 线程由受管理的线程池创建，禁止在业务代码中随意 `new Thread()`。
- 线程池必须明确用途、容量、队列、拒绝策略、上下文传播和关闭方式。
- 优先消除共享可变状态，再考虑加锁。
- `volatile` 保证可见性和部分有序性，不保证复合操作原子性。
- `Atomic` 适合单变量原子更新，不能自动保护多个状态组成的不变量。
- `synchronized` 或 Lock 用于保护临界区与复合不变量。
- 锁范围只覆盖必要状态，但不能为了缩小锁而破坏原子性。
- `ReentrantReadWriteLock`、`StampedLock` 和无锁结构必须结合竞争情况压测选择。
- 异步任务必须处理超时、取消、异常、幂等和上下文传递。
- 不默认认为 ThreadLocal 会跨线程池传播。

## 八、工具类、组件与复用

- 无状态纯函数可以放入小型工具类。
- 有配置、外部资源、重试、监控或生命周期的能力应做成 Spring Bean。
- 有业务语义的能力使用业务组件、Client、Gateway或Adapter，不放进万能 `Utils`。
- 第三方HTTP接口优先封装为 `PaymentClient`、`UserCenterClient` 等明确组件，而不是不断扩展 `HttpUtils`。
- 公共父类只承载稳定、真正共享的行为，不作为代码垃圾桶。
- 不跨业务边界抽取看似通用、实际语义不同的逻辑。

## 九、设计模式使用边界

| 模式 | 适用场景 | 不应使用 |
|---|---|---|
| 策略 | 业务类型会扩展、分支算法可替换 | 只有两三个稳定且简单的分支 |
| 模板方法 | 主流程稳定、部分步骤由子类变化 | 需要自由组合步骤或继承关系不自然 |
| 工厂 | 创建过程复杂或实现选择需要隔离 | 只是隐藏一行简单 `new` |
| 适配器 | 隔离第三方接口和内部模型 | 内外模型已经一致且不会变化 |
| 观察者/事件 | 一个业务动作触发多个相对独立后续动作 | 需要强一致、同步返回的核心步骤 |
| 装饰器 | 动态叠加职责 | 简单逻辑可以直接表达 |
| 单例 | 生命周期确实唯一且无共享状态风险 | 仅为了减少对象创建 |

原则：设计模式用于解决已经出现的结构问题，不用于展示技术栈。

## 十、分层、命名与可读性

- 类名、方法名和变量名表达业务含义，避免 `a`、`data1`、`handle()` 等模糊命名。
- 常量使用大写下划线，类型与方法遵循Java惯例。
- 注释解释“为什么”和业务约束，不重复代码字面含义。
- 废弃代码交给版本控制，不长期保留大段注释代码。
- Controller不直接编写复杂业务和数据库逻辑。
- Service不应成为包含所有规则的万能类；复杂领域规则应下沉到合适对象或组件。
- Repository/Mapper不负责业务决策。
- 转换逻辑复杂时使用专门Assembler/Mapper，但简单转换不必过度分层。

## 十一、日常开发优化顺序

1. 先保证行为正确、权限安全、事务和异常边界清晰。
2. 消除明显重复和无法理解的命名。
3. 拆分职责混乱的方法，减少无意义嵌套。
4. 修正循环查库、重复远程调用和N+1查询。
5. 保护对象状态，减少无约束setter和可变数据暴露。
6. 在真实变化点引入接口、策略、适配器或事件。
7. 通过测试确保重构不改变行为。
8. 使用监控、慢SQL、Profiler或压测定位真实性能瓶颈。
9. 优化后使用相同口径复测。

## 十二、Codex执行检查清单

### 开发前

- [ ] 先读取项目 `AGENTS.md`、现有代码风格和测试方式。
- [ ] 明确业务规则、异常情况、权限边界和数据规模。
- [ ] 判断任务是简单CRUD、复杂领域逻辑还是性能问题。

### 开发中

- [ ] 使用最简单、清晰且满足当前需求的设计。
- [ ] 不为假设需求提前增加接口、设计模式、缓存或异步化。
- [ ] 避免循环查库、无界重试、无界队列和隐式共享状态。
- [ ] 重要状态变化具有业务语义和校验。
- [ ] 对外接口最小化，内部实现不无意义公开。

### 完成前

- [ ] 编译、静态检查和相关测试通过。
- [ ] 检查异常、日志、事务、并发和权限边界。
- [ ] 性能结论有证据，不能把推测写成收益。
- [ ] 说明重要取舍及未覆盖风险。
