# day05-MQ面试练习-逐句修正记录

> Hermes 逐句拷问，四轮渐进：概念 → 机制 → 陷阱 → 设计。

## 第一轮：概念层 — 消息不丢失三角色保证

### 问题
RabbitMQ 里，消息不丢失从三个角色各自的视角，靠什么机制保证？

### 我的原始回答
> 生产者主要是做一个从事（confirm）机制和一个回执机制，MQ中间件做的是一个持久化，包括交换机队列以及消息的持久化。消费者也是做的一个回执以及从事（重试）策略。

### 逐句修正

| 我说的 | 正确说法 |
|--------|---------|
| "从事机制" | **confirm**（康-佛-姆），生产者确认 |
| "回执机制"（生产者端） | confirm 确认到达 Broker + **return** 处理不可路由退回，是两个独立回调 |
| "回执"（消费者端） | 消费者端是 **ack** / **nack**，不是"回执" |
| "从事策略" | **重试**（chóng shì） |

### 正确版本（30秒标准答案）

> 消息不丢失从三个角色保证：
> - **生产者端**：confirm 确认消息到达 Broker，return 处理不可路由退回
> - **Broker 端**：三层持久化——交换机 durable、队列 durable、消息 persistent，**缺一不可**（AND 关系）
> - **消费者端**：手动 ack + nack 重试 + 重试上限后进死信 + prefetch 限流防止本地堆积

### ⚠️ confirm 和 return 搞反了（反复犯错）

| | confirm | return |
|---|---|---|
| **触发条件** | 消息到达**交换机** | 交换机**找不到队列** |
| **含义** | "收到了" ✅ | "没地方投，退给你" ❌ |
| **confirm 返回值** | ACK（交换机在）/ NACK（交换机不存在） | 不返回 ACK/NACK——直接退回消息体 |

**核心**：交换机在但队列不在 → confirm 返回 **ACK**（交换机确实收到了）+ return 回调触发退回。confirm 只管"到没到交换机"，return 只管"有没有队列接"。

### 🪤 死信队列三个进入条件
1. 消息 TTL 过期
2. 队列满了
3. 消费者 reject/nack + `requeue=false`

**reject ≠ 进死信**，必须满足 `reject + requeue=false + 队列配了 DLX`。

## 第二轮：机制层 — 延迟队列（TTL + DLX）

### 问题
下单 30 分钟未支付自动取消，用 RabbitMQ 怎么做？

### 我的原始回答
> 需要开启定时任务，先将消息设置一个时间值存入死信交换机，然后设置不同的时间定期轮询。

### ❌ 全错
- 不需要定时任务
- 不需要轮询
- 消息不是直接发到死信交换机

### 正确方案

```
生产者（订单服务）→ 延迟队列（配 TTL=30min，指定 DLX）
                        │ 延迟队列无消费者，纯等
                        │ 30 分钟后 TTL 到期
                        ▼
              Broker 自动 → 死信交换机 → 取消队列
                                            │
                                            ▼
                              消费者（订单服务另一监听器）
                              查订单状态 → 未支付 → 取消
```

**关键**：
- 消息不直接发 DLX，发到**延迟队列**
- 投递动作是 **Broker 自己做的**，不是代码
- 延迟队列和取消队列是**两条不同的队列**
- 死信交换机**不存消息**——只是路由中转

## 第三轮：陷阱层 — 队头阻塞

### 问题
死信队列（TTL+DLX）能用 `rabbitmq_delayed_message_exchange` 插件替代吗？有什么区别？

### 核心考点：队头阻塞（Head-of-Line Blocking）

> TTL+DLX 的消息在**队列层**等待，队列是 FIFO。如果第一条 30 分钟、第二条 5 分钟，第二条会被堵在第一条后面。

| | TTL + DLX | Delayed Message 插件 |
|---|---|---|
| **延迟在哪** | 队列层 | **交换机层** |
| **FIFO 问题** | ❌ 队头阻塞 | ✅ 交换机内部按到期时间排序 |
| **灵活性** | 队列级 TTL | **每条消息**不同 `x-delay` |
| **维护成本** | 无需装插件 | 需装插件 |

### 面试标准答案

> TTL+DLX 的延迟在队列层，FIFO 导致队头阻塞。延迟交换机插件把延迟从队列提升到交换机——消息在交换机内部独立计时，到期才投到队列。每条消息可设不同的 `x-delay`，互不干扰。

## 第四轮：设计层 — Fanout + 原子性

### 问题
用户注册成功后，发短信、发邮件、送积分，三个任务怎么设计？

### 正确方案

> Fanout 交换机广播到短信、邮件、积分三个队列。

### 🚨 致命漏洞：写库和发 MQ 不是原子操作

写库成功但发 MQ NACK → 用户注册了但没收到验证码。

**解决**：开启 publisher-confirm，NACK 重试三次，还失败记日志 + 定时任务重试兜底。
