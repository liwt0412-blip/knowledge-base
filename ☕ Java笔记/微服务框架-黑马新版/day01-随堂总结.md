# 微服务 Day1 随堂总结

## 1. 认识微服务

### 1.1 单体架构

- **优点**：简单、部署方便，适合小型项目或访问量不大的项目
- **缺点**（前提：业务越来越复杂、用户量越来越多，代码量激增）：
  - 团队协作困难（冲突）
  - 代码边界模糊（编写重复功能）
  - 项目构建速度变慢
  - 项目可用性变差

### 1.2 微服务架构

- **概念**：是一种软件架构风格，由多个小型项目构成的大型复杂项目
- **特点**：
  - 粒度小，单一职责
  - 团队自治：各是各的团队
  - 服务自治：从开发、测试、部署、运维、数据库，都是独立的

### 1.3 SpringCloud

- 是经过了各大企业长久时间检验的一种功能强大的微服务技术栈
- **作用**：用来解决微服务拆分带来的各种问题
- 是很多优秀开源组件的集合体（都是解决微服务问题的）

## 2. 服务拆分

### 2.1 项目环境搭建

- 导入 CentOS（Docker）、安装 MySQL
- 前端环境：Nginx
- 后端环境：hmall

### 2.2 熟悉黑马商城业务

- 登录
- 用户
- 商品
- 购物车：查询购物车
- 订单：下单
- 支付：余额支付

### 2.3 拆分原则

- **原则**：高内聚、低耦合
- **方式**：根据**业务**纵向拆分，根据**公共功能**横向拆分

（具体拆分方案见 [[day01-微服务01]]）

### 2.5 远程调用

- **本质**：就是利用 Java 发起 HTTP 请求
- **方式**：
  - `HttpUrlConnection`（JDK 内部，不支持 HTTP 连接池）
  - `HttpClient`（Apache 的，支持 HTTP 连接池）
  - `RestTemplate`（Spring 自带的）

## 3. 服务治理

### 3.1 注册中心原理（重点）

1. **服务提供者**启动，注册服务至注册中心
2. **注册中心**保存提供者实例信息（服务名、IP、端口等）
3. **服务消费者**启动，调用时根据服务名去注册中心拉取提供者的地址列表，并保存在本地一份
4. 服务提供者运行期间基于**心跳机制**上报自己的健康情况：
   - 当心跳不正常时，注册中心会剔除服务，并将最新地址列表推送给消费者
   - 当提供者有新的实例注册到注册中心，注册中心也会将最新地址列表推送给消费者

> **注意**：
> - 服务提供者、消费者角色是相对的
> - 举例：家政中介

### 3.2 认识 Nacos

- 市面上主流的注册中心产品：
  - Alibaba 的 Nacos
  - Netflix 的 Eureka
  - SpringCloud 的 Consul
- 支持当做**注册中心**和**配置中心**

### 3.3 服务注册（两步）

1. 导注册中心依赖：`spring-cloud-starter-alibaba-nacos-discovery`
2. 加配置：`spring.cloud.nacos.server-addr: 192.168.150.101:8848`

### 3.4 服务发现（三步）

1. 导注册中心依赖：`spring-cloud-starter-alibaba-nacos-discovery`
2. 加配置：`spring.cloud.nacos.server-addr: 192.168.150.101:8848`
3. 远程调用时，使用 API 根据**服务名称**拉取地址列表，基于**负载均衡算法**挑选一个实例，发起访问

```java
// DiscoveryClient API
List<ServiceInstance> instances = discoveryClient.getInstances("item-service");
ServiceInstance instance = instances.get(RandomUtil.randomInt(instances.size()));
```
