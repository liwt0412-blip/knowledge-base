# day05 MQ高级 — 面试问答

> 从第一性原理出发，10 道题逐层深入。核心问题只有一个：**消息如何从 A 可靠送达 B，且 B 只处理一次。**

---

## 第一题：消息可靠性全链路

**问**：MQ 消息可能在哪几个环节丢失？用什么机制保障？

**答**：三个环节，每个环节都有对应的保障机制。

```
发送端 ──→ Broker ──→ 消费端
  │          │          │
重试+确认   持久化     ACK+重试
```

| 环节 | 丢失场景 | 保障机制 |
|---|---|---|
| 发送端 | 网络断、连不上MQ、路由失败 | 生产者重试 + Publisher Confirm + Publisher Return |
| Broker | MQ宕机，内存消息丢失 | 交换机/队列/消息持久化 + LazyQueue |
| 消费端 | 消费者拿到消息后崩了 | Consumer Ack + 本地重试 + 失败处理策略 |

> **第一性原理**：三个环节是**串联**的，一个断全链断。不能只管 Broker 持久化就觉得万事大吉。

---

## 第二题：生产者确认机制

**问**：Publisher Confirm 和 Publisher Return 有什么区别？什么时候用哪个？

**答**：

| | Publisher Confirm | Publisher Return |
|---|---|---|
| 触发条件 | 消息到 MQ 后的投递结果 | 消息到 MQ 但**路由失败** |
| 回执内容 | ACK（成功）/ NACK（失败） | 异常信息（exchange、routingKey、原始消息） |
| 用途 | 知道消息投递是否成功 | 排查路由错误 |

**关键细节**：
- 路由失败时会**同时**收到 Return（异常详情）和 Confirm ACK（投递成功）——因为消息确实到了 MQ，只是没找到队列
- 交换机名字写错 → 只收到 NACK，不会触发 Return
- `correlated` 模式是 MQ **异步回调**，不阻塞；`simple` 是同步等待

> **注意**：开启生产者确认**消耗 MQ 性能**，只有高可靠性场景才开，且只需处理 ConfirmCallback 的 NACK 即可。

## 第三题：持久化的代价

**问**：消息持久化是怎么做的？有什么代价？

**答**：

三层持久化：**交换机持久化（Durable）+ 队列持久化（Durable）+ 消息持久化（delivery_mode=2）**

**代价**：
1. 写磁盘 = IO 开销，性能下降
2. **不是逐条写**——MQ 约 100ms 批量刷盘一次，这 100ms 窗口内 MQ 宕机，消息还是丢
3. 开启 Confirm 后，MQ 在持久化完成后才发 ACK，所以 ACK 会有延迟 → 必须用异步 Confirm（correlated 模式）

> **第一性原理**：可靠性和性能是**零和博弈**。持久化 = 拿性能换可靠性。清楚自己要什么，不要两个都要。

## 第四题：LazyQueue

**问**：LazyQueue 解决什么问题？和普通队列的区别？

**答**：

**问题**：消息堆积 → 内存爆 → MQ 触发 PageOut（内存刷盘）→ 阻塞所有生产者请求。

**区别**：

| | 普通队列 | LazyQueue |
|---|---|---|
| 存储位置 | 默认内存 | **直接存磁盘** |
| 消费方式 | 内存中直接取 | 从磁盘加载到内存（懒加载） |
| 适用场景 | 消息量可控 | 可能大量堆积 |

RabbitMQ 3.12 之后 LazyQueue 已是默认。声明时加 `x-queue-mode: lazy`，或 `QueueBuilder.durable("name").lazy().build()`。

## 第五题：消费者确认——ACK/NACK/Reject 到底什么区别

**问**：消费者三种回执分别什么语义？SpringAMQP auto 模式下怎么判断？

**答**：

| 回执 | 语义 | 消息命运 |
|---|---|---|
| ACK | 处理成功 | 从队列删除 |
| NACK | 处理失败，重新投递 | 重新入队，再投一次 |
| Reject | 处理失败，**放弃** | 直接删除（或进死信） |

**SpringAMQP auto 模式的判断逻辑**：

| 异常类型 | 回执 | 原因 |
|---|---|---|
| `RuntimeException`（业务异常）| **NACK** | 可能下次就成功了（余额到账了、库存恢复了） |
| `MessageConversionException`（消息格式异常）| **Reject** | 重试一万次也解不了，不如丢弃 |

> **面试关键**：不能笼统说"消费异常会重试"，必须分清异常类型。业务异常 = nack，格式异常 = reject。

## 第六题：重试耗尽后的三种策略

**问**：消费者本地重试 3 次后都失败了，消息去哪了？三种策略分别是什么？

**答**：

| 策略 | 行为 | 适用场景 |
|---|---|---|
| `RejectAndDontRequeueRecoverer` | 丢弃消息 | 不重要的日志、通知 |
| `ImmediateRequeueMessageRecoverer` | 重新入队继续重试 | 几乎不用（可能死循环） |
| `RepublishMessageRecoverer` | 投到指定死信交换机，等人工处理 | **推荐**，高可靠性场景 |

**推荐做法**：
1. 定义 `error.direct` 交换机 + `error.queue` 队列
2. 配 `RepublishMessageRecoverer` → 失败消息进 error.queue
3. 运维定期查 error.queue，人工介入

> **第一性原理**：重试不是目的，最终把消息处理掉才是目的。重试耗尽后消息不能消失——必须有一条退路。

## 第七题：幂等性——为什么异步通信必须有

**问**：为什么异步通信必须有幂等性？怎么实现？

**答**：

**为什么必须有**：
异步 = 发完就走 = 没有"成功/失败"反馈 → 不知道对方收没收到 → **必然重试** → **必然可能重复消费**。这不是 bug，是异步通信的**固有约束**。

**两种方案**：

| 方案 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| 唯一消息 ID | 每条消息带 UUID，消费前查库判重 | 通用，任何业务都能用 | 需要额外建表 |
| 业务状态判断 | `WHERE id=? AND status=1` 原子更新 | 零额外存储 | 只适用于状态流转场景 |

**必须注意的坑**：先查再改是两步，有并发风险。正确写法是**一条 SQL 原子化**：

```sql
UPDATE `order` SET status=2 WHERE id=? AND status=1
```

如果用 MyBatis-Plus：`lambdaUpdate().set(...).eq(Order::getId, id).eq(Order::getStatus, 1).update()`

## 第八题：兜底方案的设计思想

**问**：MQ 消息可靠性做到极致，为什么还需要兜底方案？怎么设计？

**答**：

**为什么需要**：三个环节的保障机制都有概率失败——生产者确认有 100ms 刷盘窗口、Broker 可能宕机、消费者可能崩。多道防线叠加后失败概率极低但**不是零**。

**兜底方案的设计思路**：从 **push 切到 pull**。

```
MQ 通知（push）          定时任务轮询（pull）
     ↓                          ↓
  主路径                    兜底路径
  实时性好                   最终一致性保证
```

**实现**：交易服务每 20 秒定时查支付状态，发现已支付就更新订单。即便 MQ 通知彻底失败，最多 20 秒后状态也能一致。

> **第一性原理**：没有 100% 可靠的单通道。双通道（push + pull）才是最终一致性的正确解法。

## 第九题：死信+TTL 延迟消息的坑

**问**：用死信交换机 + TTL 实现延迟消息有什么问题？

**答**：

**核心问题**：RabbitMQ 只在消息到**队首**时才检查 TTL。

```
队列：[msg1(TTL=5s), msg2(TTL=1s), msg3(TTL=1s)]
                    ↑
              只有队首的 msg1 被检查
              msg2 即使 TTL 只有 1 秒，也要等 msg1 过期后才能被处理
```

**后果**：延迟时间**不准确**，消息堆积时误差可能很大。

**替代方案**：延迟消息插件（`rabbitmq_delayed_message_exchange`），但只适合**短延迟**（秒/分钟级），长延迟（小时级）该用定时任务。

## 第十题：串联全貌——从第一性原理串起来

**问**：用一句话串起 day05 的所有知识点。

**答**：

MQ 的本质是异步管道。管道断在发送端 → **重试+确认**；断在 Broker → **持久化+LazyQueue**；断在消费端 → **ACK+重试+死信兜底**。异步没有反馈 → 必然重试 → 必须**幂等**。所有机制加起来仍有概率失败 → **定时任务 pull 兜底**。延迟消息是用 TTL 在时间维度上调度，但有精度缺陷，长延迟回归定时任务。

## 踩坑清单（自检用）

- [ ] 生产者重试是阻塞的 → 高性能场景要禁用或异步
- [ ] ConfirmCallback 只处理 NACK 就行，ReturnCallback 是排查路由用的
- [ ] 持久化不是逐条的，有 100ms 窗口
- [ ] ACK/NACK/Reject 语义不同 → 取决于异常类型，不能笼统说"会重试"
- [ ] 重试耗尽后消息不能消失 → 走 RepublishMessageRecoverer 进死信
- [ ] 幂等的坑：先查后改是两步，要用 WHERE 条件原子化
- [ ] 死信+TTL 的延迟不准 → 队首检查机制
- [ ] 延迟消息插件不适合长延迟 → 小时级用定时任务
