# 微服务 Day4 随堂总结

## 1. 事务基础

### 1.1 概念

将一组数据库的增删改操作视为一个最小单元，要么一起成功，要么一起失败。

### 1.2 ACID 特性

| 特性 | 说明 | 例子/原理 |
|---|---|---|
| **原子性** (Atomicity) | 事务是不可分割的操作单元，要么全部执行成功，要么全部失败回滚 | 转账操作中，扣款和收款必须同时成功或失败。通过 Undo Log 实现回滚 |
| **一致性** (Consistency) | 事务执行前后，数据库必须从一个一致状态转移到另一个一致状态 | 转账前后，双方账户总额应保持不变。依赖原子性、隔离性、持久性共同保障 |
| **隔离性** (Isolation) | 事务之间相互隔离，防止数据不一致，有 4 种隔离级别 | 见下表 |
| **持久性** (Durability) | 事务提交后，对数据的修改是永久性的，即使系统故障也不丢失 | 通过 Redo Log 或写入磁盘保障 |

**隔离级别：**

| 级别 | 说明 | |
|---|---|---|
| 读未提交 (Read Uncommitted) | 可能读到脏数据 | |
| 读已提交 (Read Committed) | 避免脏读，但可能不可重复读 | Oracle 默认级别 |
| 可重复读 (Repeatable Read) | 避免脏读、不可重复读，可能幻读 | MySQL 默认级别（MVCC） |
| 串行化 (Serializable) | 完全隔离，性能最低 | |

## 2. Spring 事务

### 2.1 使用方式

- **编程式事务**：通过编码方式控制事务的开启、提交、回滚，比较灵活，但对业务代码有侵入
- **声明式事务**：通过 `@Transactional` 注解声明事务

### 2.2 核心组件

| 组件 | 说明 |
|---|---|
| `TransactionManager` | 事务管理的核心接口 |
| `TransactionDefinition` | 定义事务属性：隔离级别、传播行为、超时时间、只读 |

### 2.3 声明式事务原理：AOP

1. 前置通知：开启事务
2. 返回后通知：提交事务
3. 异常通知：回滚事务

### 2.4 传播行为

| 传播行为 | 说明 |
|---|---|
| **REQUIRED**（默认） | 当前有事务则加入，没有则新建 |
| **REQUIRES_NEW** | 总是新建事务，挂起当前事务（独立提交/回滚） |
| SUPPORTS | 当前有事务则加入，没有则以非事务运行 |
| NOT_SUPPORTED | 以非事务运行，挂起当前事务 |
| MANDATORY | 必须在事务中运行，否则抛异常 |
| NEVER | 必须在非事务中运行，否则抛异常 |
| NESTED | 嵌套事务（基于保存点实现），外部回滚会导致内部回滚，但内部可独立回滚 |

### 2.5 失效场景

1. 异常被 try-catch 捕获且未往外抛出
2. 业务方法抛出的异常与 `@Transactional` 的 `rollbackFor` 声明的异常不匹配（默认为 RuntimeException）
3. 同类中没有事务的方法调用有事务的方法
4. 方法被非 public 权限修饰符修饰
5. 方法被 final 修饰
6. 设置了错误的传播行为
7. 多线程环境

## 3. 分布式事务

### 3.1 概念

一个业务横跨多个微服务或多个数据源，每个微服务都有自己的分支事务，要保证所有分支事务一起成功或失败。

### 3.2 解决思路

引入第三方协调者，记录每个分支事务执行结果，分二阶段：
- **一阶段**：执行分支事务
- **二阶段**：协调者引导一起提交或回滚

### 3.3 Seata

阿里巴巴提供的一套开源的优秀分布式事务解决方案。

**角色：**

| 角色 | 全称 | 职责 |
|---|---|---|
| **TM** | Transaction Manager（事务管理器） | 定义全局事务的范围、开启/提交/回滚全局事务 |
| **TC** | Transaction Coordinator（事务协调者） | 维护全局和分支事务的状态，协调全局事务提交或回滚（需先搭建） |
| **RM** | Resource Manager（资源管理器） | 管理分支事务，与 TC 交谈以注册分支事务和报告分支事务的状态 |

**微服务集成 Seata 步骤：**

1. 导入依赖：`spring-cloud-starter-alibaba-seata`
2. 加入配置：告诉 RM 和 TC 的位置
3. 指定数据源代理模式：XA / AT / TCC / SAGA
4. 全局事务的入口标记 `@GlobalTransactional` 注解

### 3.4 模式对比（都是二阶段提交）

#### XA 模式

**一阶段：**
1. TM 开启全局事务，通知 TC
2. TM 调用分支事务 RM
3. RM 分支事务将自己注册到 TC
4. RM 执行本地 SQL，**但不提交**
5. RM 将本地事务执行结果告知 TC

**二阶段：**
1. TM 结束全局事务，通知 TC
2. TC 核查每一个分支事务的执行结果：
   - 都成功 → 一起提交本地事务
   - 有一个失败 → 全部回滚

| 优点 | 缺点 |
|---|---|
| 强一致性、实现简单 | 执行完事务不提交，相互等待产生资源阻塞，效率低；依赖关系型数据库才能实现 |

---

#### AT 模式

**一阶段：**
1. TM 开启全局事务，通知 TC
2. TM 调用分支事务 RM
3. RM 分支事务将自己注册到 TC
4. RM 执行本地 SQL，**直接提交**，且提交**前后**记录快照日志（undo_log 表）
5. RM 将本地事务执行结果告知 TC

**二阶段：**
1. TM 结束全局事务，通知 TC
2. TC 核查每一个分支事务的执行结果：
   - 都成功 → 删除 undo_log 快照
   - 有一个失败 → 根据 undo_log 的前置快照恢复数据，删除快照

| 优点 | 缺点 |
|---|---|
| 没有资源阻塞，效率高；不依赖关系型数据库 | 弱一致，最终一致性 |
