# Seata 分布式事务 · 面试问答

> 2026-07-03 Hermes 逐题考核记录。6 题 + 追问，含原始回答、得分、修正版答案、必背总结。

## 题目与回答

### 第 1 题：分布式事务概念（6→8/10）

**问**：什么叫分布式事务问题？一个下单业务跨了订单服务、库存服务、购物车服务三个微服务，各自有数据库——为什么各自加 `@Transactional` 不够用？

**原始回答（6/10）**：各个微服务的事务相互独立，`@Transactional` 只对自己生效。

**修正版**：

> 分布式事务问题就是：一个业务跨多个微服务操作不同数据库，但 `@Transactional` 只能管自己微服务的本地事务，管不了别人。比如下单业务——订单服务创建订单、购物车服务清空购物车、库存服务扣减库存。每个服务的本地事务叫**分支事务**，整个业务叫**全局事务**。各自加 `@Transactional` 的话，库存扣减失败了订单照样提交、购物车照清空——ACID 的原子性和一致性全丢了。所以必须引入全局事务机制，让三个分支事务一起成功或一起失败。

### 第 2 题：TC / TM / RM 角色（5→8/10）

**问**：Seata 里 TC、TM、RM 三个角色分别是什么？各自干嘛的？

**修正版**：

| 角色 | 全称 | 一句话 |
|------|------|--------|
| **TC** | Transaction Coordinator | 事务协调者，独立部署的中间服务，维护全局和分支事务状态，决定最终提交还是回滚 |
| **TM** | Transaction Manager | 事务管理器，嵌在业务代码里，`@GlobalTransactional` 标注的方法就是 TM 的管辖范围 |
| **RM** | Resource Manager | 资源管理器，每个微服务里的，管理自己的分支事务，向 TC 注册和汇报 |

**口诀**：TC 是裁判，TM 是发令员，RM 是运动员。

### 第 3 题：XA vs AT 两阶段（4→8/10）

**问**：Seata 的 XA 模式和 AT 模式，各自的两个阶段做了什么？核心区别是什么？

**修正版**：

**XA 模式：**

| 阶段 | 做什么 |
|------|--------|
| 一阶段 | RM 向 TC 注册 → 执行业务 SQL → **不提交，锁住资源** → 报告状态给 TC |
| 二阶段 | TC 检查 → 都成功通知全体**提交**，有失败通知全体**回滚**（数据库原生回滚） |

**AT 模式：**

| 阶段 | 做什么 |
|------|--------|
| 一阶段 | RM 注册 → 拍改前快照(before-image) → 执行 SQL → 拍改后快照(after-image) → 存 undo_log → **直接提交**，释放锁 → 报告状态 |
| 二阶段 | 全部成功 → **删 undo_log**；有失败 → 拿 before-image **恢复数据** → 删快照 |

**核心区别：**

| 维度 | XA | AT |
|------|-----|-----|
| 一阶段 | 不提交，锁资源 | 提交 + 记 undo_log，不锁资源 |
| 回滚方式 | 数据库原生回滚 | undo_log 快照恢复 |
| 一致性 | 强一致 | 最终一致 |
| 性能 | 差（资源阻塞） | 好 |
| 适用 | 强一致场景 | 企业 90% 场景 |

### 第 4 题：@Transactional vs @GlobalTransactional（7/10）

**问**：`@Transactional` 和 `@GlobalTransactional` 有什么区别？什么时候用哪个？

**修正版**：

> `@Transactional` 是 Spring 的事务注解，管的是**单个数据库的本地事务**——一个服务、一个数据源，ACID 在本地闭环。
>
> `@GlobalTransactional` 是 Seata 的注解，标记的是**分布式事务的入口方法**。加上它之后，TM 会从这里开启全局事务，协调下游所有分支事务一起提交或回滚。
>
> 单服务用 `@Transactional`，跨服务用 `@GlobalTransactional`。而且用了 `@GlobalTransactional` 的方法里，下游的 `@Transactional` 还是要留着的——它们管各自的分支事务。

---

### 第 5 题：注解加在哪里（8/10）

**问**：`@GlobalTransactional` 应该加在哪个服务的方法上？

**修正版**：

> 加在全局事务的**入口方法**上，也就是最先发起整个业务流程的那个服务。以下单业务为例，由订单服务（trade-service）发起，所以加在 `OrderServiceImpl.createOrder()` 方法上。

### 第 6 题：AT 回滚全流程（5→7/10）

**问**：AT 模式，下单业务三个服务（订单创建成功、库存扣减失败、购物车被清空），数据怎么恢复？

**修正版**：

**一阶段：**

| 服务 | 动作 | 结果 |
|------|------|------|
| 订单 RM | 注册 → 拍快照（空） → `INSERT` 订单 → 拍快照（订单记录） → **提交** → 报告 TC：成功 | 数据库多了一条订单 |
| 购物车 RM | 注册 → 拍快照（有4条） → `DELETE` 购物车 → 拍快照（空） → **提交** → 报告 TC：成功 | 购物车被清空 |
| 库存 RM | 注册 → 拍快照 → `UPDATE` 库存 → 库存不足**抛异常** → 报告 TC：失败 | 数据没变 |

**二阶段：**

TC 发现有失败 → 通知订单和购物车回滚：

- 订单 RM：拿 before-image → `DELETE` 那条订单 → 删快照 ✅
- 购物车 RM：拿 before-image → `INSERT` 回 4 条购物车数据 → 删快照 ✅
- 库存 RM：本来就没提交，不用回滚

---

### 追问：undo_log（6/10）

**问**：AT 模式的快照存哪了？

**修正版**：

> 存在 `undo_log` 表里，每个参与全局事务的微服务数据库各有一张。AT 一阶段提交前，RM 自动把 before-image 和 after-image 写进这张表；二阶段成功就删记录，失败就拿 before-image 回滚。

## 必背答案（面试一句话版）

| 问题 | 一句话 |
|------|--------|
| XA vs AT 区别 | XA 一阶段不提交锁资源，强一致；AT 一阶段提交 + undo_log，最终一致 |
| AT 怎么回滚 | 一阶段拍快照存 undo_log，二阶段失败拿快照反着执行 SQL |
| @GlobalTransactional | 加在全局事务入口方法上，取代 @Transactional 做分布式事务标记 |
