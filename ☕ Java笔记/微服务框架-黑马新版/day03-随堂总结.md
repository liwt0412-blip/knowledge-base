# 微服务 Day3 随堂总结

## 1. 配置中心

### 1.1 作用

集中性地管理各个微服务的共享配置，并且实现**业务**配置的热更新。

### 1.2 配置中心化管理

将各个微服务的共享配置抽取成不同的文件，放到配置中心：

1. **中心化管理**：将 MyBatis、日志、Swagger、Feign 相关配置抽取成四个文件放到配置中心
2. **拉取配置**：
   - 导入依赖：`spring-cloud-starter-alibaba-nacos-config`、`spring-cloud-starter-bootstrap`
   - 新建 `bootstrap.yml`，存放服务名、配置中心地址、共享配置文件名、后缀
   - **原理**：项目启动时先加载 bootstrap 内容 → 拉取配置中心配置 → 和本地 `application.yaml` 合并 → 启动项目
   - 修改 `application.yaml`，去除已经共享的配置即可

### 1.3 配置热更新

- **作用**：实现修改配置而不用重启项目的效果（注意：热更新只能对**业务**配置生效）
- **方式**：
  1. `@ConfigurationProperties(prefix="xx.xx")`：直接就能实现热更新
  2. `@Value` + `@RefreshScope`（打在使用 `@Value` 的所在类上面）

### 1.4 动态路由（网关路由热更新）

1. 编写动态路由配置类
2. 将路由表 JSON 文件托管至配置中心

## 2. 服务保护

### 2.1 雪崩问题

- **概念**：由于某一个微服务故障导致的整个微服务链路都级联失败的现象
- **产生原因**：
  - 前提：微服务发生了相互调用
  - 原因 1：**服务提供者**发生了故障或阻塞（代码、网络、并发太高）
  - 原因 2：**服务调用者**没有做好故障隔离，导致自身被传染发生故障
  - 后果：调用链路服务级联失败，导致整个集群故障
- **解决思路**：
  - **服务提供者**：尽量避免故障（代码健壮、网络畅通、能抗高并发）
  - **服务调用者**：做好远程调用异常的后备方案，避免传染故障并扩散

### 2.2 解决方案

| 方案 | 目标角色 | 说明 |
|---|---|---|
| **请求限流** | 服务提供者 | 对接口访问的请求并发量进行限制，避免服务因流量激增出现故障（流量整形） |
| **线程隔离** | 服务调用者 | 通过限定每个业务能使用的线程数量将故障业务隔离，避免故障扩散（舱壁模式） |
| **失败处理（fallback）** | 服务调用者 | 给业务编写调用失败时的兜底逻辑。当调用出现故障时走 fallback，而不是直接抛异常 |
| **服务熔断** | 服务调用者 | 由断路器统计请求的异常比例或慢调用比例，超出阈值则熔断该业务，拦截该接口的请求。熔断期间所有请求快速失败，全走 fallback |

### 2.3 技术栈

- **Sentinel**（Alibaba）
- **Hystrix**（Netflix）

### 2.4 Sentinel

**概念**：阿里巴巴开源的一款微服务流量控制组件。

**核心两部分：**
1. Sentinel 客户端核心库（微服务导入依赖）
2. Sentinel 控制台，页面方式配置服务保护规则

**集成使用：**

1. **Sentinel 搭建**：
   ```bash
   java -Dserver.port=8090 -Dcsp.sentinel.dashboard.server=localhost:8090 \
        -Dproject.name=sentinel-dashboard -jar sentinel-dashboard.jar
   ```
2. **微服务集成**：
   - 导依赖：`spring-cloud-starter-alibaba-sentinel`
   - 加配置：
     ```yaml
     spring:
       cloud:
         sentinel:
           transport:
             dashboard: localhost:8090
     ```
3. 访问需要进行保护的链接，让其簇点链路出现在 Sentinel 控制台

#### 请求限流

- **作用**：限制某一个接口的 QPS
- **测试**：针对簇点链路在控制台配置 QPS 即可，查看 JMeter 和 Sentinel 监控页面

#### 线程隔离

- **作用**：限制某一个远程调用业务能够使用的线程数（注意：**线程数 ≠ QPS**）
- **配置**：开启 Feign 对 Sentinel 的支持，让远程调用的簇点链路出现
- **测试**：针对簇点链路在控制台配置并发线程数，查看 JMeter 和 Sentinel 监控页面

#### Fallback

- **作用**：在远程调用时，一旦出现异常情况，调用者提前编写好的一段兜底逻辑，对调用者更加友好
- **编写方式**：

| 方式 | 特点 | 推荐 |
|---|---|---|
| Fallback Class | 不能捕获调用出现的异常信息 | 不推荐 |
| FallbackFactory | 能够捕获调用出现的异常信息 | ✅ 更推荐 |

**FallbackFactory 步骤：**
1. 为 `ItemClient` 编写降级类 `ItemClientFallbackFactory`：查询商品集合返回空集合，扣减库存抛出异常
2. 在 `DefaultFeignConfig` 中注册降级类
3. 在 `ItemClient` 的 `@FeignClient` 上指定降级类

**测试：**
- 方式一：**线程隔离**后，被拒绝的请求会走降级而不会报错，异常率为 0。查询购物车会返回旧数据（降级返回的是空集合，没有最新数据）
- 方式二：**提供者抛异常**后，调用者也会走降级

#### 熔断

- **作用**：当发现请求异常时，只要达到指定阈值，直接快速失败，防止资源进一步浪费
- **断路器原理**：close → open → half-open（三态转换）
- **熔断配置方式**：

| 策略 | 说明 | 测试方法 |
|---|---|---|
| **慢调用** | 统计一定时间范围内，至少指定数量的请求，当有指定比例的请求数超过了指定阈值，就熔断指定秒数 | 提供者休眠 480ms → 配置慢调用 → 运行 JMeter 熔断测试 |
| **异常比例** | 类似 | — |
| **异常数** | 类似 | — |
