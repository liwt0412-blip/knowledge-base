# 微服务公共 API 契约与 Feign 降级边界

## 一、公共 API 模块解决什么问题

当多个微服务都要调用同一个能力时，将 FeignClient、跨服务 DTO 和 fallback 抽取到公共 API 模块。调用方只依赖契约，不复制 URL、参数名、响应类型和降级代码。

```text
服务提供方：实现 HTTP 接口
公共 API 模块：声明 FeignClient、DTO、fallback、自动配置
服务调用方：注入 FeignClient 并组织本地业务
```

“存在公共客户端”只表示具备复用入口；至少一个业务服务真正注入并调用后，才算发生了实际复用。

## 二、Feign 契约必须与服务提供方精确一致

- 服务名、公共路径、HTTP 方法和子路径必须一致；
- `@RequestParam`、`@PathVariable` 显式写参数名，避免编译参数信息差异；
- 集合查询使用批量参数，避免循环内产生 N 次 Feign 调用；
- 返回类型按照服务间真实响应声明，不能照搬网关对客户端的包装格式；
- 共享 DTO 表达跨服务契约，不直接暴露服务内部 PO、Mapper 对象或数据库字段实现。

如果业务服务通过 `ResponseBodyAdvice` 只对网关请求包装 `R<T>`，那么内部 Feign 请求应直接声明 `List<DTO>`、`DTO` 或基础类型，而不是 `R<List<DTO>>`。判断依据是运行中的包装条件和现有调用链，不是“项目统一响应体”这句抽象描述。

## 三、fallback 由业务重要性决定

| 调用类型 | 推荐降级 | 原因 |
|---|---|---|
| 非核心展示查询 | 空集合、默认展示或局部缺省 | 不阻断主页面，但必须记录异常和监控降级率 |
| 可稍后重试的读取 | 缓存、旧值或明确的暂不可用状态 | 避免把旧值伪装成实时值 |
| 扣款、库存、权限校验等关键命令 | 失败并中止主流程 | 返回成功或默认值会破坏业务正确性 |
| 真假语义敏感的查询 | 显式降级状态，而非简单 false/空集合 | 否则调用方无法区分“确实为空”和“远程服务失败” |

`FallbackFactory` 应记录完整 `Throwable`，而不只记录错误消息。返回空集合虽然便于页面降级，但语义上具有二义性，应配合日志、指标和告警，不能把降级结果当作真实业务事实写回数据库。

## 四、依赖当前用户的接口有调用场景边界

查询“当前用户”的远程接口依赖请求头和 `UserContext` 透传，只适合同步用户请求链路：

```text
客户端 → 网关 → 服务A → Feign → 服务B
```

定时任务、MQ 消费者和离线批处理通常没有当前用户上下文，不能直接复用这类接口。后台场景应设计显式传入用户 ID 的内部接口，并增加服务身份校验和权限边界，不能伪造 ThreadLocal 用户。

## 五、公共客户端的自动装配检查清单

- [ ] FeignClient 是否位于统一扫描包下？
- [ ] 公共 API 模块是否通过自动配置启用 `@EnableFeignClients`？
- [ ] fallback/fallbackFactory 是否已经注册为 Spring Bean？
- [ ] 用户信息、请求 ID 等请求头是否由拦截器透传？
- [ ] Sentinel/Feign 降级开关是否在目标环境生效？
- [ ] 参数名、集合编码和返回类型是否与 Controller 一致？
- [ ] 调用方是否采用一次批量调用和 Map/Set 回填，避免 N+1？
- [ ] 降级值是否会被误当成真实数据持久化或参与关键决策？
- [ ] 新增公共客户端后，依赖服务是否重新构建并重启？
