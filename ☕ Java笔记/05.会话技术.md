# 会话技术


## Cookie技术

### 用HttpServletResponse/HttpServletrequest接收

### 在响应头中设置Cookie

#### response.addCookie(new Cookie(String name，String value))

### 在请求头中获取Cookie

#### 获取所有Cookie列表

##### Cookie[] cookies = request.getCookies();

#### 增强for遍历Cookie列表找到对应Cookie名的值

##### for (Cookie cookie : cookies) {
            if (cookie.getName().equals("id")) {
                System.out.println("id:" + cookie.getValue());
            }
        }

### 优点

#### HTTP协议中支持的技术

### 缺点

#### 移动端APP无法使用Cookie

#### 不安全，用户可以自己禁用Cookie

#### Cookie不能跨域

### Cookie会话跟踪原理

#### 底层涉及两个头：响应头Set-Cookie和请求头Cookie。当我们在服务器端设置Cookie之后，它会自动在响应头中将Cookie的数据响应给客户端浏览器，而浏览器一旦见到这个响应头是Set-Cookie，它就会自动将Cookie保存在浏览器本地。并且在后续的每一次请求当中，它都会将当前服务器所涉及的Cookie通过请求头一并携带到服务器端。

## Session技术

### 用HttpSession接收

### 往Session中存储数据

#### session.setAttribute(String name, Object value)

### 从Session在获取数据

#### session.getAttribute(String name)

### 优点

#### 存储在服务器端，数据相对安全

### 缺点

#### 服务器集群环境下无法直接使用Session

#### 由于底层是基于Cookie实现的，所以具备Cookie的所有缺点

### Session会话跟踪原理

#### Session的底层就是基于Cookie实现的，只不过在Cookie当中所存储的值就是JsessionId

## 令牌方案

### 优点

#### 支持PC端、移动端

#### 解决集群环境下的认证问题

#### 减轻服务器端存储压力

### 缺点

#### 如何生成令牌、如何存储令牌、以及在后续的存储当中携带令牌、校验令牌，这些都需要程序员手动实现，编码繁琐

### JWT令牌介绍

#### 全称：JSON Web Token

#### 组成

##### 第一部分

###### Header(头)，记录令牌类型、签名算法等

###### {"alg:"HS256","type":"JWT"}

##### 第二部分

###### Payload(有效载荷)，携带一些自定义信息、默认信息等

###### {"id":"1","username","Tom"}

##### Base64：是一种基于64个可打印字符来表示二进制数据的编码方式

##### 第三部分

###### Signature(签名)，防止Token被篡改、确保安全性。将Header、payload融入，并加入指定密钥，通过指定签名算法计算而来

### 使用过程

#### 1. 引入jjwt依赖

##### <dependency>
       <groupId>io.jsonwebtoken</groupId>
       <artifactId>jjwt</artifactId>
       <version>0.9.1</version>
</dependency>

#### 2. 调用官方提供的工具类Jwts来生成或解析jwt令牌

##### 指定加密算法和密钥，生成令牌

###### Jwts.builder().signWith(SignatureAlgorithm.HS256, "carrotking")

##### 添加自定义信息

###### .addClaims(Map map)

##### 设置过期时间

###### .setExpiration(Date date)

##### 压缩成令牌

###### .compact()

#### 3.解析Jwt令牌

##### 调用Jwts静态方法，指定密钥

###### Jwts.parser().setSigningKey("carrotking")

##### 解析令牌

###### .parseClaimsJws(token)

##### 获取自定义信息，返回Claim对象

###### .getBody()

##### 将Claim对象转成字符串，方便比较

###### .toString()

## Filter过滤器

### 是JavaWeb三大组件（Servlet、Filter、Listener）之一

### 一般完成一些通用的操作，比如：登录校验、统一编码处理、敏感字符处理等

### 使用方法

#### 1. 定义Filter类，实现Filter接口，并实现其所有方法

##### 初始化方法

###### init(FilterConfig filterConfig)

##### 拦截到请求后执行的方法

###### doFilter(ServletRequest servletRequest, ServletResponse servletResponse, FilterChain filterChain)

###### 1. 先将请求对象和响应对象强转为Http请求/响应对象

###### 2. 放行方法：filterChain.doFilter(servletRequest, servletResponse).

##### web服务器关闭时执行的销毁方法

###### destroy()

#### 2. 配置Filter，在Filter类上加@WebFilter(urlPatterns = "/*")注解，配置拦截路径。引导类上加@ServletComponentScan开启Servlet组件支持

### 实现步骤

#### 1. 将请求对象和响应对象强转为Http请求/响应对象

##### HttpServletRequest httpServletRequest = (HttpServletRequest) servletRequest;

###### HttpServletResponse httpServletResponse = (HttpServletResponse) servletResponse;

#### 2. 判断是否是登录请求，如果是，就放行

##### if (httpServletRequest.getRequestURI().contains("login")) {
            filterChain.doFilter(servletRequest, servletResponse);
            return;
        }

#### 3. 获取请求头中的token

##### httpServletRequest.getHeader("token")

#### 4. 如果token不存在，设置响应头状态为401

##### httpServletResponse.setStatus(HttpServletResponse.SC_UNAUTHORIZED)

#### 5. 解析token，如果解析失败，设置响应头状态为401（注意要用try-catch包裹）

#### 6. 放行

### 过滤器链

#### 一个web应用中，可以配置多个过滤器，这多个过滤器就形成了一个过滤器链

#### 顺序：优先级是按照过滤器类名（字符串）的自然排序

## Interceptor拦截器

### 使用方法

#### 1. 定义拦截器，实现HandlerInterceptor接口，并实现其所有方法

##### 目标资源运行前运行

###### boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler)

###### 返回true：放行，返回false：不放行

##### 目标资源运行后运行

###### void postHandle(HttpServletRequest request, HttpServletResponse response, Object handler, ModelAndView modelAndView)

##### 视图渲染完毕后运行

###### void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex)

#### 2. 定义一个配置类实现WebMvcConfigurer接口

#### 2. 在配置类里注册拦截器

##### 注入拦截器bean对象

###### TokenInterceptor tokenInterceptor

##### 重写addInterceptors方法

###### public void addInterceptors(InterceptorRegistry registry) {      registry.addInterceptor(tokenInterceptor).addPathPatterns("/**").excludePathPatterns("/login"); 
    }

## Filter于Interceptor的区别

### 1. 接口规范不同：过滤器需要实现Filter接口，而拦截器需要实现HandlerInterceptor接口

### 2. 拦截范围不同：过滤器会拦截所有资源，而拦截器只会拦截Spring环境中的资源
