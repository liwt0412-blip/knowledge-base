# 聚合查询、时间区间与批量数据补全

## 一、先区分四类对象

| 类型 | 主要职责 | 是否适合承接临时聚合字段 |
|---|---|---|
| PO/Entity | 映射数据库中的一条持久化记录 | 默认不适合 |
| DTO/Projection | 承接查询参数、远程契约或聚合结果 | 适合 |
| VO | 描述对外接口返回 | 适合最终展示字段 |
| Mapper Result | 描述特定 SQL 的返回形状 | 适合专用聚合 DTO |

对于 `GROUP BY`、`COUNT`、`SUM` 这类查询，优先定义或复用专用 DTO：

```java
public class IdAndNumDTO {
    private Long id;
    private Integer num;
}
```

```sql
SELECT lesson_id AS id, COUNT(1) AS num
FROM learning_record
GROUP BY lesson_id
```

只有当某个派生字段稳定属于领域模型、被多处复用，并且放入实体不会模糊持久化边界时，才考虑使用 `@TableField(exist = false)`。

## 二、全局统计与分组统计解决不同问题

```java
selectCount(...)
```

适合返回一个全局总数，例如“本周一共完成多少小节”。

```sql
GROUP BY lesson_id
```

适合返回多组数据，例如“每个学习计划本周分别完成多少小节”。

如果页面同时需要顶部汇总和当前页明细，可采用：

1. 标量 `COUNT` 查询全局数量。
2. 分组查询只传当前页业务 ID，避免统计无关数据。

当用户数据规模较小，且分组结果本来就要覆盖全部数据时，也可以对分组结果求和以减少一次 SQL。选择依据是统计范围和实际数据量，不是机械追求 SQL 次数最少。

## 三、Map 为什么可以直接匹配统计值

将聚合结果转换为：

```text
业务ID -> 数量
```

例如：

```java
Map<Long, Integer> countMap = IdAndNumDTO.toMap(counts);
Integer count = countMap.getOrDefault(lesson.getId(), 0);
```

成立的前提是 SQL 返回的 `id` 与循环对象使用的 ID 属于同一个关联键。若 SQL 是 `lesson_id AS id`，匹配键就必须是 `LearningLesson.id`，不能误用 `courseId`。

## 四、时间统计默认使用半开区间

推荐：

```text
finish_time >= 本周一 00:00
finish_time <  下周一 00:00
```

不推荐把相邻周期都写成两端包含的 `BETWEEN`。左闭右开的半开区间具有三个优点：

- 相邻周期没有重叠。
- 不依赖数据库支持到多少毫秒或纳秒。
- 月、周、日统计可以使用同一种模板。

## 五、本地业务数据与远程补全字段分开

列表接口先分页查询本地归属数据，再提取当前页全部外部 ID，一次批量调用远程服务：

```text
本地分页
  → 提取当前页全部 courseId
  → 批量调用课程服务
  → 转换为 courseId -> CourseDTO 的 Map
  → 组装 VO
```

适合远程补全的字段包括课程名称、封面、课程总小节数；用户归属、学习状态、学习进度等应由本地服务负责。禁止在循环中逐条远程调用。

## 六、完成前检查清单

- [ ] 聚合结果是否使用了与 SQL 返回形状一致的 DTO/Projection？
- [ ] Map 的键与当前循环对象使用的是不是同一个业务 ID？
- [ ] 全局统计与当前页统计的范围是否明确？
- [ ] 时间区间是否采用左闭右开？
- [ ] 分组查询是否可以限制为当前页 ID？
- [ ] 远程字段是否一次批量查询，而不是循环调用？
- [ ] 远程服务返回空时，是降级、返回空字段还是中断接口，是否有明确契约？
