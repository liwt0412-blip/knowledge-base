# 热冷分离：实时榜 Redis 与历史归档表的双数据源查询

> 来源：天机学堂学霸积分榜接口（[[🖥 项目笔记/天机学堂/10-tj-learning积分服务实现]]，2026-07-21）。
> 配套阅读：[[高频最新状态的Redis合并写模式]]（写路径的热数据合并写，与本文一读一写互补）。

## 模式

排行榜、账单、统计类数据的共同结构：**当前周期数据高频变动**（适合 Redis ZSet/Hash 实时维护），**历史周期数据不再变动**（适合定时任务归档进 MySQL）。查询接口按"是否当前周期"路由数据源：

```text
请求（周期标识 period）
  ├─ period 为空/默认 或 等于当前周期 → 查 Redis 实时榜（ZREVRANGE/ZSCORE/ZREVRANK）
  └─ 指定历史周期                      → 查 MySQL 归档表（周期 + 唯一键归档）
```

## 关键设计要点

1. **路由条件必须是"等于当前周期"而非"缺省"**：用户显式指定当前周期 id 时也要走 Redis，走归档表会查不到（当前周期还没归档）。
2. **归档范围按"历史还要回答什么问题"来定，两种变体**：
   - 历史只需头部展示 → 归档只存前 N，历史分支天然截断；
   - 历史还要回答"我排第几"（任意名次）→ 必须**全量归档**，展示截断由查询层负责（分页窗口 ∩ [0,N)）。注意此时归档表的"名次"列容量要按用户量选型（如 tinyint 最大 255，名次超界即写入溢出）。"页面只显示前 N"不等于"数据只存前 N"。
3. **"我的排名"双分支同源语义**：实时分支用 `ZREVRANK`+`ZSCORE`，历史分支查归档表本人行；查不到统一返回"未上榜"语义（如 `rank=null, points=0`），并写进接口契约。
4. **归档幂等**：归档表以（周期, 用户）唯一键 + INSERT IGNORE，重复执行/补跑不产生重复行；破坏性清理（清流水、删旧榜 key）只在正当结算窗口执行。
5. **展示增强可降级**：榜单本体数据在本服务，姓名/头像等跨服务批量回填（一次调用防 N+1），降级时显示兜底文案，不阻断榜单返回。

## 适用条件与失效边界

- 适用：数据按自然周期（日/周/月赛季）滚动、历史只读、只需要保留头部 N 名的场景。
- 失效边界：需要查询任意历史明细的场景不适用（即使全量归档也只存名次与总分，没有行为明细）；实时数据强一致要求极高的场景需评估 Redis 与落库的最终一致窗口。

## 验证方法

- 当前周期：制造一条实时数据后立即查询，榜单应包含且排名正确；
- 历史周期：手动执行一次归档任务后按周期 id 查询，与归档前 Redis 榜快照一致；
- 边界：分页窗口超出 N 返回空列表；未上榜用户返回约定的"未上榜"结构。
