# 事件驱动积分发放的幂等、防冤扣与重放清单

> 来源：天机学堂积分服务评审修复（[[🖥 项目笔记/天机学堂/10-tj-promotion积分服务实现]]，双向链接）。
> 适用条件：任何"行为事件（MQ/回调）驱动积分、计数、余额类数值增减"的链路，尤其是加分与扣分对称存在的场景。

## 四条规则

### 1. 扣减必须凭原发放记录，不做"无条件对称扣减"

事件"内容被删除"不等于"内容曾产生过数值"。处于中间态（待审核、未入账、上月已归档）的对象被删除时，无条件扣减会冤扣。

- 做法：扣减前查当期的发放流水，存在才扣；查不到跳过。
- 取舍：跨周期删除（流水已清）因此不扣——防冤扣优先于防刷，不要用"固定扣一笔"近似。
- 验证：构造"从未入账的对象被删除"的消息，断言数值不变、无扣减流水。

### 2. 幂等键必须"失败回滚"，否则死信重放被静默吞掉

先 SETNX 幂等键再执行业务是常见写法，但业务抛异常后消息进死信队列，人工重放时 SETNX 返回 false，消息被当作重复投递丢弃——重放机制形同虚设。

- 做法：业务异常分支里先删幂等键再向上抛出。
- 失效边界：仅适用于"幂等键只是前置减负、DB 唯一约束做最终裁决"的结构；若幂等键本身就是唯一事实来源，不能删。
- 验证：第一次消费制造业务异常，人工重放同一条消息，断言能正常入账且只入一次。

### 3. 唯一约束要"先行裁决"：先落流水，再动缓存/榜

先更新 Redis（ZSet/Hash）最后写库的顺序下，一旦 Redis 侧幂等键丢失（重启、逐出），重复消息会把缓存数值双加，唯一约束只挡了流水没挡住数值。

- 做法：调整为先写流水（捕获 DuplicateKeyException 直接返回），唯一键通过后再更新缓存。
- 代价：正常路径多一次先行 DB 写，与"普通 MQ + 死信兜底"的可靠性档位匹配。
- 验证：删掉 Redis 幂等键后重发同一消息，断言流水、榜、当日累计均未双增。

### 4. 置位类操作（Bitmap/状态位）失败要回滚，给用户留重试路

先置位后办业务的链路（如签到 SETBIT 后发积分），业务失败会导致"位已置、事未办"，用户当日再也无法重试，数值永久丢失。

- 做法：业务异常分支里把位回滚（setBit false）再抛出。
- 验证：模拟发积分异常后重试签到，断言可以重新完成且只入账一次。

## 配套经验

- 事件体只带"谁、什么对象、什么类型"，**分值/规则一律由消费端按映射裁定**，不信任生产侧传值，防止规则被绕过。
- 未来渠道的预留方式：消费队列提前声明（空转无成本），生产侧上线发消息即生效，消费侧零改动。
- 周期结算任务（月结/归档）的破坏性步骤（清表、删 key）要加"正当窗口"防御，避免月中人工补跑误清当期数据。
