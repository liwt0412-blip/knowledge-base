# Bitmap 标志位存储模式：连签统计与验重

> 来源：天机学堂签到（SETBIT 判重 + BITFIELD 连签 + TTL 滑动续期）与优惠券兑换码验重（[[🖥 项目笔记/天机学堂/10-tj-learning积分服务实现]]）。
> 配套：[[Redis的常用命令]] 第（六）节查命令；[[事件驱动积分发放的幂等、防冤扣与重放清单]] 第 4 条防"位已置、事未办"。本文只讲**选型与键设计**。

## 核心：四问判断框架

判断一个业务能不能用位图，四个问题全过才适用：

1. **状态是二元的吗？** 1 bit 只表达"是/否"（签了/没签、兑了/没兑、已读/未读）。要存时间、次数、IP 等元数据 → 不适用，或位图只存标志、元数据另配存储。
2. **每个状态能映射到稠密、有界的整数序号吗？** offset 必须是整数：天号、序列号、用户 id。随机 UUID、稀疏无界编号 → offset 爆炸（`SETBIT key 1200000000 1` 直接开辟约 143MB）。
3. **数据量级大到值得压缩吗？** 小系统一张 MySQL 表更简单，别过度设计。位图价值在海量：1 亿条签到记录 MySQL 数 GB + 索引，Bitmap 仅约 12MB，读写 O(1)。
4. **查询模式是按位读/统计/位运算吗？** `BITCOUNT`（总数）、`BITOP AND/OR`（交集并集）、`BITFIELD`（连续片段）→ 适用；"按条件筛选列表再分页"→ 那是 MySQL/ES 的活。

口诀：**二元状态 + 稠密序号 + 海量数据 + 位运算查询**——四条全中用位图；中两三条位图配合其他存储；只中一条老老实实用表。

## 关键设计决策：键的维度决定能回答什么问题

先列业务要回答的问题，再定键的维度：

| 业务问题 | 键设计 | offset | 命令 |
|---|---|---|---|
| 这人本月哪几天签了、连签几天 | `sign:uid:{userId}:{yyyyMM}` | 天号-1 | SETBIT / BITFIELD |
| 今天有多少人活跃 | `active:{yyyyMMdd}` | userId | SETBIT / BITCOUNT |
| 两人共同签到天数、N 天全勤 | 多键 | 同维度序号 | BITOP AND |
| 这个兑换码用没用过 | `coupon:code:{批次}` | 序列号 | SETBIT / GETBIT |

两个方向都要查（如"这人哪天签了" + "今天谁签了"）→ 按查询方向各建一套键双写，或接受单方向扫描的代价。

## 项目验证过的技法

- **BITFIELD 低位连 1 算连续天数**：`BITFIELD key GET u{day} 0` 一次取出十进制数，循环"与 1 位与 → 右移"，O(天数) 得连签天数；同一片段逐位拆分即得签到日历 0/1 数组。
- **TTL 滑动续期代替历史清理**：每次签到把 key 续期（如 90 天），不活跃用户的 key 自然过期，省掉定时清理任务——用 TTL 白嫖生命周期管理。
- **SETBIT 返回值即判重**：`SETBIT` 返回旧值，旧值=1 即重复操作，天然原子判重，无需先 GETBIT。
- **月度分键把周期折进 key**：`{userId}:{yyyyMM}` 让"当月"边界由 key 天然表达，跨月统计就是多键各取再合并。

## 适用边界（不满足时回头查四问框架）

- 位上挂不了元数据：只有"是否"，没有"何时、何地、几次"；
- offset 必须稠密有界，否则内存失控；
- 按 user 分键时反查"某天所有签到的人"需扫全部 key——查询方向必须在键设计阶段定死；
- 布隆过滤器本质是同一思想（判断"一定不存在/可能存在"），需要"海量存在性预判"时优先考虑它而不是自己造位图。
