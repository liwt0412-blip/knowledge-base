# RAG 系统上线后的效果评估与监控

> 面试逐字稿版本，可直接背诵。
> 
> **使用边界（2026-07-23 核验分层，背诵前必读）：**
> - 已落地：反馈入口的方案设计（有用/没用 + 结构化负反馈原因）；代码已在学习项目 heima-ai 完整实现，石化项目代码尚未核实，不说"已上线运行"。
> - 设计预案（只能讲"设计思路/规划方向"，不能说已经在跑）：隐式反馈采集、裁判模型自动打分、bad case 池每日导出、每周 200 条评测集、上线门禁。

> - RAG 评估指标的标准深挖材料已迁移至 [[RAG面试追问三层回答]] 第八章（题 28-32），本文只作在线/离线双线思路的补充。

## 问题

RAG 系统上线后，你怎么知道它好不好用？怎么持续监控？

## 回答

我分两条线来做：在线实时监控 + 离线定期评估。

### 一、在线监控

**1. 用户反馈闭环**

产品界面上有 👍👎 按钮，用户点踩的时候弹原因选项——答非所问、内容错误、回答太慢。每天看踩赞比，踩率突然飙升就拉出来看是哪个知识库片段出问题了。

**2. 隐式反馈**

大部分用户懒得点反馈，答得不好直接关页面。所以我们还抓隐式信号：用户复制了回答 → 可能不满意自己拿去改了；同一个问题短时间内重复问 → 第一次没答好；用户秒关页面 → 回答没用。这些不需要用户点按钮，自动就抓到了。

**3. 裁判模型自动打分**

每次生成回答后，另一个大模型从三个维度打分：

| 维度 | 检查内容 |
|------|---------|
| 相关性 | 有没有跑题 |
| 忠实度 | 有没有脱离文档瞎编（幻觉检测） |
| 完整性 | 有没有漏关键信息 |

低于阈值就告警。石化行业操作规程一个字错了都是事故，**忠实度这个指标盯得最紧**。

### 二、离线评估

在线看的是"现在有没有出问题"，离线看的是"系统有没有在变好"。

每周跑一次评测集——提前标注好 200 条问答对，对比新旧版本：

- **检索指标**：Recall@K（正确答案在召回的第几条）、MRR（平均倒数排名）
- **生成指标**：答案正确率、事实一致性

上新模型之前必须先跑一遍，分数不降级才能上线。

### 三、闭环

点踩的自动入库到 bad case 池。每天早上一键导出分析：

- 文档切分的问题 → 修切分策略
- 相关性阈值太松 → 调阈值
- 文档本身写得太差 → 通知业务方更新知识库

"发现问题 → 定位原因 → 修复上线"的闭环跑通了，系统才敢说在持续变好。

### 结合石化项目这样讲

> "评估这块我分在线和离线两条线想过完整方案。在线侧，回答反馈入口我做了方案设计：有用/没用加结构化负反馈原因，落库后能和引用来源表 JOIN 分析是哪类问题；方案我在项目里完整实现过。离线侧，评测集加检索命中率和忠实度对比是标准做法，我的规划是上任何检索或模型变更前跑一遍固定问题集。裁判模型打分、bad case 池自动分析这些是下一步的演进方向，目前还没有生产运行数据，我不把它说成已经在线上跑。"
