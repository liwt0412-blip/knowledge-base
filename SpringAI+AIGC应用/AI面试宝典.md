## 模块一 · SpringAI 框架

> 覆盖题目：01 SpringAI 是什么 / 02 SpringAI vs SpringAI Alibaba / 03 三大架构

### 1.1 SpringAI 是什么

**一句话**：Spring 生态里面向 AI 应用的新框架，用 Spring 的风格简化 AI 功能集成。

```
SpringAI 核心能力
├── 统一 Portable API → 一套代码对接不同大模型厂商
├── 聊天补全 / 图片生成 / 语音识别
├── 向量数据库集成
├── RAG（检索增强生成）
└── 快速搭建企业级 AI 应用
```

### 1.2 SpringAI vs SpringAI Alibaba

| 对比维度 | Spring AI | Spring AI Alibaba |
|---------|-----------|-------------------|
| 定位 | Spring 官方 AI 框架 | 阿里基于 Spring AI 的实现增强 |
| 目标 | 统一对接各种大模型 API，屏蔽底层差异 | 深度适配阿里云灵积（通义系列） |
| 核心抽象 | ChatClient / Embedding / Tool Calling | 同 Spring AI + 阿里云最佳实践 |
| 支持厂商 | OpenAI / Azure / Ollama 等多家 | 专注阿里云生态 |
| 额外能力 | — | 知识库检索、多模态、Prompt 模板 |
| 类比 | **标准接口** | **阿里云的专业实现** |

> 🧠 **口诀**：Spring AI 是 JDBC 标准，Spring AI Alibaba 是 MySQL 驱动

### 1.3 SpringAI 支持的三大架构

```
SpringAI 三大架构
│
├── ① 模型接入架构
│   可插拔的 ChatClient / EmbeddingClient
│   一套代码切换 OpenAI / Azure / HuggingFace / Ollama
│
├── ② RAG 检索增强生成架构
│   DocumentReader → TextSplitter → VectorStore
│   读取 → 分块 → 向量化 → 检索 → 喂给 LLM
│   场景：私有知识库问答
│
└── ③ Agent 智能体与函数调用架构
    把 Java 函数注册给模型
    模型自主决策何时调用工具、处理结果
    场景：自动化任务链
```

## 模块二 · AI 核心能力

> 覆盖题目：04 ToolCalling / 05 RAG / 06 MCP vs ToolCalling / 07 System Prompt / 14 Token 超限 / 21-22 模型微调

### 2.1 ToolCalling 调用原理

```
ToolCalling = 让模型像人一样"使用工具"

┌─────────────────────────────────────────────────────┐
│  Step 1：定义工具                                    │
│  提前告诉模型有哪些工具、叫什么、要什么参数（JSON Schema）│
├─────────────────────────────────────────────────────┤
│  Step 2：模型决策                                    │
│  用户提问 → 模型判断是否需要工具                      │
│  需要 → 返回"函数调用请求"（工具名 + 参数 JSON）       │
│  不需要 → 直接回答                                   │
├─────────────────────────────────────────────────────┤
│  Step 3：执行并反馈                                  │
│  你的代码执行函数 → 拿到结果 → 丢回给模型              │
│  模型结合工具结果 → 生成最终自然语言回答               │
└─────────────────────────────────────────────────────┘
```

> 🧠 **关键认知**：模型不亲自执行代码，只负责"决定叫谁、传什么参数"，执行由我们代码完成

### 2.2 RAG 实现原理

```
RAG = Retrieval（检索） + Augmented（增强） + Generation（生成）

┌─ 离线阶段：建库 ────────────────────────────────────┐
│  企业文档 → 切块 → Embedding 模型 → 向量 → 向量库    │
└─────────────────────────────────────────────────────┘
                         │
┌─ 在线阶段：问答 ────────────────────────────────────┐
│  用户提问 → 向量化 → 在向量库语义检索 → 找最相近文档块 │
│       │                                              │
│       ▼                                              │
│  文档块 + 原始问题 → 拼成 Prompt → 送给大模型          │
│       │                                              │
│       ▼                                              │
│  模型基于资料回答（不是凭空编造）                      │
└─────────────────────────────────────────────────────┘
```

> 🧠 **核心价值**：让模型"有据可依"，利用私域知识又不丢失表达能力，答案可控不跑偏

### 2.3 MCP vs ToolCalling — 类比理解

| 对比维度 | Tool Calling | MCP（Model Context Protocol） |
|---------|-------------|------|
| 是什么 | 模型**内部能力** | 开放**协议标准** |
| 做什么 | 模型输出结构化 JSON 选工具、传参数 | 定义客户端与工具服务器的通信接口 |
| 谁定的 | 各模型厂商自己实现 | Anthropic 推出的开放协议 |
| 类比 | 模型"会点菜" | 规定了"厨房、服务员、菜单"怎么协作 |
| 解决什么 | 模型能调用工具 | 不同模型和工具之间**即插即用**，不再需要胶水代码 |

> 🧠 **一句话**：ToolCalling 是能力，MCP 是协议。有 MCP 后，换模型换工具都即插即用

### 2.4 系统提示词（System Prompt）组成

```
一条完整的 System Prompt
│
├── ① 角色设定     — 告诉 AI 它是谁、什么风格（"你是一个Java面试官"）
├── ② 任务描述     — 要做什么事（"模拟技术面试，追问并发知识"）
├── ③ 输出格式要求 — 结构、字数、是否分点
├── ④ 约束条件     — 不能说的内容、必须遵守的规则
└── ⑤ Few-shot 示例 — 给几个范例让 AI 照着来
```

> 🧠 **口诀**：角色 + 任务 + 格式 + 约束 + 示例，五件套

### 2.5 Token 超限处理策略

```
Token 超限怎么办？
│
├── 首选：换长上下文模型（128K 甚至 1M token）
│
├── 滑动窗口：只保留最近几轮对话，丢弃最早消息
│
├── 动态摘要：把早期对话压缩成简短摘要嵌进 Prompt
│   既省 token 又能保留关键信息
│
├── RAG 思路：历史对话存向量库，每次检索最相关片段
│   避免暴力全量输入
│
└── ⚠️ 大忌：简单粗暴截断 → 丢上下文 → 回答跑偏
```

### 2.6 为什么需要模型微调

| 不用微调 | 微调后 |
|---------|--------|
| 通用模型，学的是互联网通用数据 | 变成你的专业助手 |
| 学的是语言规律，不是你业务的逻辑 | 学会你领域的用词、规则、套路 |
| 面对客服/代码审查/领域文档可能答非所问 | 在你的任务上表现飙升 |
| — | 成本低、见效快，不用从零烧显卡 |

> 🧠 **一句话**：微调 = 在预训练好的大脑上，用你的业务数据再教一教

### 2.7 不准确数据的微调策略

```
数据不准确时怎么微调？

① 基于预训练模型接着训练（别从头来）
② 学习率调得特别小 → 防止遗忘预训练知识
③ 加正则化 / Dropout / 对抗训练 → 别太纠结噪声样本
④ 先自动清洗数据，筛掉明显错误
⑤ 用集成模型投票纠正标注 → 再拿去微调

核心：借力预训练 + 小步调整 + 抗噪手段
```

## 模块三 · AI 项目架构实战

> 覆盖题目：08 AI模块设计 / 09 系统集成 / 10 前端交互 / 13 会话记忆 / 12 上下文打断

### 3.1 AI 模块三层架构（实战案例）

```
项目：智能客服系统（Java + Spring Boot + OpenAI）

┌─ 第一层：接入层 ─────────────────────────────────────┐
│  线程池处理用户请求                                    │
│  核心线程 = CPU核数×2，最大线程 = 100                  │
│  队列：LinkedBlockingQueue                            │
│  拒绝策略：CallerRunsPolicy（压力回传，不丢请求）       │
│  临时线程空闲超时：60s（避免频繁创建销毁）               │
└──────────────────────────────────────────────────────┘
                         │
┌─ 第二层：调度层 ─────────────────────────────────────┐
│  基于 AQS 的信号量限流（OpenAI 有 RPM 上限）            │
│  重写 tryAcquireShared / tryReleaseShared            │
│  state 控制并发令牌数                                 │
│  没拿到令牌 → AQS FIFO 队列阻塞                       │
│  超时 → 降级返回预设话术                              │
│  volatile 保证令牌数多线程可见                         │
└──────────────────────────────────────────────────────┘
                         │
┌─ 第三层：调用与缓存层 ────────────────────────────────┐
│  双重检查锁单例 → 本地 LoadingCache 缓存常用答案        │
│  单例用 volatile 防指令重排                           │
│  锁用 AQS 的 ReentrantLock                           │
└──────────────────────────────────────────────────────┘

成果：支撑日活 10万+，平均响应 < 2s

串联知识点：CAS + volatile + 线程池 + AQS 全部用上
```

> 🧠 **记忆口诀**：接入层扛流量（线程池），调度层控并发（AQS限流），缓存层省成本（DCL+Cache）

### 3.2 AI 系统集成步骤

```
Step 1: 明确业务场景 → 推荐 / NLP / 图像识别？

```

### 3.3 前端交互 — 流式输出问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| HTML/Markdown 截断 | 按字符输出，标签未闭合 | 后端加边界标记，按完整 token 输出 |

### 3.4 会话记忆实现

```
智能助手会话记忆方案
│
├── 短期记忆：滑动窗口保留最近 N 轮对话
│
├── 长期记忆：
│   ├── 历史对话 → 摘要压缩 → 存入上下文
│   └── 历史对话 → 向量库 → 检索最相关片段注入
│
└── 结构化记忆：
    用户画像 / 偏好 / 历史关键信息 → 持久化存储
    每次对话时注入 System Prompt
```

### 3.5 文档分割 — 如何不破坏上下文（RAG 分块策略）

在 RAG 中，文档需要切分成小块才能存入向量库，但切分不当会**把一句完整的话或一个完整语义拦腰斩断**，导致检索时拿到的片段缺少上下文，LLM 理解偏差。

**四种分块策略**

| 策略 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **固定大小分块** | 按 token 数切（如 512 token/块） | 简单，成本低 | 经常切断句子和段落 |
| **语义分块** | 按段落、句子边界自然切分 | 保持语义完整 | 块大小不均匀 |
| **重叠窗口** | 相邻块之间有 10%~20% 内容重叠 | 减少边界信息丢失 | 存储冗余 |
| **层级分块** | 大块（文档级）→ 中块（段落级）→ 小块（句子级），检索时多粒度匹配 | 兼顾宏观和微观上下文 | 实现复杂 |

**实战：父文档检索（Parent Document Retriever）**

```
文档
  │
  ▼
切成大块（如 2000 token）← 喂给 LLM 时用（上下文完整）
  │
  ▼
大块再切成小块（如 300 token）← 存入向量库检索用（匹配精准）
  │
  ▼
检索时：用小块做语义匹配 → 命中小块 → 返回它所属的大块给 LLM

效果：小块精准定位 + 大块保留上下文，两全其美
```

> 🧠 **口诀**：固定快但碎，语义整但偏，叠窗口补边界，父文档小块搜大块喂

## 模块四 · 部署与运维

> 覆盖题目：11 训练条件 / 15 AI 部署 / 16-20 Ollama 生态

### 4.1 AI 训练条件与硬件

```
训练流程：数据准备 → 模型设计 → 训练 → 评估

硬件核心：GPU
├── NVIDIA A100 / V100（显存要大，否则大模型跑不起来）
├── 大数据量 → 大内存 + 高速 SSD
├── 小规模实验 → 单卡
└── 大规模训练 → 多机多卡并行 + NCCL 通讯

训练过程：喂标注数据 → 反向传播更新参数 → 反复迭代至收敛
```

### 4.2 AI 部署方案

| 部署方式 | 工具 | 模型格式 |
|---------|------|---------|
| 通用推理 | TensorFlow Serving / PyTorch Serve | SavedModel / TorchScript |
| 大语言模型 | **vLLM**（高性能推理引擎） | HuggingFace 权重 |
| 跨平台 | ONNX Runtime | ONNX |
| 本地轻量 | Ollama | GGUF |

### 4.3 Ollama 是什么

```
Ollama = 本地运行大模型的"一键工具"

能力：
├── 模型下载、管理、推理全封装
├── 一行命令启动本地模型服务
├── 支持 Llama / Gemma / Mistral 等开源模型
├── 提供 REST API + 命令行交互
├── 无需联网，不依赖云端 API

适用场景：
├── 本地开发测试
├── 隐私敏感场景
└── 离线玩模型
```

### 4.4 Ollama 部署与硬件

| 维度 | 说明 |
|------|------|
| 部署方式 | 本地安装 Ollama → 拉取模型 → 启动服务 → 当"黑盒子"用 |
| 硬件 | 普通 GPU 服务器，几张 A100 或 A10 |
| 模型规格 | 13B / 70B 推理够用 |
| 用途 | 纯推理，不搞训练，压力不大 |

### 4.5 Ollama 数据老化 → 解决方案

```
问题：Ollama 本地模型知识有截止日期，数据老旧

解决方案（由简到强）：

① RAG（最主流）
   提问 → 查最新文档 / 数据库 / 搜索引擎
   → 把新资料 + 问题拼成 Prompt → 喂给模型
   模型基于实时信息回答

② 联网插件 / Tool Calling
   模型主动搜索网页，拿到最新内容再回答
```

## 答-01 · SpringAI 框架是什么

Spring AI 是 Spring 生态里专门面向人工智能应用的新框架，主要目的是简化 AI 功能的集成。

它提供了一套统一的 Portable API，让你可以方便地接入不同大模型厂商的服务，比如聊天补全、图片生成和语音识别等，不用自己处理繁琐的对接细节。

同时它也集成了向量数据库、检索增强生成等功能，能帮你快速搭建企业级 AI 应用。

---

## 答-02 · SpringAI 与 SpringAI Alibaba 的区别

**Spring AI** 是 Spring 官方推出的 AI 应用开发框架，目标是统一对接各种大模型厂商的 API，屏蔽底层差异，提供类似 ChatClient、Embedding、Tool Calling 这些抽象，让你能快速开发 AI 功能。它本身支持很多模型供应商，比如 OpenAI、Azure、Ollama 等。

**Spring AI Alibaba** 是阿里基于 Spring AI 做的一个实现增强，专为阿里云灵积模型服务（通义系列）深度适配。它在底层自动对接阿里云的认证、计费等，还沉淀了阿里在企业级 AI 场景的最佳实践，比如知识库检索、多模态、Prompt 模板等，相当于是 Spring AI 在阿里云生态下的落地版本。

你可以理解成 **Spring AI 是标准接口，而 Spring AI Alibaba 是阿里云这个具体供应商的专业实现**。

---

## 答-03 · SpringAI 支持的架构有哪些

Spring AI 主要是给 Spring 开发者提供一套统一的抽象，来对接各种 AI 模型和编排流程。它支持的"架构"核心有三类：

- **模型接入架构**：通过可插拔的 `ChatClient`、`EmbeddingClient` 等，接 OpenAI、Azure、HuggingFace、Ollama 等不同厂商的模型，你可以一套代码切换底层 AI 服务。
- **RAG 检索增强生成架构**：内置 `DocumentReader`、`TextSplitter`、`VectorStore`，这套组件实现"读取文档 → 分块 → 存入向量库 → 检索出相关片段喂给 LLM"的完整流程，专门应对私有知识库问答场景。
- **Agent 智能体与函数调用架构**：允许你把 Java 函数注册给模型，让模型自主决策何时调用外部工具、如何处理结果，实现更复杂的自动化任务链。

简单说就是：用 Spring 风格帮你快速搭出调用大模型、做企业知识库 Q&A、搞智能体协作的应用。

---

## 答-04 · ToolCalling 调用原理

Tool Calling 就是让模型能像人一样"使用工具"来解决它自己完成不了的任务，比如查天气、算数学题、调 API。

过程分三步：

1. **定义工具**：提前告诉模型有哪些工具可用，每个工具叫啥、干嘛的、要什么参数，用 JSON Schema 描述清楚。
2. **模型决策**：当用户提问时，模型判断是否需要呼叫某个工具。如果需要，它不直接回答，而是返回一个"函数调用请求"，里面包含工具名和参数的值（都是结构化 JSON）。
3. **执行并反馈**：你的程序拿到这个请求，真正去执行对应函数，得到结果（比如天气数据），再把结果作为新消息丢回给模型。模型结合工具结果生成最终的自然语言回答。

这样就实现了模型和外部世界的交互，整个过程**模型不亲自执行代码，只负责"决定叫谁、传什么参数"，执行由我们代码完成**。

---

## 答-05 · RAG 的实现原理

RAG，就是把检索和生成拼在一起，让大模型回答问题时，先查知识库，再根据查到的资料去生成内容。

它的流程大致三步：

1. **离线建库**：先把企业自己的文档、资料切块，用嵌入模型转成向量存进向量库。
2. **在线检索**：用户提问时，把问题也转成向量，在库里面找语义最相近的几个文档块。
3. **增强生成**：把找到的文档块和原始问题，一块儿组成提示词送给大模型，让它基于这些资料去回答，不是凭空瞎编。

这样既利用了私域知识的实时性，又保留了模型的表达和总结能力，答案是有限制的，不会太放飞。

---

## 答-06 · MCP 与 ToolCalling 有什么区别

**MCP** 是 Anthropic 推出的一种开放协议，全称 Model Context Protocol，它定义了一套客户端与工具服务器之间的标准通信接口。你可以把它想象成工具调用的"USB 协议"，目标是让不同模型和应用都能用统一的方式连接各种外部工具、数据源，而不用每个工具单独适配。

**Tool Calling**（工具调用）是模型本身的内部能力，就是模型能根据用户意图，输出一个结构化的 JSON，选好要调哪个函数、传什么参数。它只是一种能力格式，但不规定工具怎么提供、怎么执行。

打个比方：**Tool Calling 是模型"会点菜"，MCP 是规定了"厨房、服务员、菜单"怎么协作**。有了 MCP，随便换个模型或工具，只要都支持 MCP，就能即插即用，不像以前每个平台都要自己写胶水代码。

## 答-07 · 系统提示词一般有哪些组成部分

通常包括这么几块：

1. **角色设定**：告诉 AI 它是谁、有什么能力、以什么风格说话。
2. **任务描述**：直接说清楚要做什么事，比如"回答问题"还是"总结文本"。
3. **输出格式要求**：规定回答的结构、字数、要不要分点。
4. **约束条件**：比如不能说的内容、必须遵守的规则。
5. **Few-shot 示例**：给几个范例让 AI 照着例子来。

这些组合起来，就是一条完整的系统提示词。

## 答-08 · 聊一聊你项目中的 AI 模块

在我之前的一个智能客服系统中，AI 模块主要是负责跟大模型交互，给用户返回自动化的答案。当时我们用的是 Java + Spring Boot，对外接的是 OpenAI 的接口，实际模块设计我分了三层：

**第一层是接入层**，我用了一个线程池来处理所有用户请求，核心参数配置参考了业务场景：核心线程数设成 CPU 核数的两倍，最大线程数设成一百，队列用的是 LinkedBlockingQueue，拒绝策略选的是 CallerRunsPolicy，这样能在流量洪峰时把压力回传给调用方，不至于丢请求。因为大模型调用耗时较长，我特意把临时线程的空闲超时调大到 60 秒，避免频繁创建销毁。

**第二层是调度层**，我做了一个基于 AQS 的信号量限流，因为 OpenAI 有每分钟请求上限。我重写了 AbstractQueuedSynchronizer 的 tryAcquireShared 和 tryReleaseShared，用 state 变量控制并发令牌数，没拿到令牌的线程会进入 AQS 的 FIFO 队列阻塞等待，超时就直接降级返回预设话术。这里 volatile 也被用在令牌数的可见性上，保证多个工作线程能实时看到最新余量。

**第三层是实际调用与缓存层**，我用双重检查锁的单例来维护一个本地 LoadingCache，缓存常用问题的答案，减少大模型调用成本。单例对象用 volatile 修饰防止指令重排，锁对象用 AQS 的 ReentrantLock 实现。

整体下来，项目里之前聊到的 CAS、volatile、线程池、AQS 这些知识全都串联起来了，效果就是支撑了日活十万级用户，平均响应时间控制在两秒以内。

## 答-09 · AI 系统是如何集成到项目中的

集成 AI 系统到项目里，一般分这么几步走：

1. 首先得**明确业务场景**，是推荐、NLP 还是图像识别？
2. **选型**：是用现成的云 API，比如 OpenAI、通义千问这些大模型接口，还是自己部署开源模型。如果用云 API，直接调接口就行了，重点在做好 Prompt 工程、接口封装、异常处理和成本控制。要是自部署，就得上模型服务化，比如用 FastAPI 或 Triton Inference Server 把模型包成 REST 或 gRPC 接口，再考虑并发、GPU 调度。
3. 在业务代码里，通过 **A/B 测试、功能开关、缓存、熔断**这些手段保证稳定性，同时监控调用延迟、token 消耗、准确率这些指标。
4. 最后**持续迭代**，根据线上反馈微调模型或调整策略。整个过程 DevOps 和 MLOps 工具链都会用上。

## 答-10 · AI 项目中，在与前端进行交互时，遇到过什么问题

在 AI 项目里跟前端交互，我遇到最多的问题就是**流式输出的数据格式和处理方式**：前端拿到的是一段一段不完整的文本，需要实时拼接显示，还要处理乱码和缓冲问题。

刚开始我们直接按字符输出，前端渲染会出现 HTML 标签被截断，比如一个 Markdown 的粗体符号还没闭合，前端渲染就崩了。后来我们在后端输出时加了边界标记，按完整 token 输出，前端按标记缓存拼接，防止截断。

还有一个常见问题是**异常断连和重试策略**：用户请求中断或网络抖动，前后端需要约定好心跳和断点续传的机制。

## 答-11 · 你项目中的 AI 怎么训练的，要什么条件（硬件）

我们用的是深度学习模型，训练主要分数据准备、模型设计、训练和评估几步。

硬件条件最核心的是 **GPU**，比如 NVIDIA 的 A100、V100 这些，显存要大，否则大模型跑不起来。如果数据量大还得配大内存和高速固态硬盘。

## 答-12 · 分割的话有可能把上下文打断怎么解决

这个问题在 RAG 系统里非常关键——文档切分（Chunking）时如果一刀切在句子中间，就会把一个完整语义拦腰斩断，后续检索到的片段缺少上下文，LLM 就容易理解偏差或答非所问。

**解决方案**：

1. **语义分块**：不要按固定字符数硬切，而是按段落、句子等自然边界来切分，保持每个块内的语义完整。可以借助 NLP 的句子分割器或 Markdown 的结构（标题、段落）来做切分依据。

2. **重叠窗口（Overlap）**：相邻的两个块之间保留 10%~20% 的内容重叠，这样即使某个关键信息刚好落在上一块的边界上，下一块的头部也会包含它，减少边界信息丢失。

3. **父文档检索（Parent Document Retriever）**：这是目前最推荐的方案。文档先切成大块（如 2000 token），大块再切成小块（如 300 token）。小块存入向量库用于语义匹配（检索精准），但返回给 LLM 时不是返回命中的小块，而是返回它所属的那个大块（上下文完整）。相当于"小块精准定位，大块完整喂给 LLM"。

4. **层级/多粒度索引**：同时维护文档级、段落级、句子级等多层索引，检索时根据问题类型选择合适粒度，或者多粒度结果综合排序。

5. **上下文扩展**：检索到目标块后，自动把它的前一块和后一块也一并取出来，拼在一起喂给 LLM，相当于动态补全上下文。

实际项目中这几种手段通常组合使用：**语义分块 + 重叠窗口 + 父文档检索**，基本就能把上下文打断的问题降到很低。

## 答-13 · 智能助手的会话记忆是如何做的

（此题答案原为引导回 Java 面试方向，以下为会话记忆通用方案）

智能助手的会话记忆通常从以下几个层面实现：

1. **短期记忆**：滑动窗口保留最近 N 轮对话，直接拼在上下文里。
2. **长期记忆**：将历史对话做摘要压缩后嵌入 Prompt；或将历史对话存入向量库，每次检索最相关片段注入当前上下文。
3. **结构化记忆**：提取用户画像、偏好、历史关键信息做持久化存储，每次对话时注入 System Prompt。

## 答-14 · 如果对话时上下文的 token 超过了大模型输入限制，怎么办

超过 token 限制时，可以先看看能不能换长上下文的模型，但现在很多模型 128K 甚至 1M token 倒也不容易超。

如果实在超了，通常用**滑动窗口**，只保留最近几轮对话，丢弃最早的消息。更细一点可以做**动态摘要**，把更早的对话压缩成简短摘要嵌进提示里，既省 token 又能保留关键信息。

还有用 **RAG** 那套，把历史对话存向量库，每次生成时检索最相关的几段塞进去，避免暴力全量输入。

关键就是**别简单粗暴截断**，容易丢上下文导致回答跑偏。

## 答-16 · Ollama 是什么，有什么作用

Ollama 是一个开源工具，让你能在自己电脑上**本地运行大语言模型**，比如 Llama、Gemma、Mistral 这些，不需要联网也不用依赖云端 API。

它把模型下载、管理、推理全部封装好了，你装好之后一行命令就能启动一个本地模型服务，还能通过 REST API 或命令行交互。

## 答-19 · 你们 Ollama 大模型里面都是老数据，是怎么解决的

大模型知识确实有截止日期，没办法实时更新，这是所有离线大模型的共同特点。但实际应用中，可以通过几种方式来解决"数据老"的问题：

1. **最主流的是 RAG**（检索增强生成）。通俗讲就是在提问时，系统先拿着问题去查最新的文档、数据库或搜索引擎，把查到的新资料和问题拼在一起喂给模型，这样回答时就能用到实时信息了。
2. **给模型配联网插件或工具调用能力**，让模型能主动搜索网页，拿到最新内容再回答。
3. 如果业务场景固定，也可以**定期用新数据对模型做微调或增量训练**，让它的知识库变新。
4. 如果是在 Ollama 里跑模型，可以搭配 **LangChain、LlamaIndex** 这类框架，轻松实现本地知识库查询或联网搜索，直接弥补数据老化的问题。

## 答-20 · 你的 MaxKB 和 Ollama 是怎么联系起来的

简单说，**MaxKB 负责"知识库的构建与检索"，Ollama 负责"提供本地大模型的推理能力"，两者通过 API 连起来**。

MaxKB 在后台配置模型时，可以选择"Ollama"作为模型供应商，填上 Ollama 的服务地址和模型名，之后用户提问时，MaxKB 会先把问题拿去做文档检索，拼成 Prompt，再发给 Ollama 的 API，Ollama 本地推理完把结果返回，MaxKB 再展示给用户。

整个流程是：**问题 → MaxKB 检索相关文档 → 组合上下文 → 调用 Ollama 推理 → 输出答案**。

## 答-21 · 对于不准确的数据如何进行模型的微调

简单说，就是当你手里有一些不太精确、有噪声或者标注不完全正确的数据时，直接训练效果肯定差。微调思路一般是几步走：

1. 先用一个已经在高质量数据上**预训练好的模型**，比如 BERT、GPT 这类，然后用你的"不准确数据"接着训练，但别从头开始。
2. **学习率调得特别小**，防止模型把预训练学到的知识忘光。
3. 数据要是不准确，最好加**正则化、丢弃一些特征（Dropout）、或者用对抗训练**这些东西，让模型别太纠结那些噪声样本。
4. 还有一种办法是**先自动筛一遍数据**，清洗掉错误比较明显的，或者用集成模型投票来纠正标注，再拿去微调。

总之核心就是"**借力预训练模型 + 小步调整 + 抗噪手段**"。

## 答-22 · 为什么需要模型微调

微调说白了，就是**让通用模型变成你的专业助手**。

预训练模型用的都是互联网上海量的通用数据，学到的是一般性的语言规律，不是你业务的特定逻辑。直接拿过来面对你的客服问答、代码审查或者领域文档分析，很可能答非所问。

微调就是用你自己的少量业务数据，在人家已经训练好的大脑上再教一教，让模型学会你这一亩三分地的用词习惯、规则和套路。这样做既不用自己从零烧显卡，又能让模型在你的任务上表现飙升，成本低、见效快，所以公司里基本都走这条路。

## 答-23 · 你都知道什么大模型

大模型的话，现在比较主流的有：

- **海外闭源**：OpenAI 的 GPT 系列（GPT-4），谷歌的 Gemini
- **国内闭源**：阿里的通义千问、百度的文心一言、讯飞的星火
- **国内开源**：清华的 ChatGLM
- **海外开源**：Meta 的 LLaMa

各有各的特点，有的擅长推理，有的长上下文，有的多模态能力强，主要看应用场景选型。
