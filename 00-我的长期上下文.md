# 00-我的长期上下文

> 用途：这是给所有 Agent 共用的长期上下文入口，不绑定 Codex、Claude、ChatGPT 或任何单一工具。  
> 原则：只写长期稳定、会反复影响决策的信息；不写临时任务、短期情绪、一次性细节、未经确认的猜测。  
> 使用方式：任何 Agent 开始处理我的长期任务前，应先读取本文件，再读取本文件引用的专题文档。

---

## 0. 多 Agent 使用协议

### 0.1 Agent 开始工作前

任何 Agent 如果要处理以下任务，应先读取本文件：

- 整理知识库
- 修改面试材料
- 生成长期规划
- 重构项目文档
- 管理个人上下文
- 延续上一次 Agent 的工作

读取顺序：

1. 先读本文件。
2. 再读本文件指向的专题入口。
3. 最后再读任务相关的具体文件。

### 0.2 Agent 写入本文件前

写入前必须判断信息是否满足至少两条：

- 未来一个月以上大概率仍然有效。
- 会反复影响后续 Agent 的判断。
- 和我的身份、目标、项目、知识库结构、表达红线、协作偏好有关。
- 如果不写，下次 Agent 容易误判或重复沟通。

不满足这些条件的内容，不应写入本文件。

### 0.3 修改规则

- 不要把本文件变成流水账。
- 不要大段粘贴临时聊天记录。
- 不要写未经我确认的事实。
- 如果是推断，必须标注“推断”。
- 如果是过期信息，移动到“历史记录”或删除，不要和当前事实混在一起。
- 大改本文件前，应说明改动意图。

### 0.4 冲突处理

如果本文件和其他笔记冲突：

1. 以最近更新、且更具体的专题文档为准。
2. 如果冲突影响面试口径、项目真实性或文件移动，先向我确认。
3. 不要自行把高风险口径写成事实。

## 1. 当前主目标

当前主目标是围绕 Java + AI 应用方向准备面试，把知识库中的项目材料和技术笔记整理成真实、抗追问、可表达的面试内容。

当前面试准备不追求堆材料，而是优先解决：

- 项目主线是否讲得清楚。
- 个人职责是否边界清晰。
- 技术回答是否能承受追问。
- 数字、时间、团队规模、技术栈是否自洽。

## 6. 面试表达红线

面试材料必须遵守以下红线：

- 数字必须自洽，不能前后矛盾。
- 可以使用区间表述，但不能用虚假精确数字制造可信度。
- 不做假演进，不编造“从 A 架构演进到 B 架构”的经历。
- JDK、Spring、Spring AI、Python 等技术栈差异必须能解释。
- 不能把 3 人后端团队包装成自己承担 4 个大模块。
- 不能把没直接负责的模块写成自己主导。
- 面试回答要先抗追问，再考虑润色。

## 7. Agent 协作偏好

- Agent 应先读文件再判断，不要凭空发挥。
- 重要或批量修改前，先从第一性原理说明判断逻辑和改动方案。
- 大规模改动前先确认；用户明确授权后再执行。
- 修改知识库时优先归档，不要直接删除。
- 面试材料先质疑真实性和抗追问性，再做表达润色。
- 对高风险项目经历要标明“直接负责 / 参与联调 / 了解架构 / 平台已有”。
- 面试练习中，如果技术术语或英文发音明显错误，应直接纠正。
- 使用技能或工具时，应遵守该技能/工具的完整工作流，不把技能当普通参考文档。
- 每次长对话结束或阶段性任务完成后，Agent 应主动判断本轮是否产生需要写入长期上下文的信息；只给出候选项，不默认写入，等待用户确认。

## 8. 常用工作流

- Agent 接力：先读本文件，再读 `00-当前主线.md`，需要导航时再读 `00-知识库地图.md`。
- 配置新 Agent：把 `00-Agent启动提示词.md` 里的通用启动提示词复制到对应 Agent 的项目说明或系统提示中。
- 面试材料整理：先读面试作战首页，再读具体项目/技术追问文件，最后再修改当前主线材料。
- 长期记忆写入：只有用户明确触发，或 Agent 主动建议且用户确认后，才写入本文件。

## 9. 不应写入长期上下文的内容

以下内容不要写入本文件：

- 一次性的命令输出。
- 临时网络问题。
- 当天情绪。
- 还没确认的项目细节。
- 大段未提炼的聊天记录。
- 可从具体文件中直接读取的全文内容。
- 只对当前会话有意义的中间过程。

这些内容可以写入具体专题文档、任务记录或临时笔记，但不要污染长期上下文。
